# Vault Retrieval Architecture v1

이 문서는 vault 전체를 다시 쓰기 위한 문서가 아니다.  
보안 교육이 끝난 뒤에도 까먹은 개념, 머리로만 아는 실습, 오래된 프로젝트 기록을 다시 꺼내 쓰기 위한 검색·회상 구조를 정의한다.

## 최종 목적

이 vault의 최종 목적은 보안 교육과 실습 경험을 장기 보조기억으로 보존하는 것이다.

학습이 끝난 뒤 사용자는 세 가지 상태를 오간다.

| 상태 | 의미 | vault가 해야 할 일 |
|---|---|---|
| 몸에 밴 지식 | 바로 수행 가능 | 세부 설명보다 확인용 체크포인트만 있으면 된다. |
| 머리로만 아는 지식 | 이해는 했지만 실행이 가물가물함 | 개념 노트와 실습 노트로 빠르게 복구 가능해야 한다. |
| 까먹은 지식 | 다시 찾아야 함 | MOC, source, RAW, 정리본의 경로가 분명해야 한다. |

## 핵심 원칙

1. MOC는 전체 파일 목록이 아니라 탐색 지도다.
2. 실제 탐색 경로는 `Home → 영역 MOC → 주제/프로젝트 MOC → 실제 노트`에 둔다.
3. `LLM_AGENT_INDEX.md`는 경로를 대체하지 않는다. AI가 실제 MOC를 오해하지 않게 하는 운영 규칙이다.
4. 현재진행 RAW는 완벽히 정리하지 않는다. 완료되거나 다시 찾을 가치가 생긴 단위부터 안정화한다.
5. RAW, source, stable note, restart point는 분리한다.
6. AI가 헤맨 지점은 실제 MOC 또는 source catalog에 먼저 반영하고, 필요하면 agent index에 오판 방지 규칙을 추가한다.
7. 전체 inventory와 frontmatter 통일은 즉시 전수 적용하지 않는다. 검증된 구조가 생긴 뒤 단계적으로 적용한다.

## 구조 계층

| Layer | 역할 | 대표 위치 | 운영 기준 |
|---|---|---|---|
| Entry | vault의 공식 정문 | [[00_index/Home]] | 얇게 유지한다. 세부 규칙을 길게 넣지 않는다. |
| Area MOC | 큰 영역 라우팅 | `10_학습 노트/00_학습노트_목차.md`, `20_팀 프로젝트/00_프로젝트_목차.md` | 주제/프로젝트 MOC로 연결한다. |
| Subject or Project MOC | 실제 탐색 경로 | 각 `00_*_목차.md` | concrete note, restart point, 정리 대기만 관리한다. |
| Stable note | 장기 기억 단위 | 개념 노트, 실습 노트, troubleshooting | 나중에 읽어도 독립적으로 이해되어야 한다. |
| RAW and source | 원자료와 증거 | RAW 메모, PDF, 캡처, 진단 원문 | 정리본과 같은 위상으로 취급하지 않는다. |
| Agent operating layer | AI 오판 방지 | [[00_index/LLM_AGENT_INDEX]] | 실제 경로를 대신 만들지 않는다. |
| Future inventory | 전체 파일 추적 | 미정 | 필요할 때 생성한다. 사람이 매일 읽는 MOC가 아니다. |

## Note type 기준

| Type | 정의 | 정리 기준 |
|---|---|---|
| `moc` | 탐색 지도 | 링크는 핵심 경로만. 파일 목록화 금지. |
| `concept` | 개념 복구용 노트 | 정의, 오해 포인트, 확인 질문을 포함한다. |
| `lab` | 실습 복구용 노트 | 명령, 관찰, 오류, 검증 결과를 분리한다. |
| `troubleshooting` | 문제 해결 기록 | 증상, 원인, 해결, 재발 방지 기준을 둔다. |
| `raw` | 수업 중 빠른 메모 | 보존한다. 정리본으로 단정하지 않는다. |
| `source` | PDF, 캡처, 원문 로그 | 범위와 출처 확인용이다. |
| `project` | 팀 프로젝트 운영/결과 | 결과물, 증거, 스크립트, 원문 로그를 구분한다. |

## 상태 기준

| Status | 의미 | 다음 행동 |
|---|---|---|
| `current` | 현재진행 중 | 최소 진입점만 유지하고 잦은 재정리는 피한다. |
| `stable` | 다시 읽을 수 있는 정리본 | MOC에 연결하고 필요하면 review 질문을 붙인다. |
| `raw` | 아직 흡수되지 않음 | source로 보존한다. 필요할 때만 요약한다. |
| `legacy` | 예전 방식으로 만든 노트 | 자주 찾거나 혼선을 일으킬 때만 재구성한다. |
| `stale` | 현재 기준과 충돌 가능 | 최신성 또는 현재 수업 흐름을 확인한다. |
| `archived` | 완료된 과거 기록 | 현재 상태로 단정하지 않는다. |

## 작업 우선순위

정리 대상은 파일 수가 아니라 회상 가치와 오판 비용으로 정한다.

| 우선순위 | 대상 | 이유 |
|---:|---|---|
| 1 | 완료된데 다시 찾을 가능성이 높은 영역 | 장기 보조기억 효과가 가장 크다. |
| 2 | AI가 실제로 헤맨 영역 | 오판과 토큰 낭비가 반복된다. |
| 3 | RAW와 정리본이 섞인 영역 | 근거와 결론이 혼동된다. |
| 4 | 구버전 수업 흐름과 현재 수업 흐름이 섞인 영역 | 잘못된 재시작 지점을 만들 수 있다. |
| 5 | 미숙한 시절의 큰 덩어리 노트 | 쪼개야 할 가능성이 높지만, 실제 재사용 가치가 있을 때 한다. |

현재진행 중인 RAW 영역은 원칙적으로 대공사하지 않는다. 단, 진입점이 없어 AI나 사용자가 반복해서 헤매면 얇은 MOC 링크만 추가한다.

## Retrieval flow

까먹은 내용을 다시 찾을 때의 기본 흐름은 다음과 같다.

```text
질문 또는 기억 단서
→ Home
→ 영역 MOC
→ 주제/프로젝트 MOC
→ stable note
→ lab note
→ RAW/source
```

AI에게 맡길 때도 같은 흐름을 사용한다. AI가 먼저 RAW나 PDF 전체를 읽게 하지 않는다.

## AI 운영 원칙

AI는 다음 기준으로 움직여야 한다.

1. `AGENTS.md`와 실제 repo 상태를 먼저 확인한다.
2. 넓은 vault 탐색이나 MOC/index 작업이면 [[00_index/LLM_AGENT_INDEX]]를 읽는다.
3. 실제 경로는 MOC에서 확인한다.
4. `LLM_AGENT_INDEX.md`에만 있는 경로는 안정 경로로 간주하지 않는다.
5. RAW를 stable note로 오인하지 않는다.
6. 링크 또는 완료 여부를 말할 때는 실제 파일과 명령 출력으로 확인한다.
7. 현재진행 영역은 정리보다 보존과 최소 라우팅을 우선한다.

## No shadow routing

중요한 경로가 `LLM_AGENT_INDEX.md`에만 있고 실제 MOC에 없다면 shadow route로 본다.

처리 순서는 다음과 같다.

1. 실제 MOC에 얇은 진입점을 만들지 판단한다.
2. source/RAW 위치가 필요하면 관련 자료 MOC나 source catalog에 둔다.
3. 그 뒤 `LLM_AGENT_INDEX.md`에는 오판 방지 규칙만 남긴다.

## 완료 조건

이 구조가 제대로 작동한다고 볼 수 있는 조건은 다음과 같다.

- 사용자가 까먹은 주제를 MOC 경로로 다시 찾을 수 있다.
- AI가 Home과 MOC를 따라가며 stable note와 RAW/source를 구분한다.
- 현재진행 영역은 지저분해도 최소 진입점이 있다.
- 완료된 영역은 재시작 지점과 대표 정리본이 있다.
- 오래된 수업 흐름과 현재 수업 흐름이 명시적으로 분리된다.
- 전체 파일이 MOC에 모두 들어가 있지 않아도, 비노출 이유를 설명할 수 있다.
