# 세션과 쿠키

source:
- [[40_자료/강의 자료/5-20_웹보안.pdf|5-20 웹보안]], p.49-61
- [OWASP Session Management Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Session_Management_Cheat_Sheet.html)
- [MDN Set-Cookie](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Set-Cookie)

## 한 줄 요약

웹 세션은 **HTTP 요청 여러 개를 같은 사용자 흐름으로 묶기 위해 서버가 발급한 Session Token으로 사용자를 다시 식별하는 구조**다.

쿠키는 이 Session Token을 브라우저에 저장하고, 이후 같은 사이트 요청에 자동으로 붙여 보내게 하는 대표적인 전달 방식이다.

---

## 먼저 잡아야 할 핵심

- HTTP 요청은 기본적으로 서로 독립적이다.
- 로그인 상태를 유지하려면 서버가 “이 요청이 방금 로그인한 그 사용자에게서 온 요청”이라는 근거를 가져야 한다.
- 이때 서버가 발급하는 식별자가 Session Token 또는 Session ID다.
- 브라우저는 보통 이 token을 Cookie로 저장하고, 이후 요청마다 `Cookie` header에 담아 보낸다.
- 서버는 token 값 자체가 아니라, token이 가리키는 서버 측 session record를 보고 사용자와 권한을 판단한다.
- 따라서 Session Token은 로그인 이후에는 사실상 임시 신분증처럼 동작한다.
- Session Token이 탈취되면 [[10_학습 노트/시스템보안/웹보안/Web Session Hijacking|Web Session Hijacking]]으로 이어질 수 있다.

---

## “너랑 나도 세션인가?”

일반적인 말로는 맞다. 어떤 상호작용이 시작해서 끝날 때까지 이어지는 구간도 session이라고 부를 수 있다.

하지만 웹보안에서 말하는 **Web Session**은 더 좁은 뜻이다.

```text
일반 session
= 시작부터 종료까지 이어지는 한 번의 상호작용 구간

Web Session
= HTTP 요청들이 끊겨 있어도, 서버가 Session Token으로 같은 사용자 흐름이라고 연결해 주는 상태 관리 구조
```

즉 핵심은 “대화가 이어진다”가 아니라, **서버가 stateless HTTP 요청들을 같은 사용자 상태로 묶어 관리한다**는 점이다.

---

## Web Session이 필요한 이유

HTTP는 요청과 응답이 한 번 오가면 끝나는 구조다. 브라우저가 다음 페이지를 요청할 때, 서버는 기본 HTTP만으로는 그것이 방금 로그인한 사용자와 같은 사람인지 알 수 없다.

그래서 웹 애플리케이션은 다음 정보를 어딘가에 유지해야 한다.

| 유지해야 하는 정보 | 예시 |
|---|---|
| 사용자가 누구인가 | 로그인한 사용자 ID |
| 어떤 권한을 가지는가 | 일반 사용자, 관리자 |
| 어떤 상태인가 | 장바구니, 언어 설정, 진행 중인 작업 |
| 언제 만료되는가 | idle timeout, absolute timeout |

이 상태 전체를 브라우저에 다 넣는 것이 아니라, 보통 브라우저에는 식별자만 두고 실제 의미는 서버 측 session store에 둔다.

---

## 일반적인 Session Management 흐름

```text
1. 사용자가 웹 사이트에 처음 접근한다.
2. 서버가 Session Token을 생성한다.
3. 서버는 token과 session state를 mapping해 저장한다.
4. 서버는 token을 응답으로 브라우저에 전달한다.
5. 브라우저는 token을 저장한다.
6. 이후 요청마다 브라우저가 token을 서버에 보낸다.
7. 서버는 token을 보고 해당 session state를 찾는다.
8. 로그아웃, timeout, 브라우저 종료, 서버 정책에 따라 session을 폐기한다.
```

PDF p.52의 그림은 이 흐름을 “Session Token 생성 -> 저장 -> 요청마다 포함 -> timeout 또는 종료 시 파기”로 보여준다.

---

## Session Token은 무엇이어야 하나

Session Token은 클라이언트 입장에서는 의미 없는 랜덤 식별자여야 한다.

| 원칙 | 이유 |
|---|---|
| 예측 불가능해야 함 | 추측으로 다른 사용자의 session을 맞히면 안 된다. |
| 충분한 entropy가 있어야 함 | brute force로 유효한 token을 찾기 어렵게 한다. |
| 내부 정보를 담지 않아야 함 | token을 보고 사용자 ID, 권한, 로직을 알 수 있으면 안 된다. |
| 서버 측 state와 연결되어야 함 | 권한과 사용자 정보는 서버에서 관리하는 편이 안전하다. |
| 로그인/권한 변경 시 재발급해야 함 | session fixation과 권한 상승 전후 혼선을 줄인다. |

PDF p.49는 ASP, JSP, PHP 계열 session ID 이름을 예로 든다. 다만 이름 자체는 구현 기술을 드러낼 수 있으므로, 실제 보안 관점에서는 token 값의 안전성과 cookie 속성, 서버 측 session 관리가 더 중요하다.

---

## 웹 서버가 여러 대일 때

PDF p.49는 “웹 서버가 물리적으로 여러 대인 경우 WAS에서 발행한 Session ID만으로 인증 불가”라고 설명한다.

의미는 이렇다.

```text
WEB1이 Session Token 발급
-> 다음 요청이 WEB2로 감
-> WEB2가 그 token에 해당하는 session state를 모르면 로그인 상태를 판단할 수 없음
```

그래서 다중 서버 환경에서는 다음 중 하나가 필요하다.

| 방식 | 의미 | 주의 |
|---|---|---|
| 공유 session store | Redis, DB 등 공용 저장소에 session state 저장 | 저장소 장애와 만료 정책 관리 필요 |
| session replication | WAS끼리 session state 복제 | 복잡도와 동기화 비용 증가 |
| sticky session | 같은 사용자를 같은 WEB으로 라우팅 | 해당 WEB 장애 시 세션 유지가 어려울 수 있음 |
| stateless signed token | 서버가 서명한 token으로 상태 일부 검증 | 탈취 시 만료 전까지 위험. 민감정보를 담으면 안 됨 |

이 노트의 범위에서는 “Session Token은 브라우저에 있지만, session의 의미는 서버가 관리해야 한다”는 점만 잡으면 된다.

---

## Session Token 저장 방식

PDF는 세 가지 방식을 다룬다.

| 방식 | 설명 | 보안 관점 |
|---|---|---|
| Hidden Field | HTML form의 hidden input에 token 저장 | 소스 보기와 요청 조작으로 노출·변조 가능. 중요한 token 저장에 부적절 |
| URL Rewriting | URL에 token을 붙임 | 로그, 히스토리, 북마크, Referer로 노출되기 쉬움 |
| Cookie | 브라우저가 저장하고 요청마다 자동 전송 | 가장 일반적. 속성 설정과 session 관리가 중요 |

Hidden Field와 URL Rewriting은 “브라우저 화면에 안 보인다” 또는 “쿠키가 없어도 된다”는 장점보다 노출·변조 위험이 더 크다. 기본값으로는 Cookie 기반 session management를 우선 이해한다.

---

## Cookie란 무엇인가

Cookie는 서버가 브라우저에 저장시킬 수 있는 작은 `name=value` 데이터다.

흐름은 보통 다음과 같다.

```text
서버 응답: Set-Cookie: session_id=...
브라우저 저장
다음 요청: Cookie: session_id=...
서버가 session_id로 사용자 상태 확인
```

Cookie 자체가 곧 session은 아니다. Cookie는 session token을 실어 나르는 대표적인 저장·전달 수단이다.

---

## Session Cookie와 Persistent Cookie

| 구분 | Session Cookie | Persistent Cookie |
|---|---|---|
| 만료 기준 | 보통 브라우저 세션 종료 또는 서버 정책 | `Expires` 또는 `Max-Age` |
| 용도 | 로그인 유지, 서버 이용 중 사용자 상태 유지 | 재방문 기억, 설정 유지, Remember Me |
| 위험 | 탈취되면 현재 로그인 상태가 재사용될 수 있음 | 오래 남으므로 탈취 시 피해 시간이 길어질 수 있음 |
| 주의 | 브라우저 session restore 기능 때문에 닫아도 복원될 수 있음 | 민감한 인증 상태를 오래 유지하면 위험 |

PDF는 Session Cookie를 브라우저 cache에 임시 저장한다고 설명한다. 현대적으로는 “`Expires`나 `Max-Age`가 없는 cookie는 session cookie로 취급되며, 브라우저 세션 종료 시 제거되는 것이 기본” 정도로 이해하는 편이 안전하다. 다만 브라우저의 session restore 기능 때문에 항상 즉시 삭제된다고 단정하면 안 된다.

---

## Cookie 보안 속성

Session Cookie는 다음 속성과 같이 봐야 한다.

| 속성 | 역할 | 주의 |
|---|---|---|
| `Secure` | HTTPS 요청에서만 cookie를 보내게 함 | HTTP 평문 전송을 막는 데 필요하지만, JavaScript 접근을 막지는 않는다. |
| `HttpOnly` | `document.cookie` 같은 JavaScript 접근을 막음 | XSS 자체를 막지는 못한다. XSS가 사용자 권한 요청을 보내는 문제는 남는다. |
| `SameSite` | cross-site 요청에 cookie가 붙는 범위를 제한 | CSRF 위험을 줄이는 데 중요하다. |
| `Domain` | 어떤 host/subdomain에 cookie를 보낼지 결정 | 넓게 잡으면 하위 도메인 위험이 커질 수 있다. |
| `Path` | 어떤 URL path에 cookie를 보낼지 결정 | 보낼 범위를 줄일 뿐, 보안 경계로 보면 안 된다. |
| `Expires` / `Max-Age` | cookie 만료 시점 결정 | 장기 cookie는 탈취 시 피해 시간이 길어진다. |
| `__Host-` prefix | host-only, `Secure`, `Path=/` 조합 강제 | 지원 브라우저에서 origin 경계에 가깝게 제한하는 데 도움 |

Session Cookie의 실무 기본값은 보통 다음 방향이다.

```text
Secure + HttpOnly + SameSite + 짧은 만료 + 서버 측 session state
```

|---|---|
| Session은 시작부터 끝까지를 의미 | 일반 session 의미다. Web Session은 HTTP 요청들을 같은 사용자 상태로 묶는 기술 구조다. |
| Session Token은 서버와 클라이언트 양쪽에 저장 | 클라이언트에는 token, 서버에는 token에 대응하는 session state가 저장된다고 이해한다. |
| Hidden Field는 화면에 보이지 않음 | 화면에 안 보일 뿐 소스와 요청에서는 보이므로 중요한 값 저장에 부적절하다. |
| URL Rewriting은 쿠키가 없을 때 사용 | token이 URL에 노출되어 로그, 히스토리, Referer로 새기 쉬워 기본 선택으로 부적절하다. |
| Cookie는 발행한 사이트에서만 읽음 | Domain, Path, SameSite, HttpOnly 같은 속성에 따라 전송·접근 범위가 달라진다. |

## 오해하기 쉬운 지점

- Session Token은 “로그인 정보 전체”가 아니라 서버 측 session state를 찾기 위한 식별자다.
- Cookie는 session의 동의어가 아니다. Cookie는 token 저장·전달 방식이다.
- Session Cookie라고 해서 무조건 브라우저 종료 즉시 사라진다고 단정하면 안 된다.
- Hidden Field는 보이지 않을 뿐 안전한 저장소가 아니다.
- URL에 token을 넣으면 로그, 히스토리, 공유 링크, Referer로 새기 쉽다.
- `HttpOnly`는 cookie 탈취를 줄이지만 XSS 자체를 제거하지 않는다.
- `Secure`는 HTTPS 전송 제한이지, 클라이언트 내부 접근 방지가 아니다.

## 확인 질문

- HTTP가 stateless라면 서버는 로그인 상태를 어떻게 이어서 판단하는가?
- Session Token과 서버 측 session state는 무엇이 다른가?
- Cookie가 session과 같은 말이 아닌 이유는 무엇인가?
- Hidden Field와 URL Rewriting이 session token 저장 방식으로 위험한 이유는 무엇인가?
- `Secure`, `HttpOnly`, `SameSite`는 각각 어떤 공격면을 줄이는가?
- 왜 `세션과 쿠키`를 이해해야 XSS, CSRF, Session Hijacking을 제대로 구분할 수 있는가?
