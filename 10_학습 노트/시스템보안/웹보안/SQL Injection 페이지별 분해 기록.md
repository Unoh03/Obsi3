## Page Digest

### p.107 - SQL의 위치와 분류

#### PDF에서 실제로 다룬 내용

- SQL은 `Structured Query Language`의 약자다.
- PDF는 SQL을 `DDL`, `DML`, `DCL`을 포함하는 Database 관리용 언어로 설명한다.
- `DDL`: 데이터 정의어
- `DML`: 데이터 조작어
- `DCL`: 데이터 제어어

#### 핵심 개념

SQL은 데이터베이스에 명령을 내리는 언어다. SQL Injection을 이해하려면 먼저 애플리케이션이 사용자 입력값을 받아 SQL 문자열을 만들고, 그 SQL이 DB에서 실행된다는 흐름을 잡아야 한다.

#### SQL/SQLi 흐름에서 이 페이지의 역할

이 페이지는 공격 설명이 아니라 전제 지식이다. 뒤에서 나오는 `SELECT`, `INSERT`, `UPDATE`, `DELETE`, `UNION`은 SQL Injection payload가 영향을 주는 SQL 명령의 종류이므로, 여기서는 “SQL은 DB를 조회/변경/정의/제어하는 명령 언어”라는 큰 틀을 잡는다.

#### 나중에 노트화할 때 후보 위치

- `SQL Injection을 위한 SQL 기초`
  - SQL의 의미
  - DDL / DML / DCL 구분
  - 웹 애플리케이션이 SQL을 만드는 위치

#### 확인 필요 / 헷갈리는 점

- PDF는 `DDL과 DML을 포함`한다고 적은 뒤 `DCL`도 함께 나열한다. 최종 노트에서는 SQL의 하위 분류를 `DDL / DML / DCL`로 정확히 정리하는 편이 자연스럽다.
- SQL Injection 실습에서는 주로 `DML`, 특히 `SELECT`가 먼저 중요해진다. `DDL`, `DCL`은 전체 SQL 범주로만 짧게 다뤄도 충분할 수 있다.

#### 이전/다음 페이지와의 연결

- 다음 p.108은 `SELECT` 문이다.
- SQL Injection에서 가장 먼저 악용되는 위치가 로그인 조회 쿼리 같은 `SELECT`이므로, p.108부터 공격 흐름과 직접 연결되기 시작한다.

### p.108 - SELECT 문과 WHERE 조건

#### PDF에서 실제로 다룬 내용

- `SELECT` 문을 다룬다.
- 기본 형식:
  - `SELECT <필드명> FROM <테이블명> WHERE <조건>`
- 의미:
  - `<테이블>`에서 `<조건>`에 맞는 레코드를 검색한다.
  - 검색된 레코드 중 `<필드>`에 해당하는 값만 출력한다.
- 예시:
  - `SELECT * FROM member WHERE name='이명수'`

#### 핵심 개념

`SELECT`는 DB에서 데이터를 조회하는 명령이다. 여기서 가장 중요한 부분은 `WHERE <조건>`이다. 웹 로그인, 검색, 게시글 조회 같은 기능은 사용자가 입력한 값을 `WHERE` 조건에 넣어 필요한 레코드를 찾는다.

#### SQL/SQLi 흐름에서 이 페이지의 역할

SQL Injection은 사용자가 입력한 값이 `WHERE` 조건의 일부로 들어갈 때 자주 발생한다. 특히 로그인에서는 보통 `id`와 `pw` 조건을 만족하는 레코드가 있는지 조회하므로, 공격자는 `WHERE` 조건을 항상 참이 되게 만들거나 뒤 조건을 주석 처리하려고 한다.

#### 나중에 노트화할 때 후보 위치

- `SQL Injection을 위한 SQL 기초`
  - `SELECT`
  - `FROM`
  - `WHERE`
  - `*`의 의미
- `SQL Injection 개념과 인증 우회`
  - 로그인 쿼리에서 `WHERE` 조건이 조작되는 이유

#### 확인 필요 / 헷갈리는 점

- PDF의 예시는 단순 조회다. 최종 노트에서는 이 예시를 로그인 쿼리와 연결해 “조회 결과가 있으면 인증 성공”이라는 구조까지 확장해야 한다.
- `SELECT *`는 모든 컬럼을 가져온다는 의미라 실습에서는 편하지만, 실무에서는 필요한 컬럼만 조회하는 편이 일반적으로 낫다.

#### 이전/다음 페이지와의 연결

- 이전 p.107은 SQL의 큰 분류였다.
- p.108은 SQL Injection의 핵심 표면인 `SELECT ... WHERE ...` 구조를 만든다.
- 다음 p.109는 집계함수다. 직접적인 인증 우회보다는 나중에 `COUNT`, 결과 개수 판단, schema 탐색 쪽 이해에 보조적으로 연결될 수 있다.

### p.109 - 집계함수

#### PDF에서 실제로 다룬 내용

- `Aggregate function`, 즉 집계함수를 다룬다.
- 기본 형식:
  - `SELECT 집단함수(<필드명>) FROM <테이블명>`
- 집단함수 종류:
  - `COUNT`: 반환 레코드의 개수
  - `SUM`: 값의 합계
  - `AVG`: 값의 평균
  - `MAX`: 값의 최대값
  - `MIN`: 값의 최소값
- 예시:
  - `SELECT COUNT(*) FROM member`

#### 핵심 개념

집계함수는 여러 행을 대상으로 계산 결과 하나를 만드는 함수다. `COUNT(*)`는 조건에 맞는 행이 몇 개인지 세는 대표적인 함수다.

#### SQL/SQLi 흐름에서 이 페이지의 역할

이 페이지는 SQL Injection의 직접 payload보다 SQL 조회 결과를 해석하는 기반에 가깝다. 로그인 로직은 명시적으로 `COUNT(*)`를 쓰지 않아도, 내부적으로는 “조건에 맞는 레코드가 존재하는가”를 판단한다. 따라서 `COUNT`는 인증 판단, 데이터 존재 여부 확인, 결과 개수 확인과 연결될 수 있다.

#### 나중에 노트화할 때 후보 위치

- `SQL Injection을 위한 SQL 기초`
  - 집계함수의 역할
  - `COUNT(*)`와 레코드 존재 여부
- `SQL Injection 개념과 인증 우회`
  - 로그인 성공/실패가 조회 결과 개수와 연결되는 구조

#### 확인 필요 / 헷갈리는 점

- 이 페이지는 SQLi 실습의 핵심 페이지는 아니다. 최종 노트에서는 길게 설명하기보다 `COUNT` 중심으로만 가져가도 될 가능성이 높다.
- `SUM`, `AVG`, `MAX`, `MIN`은 SQL 기본 문법으로는 필요하지만 SQLi 입문 흐름에서는 비중을 낮춰도 된다.

#### 이전/다음 페이지와의 연결

- 이전 p.108은 `SELECT ... WHERE`로 레코드를 조회하는 구조를 다뤘다.
- p.109는 조회된 레코드들을 계산하는 함수다.
- 다음 p.110은 `GROUP BY / HAVING`으로, 집계함수 결과를 그룹 단위로 다루는 흐름이다.

### p.110 - GROUP BY / HAVING

#### PDF에서 실제로 다룬 내용

- `GROUP BY / HAVING`을 다룬다.
- 기본 형식:
  - `SELECT 집단함수(<필드명>) FROM <테이블명> GROUP BY <필드명> HAVING <조건>`
- 의미:
  - 출력된 레코드를 그룹으로 묶는다.
  - 각 그룹에 대한 요약 값을 계산한다.
- 예시:
  - `SELECT age, count(age) FROM member GROUP BY age HAVING age<=30`
  - 30세 이하만 그룹화하여 출력하는 예시로 설명한다.

#### 핵심 개념

`GROUP BY`는 같은 값을 가진 행들을 묶고, `HAVING`은 그룹화된 결과에 조건을 거는 문법이다. `WHERE`가 개별 행을 필터링한다면, `HAVING`은 집계 결과나 그룹에 조건을 거는 쪽에 가깝다.

#### SQL/SQLi 흐름에서 이 페이지의 역할

이 페이지는 SQL Injection 초반의 인증 우회와 직접 연결되지는 않는다. 다만 PDF p.132 대응책에서 `group by`, `having` 같은 SQL 키워드 필터링이 언급되므로, SQLi 방어/필터링 파트에서 “SQL 키워드 일부”로 다시 등장할 가능성이 있다.

#### 나중에 노트화할 때 후보 위치

- `SQL Injection을 위한 SQL 기초`
  - `GROUP BY`
  - `HAVING`
- `SQL Injection 방어`
  - keyword filtering 예시로 등장할 수 있지만, 핵심 방어로 두면 안 됨

#### 확인 필요 / 헷갈리는 점

- 입문 수준에서는 `WHERE`와 `HAVING`의 차이가 헷갈릴 수 있다. 최종 노트에 넣는다면 `WHERE = 행 필터`, `HAVING = 그룹 결과 필터` 정도로 짧게 정리하면 충분하다.
- SQL Injection 실습에서 `GROUP BY / HAVING`을 payload로 깊게 다루는지는 이 PDF 범위만으로는 아직 확실하지 않다.

#### 이전/다음 페이지와의 연결

- 이전 p.109의 집계함수와 연결된다.
- 다음 p.111은 `UNION`이다. `UNION`은 뒤의 SQL Injection 정보 추출에서 중요하게 재등장하므로 p.111부터 다시 중요도가 올라간다.

### p.111 - UNION

#### PDF에서 실제로 다룬 내용

- `UNION`을 다룬다.
- 기본 형식:
  - 첫 번째 `SELECT`
  - `UNION`
  - 두 번째 `SELECT`
- 의미:
  - 두 테이블의 검색 결과를 합쳐서 반환한다.
- 조건:
  - 합쳐질 레코드의 개수와 타입이 맞아야 한다.
- 예시:
  - `SELECT name, age FROM member UNION SELECT strname, intReadno FROM board`

#### 핵심 개념

`UNION`은 여러 `SELECT` 결과를 하나의 결과처럼 합치는 문법이다. 단, 합치려는 두 결과의 컬럼 개수와 데이터 타입이 맞아야 한다.

#### SQL/SQLi 흐름에서 이 페이지의 역할

`UNION`은 뒤에서 SQL Injection 정보 추출에 직접 연결된다. 공격자가 원래 애플리케이션이 실행하는 `SELECT` 뒤에 자신이 원하는 `SELECT`를 붙일 수 있으면, 원래 화면에 다른 테이블의 값을 끼워 넣어 출력하게 만들 수 있다. 이때 컬럼 개수와 타입을 맞춰야 하므로, `UNION`의 제약 조건 자체가 공격 과정의 핵심 체크포인트가 된다.

#### 나중에 노트화할 때 후보 위치

- `SQL Injection을 위한 SQL 기초`
  - `UNION`의 의미
  - 컬럼 개수/타입 일치 조건
- `Error/UNION 기반 정보 추출과 Schema 파악`
  - `UNION` 기반 컬럼 개수 확인
  - 다른 테이블 데이터 끼워 넣기

#### 확인 필요 / 헷갈리는 점

- PDF p.111은 정상 SQL 문법으로 `UNION`을 설명한다.
- 뒤 p.123에서 `UNION`을 이용해 mismatch를 발생시키고 필드 개수/타입을 파악하는 흐름이 나올 예정이므로, 최종 노트에서는 p.111과 p.123을 강하게 연결해야 한다.

#### 이전/다음 페이지와의 연결

- 이전 p.110까지는 집계/그룹화 문법이었다.
- p.111부터는 SQL Injection 정보 추출 파트와 직접 이어지는 핵심 기초다.
- 다음 p.112는 `INSERT`다. `INSERT`는 게시글 작성, 회원가입처럼 DB에 행을 추가하는 상태 변경 기능과 연결된다.

### p.112 - INSERT

#### PDF에서 실제로 다룬 내용

- `INSERT`를 다룬다.
- 기본 형식:
  - `INSERT INTO <테이블명> (컬럼명, ) VALUES (값, )`
- 의미:
  - 테이블에 레코드를 추가한다.
- 예시:
  - `INSERT INTO member (user_id, user_pw, name, nickname, age) VALUES ('hacker', 'hacker', 'hacker', 'hacker', 1)`

#### 핵심 개념

`INSERT`는 DB 테이블에 새 행을 추가하는 명령이다. 웹에서는 회원가입, 게시글 작성, 댓글 작성, 로그 저장 같은 기능에서 자주 쓰인다.

#### SQL/SQLi 흐름에서 이 페이지의 역할

SQL Injection 입문에서는 `SELECT`를 이용한 인증 우회가 먼저 나오지만, SQL Injection의 영향은 조회에만 그치지 않는다. 애플리케이션이 동적 SQL로 `INSERT`를 만들고 사용자 입력을 안전하게 처리하지 않으면, 저장되는 값 조작이나 쿼리 구조 변조가 문제가 될 수 있다.

#### 나중에 노트화할 때 후보 위치

- `SQL Injection을 위한 SQL 기초`
  - `INSERT`
  - 테이블/컬럼/값의 대응
- `SQL Injection 영향`
  - 데이터 추가/오염 가능성

#### 확인 필요 / 헷갈리는 점

- PDF 예시는 `hacker` 값을 회원 테이블에 넣는 형태라 공격 예시처럼 보일 수 있다. 최종 노트에서는 “문법 예시”와 “공격 영향”을 분리해서 설명해야 한다.
- SQLi 대응 노트에서는 `INSERT`도 parameterized query가 필요하다는 쪽으로 연결하면 좋다.

#### 이전/다음 페이지와의 연결

- 이전 p.111은 조회 결과를 합치는 `UNION`이었다.
- p.112는 새 레코드 추가다.
- 다음 p.113은 기존 레코드를 바꾸는 `UPDATE`다.

### p.113 - UPDATE

#### PDF에서 실제로 다룬 내용

- `UPDATE`를 다룬다.
- 기본 형식:
  - `UPDATE <테이블> SET <필드명>=<값> WHERE <조건>`
- 의미:
  - `<조건>`에 맞는 레코드의 `<필드>` 값을 `<변경될 값>`으로 변경한다.
- 예시:
  - `UPDATE member SET name='박명수' WHERE name='이명수'`

#### 핵심 개념

`UPDATE`는 이미 존재하는 행의 값을 바꾸는 명령이다. 핵심은 `SET`과 `WHERE`다. `SET`은 무엇을 바꿀지, `WHERE`는 어떤 행을 바꿀지 결정한다.

#### SQL/SQLi 흐름에서 이 페이지의 역할

SQL Injection이 `UPDATE` 쿼리에 영향을 주면 데이터 변조로 이어질 수 있다. 특히 `WHERE` 조건이 잘못되면 의도한 한 행이 아니라 여러 행이 바뀔 수 있다. 웹 애플리케이션에서는 회원정보 변경, 비밀번호 변경, 권한 변경, 게시글 수정 같은 기능과 연결된다.

#### 나중에 노트화할 때 후보 위치

- `SQL Injection을 위한 SQL 기초`
  - `UPDATE`
  - `SET`
  - `WHERE`
- `SQL Injection 영향`
  - 데이터 변조
  - 조건 조작으로 영향 범위 확대

#### 확인 필요 / 헷갈리는 점

- `UPDATE`에서 `WHERE`가 없거나 조작되면 전체 행에 영향을 줄 수 있다. 최종 노트에서 “WHERE 조건은 조회뿐 아니라 변경 범위도 결정한다”는 식으로 강조하면 좋다.
- PDF의 SQLi 본문은 로그인 인증 우회와 데이터 추출 중심이므로 `UPDATE` 공격은 깊게 다루지 않을 가능성이 있다.

#### 이전/다음 페이지와의 연결

- 이전 p.112는 행 추가인 `INSERT`였다.
- p.113은 행 수정인 `UPDATE`다.
- 다음 p.114는 행 삭제인 `DELETE`로, SQL 기본 문법 파트의 마지막으로 보인다.

### p.114 - DELETE

#### PDF에서 실제로 다룬 내용

- `DELETE`를 다룬다.
- 기본 형식:
  - `DELETE FROM <테이블명> WHERE <조건>`
- 의미:
  - `<테이블명>`에서 `<조건>`에 맞는 레코드를 삭제한다.
- 예시:
  - `DELETE FROM member WHERE name = 'tester'`

#### 핵심 개념

`DELETE`는 조건에 맞는 행을 삭제하는 명령이다. `UPDATE`와 마찬가지로 `WHERE` 조건이 삭제 범위를 결정한다.

#### SQL/SQLi 흐름에서 이 페이지의 역할

SQL Injection의 영향 중 `Database 조작, 파괴`를 이해하는 기반이다. 공격자가 `DELETE` 같은 변경 쿼리의 조건을 조작하거나, 다중 구문 실행이 가능한 환경에서 삭제 구문을 끼워 넣을 수 있으면 데이터 손실로 이어질 수 있다.

#### 나중에 노트화할 때 후보 위치

- `SQL Injection을 위한 SQL 기초`
  - `DELETE`
  - 삭제 조건으로서의 `WHERE`
- `SQL Injection 영향`
  - 데이터 삭제/파괴 가능성

#### 확인 필요 / 헷갈리는 점

- 이 PDF 범위에서 실제로 `DELETE`를 이용한 SQLi 실습까지 가는지는 아직 불확실하다.
- 최종 노트에서는 `DELETE`를 길게 다루기보다 “SQLi 영향이 조회를 넘어 데이터 변경/삭제까지 갈 수 있다”는 근거로 쓰는 편이 적절해 보인다.

#### 이전/다음 페이지와의 연결

- p.107-114는 SQL 기본 문법 구간으로 정리된다.
- 다음 p.115부터 `SQL Injection 이란?`, `Main Cause`, `Impact`가 나오므로 본격적인 SQL Injection 개념 구간이 시작된다.

### p.115 - SQL Injection 정의, 원인, 영향

#### PDF에서 실제로 다룬 내용

- `SQL Injection 이란?`
  - 사용자가 서버에 제출한 데이터가 SQL Query로 사용되어 Database나 시스템에 영향을 주는 공격기법
- `Main Cause`
  - 사용자 입력 데이터를 이용해 동적으로 Query 구문을 완성하는 페이지에서 발생할 수 있다.
  - 사용자 입력이 적절한 검증 없이 DB 쿼리문의 일부로 포함될 때 발생한다.
- `Impact`
  - 웹 애플리케이션 인증 우회
  - Database 덤프, 조작, 파괴
  - 시스템 커맨드 실행, 주로 MS-SQL에서 발생
  - 시스템 주요 파일 노출
- PDF 하단 설명:
  - SQL Injection은 웹 해킹으로 분류하지만 Database 해킹 기법이다.

#### 핵심 개념

SQL Injection의 핵심은 “사용자 입력이 SQL 문법으로 해석되는 것”이다. 사용자는 원래 데이터만 입력해야 하지만, 서버가 그 값을 문자열 결합으로 SQL에 끼워 넣으면 입력값 일부가 SQL 명령 구조를 바꿀 수 있다.

#### SQL/SQLi 흐름에서 이 페이지의 역할

이 페이지는 SQL Injection 본론의 기준점이다. 앞 p.107-114에서 배운 SQL 문법이 이제 공격 표면으로 바뀐다. 특히 `동적 Query 생성`, `입력값 검증 누락`, `DB 쿼리 일부로 포함`이 원인 축이다.

#### 나중에 노트화할 때 후보 위치

- `SQL Injection 개념과 인증 우회`
  - SQLi 정의
  - 발생 원인
  - 영향
- `SQL Injection 방어`
  - 동적 문자열 쿼리가 위험한 이유
  - 입력 검증만으로 충분하지 않은 이유

#### 확인 필요 / 헷갈리는 점

- PDF는 원인에서 `검증`을 강조한다. 최종 방어 노트에서는 이것을 그대로 두되, 현대적 1차 방어는 `Prepared Statement / Parameterized Query`라는 점을 보강해야 한다.
- “SQL Injection은 웹 해킹으로 분류하나 Database 해킹 기법”이라는 표현은 강의 의도상 중요해 보인다. 사용자는 웹 페이지를 통해 공격하지만 실제 피해 대상은 DB라는 뜻으로 풀어야 한다.
- `시스템 커맨드 실행`, `시스템 주요 파일 노출`은 DBMS 기능/권한/설정에 따라 가능성이 달라진다. 일반화하지 말고 조건부 영향으로 써야 한다.

#### 이전/다음 페이지와의 연결

- 이전 p.107-114는 SQL 문법 기초였다.
- p.115는 그 문법이 사용자 입력으로 조작될 때 SQL Injection이 된다는 개념 전환이다.
- 다음 p.116은 웹 애플리케이션의 일반적인 인증 절차를 보여주며, SQLi가 인증 우회로 이어지는 구조를 설명할 것으로 보인다.

### p.116 - 웹 애플리케이션 인증 절차

#### PDF에서 실제로 다룬 내용

- 웹 애플리케이션의 일반적인 인증 절차를 5단계로 설명한다.
  1. ID / Password 입력
  2. SQL Query 생성
  3. Database에 쿼리 전송
  4. Database에서 Query 실행
  5. 반환되는 Return 값에 따라 인증 여부 판단
- 슬라이드 그림은 `SQL Query 생성`과 `회원정보 DB`의 관계를 보여준다.

#### 핵심 개념

로그인은 단순히 화면에서 ID/PW를 비교하는 과정이 아니라, 서버가 입력값으로 SQL 쿼리를 만들고 DB에 질의한 뒤 반환 결과로 인증 여부를 판단하는 과정이다.

#### SQL/SQLi 흐름에서 이 페이지의 역할

SQL Injection 인증 우회는 이 5단계 중 2번과 5번 사이를 노린다. 사용자가 입력한 ID/PW가 SQL Query 생성에 들어가고, 조작된 쿼리의 반환값이 인증 성공처럼 보이면 서버는 공격자를 정상 사용자로 판단할 수 있다.

#### 나중에 노트화할 때 후보 위치

- `SQL Injection 개념과 인증 우회`
  - 로그인 인증 흐름
  - SQL Query 생성 지점
  - DB 반환값과 인증 성공/실패 판단

#### 확인 필요 / 헷갈리는 점

- “Return 값”은 DB가 반환한 레코드셋 또는 조회 결과로 이해하면 된다. 최종 노트에서는 `반환되는 Return 값`을 `조건에 맞는 사용자 레코드가 존재하는가`로 풀어 쓰는 편이 초보자에게 더 명확하다.
- 이 흐름은 SQLi뿐 아니라 인증 로직 전반의 약점 이해에도 중요하다.

#### 이전/다음 페이지와의 연결

- 이전 p.115는 SQLi의 원인과 영향을 설명했다.
- p.116은 그 원인이 로그인 인증 절차에서 어디에 끼어드는지 보여준다.
- 다음 p.117은 SQL Injection 가능성을 확인하기 위한 특수 문자 삽입을 다룬다.

### p.117 - SQL Injection 가능성 확인 문자

#### PDF에서 실제로 다룬 내용

- SQL 구문에 영향을 줄 수 있는 문자들을 다양한 방법으로 삽입하여 결과를 확인한다고 설명한다.
- 예시 문자:
  - `'`
  - `"`
  - `;`
  - `--`
  - `#`
  - `/* */`
- DBMS 예시:
  - MSSQL
  - MySQL
  - ORACLE

#### 핵심 개념

SQL Injection 가능성 확인은 입력값이 데이터로만 처리되는지, 아니면 SQL 문법의 일부로 해석되는지를 보는 과정이다. 따옴표, 주석, 구문 종료 문자 등은 SQL 문자열 구조를 깨거나 뒤 조건을 무력화할 수 있기 때문에 반응 확인에 쓰인다.

#### SQL/SQLi 흐름에서 이 페이지의 역할

이 페이지는 실제 공격 전 “입력값이 SQL 구문에 영향을 주는가”를 확인하는 진단 단계다. 예를 들어 `'` 하나만 넣어도 SQL syntax error가 발생한다면, 입력값이 SQL 문자열 내부에 직접 들어갔을 가능성을 의심할 수 있다.

#### 나중에 노트화할 때 후보 위치

- `SQL Injection 개념과 인증 우회`
  - SQLi 가능성 확인
  - 특수문자와 SQL 구문 영향
- `SQL Injection 방어`
  - 에러 메시지 노출 제한
  - 입력값을 SQL 문법으로 해석시키지 않는 방식

#### 확인 필요 / 헷갈리는 점

- 이 문자들은 DBMS마다 의미가 조금 다를 수 있다. 예를 들어 주석 문법은 MySQL, MSSQL, Oracle에서 차이가 있다.
- 최종 노트에서는 문자 목록을 “외워서 넣는 payload”처럼 보이게 하기보다, `문자열 종료`, `구문 분리`, `주석 처리`라는 역할별로 재구성하는 게 낫다.
- 실습/방어 맥락을 유지해야 하며, 실제 타깃 악용 절차처럼 길게 쓰지 않는다.

#### 이전/다음 페이지와의 연결

- 이전 p.116은 로그인 인증 흐름을 설명했다.
- p.117은 그 흐름에서 입력값이 SQL에 영향을 줄 수 있는지 확인하는 단계다.
- 다음 p.118부터 인증 우회 예시가 본격적으로 나올 것으로 보인다.

### p.118 - 취약한 인증 방식과 동적 SQL

#### PDF에서 실제로 다룬 내용

- `인증우회`를 다룬다.
- 일반적인 애플리케이션의 취약한 인증 방식:
  - 사용자로부터 ID/PW를 입력받는다.
  - 해당 ID와 PW 쌍이 일치하는 레코드가 존재하는지 검사한다.
  - 레코드가 존재하면 인증 성공
  - 레코드가 존재하지 않으면 인증 실패
- 취약한 코드 예시:
  - `id = request("user_id")`
  - `password = request("user_pw")`
  - 입력값을 문자열 결합으로 `strSQL`에 넣는다.
  - `DBconn.execute(strSQL)`로 쿼리를 실행한다.
  - 결과 레코드가 있으면 `chkUser = true`

#### 핵심 개념

취약점의 핵심은 “ID/PW를 DB에서 확인한다”가 아니다. 문제는 사용자 입력값을 SQL 문자열에 그대로 이어 붙여 동적 SQL을 만든다는 점이다. 이 경우 입력값이 데이터가 아니라 SQL 구조를 바꾸는 문법으로 해석될 수 있다.

#### SQL/SQLi 흐름에서 이 페이지의 역할

이 페이지는 인증 우회의 취약한 코드 구조를 보여준다. `where user_id='입력값' and user_pw='입력값'` 형태는 SQL Injection 입문에서 가장 전형적인 예시다. 공격자는 `user_id`나 `user_pw` 위치에 조건식/주석 등을 끼워 넣어 결과 레코드가 존재하도록 만들 수 있다.

#### 나중에 노트화할 때 후보 위치

- `SQL Injection 개념과 인증 우회`
  - 취약한 로그인 로직
  - 동적 SQL 문자열 결합
  - 레코드 존재 여부와 인증 성공
- `SQL Injection 방어`
  - 문자열 결합 쿼리의 위험
  - parameterized query의 필요성

#### 확인 필요 / 헷갈리는 점

- PDF의 코드 예시는 ASP/VBScript 계열로 보인다. 최종 노트에서는 언어 문법보다 구조를 설명해야 한다.
- `if not Rs.eof then chkUser = true`는 “조회 결과가 하나라도 있으면 로그인 성공”이라는 뜻으로 풀어야 초보자가 이해하기 쉽다.
- 이 예시는 password hashing, rate limiting, MFA 같은 현대 인증 요소를 다루지 않는다. SQLi 설명에서는 범위를 인증 우회 구조에 한정하는 게 좋다.

#### 이전/다음 페이지와의 연결

- 이전 p.117은 SQL 구문에 영향을 주는 문자로 가능성을 확인하는 단계였다.
- p.118은 그 문자가 들어갈 수 있는 취약한 로그인 코드 구조를 보여준다.
- 다음 p.119는 인증에 사용되는 SQL Query와 `WHERE` 절을 항상 참으로 만드는 핵심 원리를 설명할 것으로 보인다.

### p.119 - 인증 우회의 핵심: WHERE 조건 조작

#### PDF에서 실제로 다룬 내용

- 인증에 사용되는 SQL Query:
  - `SELECT * FROM user_table WHERE id='사용자입력값' and pw='사용자입력값'`
- 입력된 `id`와 `pw`가 만족하는 레코드셋을 가져온다고 설명한다.
- SQL Injection을 이용한 인증 우회 핵심:
  - `WHERE`절 이하의 조건이 항상 참이 되도록 한다.
  - 쿼리 구문이 에러가 나지 않도록 해야 한다.
  - SQL Injection에 사용할 입력값은 쿼리문에 따라 달라질 수 있다.
- 예시:
  - `SELECT * FROM user_table WHERE 1=1`: 모든 행이 반환
  - `SELECT * FROM user_table WHERE 1=2`: 어떤 레코드셋도 반환되지 않음

#### 핵심 개념

로그인 쿼리의 본질은 `WHERE` 조건을 만족하는 사용자 레코드가 있는지 확인하는 것이다. SQL Injection 인증 우회는 이 조건을 조작해서 원래 ID/PW가 맞지 않아도 레코드가 반환되게 만드는 방식이다.

#### SQL/SQLi 흐름에서 이 페이지의 역할

이 페이지는 인증 우회의 원리를 가장 직접적으로 설명한다. 중요한 것은 특정 payload 암기가 아니라, `WHERE` 절의 논리값이 인증 결과를 결정한다는 점이다. 조건이 항상 참이면 많은 레코드가 반환되고, 취약한 로그인 코드는 그 자체를 인증 성공으로 처리할 수 있다.

#### 나중에 노트화할 때 후보 위치

- `SQL Injection 개념과 인증 우회`
  - `WHERE` 조건과 인증 판단
  - `1=1` / `1=2`의 의미
  - payload는 쿼리 구조에 따라 달라진다는 점

#### 확인 필요 / 헷갈리는 점

- `1=1`은 “마법 문구”가 아니라 SQL 조건식이다. 항상 참인 조건을 만들기 위한 예시로 설명해야 한다.
- `쿼리 구문이 에러가 나지 않도록 해야 함`은 중요하다. SQLi는 SQL 문법을 깨뜨리는 것이 아니라, 공격자가 원하는 방향으로 유효한 SQL을 만들려고 하는 과정이다.
- 입력값은 원래 쿼리의 따옴표 위치, 괄호, DBMS, 주석 처리 방식에 따라 달라진다.

#### 이전/다음 페이지와의 연결

- 이전 p.118은 취약한 로그인 코드 구조를 보여줬다.
- p.119는 그 코드의 `WHERE` 조건을 어떻게 논리적으로 우회하는지 설명한다.
- 다음 p.120은 실제 입력값과 조작된 SQL 쿼리 결과를 보여준다.

### p.120 - 항상 참 조건을 이용한 인증 우회 예시

#### PDF에서 실제로 다룬 내용

- 입력 예시:
  - ID: `' OR '1'='1`
  - PW: `' OR '1'='1`
- 조작된 SQL 쿼리문:
  - `SELECT * FROM member WHERE user_id='' OR '1'='1' AND user_pw='' OR '1'='1'`
- 결과:
  - `member`의 모든 행이 반환된다.
  - 반환되는 레코드셋의 현재 포인터가 일반적으로 첫 번째 레코드에 있게 되므로 첫 번째 사용자의 권한을 갖게 된다고 설명한다.
  - 관리자 계정은 테이블의 처음이나 끝에 두지 말고 별도 관리자 테이블을 구성해야 한다고 설명한다.

#### 핵심 개념

입력값이 SQL 조건식으로 해석되면, 원래 ID/PW 검증 조건 뒤에 항상 참인 조건이 붙을 수 있다. 취약한 로그인 로직이 “레코드가 하나라도 있으면 로그인 성공”으로 처리하면, 전체 행이 반환되는 상황이 인증 우회로 이어진다.

#### SQL/SQLi 흐름에서 이 페이지의 역할

이 페이지는 p.119의 원리를 구체적인 입력값으로 보여준다. 특히 “모든 행이 반환됨”과 “첫 번째 레코드의 권한을 가질 수 있음”은 취약한 코드가 반환 결과를 어떻게 처리하는지와 연결된다.

#### 나중에 노트화할 때 후보 위치

- `SQL Injection 개념과 인증 우회`
  - 항상 참 조건 예시
  - 전체 행 반환
  - 첫 번째 레코드로 로그인될 수 있는 이유
- `오해하기 쉬운 지점`
  - 관리자 계정을 테이블 처음/끝에 두지 않는 것은 근본 방어가 아님

#### 확인 필요 / 헷갈리는 점

- PDF의 “관리자 계정을 테이블 처음이나 끝에 두지 말라”는 조언은 피해를 줄이는 보조 조치로 봐야 한다. 근본 방어는 쿼리 구조가 조작되지 않게 하는 것이다.
- 최종 노트에서는 이 예시를 그대로 길게 반복하기보다, “조건 우회 -> 전체 행 반환 -> 취약한 코드가 첫 행을 사용자로 처리” 흐름을 중심으로 정리하는 편이 좋다.
- SQL 연산자 우선순위 때문에 `AND`/`OR` 조합 결과는 DBMS와 괄호 구조에 따라 해석을 조심해야 한다. 강의 예시는 취약 구조 설명용으로 다룬다.

#### 이전/다음 페이지와의 연결

- 이전 p.119는 `WHERE 1=1` 원리를 설명했다.
- p.120은 그 원리를 로그인 입력값과 조작된 쿼리로 보여준다.
- 다음 p.121은 특정 ID의 권한으로 로그인하는 방식, 즉 전체 행이 아니라 특정 계정을 겨냥하는 예시로 이어진다.

### p.121 - 특정 ID 권한으로 로그인하기

#### PDF에서 실제로 다룬 내용

- 특정 ID의 권한으로 로그인하는 SQL Injection 예시를 다룬다.
- 사용자 ID를 획득한 경우 비밀번호 없이 입력한 사용자의 권한으로 인증을 우회할 수 있다고 설명한다.
- 예시 상황:
  - 게시판에서 관리자의 ID가 `admin`임을 확인했다.
- 입력:
  - ID: `admin'--`
  - PW: `Anything`
- 조작된 SQL 쿼리 일부:
  - `where user_id='admin'-- and passwd='Anything'`
- 결과:
  - `admin` 계정의 권한으로 로그인

#### 핵심 개념

이 예시는 `OR 1=1`처럼 모든 행을 반환시키는 방식이 아니라, 특정 ID 조건만 남기고 뒤의 비밀번호 검증 조건을 주석 처리하는 방식이다. 핵심은 `--` 이후의 `and passwd=...` 조건이 SQL에서 무시된다는 점이다.

#### SQL/SQLi 흐름에서 이 페이지의 역할

p.120이 전체 행 반환을 이용한 우회라면, p.121은 특정 계정으로 인증을 우회하는 흐름이다. 공격자가 사용자 ID를 알고 있으면, 취약한 쿼리 구조에서 비밀번호 조건을 제거해 해당 ID의 레코드만 반환되게 만들 수 있다.

#### 나중에 노트화할 때 후보 위치

- `SQL Injection 개념과 인증 우회`
  - 특정 ID 기반 인증 우회
  - 주석 처리로 뒤 조건 제거
  - 사용자 ID 노출과 공격 가능성의 연결
- `관련 노트 후보`
  - 웹 애플리케이션 정보수집
  - 로그인 Brute Force와 계정 보호

#### 확인 필요 / 헷갈리는 점

- `--` 주석은 DBMS별로 세부 문법 차이가 있다. MySQL에서는 `--` 뒤에 공백이 필요한 경우가 있으므로 최종 노트에서 “DBMS마다 주석 문법이 다를 수 있음”을 남기는 게 안전하다.
- 게시판에서 ID가 보이는 것 자체가 곧 취약점은 아니지만, SQLi와 결합되면 특정 계정 우회의 재료가 될 수 있다.
- 이 예시는 공격 절차로 자세히 확장하기보다 “뒤 조건을 주석 처리한다”는 SQL 해석 원리 중심으로 정리한다.

#### 이전/다음 페이지와의 연결

- 이전 p.120은 항상 참 조건으로 전체 행을 반환시키는 예시였다.
- p.121은 주석 처리로 특정 사용자의 비밀번호 조건을 무력화하는 예시다.
- 다음 p.122부터는 인증 우회에서 데이터 알아내기 쪽으로 넘어가며, 변환 에러를 이용한 정보 노출을 다룬다.

### p.122 - 변환 에러를 이용한 데이터 알아내기

#### PDF에서 실제로 다룬 내용

- 문자열과 숫자 데이터의 비교 연산은 형 변환 에러를 유발한다고 설명한다.
- 이때 문자열 데이터가 에러 메시지에 포함될 수 있다고 설명한다.
- `'` 같은 문자를 입력하여 에러를 유발하면 Column명이 도출될 수 있다고 설명한다.
- 예시:
  - `nuno' and user_pw=1--`

#### 핵심 개념

에러 기반 SQL Injection은 DB가 오류를 내는 과정에서 내부 정보가 에러 메시지로 노출되는 것을 이용한다. 중요한 지점은 공격자가 DB를 직접 보는 것이 아니라, 웹 애플리케이션이 DB 에러를 화면에 그대로 보여줄 때 정보가 새어 나간다는 것이다.

#### SQL/SQLi 흐름에서 이 페이지의 역할

이 페이지부터 SQL Injection의 목표가 인증 우회에서 정보 획득으로 확장된다. 인증 우회는 `WHERE` 조건 조작이 핵심이었다면, 에러 기반 정보 획득은 의도적으로 타입/문법 오류를 만들고 에러 메시지에 포함된 내부 정보를 관찰하는 흐름이다.

#### 나중에 노트화할 때 후보 위치

- `Error/UNION 기반 정보 추출과 Schema 파악`
  - 형 변환 에러
  - 에러 메시지 정보 노출
  - Column명 노출 가능성
- `SQL Injection 방어`
  - 상세 DB 에러 메시지 제한

#### 확인 필요 / 헷갈리는 점

- 에러 메시지가 사용자에게 그대로 노출되지 않는 환경에서는 이 방식의 관찰 가능성이 줄어든다.
- “Column명이 도출될 수 있다”는 말은 DBMS, 쿼리 구조, 에러 출력 방식에 따라 달라질 수 있으므로 조건부로 설명해야 한다.
- 최종 노트에서는 payload 자체보다 “타입 불일치 -> DB 에러 -> 애플리케이션이 에러 출력 -> 내부 정보 노출” 순서가 중요하다.

#### 이전/다음 페이지와의 연결

- 이전 p.121까지는 로그인 인증 우회였다.
- p.122부터는 에러를 이용해 DB 내부 정보를 알아내는 단계다.
- 다음 p.123은 `UNION`을 이용한 mismatch와 정보 획득을 다룬다.

### p.123 - UNION을 이용한 데이터 알아내기

#### PDF에서 실제로 다룬 내용

- `UNION`을 이용한 방법을 다룬다.
- `UNION`을 이용해 mismatch를 발생시키면 `SELECT` 구문에 사용된 Field 개수를 알아낼 수 있다고 설명한다.
- 문자열 데이터의 경우 그 데이터를 알아낼 수 있다고 설명한다.
- 정상적인 `UNION` 구문은 레코드셋의 필드 개수와 타입이 맞아야 한다.
- 데이터 타입 오류가 발생하면 오류가 발생한 값이 에러 메시지에 포함될 수 있다.
- 예시:
  - `ID : nuno' UNION select 'a',1--`
- 어떤 필드에서 에러가 발생했는지 알 수 없으므로 앞 필드부터 하나씩 문자로 바꾸며 테스트한다고 설명한다.
- 단점:
  - 문자열 형태의 값만 알아낼 수 있다.
  - 숫자형 데이터는 묵시적 형 변환 시 에러가 발생하지 않을 수 있어 에러를 통한 방법으로는 알아낼 수 없다.

#### 핵심 개념

`UNION` 기반 정보 획득은 원래 쿼리의 결과와 공격자가 만든 `SELECT` 결과를 합치려는 방식이다. 이때 컬럼 개수와 타입이 맞아야 하므로, mismatch와 에러 메시지를 통해 원래 쿼리 구조를 추정할 수 있다.

#### SQL/SQLi 흐름에서 이 페이지의 역할

p.111에서 배운 정상 `UNION` 문법이 여기서 SQL Injection 정보 추출 기법으로 재사용된다. 공격자는 단순히 로그인 우회를 넘어서, 원래 쿼리의 컬럼 수/타입을 파악하고 출력 가능한 위치를 찾으려 한다.

#### 나중에 노트화할 때 후보 위치

- `Error/UNION 기반 정보 추출과 Schema 파악`
  - `UNION` 조건: 컬럼 개수와 타입 일치
  - mismatch로 구조 추정
  - 에러 메시지 기반 정보 노출
- `SQL Injection을 위한 SQL 기초`
  - p.111 `UNION`과 강하게 링크

#### 확인 필요 / 헷갈리는 점

- PDF의 “제일 앞의 필드부터 하나씩 문자로 바꿔 테스트”는 실습 절차에 가까우므로 최종 개념 노트에서는 원리 중심으로 요약하고, 실습 노트가 있으면 별도로 기록하는 편이 좋다.
- 숫자형 데이터는 자동 형 변환 때문에 에러가 안 날 수 있다는 제한은 중요하다. 에러 기반 정보 획득이 항상 통하는 것은 아니다.
- `Field`는 최종 노트에서 `컬럼` 또는 `SELECT 결과 컬럼`으로 풀어 쓰는 게 좋다.

#### 이전/다음 페이지와의 연결

- 이전 p.122는 형 변환 에러를 이용한 정보 노출이었다.
- p.123은 `UNION` 조건과 mismatch를 이용해 쿼리 구조를 추정한다.
- 다음 p.124는 DB Schema 파악으로 넘어가므로, 정보 추출의 목표가 더 체계화된다.

### p.124 - DB Schema 파악과 Client-side Validation 우회

#### PDF에서 실제로 다룬 내용

- `DB Schema 파악`을 다룬다.
- SQL Injection을 이용하면 단순 인증 우회만 가능한 것이 아니라고 설명한다.
- DB Schema를 파악한 후 DB의 모든 레코드를 알아낼 수 있다고 설명한다.
- `Client Side Validation 우회`를 함께 언급한다.
- DB Schema를 파악하는 Query 구문은 일반적으로 입력 문자열 길이가 비교적 길기 때문에, 입력값 사이즈 제한이 있는 경우 이를 우회해야 한다고 설명한다.

#### 핵심 개념

DB Schema는 DB 안에 어떤 데이터베이스, 테이블, 컬럼이 있는지에 대한 구조 정보다. 공격자가 schema를 알게 되면 “어디에 어떤 데이터가 있는지”를 추정할 수 있고, 그 다음 실제 데이터 추출로 이어질 수 있다.

#### SQL/SQLi 흐름에서 이 페이지의 역할

이 페이지는 SQLi 목표가 인증 우회에서 DB 구조 파악으로 확장되는 전환점이다. 또한 이전에 배운 Client-side Validation 우회가 다시 연결된다. 긴 입력값 제한이 브라우저/JavaScript에만 있으면 proxy나 request 조작으로 우회될 수 있기 때문이다.

#### 나중에 노트화할 때 후보 위치

- `Error/UNION 기반 정보 추출과 Schema 파악`
  - schema 파악의 의미
  - Database -> Table -> Column -> Data 순서
- `Client-side Validation 우회와 Server-side Validation`
  - 입력 길이 제한 우회와 연결
- `SQL Injection 방어`
  - 서버 측 길이/형식 검증

#### 확인 필요 / 헷갈리는 점

- “DB의 모든 레코드를 알아낼 수 있다”는 표현은 가능성을 말하는 것이며, 실제 가능 여부는 DB 권한, 에러 노출, 쿼리 구조, 방어 설정에 따라 달라진다.
- 입력값 사이즈 제한 우회는 SQLi 자체라기보다 SQLi를 수행하기 위한 보조 조건이다. 최종 노트에서 주제 경계를 흐리지 않게 해야 한다.

#### 이전/다음 페이지와의 연결

- 이전 p.123은 `UNION`과 mismatch를 이용한 정보 추정이었다.
- p.124는 그 정보 추정의 목표를 DB Schema 파악으로 확장한다.
- 다음 p.125는 Database명을 알아내는 구체 흐름이다.

### p.125 - Database명 알아내기

#### PDF에서 실제로 다룬 내용

- `database()` 함수를 다룬다.
- PDF 설명:
  - `database()`는 Database 이름을 문자열로 반환하는 MySQL 함수다.
  - 문자열을 숫자와 비교하는 구문이 되면 형 변환 중 오류가 발생한다.
  - 에러 메시지를 화면에 표시하는 경우, 변환에 실패한 값인 Database 이름이 에러 메시지에 포함될 수 있다.
- 예시:
  - `' or updatexml(1, concat('[DB]=', database()), 1) –`
- 실행 흐름 설명:
  - 함수 인자들을 먼저 계산
  - 계산된 값을 함수에 전달
  - 그 다음 논리 오류/XPath 오류 발생

#### 핵심 개념

이 페이지는 DBMS 함수와 에러 메시지를 결합해 현재 Database 이름을 노출시키는 흐름을 설명한다. 핵심은 `database()`가 이름을 문자열로 만들고, 그 값이 에러 메시지에 포함되도록 유도한다는 점이다.

#### SQL/SQLi 흐름에서 이 페이지의 역할

schema 파악의 첫 단계다. 현재 연결된 Database 이름을 알면, 이후 `information_schema`에서 해당 DB의 테이블과 컬럼을 좁혀가며 찾을 수 있다.

#### 나중에 노트화할 때 후보 위치

- `Error/UNION 기반 정보 추출과 Schema 파악`
  - Database명 확인
  - DBMS 함수와 에러 메시지
  - `database()`는 MySQL 계열 함수라는 점

#### 확인 필요 / 헷갈리는 점

- PDF에 `MSQL`처럼 보이는 표기가 있으나 맥락상 MySQL 함수 `database()`를 말하는 것으로 보인다.
- `updatexml()`은 MySQL XML 함수이고, 이를 이용한 에러 유도는 DBMS/버전/설정에 따라 다르게 동작할 수 있다.
- 최종 노트에서는 구체 payload보다 “함수 결과를 에러 메시지에 노출시키는 방식”을 중심으로 정리하는 게 안전하다.

#### 이전/다음 페이지와의 연결

- 이전 p.124는 DB Schema 파악의 필요성을 설명했다.
- p.125는 그 첫 정보인 Database명을 얻는 흐름이다.
- 다음 p.126은 Table명을 알아내는 단계로 넘어간다.

### p.126 - Table명 알아내기 개요

#### PDF에서 실제로 다룬 내용

- Table명을 알아내는 방법을 다룬다.
- 에러를 유발시켜 에러 메시지에 Table 이름이 포함되게 한다고 설명한다.
- Table명을 포함한 시스템 테이블:
  - `information_schema.tables`
  - `information_schema.columns`
- 모든 테이블 이름을 알아내는 방법:
  - 알아낸 Table 이름을 하나씩 제거하면서 알아냄
  - 알아낸 Table 이름과 비교 연산을 통해 하나씩 차례대로 알아냄
  - `limit` 구문을 이용하여 검색 결과를 하나씩 추출함

#### 핵심 개념

`information_schema`는 DB 구조 정보를 담는 메타데이터 영역이다. 공격자가 이 영역에 접근할 수 있으면 사용자 테이블 이름과 컬럼 이름을 추정할 수 있다.

#### SQL/SQLi 흐름에서 이 페이지의 역할

Database명을 알아낸 뒤에는 그 DB 안의 Table명을 찾아야 한다. p.126은 Table명 추출의 출발점이며, `information_schema.tables`와 `information_schema.columns`가 schema 탐색의 핵심 자료원으로 등장한다.

#### 나중에 노트화할 때 후보 위치

- `Error/UNION 기반 정보 추출과 Schema 파악`
  - `information_schema`
  - Table명 파악
  - `LIMIT`으로 결과 하나씩 확인
- `SQL Injection 방어`
  - DB 계정 권한 최소화
  - 메타데이터 접근 제한 가능성

#### 확인 필요 / 헷갈리는 점

- `information_schema`는 MySQL에서 특히 익숙한 이름이다. 다른 DBMS는 시스템 카탈로그/메타데이터 조회 방식이 다를 수 있다.
- “알아낸 Table 이름을 하나씩 제거”라는 표현은 최종 노트에서 `이미 확인한 값을 제외하며 다음 값을 찾는 방식`으로 풀어야 이해가 쉽다.
- 이 페이지는 실습 절차 냄새가 강하므로 최종 개념 노트에서는 원리와 흐름 중심으로 정리한다.

#### 이전/다음 페이지와의 연결

- 이전 p.125는 Database명 확인이었다.
- p.126은 해당 Database 안의 Table명을 찾는 방법 개요다.
- 다음 p.127은 `information_schema.tables`를 더 구체적으로 다룬다.

### p.127 - information_schema.tables

#### PDF에서 실제로 다룬 내용

- `information_schema.tables`를 다룬다.
- 사용자 테이블 정보를 확인할 수 있다고 설명한다.

#### 핵심 개념

`information_schema.tables`는 테이블 목록을 확인하는 메타데이터 테이블이다. 공격자가 SQLi로 이 정보를 조회할 수 있으면, 실제 데이터가 들어 있는 테이블명을 찾는 데 사용할 수 있다.

#### SQL/SQLi 흐름에서 이 페이지의 역할

Table명 파악의 구체 대상이 된다. Database명을 안 뒤 `information_schema.tables`에서 해당 DB의 테이블 목록을 확인하는 흐름으로 이어진다.

#### 나중에 노트화할 때 후보 위치

- `Error/UNION 기반 정보 추출과 Schema 파악`
  - `information_schema.tables`
  - 테이블 목록 확인

#### 확인 필요 / 헷갈리는 점

- PDF에는 설명이 매우 짧다. 최종 노트에서는 p.126, p.129와 묶어 설명하는 편이 낫다.
- “사용자 테이블 정보만 확인”이라는 표현은 DBMS/쿼리 조건에 따라 달라질 수 있으므로, MySQL의 메타데이터 테이블 관점으로 제한해 설명한다.

#### 이전/다음 페이지와의 연결

- 이전 p.126은 Table명 파악 방법 전체를 설명했다.
- p.127은 그중 `information_schema.tables`를 지정한다.
- 다음 p.128은 `information_schema.columns`로 보이며, Table명뿐 아니라 Column명까지 이어진다.

### p.128 - information_schema.columns

#### PDF에서 실제로 다룬 내용

- `information_schema.columns`를 다룬다.
- DB, Table, Column 정보를 모두 얻어낼 수 있다고 설명한다.

#### 핵심 개념

`information_schema.columns`는 컬럼 단위 메타데이터를 담는다. 어떤 DB의 어떤 테이블에 어떤 컬럼이 있는지 확인할 수 있기 때문에, Table명뿐 아니라 Column명 추출에도 연결된다.

#### SQL/SQLi 흐름에서 이 페이지의 역할

Schema 파악이 더 정밀해지는 지점이다. 테이블명만 알아서는 실제 데이터를 조회하기 어렵고, 어떤 컬럼에 ID, 비밀번호, 이메일 같은 값이 있는지 알아야 데이터 추출이 가능해진다.

#### 나중에 노트화할 때 후보 위치

- `Error/UNION 기반 정보 추출과 Schema 파악`
  - `information_schema.columns`
  - Table명과 Column명 연결
  - Data 추출 전 단계

#### 확인 필요 / 헷갈리는 점

- p.128은 제목상 `Table명 알아내기(cont)`에 포함되어 있지만, 내용은 Column명 파악으로 이어지는 다리 역할도 한다.
- 최종 노트에서는 `Database명 -> Table명 -> Column명 -> Data` 순서를 명시하면 이 페이지의 위치가 선명해진다.

#### 이전/다음 페이지와의 연결

- 이전 p.127은 테이블 목록 중심의 `information_schema.tables`였다.
- p.128은 컬럼 메타데이터까지 포함하는 `information_schema.columns`다.
- 다음 p.129는 알아낸 테이블 이름을 하나씩 출력하는 흐름이다.

### p.129 - 테이블 이름을 하나씩 출력

#### PDF에서 실제로 다룬 내용

- 알아낸 테이블 이름을 하나씩 출력하는 흐름을 다룬다.
- 첫 번째 테이블 이름 알아내기
- 두 번째 테이블 이름 알아내기

#### 핵심 개념

테이블 목록은 여러 행으로 반환될 수 있다. 공격자는 한 번에 전체 목록을 보지 못하는 상황에서도 `LIMIT`이나 비교 조건 등을 이용해 결과를 하나씩 확인할 수 있다.

#### SQL/SQLi 흐름에서 이 페이지의 역할

p.126에서 언급한 `limit` 기반 하나씩 추출 흐름의 구체 단계로 보인다. schema 탐색은 한 번에 모든 정보를 얻는 과정이라기보다, DB명, 테이블명, 컬럼명을 단계적으로 좁혀가는 과정이다.

#### 나중에 노트화할 때 후보 위치

- `Error/UNION 기반 정보 추출과 Schema 파악`
  - 여러 결과를 하나씩 추출하는 방식
  - Table명 enumeration

#### 확인 필요 / 헷갈리는 점

- PDF 추출 텍스트에는 실제 쿼리나 그림 세부가 빠져 있을 수 있다. 페이지 이미지에는 쿼리 예시가 있을 가능성이 있으므로, 최종 노트 작성 전에는 p.129 스크린샷/PDF 화면을 직접 확인하는 것이 좋다.
- “하나씩 출력”은 SQLi 실습 절차와 연결되므로, 개념 노트에서는 자세한 payload 나열보다 단계적 schema enumeration의 의미를 중심으로 쓴다.

#### 이전/다음 페이지와의 연결

- 이전 p.128은 `information_schema.columns`의 역할을 설명했다.
- p.129는 테이블명을 실제로 하나씩 확인하는 단계다.
- 다음 p.130은 Column명 알아내기 구간으로 넘어간다.

### p.130 - Column명 알아내기

#### PDF에서 실제로 다룬 내용

- `information_schema.columns`를 이용한 Column명 확인을 다룬다.
- `information_schema.columns`에는 `table_name`과 `column_name`이 포함되어 있으므로, `table_name`도 알아낼 수 있다고 설명한다.

#### 핵심 개념

컬럼명은 실제 데이터 추출에 필요한 마지막 구조 정보에 가깝다. 테이블명을 알아도 컬럼명을 모르면 어떤 값을 조회해야 할지 알기 어렵다. `information_schema.columns`는 테이블과 컬럼의 대응 관계를 보여준다.

#### SQL/SQLi 흐름에서 이 페이지의 역할

Database명과 Table명을 파악한 뒤, 실제 민감 데이터가 들어 있을 가능성이 있는 컬럼명을 찾는 단계다. 예를 들어 사용자 테이블에서 `id`, `pw`, `email`, `role` 같은 컬럼이 있는지 확인하는 흐름으로 이어질 수 있다.

#### 나중에 노트화할 때 후보 위치

- `Error/UNION 기반 정보 추출과 Schema 파악`
  - Column명 파악
  - `table_name` / `column_name`
  - Table명과 Column명의 대응 관계

#### 확인 필요 / 헷갈리는 점

- p.128에서 이미 `information_schema.columns`가 등장했으므로 최종 노트에서는 p.128과 p.130을 합쳐 설명하는 편이 자연스럽다.
- `information_schema.columns`를 이용하면 Table명도 확인할 수 있다는 점 때문에, 실제 흐름은 꼭 `tables -> columns` 순서만은 아닐 수 있다.

#### 이전/다음 페이지와의 연결

- 이전 p.129는 테이블 이름을 하나씩 확인하는 흐름이었다.
- p.130은 컬럼 이름 확인으로 이어진다.
- 다음 p.131은 Table과 Column 이름을 바탕으로 실제 Data를 알아내는 단계다.

### p.131 - Data 알아내기

#### PDF에서 실제로 다룬 내용

- Table과 Column 이름을 파악했으므로 각각의 데이터도 알아낼 수 있다고 설명한다.
- 제목은 `Data 알아내기`다.

#### 핵심 개념

Schema 파악의 최종 목적은 실제 데이터 조회다. Database명, Table명, Column명을 알게 되면 어느 테이블의 어느 컬럼을 조회해야 할지 알 수 있으므로 데이터 추출 가능성이 커진다.

#### SQL/SQLi 흐름에서 이 페이지의 역할

p.124-130의 schema enumeration 흐름이 실제 데이터 추출로 이어지는 결론 페이지다. 이 페이지 자체의 텍스트는 짧지만, SQLi의 피해가 단순 로그인 우회를 넘어 정보 유출로 확장된다는 점을 보여준다.

#### 나중에 노트화할 때 후보 위치

- `Error/UNION 기반 정보 추출과 Schema 파악`
  - Schema 파악 후 데이터 조회
  - Database -> Table -> Column -> Data 순서의 마지막 단계
- `SQL Injection 영향`
  - DB 덤프/정보 유출

#### 확인 필요 / 헷갈리는 점

- PDF 추출 텍스트에는 구체 예시가 거의 없다. 슬라이드 이미지에는 쿼리 예시가 있을 수 있으므로 최종 노트 작성 전 p.131 화면 확인이 필요하다.
- 최종 노트에서는 실제 데이터 탈취 절차를 과하게 상세화하지 않고, “schema를 알면 데이터 접근 경로가 열린다”는 개념 중심으로 제한한다.

#### 이전/다음 페이지와의 연결

- 이전 p.130은 Column명 파악이었다.
- p.131은 그 결과 실제 Data를 알아낼 수 있다는 결론이다.
- 다음 p.132는 SQL Injection 대응책으로, 공격 흐름을 닫고 방어로 전환한다.

### p.132 - SQL Injection 대응책

#### PDF에서 실제로 다룬 내용

- SQL Injection 대응책을 다룬다.
- 사용자로부터 입력되는 값을 반드시 서버에서 체크해야 한다고 설명한다.
- Client Side Script로도 입력값 제한이 가능하지만, 공격자가 충분히 우회 가능하므로 반드시 서버에서 체크해야 한다고 설명한다.
- SQL Injection에 사용되는 문자를 필터링한다고 설명한다.
  - 예: `'`, `"`, `;`, `--`, `#` 등
  - 예: `select`, `insert`, `delete`, `update`, `drop`, `union`, `group by`, `having` 등
- 에러 메시지를 제한한다.
- Web 방화벽을 사용한다.

#### 핵심 개념

PDF의 대응책은 서버 측 검증, 위험 문자/키워드 필터링, 에러 메시지 제한, WAF 사용으로 구성된다. 여기서 가장 확실히 가져가야 할 점은 Client-side 검증만으로는 방어가 되지 않으며, 서버에서 다시 처리해야 한다는 것이다.

#### SQL/SQLi 흐름에서 이 페이지의 역할

이 페이지는 p.115-131에서 본 SQLi 흐름을 방어 관점으로 닫는다. 특히 p.124의 Client-side Validation 우회와 다시 연결된다. 공격자는 브라우저 제한을 우회할 수 있으므로 서버가 신뢰 경계가 되어야 한다.

#### 나중에 노트화할 때 후보 위치

- `SQL Injection 방어`
  - 서버 측 검증
  - 에러 메시지 제한
  - WAF의 보조적 역할
  - keyword/blocklist filtering의 한계
  - Prepared Statement / Parameterized Query 보강 필요

#### 확인 필요 / 헷갈리는 점

- PDF는 필터링을 강조하지만, 최종 노트에서는 필터링을 1차 방어처럼 두면 안 된다. 현대적인 핵심 방어는 SQL과 데이터를 분리하는 Prepared Statement / Parameterized Query다.
- 위험 문자/키워드 필터링은 우회 가능성이 높고 정상 입력을 깨뜨릴 수 있다. 보조 방어 또는 실습용 방어로 낮춰 설명하는 편이 맞다.
- WAF는 애플리케이션 코드를 고치는 대체재가 아니라 보조 방어다.
- `에러 메시지 제한`은 p.122-125의 error-based 정보 노출을 줄이는 대응책으로 연결된다.

#### 이전/다음 페이지와의 연결
