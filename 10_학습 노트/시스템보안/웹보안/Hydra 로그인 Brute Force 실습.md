## p.77-79 로그인 Brute Force 실습

> [!important]
> 이 구간은 정리 중에도 정보 생략 금지. 중복은 한 위치로 합치되, PDF 범위, 실제 실행 명령, 실행 결과, 옵션 해설, 비교 명령, raw 메모는 모두 보존한다.

목표는 Hydra 명령을 외우는 것이 아니라, **로그인 실패/성공 응답 차이를 이용해 ID/PW 조합을 반복 검증하는 구조와 그 방어책**을 이해하는 것이다.

### PDF 범위

| 페이지 | 주제 | 핵심 |
|---|---|---|
| p.77 | `ID/PW Brute Forcing` | ID와 Password 조합을 반복 입력한다. 사전 파일을 이용하면 Dictionary Attack이 된다. 로그인 실패/성공 응답 메시지 차이로 올바른 계정 여부를 확인한다. 예시 도구는 brutus-aet2, wwwhack, hydra 등이다. |
| p.78 | `[실습] Hydra 를 이용한 Brute Forcing` | `hydra`의 `http-post-form` 예시를 다룬다. 실제 실습에서는 대상 URL, 파라미터명, 실패/성공 판별 문자열을 확인한 뒤 명령을 구성한다. |
| p.79 | `ID/PW Brute Forcing 대응책` | 실패 메시지 정보 제한, 계정 잠금 정책, 로그인 처리 지연, IP 기반 동시접속 제어가 대응책으로 제시된다. |

### 실제 실행 요약

실제로 입력한 명령:

```bash
hydra -V -l unoh03 -x 3:3:1 172.16.0.150 -s 8080 http-post-form "/member/loginModel.php:id=^USER^&pw=^PASS^:history"
```

한 줄 해석: **ID 하나를 고정하고, 숫자 3자리 비밀번호 후보를 Hydra가 직접 만들어 `POST /member/loginModel.php`에 반복 대입한 실습**이다.

실행 결과:

| 관찰한 출력 | 의미 |
|---|---|
| `Hydra v9.6` | Kali에서 Hydra 9.6 사용 |
| `max 16 tasks per 1 server` | `-t 1`을 주지 않아 기본값인 최대 16개 task가 병렬 실행됨 |
| `1000 login tries (l:1/p:1000)` | ID 1개와 비밀번호 후보 1000개 조합을 시도 |
| `pass "000"`부터 순차 출력 | `-V` 때문에 각 시도 값이 화면에 표시됨 |
| `[8080][http-post-form] host: 172.16.0.150 login: unoh03 password: <REDACTED>` | 유효한 로그인 조합을 찾음 |

핵심은 `history`가 실패 판별 문자열로 잘 동작했다는 점이다. 실패 응답에는 `history.go(-1)`가 포함되고, 성공 응답에는 해당 문자열이 없어서 Hydra가 성공/실패를 구분할 수 있었다.

이 출력은 계정 잠금 방어 적용 전 실행 결과다. 방어 구현 후에도 같은 실습 명령을 한 번 더 실행해, 반복 로그인 시도가 서버 측 실패 횟수/잠금 상태 관리 대상으로 들어가는지 확인했다.

### 실제 명령어 분해

| 명령 조각 | 역할 | 의미 | 이번 실습에서의 의미 |
|---|---|---|---|
| `hydra` | 실행 파일 | Hydra 실행 | 로그인 시도 자동화 도구를 실행한다. |
| `-V` | 상세 출력 | 시도하는 ID/PW 조합과 진행 상황을 자세히 출력 | `000`, `001`, `002`처럼 어떤 후보를 넣는지 확인 가능하다. |
| `-l unoh03` | 단일 로그인 ID | ID를 하나로 고정하고 비밀번호만 바꿔가며 시도 | 로그인 ID는 하나만 사용한다. |
| `-x 3:3:1` | 비밀번호 자동 생성 | 비밀번호 후보를 사전 파일에서 읽지 않고 Hydra가 직접 생성한다. 형식은 `-x 최소길이:최대길이:문자셋`이다. | `000`부터 `999`까지 총 1000개 숫자 후보를 생성한다. |
| `172.16.0.150` | 대상 호스트 | 실습 WEB 서버 IP | 공격 대상이 아니라 수업용 실습 서버다. |
| `-s 8080` | 대상 포트 | 기본 포트 대신 8080 포트로 접속 | Apache가 8080에서 동작하므로 필요하다. |
| `http-post-form` | Hydra 모듈 | HTTP POST 로그인 폼 대상으로 시도 | `login.php`의 form method가 `post`이므로 사용한다. |
| `/member/loginModel.php` | 로그인 처리 경로 | 로그인 요청을 처리하는 PHP 파일 | `login.php`의 form action 값이다. |
| `id=^USER^&pw=^PASS^` | POST body | `^USER^`에는 고정 ID, `^PASS^`에는 생성된 비밀번호 후보가 들어간다. | `login.php`의 input name이 `id`, `pw`다. |
| `history` | 실패 판별 문자열 | 응답에 `history`가 있으면 로그인 실패로 판단 | `loginModel.php`의 실패 응답에 `history.go(-1)`가 포함된다. |

Hydra 명령의 마지막 큰따옴표 안은 다음 3부분으로 나뉜다.

```text
"로그인_처리_경로:POST_데이터:실패_판별_문자열"
```

이번 실습 값:

```text
"/member/loginModel.php:id=^USER^&pw=^PASS^:history"
```

### `-x 3:3:1` 상세 해석

`-x`는 사전 파일 없이 후보를 직접 생성하는 옵션이다.

```text
-x 최소길이:최대길이:문자셋
```

| 부분 | 의미 | 이번 값 | 해석 |
|---|---|---|---|
| `-x` | brute force 생성 옵션 | `-x` | 비밀번호 후보를 파일에서 읽지 않고 Hydra가 규칙에 따라 직접 만든다. |
| 첫 번째 `3` | 최소 길이 | `3` | 생성할 비밀번호 후보는 최소 3자리다. |
| 두 번째 `3` | 최대 길이 | `3` | 생성할 비밀번호 후보는 최대 3자리다. 최소와 최대가 같으므로 정확히 3자리만 시도한다. |
| 마지막 `1` | 문자셋 선택 | `1` | 숫자 문자셋을 의미한다. 즉 `0-9`만 사용한다. |
| 전체 `3:3:1` | 생성 범위 | `3:3:1` | 숫자로만 이루어진 3자리 후보를 생성한다. |
| 생성 예 | 실제 후보 | `000`, `001`, `002`, ... | 앞자리에 `0`이 붙는 값도 포함된다. |
| 총 시도 수 | 후보 개수 | `1000` | `000`부터 `999`까지라서 10 x 10 x 10 = 1000개다. |

즉 `-x 3:3:1`은 "숫자 3자리 비밀번호를 전부 만들어서 넣어봐라"에 가깝다. 사전 파일을 쓰는 `-P pass.txt`와 달리, 파일을 준비하지 않아도 정해진 규칙 안에서 가능한 값을 생성한다.

문자셋에서 자주 쓰는 기호:

| 문자셋 기호 | 의미 | 포함 문자 |
|---|---|---|
| `1` | 숫자 | `0-9` |
| `a` | 소문자 | `a-z` |
| `A` | 대문자 | `A-Z` |
| 그 외 문자 | 문자 그대로 포함 | 예: `@`, `_`, `!` |

후보 개수는 문자셋 크기와 길이에 따라 급격히 늘어난다.

```text
후보 수 = 문자셋크기^길이
```

| 옵션 | 의미 | 후보 수/예 |
|---|---|---|
| `-x 3:3:1` | 숫자 3자리만 생성 | `000` ~ `999`, 총 1000개 |
| `-x 6:6:1` | 숫자 6자리만 생성 | `000000` ~ `999999`, 총 1,000,000개 |
| `-x 4:6:a` | 소문자 4~6자리 생성 | `aaaa`, `aaab`, ... |
| `-x 8:8:aA1` | 소문자+대문자+숫자 8자리 생성 | 후보 수가 매우 커짐 |
| `-x 3:3:1@` | 숫자와 `@`를 섞은 3자리 생성 | `00@`, `0@1` 등 |

그래서 3자리 실습 계정에 `-x 6:6:1`을 쓰면 불필요하게 오래 걸린다.

### `http-post-form` 모듈 구조

웹 로그인 폼을 대상으로 할 때는 `http-post-form` 또는 `https-post-form`을 쓴다.

기본 형태:

```text
"경로:POST_데이터:실패_또는_성공_판별_문자열"
```

| 부분 | 이번 값 | 의미 | 확인 방법 |
|---|---|---|---|
| 경로 | `/member/loginModel.php` | 로그인 요청을 처리하는 서버 경로 | `login.php`의 form action 또는 Paros 요청 |
| POST 데이터 | `id=^USER^&pw=^PASS^` | 로그인 폼 body. Hydra가 `^USER^`, `^PASS^`를 바꿔 넣음 | Paros Request 탭 |
| 판별 문자열 | `history` | 응답에 이 문자열이 있으면 실패로 판단 | 실패 응답의 `history.go(-1)` |

Hydra의 HTTP form 모듈은 `^USER^`와 `^PASS^`를 치환 자리로 사용한다. Base64 인코딩이 필요한 경우에는 `^USER64^`, `^PASS64^` 형태도 지원한다.

판별 문자열은 특히 중요하다.

| 방식 | 의미 | 예 |
|---|---|---|
| `F=문자열` | 이 문자열이 있으면 실패로 판단 | `F=로그인 실패`, `F=history` |
| `S=문자열` | 이 문자열이 있으면 성공으로 판단 | `S=Welcome`, `S=Dashboard` |
| 접두사 없음 | 기본적으로 실패 조건처럼 쓰는 경우가 많음 | `history` |

이번 실습에서는 실패 응답에 `history.go(-1)`가 반복적으로 나타났으므로 `history`를 실패 판별 문자열로 사용했다.

> [!tip]
> 실패/성공 문자열을 잘못 잡으면 Hydra가 실패를 성공으로 보거나, 성공을 실패로 볼 수 있다. 이 값은 Paros, Burp, ZAP, 브라우저 개발자 도구 같은 프록시/관찰 도구로 실제 응답을 보고 정해야 한다.

### 실습 명령 읽는 순서

```bash
hydra -V -l unoh03 -x 3:3:1 172.16.0.150 -s 8080 http-post-form "/member/loginModel.php:id=^USER^&pw=^PASS^:history"
```

1. `hydra`: Hydra를 실행한다.
2. `-V`: 어떤 ID/PW를 시도하는지 화면에 자세히 보여준다.
3. `-l unoh03`: ID는 하나로 고정한다.
4. `-x 3:3:1`: 비밀번호는 숫자 3자리 후보를 직접 생성한다.
5. `172.16.0.150`: 대상 WEB 서버 IP로 보낸다.
6. `-s 8080`: 대상 포트는 8080이다.
7. `http-post-form`: HTTP POST 로그인 폼 모듈을 사용한다.
8. `/member/loginModel.php`: 로그인 처리 URL로 보낸다.
9. `id=^USER^&pw=^PASS^`: POST body에 ID/PW 후보를 넣는다.
10. `history`: 응답에 이 문자열이 있으면 실패로 본다.

### Hydra 공통 옵션 참고

> [!info]- 자주 쓰는 공통 옵션
> | 옵션 | 분류 | 의미 | 예시 | 주의점 |
> |---|---|---|---|---|
> | `-l LOGIN` | 로그인 후보 | 로그인 ID를 하나만 지정 | `-l unoh03` | 단일 계정의 비밀번호만 바꿔 시도 |
> | `-L FILE` | 로그인 후보 | 로그인 ID 후보를 파일에서 읽음 | `-L users.txt` | 여러 ID를 시도하므로 시도 횟수가 크게 늘어남 |
> | `-p PASS` | 비밀번호 후보 | 비밀번호 하나만 지정 | `-p 1234` | ID 후보가 여러 개일 때 유용할 수 있음 |
> | `-P FILE` | 비밀번호 후보 | 비밀번호 후보를 파일에서 읽음 | `-P pass.txt` | Dictionary Attack에 해당 |
> | `-C FILE` | 로그인/비밀번호 후보 | `login:pass` 형식의 조합 파일을 사용 | `-C combo.txt` | `-L`/`-P` 조합 대신 이미 만들어진 쌍을 사용 |
> | `-x MIN:MAX:CHARSET` | 비밀번호 후보 | 규칙에 따라 비밀번호 후보를 직접 생성 | `-x 3:3:1` | 후보 수가 폭발적으로 늘 수 있음 |
> | `-e nsr` | 비밀번호 후보 보강 | 빈 비밀번호, 로그인명과 같은 비밀번호, 로그인명 뒤집기 등을 추가 시도 | `-e nsr` | 실습 범위를 넘기 쉬우므로 의도 확인 필요 |
> | `-t TASKS` | 속도/병렬성 | 대상 하나당 병렬 연결 수 지정 | `-t 1`, `-t 4` | 기본값은 16. 낮추면 느리지만 서버 부하가 줄어듦 |
> | `-s PORT` | 대상 | 기본 포트 대신 지정 포트 사용 | `-s 8080` | 웹 서버가 80/443이 아닌 포트에서 열릴 때 필요 |
> | `-S` | 대상 | SSL/TLS 사용 | `-S` | `https-*` 모듈을 쓰는 편이 더 명확할 때도 있음 |
> | `-V` | 출력 | 각 시도 값을 자세히 출력 | `-V` | 출력이 매우 길어짐. 실습 관찰에는 좋음 |
> | `-v` | 출력 | verbose 출력 | `-v` | `-V`보다 덜 상세함 |
> | `-d` | 디버그 | 송수신 데이터를 확인할 때 사용 | `-d` | 실패/성공 판별 문자열 찾을 때 유용하지만 출력이 많음 |
> | `-f` | 종료 조건 | 첫 번째 유효한 조합을 찾으면 해당 대상 공격 중지 | `-f` | 실습 시간을 줄일 수 있음 |
> | `-o FILE` | 결과 저장 | 결과를 파일로 저장 | `-o result.txt` | 노트에는 민감값을 그대로 붙이지 않도록 주의 |
> | `-U` | 모듈 도움말 | 특정 서비스 모듈의 사용법 출력 | `hydra -U http-post-form` | `http-post-form` 옵션 구조 확인에 유용 |

### 예시

#### 예시 1: 6자리 비밀번호인데 앞 3자리를 알고 있는 경우

가정:

- 비밀번호는 총 6자리 숫자
- 앞 3자리는 이미 알고 있음
- 뒤 3자리만 모름
- 실습 대상은 사설 LAB 서버

전체 6자리 숫자 `000000`~`999999`를 모두 시도하면 1,000,000개다. 하지만 앞 3자리를 알고 있으면 뒤 3자리만 만들면 되므로 1,000개로 줄어든다.

```bash
hydra -V -l labuser -x 3:3:1 192.168.56.20 -s 8080 http-post-form "/member/loginModel.php:id=^USER^&pw=246^PASS^:F=history"
```

| 부분 | 의미 |
|---|---|
| `-l labuser` | ID는 `labuser`로 고정 |
| `-x 3:3:1` | 뒤 3자리 숫자 후보를 생성 |
| `pw=246^PASS^` | 비밀번호 앞 3자리는 `246`으로 고정하고, 뒤 3자리에 Hydra 후보를 붙임 |
| `F=history` | 응답에 `history`가 있으면 실패로 판단 |

실제 시도되는 비밀번호 형태:

```text
246000
246001
246002
...
246999
```

#### 예시 2: 이번 실습과 반대에 가까운 복잡한 상황

이번 실습은 `단일 ID`, `숫자 3자리 생성`, `HTTP`, `실패 문자열 기준`이었다. 반대에 가깝게 잡으면 다음처럼 된다.

가정:

- HTTPS 로그인 폼
- ID 후보가 여러 개
- 비밀번호는 생성하지 않고 wordlist 사용
- 로그인 실패 문자열보다 성공 문자열이 더 안정적임
- 세션 쿠키와 Referer 헤더가 필요함
- 대상은 사설 LAB 서버

```bash
hydra -L users.txt -P passwords.txt -t 4 -f -s 8443 192.168.56.30 https-post-form "/auth/login:username=^USER^&password=^PASS^&submit=Login:H=Cookie\: PHPSESSID=<LAB_SESSION>:H=Referer\: https\://192.168.56.30\:8443/auth/login:S=Dashboard"
```

| 부분 | 의미 |
|---|---|
| `-L users.txt` | ID 후보를 파일에서 읽음 |
| `-P passwords.txt` | 비밀번호 후보를 파일에서 읽음 |
| `-t 4` | 병렬 task를 4개로 제한 |
| `-f` | 유효한 조합을 찾으면 중지 |
| `-s 8443` | HTTPS 서비스가 8443 포트에서 동작 |
| `https-post-form` | HTTPS POST 로그인 폼 모듈 사용 |
| `/auth/login` | 로그인 처리 경로 |
| `username=^USER^&password=^PASS^&submit=Login` | POST body |
| `H=Cookie\: ...` | 요청에 실습용 세션 쿠키 헤더 추가 |
| `H=Referer\: ...` | Referer 헤더 추가 |
| `S=Dashboard` | 응답에 `Dashboard`가 있으면 성공으로 판단 |

이 예시는 구조 이해용이다. 실제 서비스가 CSRF token처럼 매 요청마다 바뀌는 값을 요구하면 Hydra 단독 명령만으로는 잘 맞지 않을 수 있다. 그 경우에는 먼저 Paros/Burp/ZAP으로 요청 구조와 토큰 갱신 방식을 확인해야 한다.

## p.79 계정 보호 대책 구현

### 세션 기반 시도 - 실패 횟수와 세션

로그인 버튼을 눌러 `POST /member/loginModel.php` 요청이 갈 때마다 `loginModel.php`는 위에서 아래로 다시 실행된다.

```text
로그인 시도 1번 = loginModel.php 1번 실행
로그인 시도 2번 = loginModel.php 다시 실행
로그인 시도 3번 = loginModel.php 다시 실행
```

그래서 일반 PHP 변수는 요청이 끝나면 사라진다.

| 저장 위치 | 예 | 다음 로그인 시도 때 유지되는가 | 의미 |
|---|---|---|---|
| 일반 변수 | `$fail_count` | 아니오 | 이번 요청 안에서만 살아 있다. |
| 세션 변수 | `$_SESSION['fail_count']` | 예 | 같은 사용자 세션에서 다음 요청에도 값을 다시 꺼내 쓸 수 있다. |

따라서 로그인 실패 횟수나 차단 해제 시간처럼 요청 사이에 유지되어야 하는 값은 일반 변수보다 세션에 저장하는 편이 자연스럽다.

현재 구현 사고 흐름:

1. `session_start()`로 세션을 시작한다.
2. `fail_count`가 처음 없으면 초기값을 만든다.
3. 로그인 성공/실패를 판정하기 전에 차단 상태인지 먼저 검사한다.
4. 차단 중이면 DB 조회나 로그인 판정까지 가지 않고 바로 차단 응답을 보낸다.
5. 실패하면 실패 횟수를 증가시킨다.
6. 실패 횟수가 기준 이상이면 차단 해제 예정 시간을 저장한다.
7. 이후 로그인 요청이 올 때마다 현재 시간과 차단 해제 예정 시간을 비교한다.
8. 아직 시간이 안 지났으면 계속 차단하고, 시간이 지났으면 차단을 해제한다.
9. 로그인에 성공하면 실패 횟수를 초기화한다.

중요한 점:

- `while`로 5분을 기다리게 만들면 안 된다. PHP 요청이 계속 붙잡히고 서버 자원을 낭비한다.
- 웹/PHP는 백그라운드 타이머처럼 계속 실행되는 구조가 아니다.
- "5분 지나면 풀어준다"는 말은 실제 구현에서는 "해제 예정 시간을 저장하고, 다음 요청마다 현재 시간과 비교한다"는 뜻이다.
- 차단 상태 검사 코드는 로그인 성공/실패 판정보다 위에 있어야 한다.
- 실패 횟수 증가는 `<script>` 안이 아니라 PHP 코드 영역에서 해야 한다.

### 세션 기반 방어의 한계

중요한 재검토 포인트: `$_SESSION['fail_count']` 방식은 **같은 브라우저 세션 안에서만** 실패 횟수를 기억한다.

따라서 브라우저를 끄거나, 쿠키/세션을 지우거나, 새 세션으로 접근하면 다음처럼 우회될 수 있다.

```text
기존 세션에서 로그인 실패 5회
→ $_SESSION['fail_count'] 증가
→ 세션 기준으로 잠금
→ 브라우저 종료 또는 세션 쿠키 삭제
→ 새 세션 발급
→ $_SESSION['fail_count'] 없음
→ 다시 0부터 시도 가능
```

즉 세션 기반 실패 횟수 제한은 "현재 세션에서만" 동작하는 약한 방어다. Brute Force 방어로 쓰려면 실패 횟수와 잠금 시간을 **브라우저가 아니라 서버가 기억해야 한다.**

더 적절한 저장 기준:

| 기준 | 설명 | 장점 | 한계 |
|---|---|---|---|
| 계정 ID 기준 | 특정 ID의 실패 횟수와 잠금 시간을 서버에 저장 | 브라우저를 바꿔도 같은 계정 공격을 막기 쉬움 | 공격자가 여러 ID를 바꾸면 분산될 수 있음 |
| IP 기준 | 특정 IP의 실패 횟수와 잠금 시간을 서버에 저장 | 같은 출발지에서 반복 시도하는 공격을 막기 쉬움 | NAT/공용 IP 환경에서 정상 사용자까지 영향 가능 |
| ID + IP 조합 기준 | 특정 ID와 특정 IP 조합으로 실패 횟수 저장 | 계정 기준과 IP 기준의 균형 | 구현이 조금 더 복잡함 |

실습 앱에서 다음 단계로 고려할 방식:

- `member` 테이블에 `fail_count`, `rest_time` 컬럼 추가
- 또는 별도 `login_fail` 테이블 생성
- 로그인 실패 시 DB에 실패 횟수 저장
- 기준 횟수 이상이면 DB에 잠금 해제 예정 시간 저장
- 로그인 시도마다 DB에 저장된 잠금 상태를 먼저 확인

이번 깨달음:

```text
$_SESSION['fail_count']는 브라우저/세션을 닫으면 초기화될 수 있다.
Brute Force 방어는 서버가 기억해야 한다.
```

### DB 기반 자체 구현 - `login_fail` 테이블로 전환한 시행착오

이 구간은 최종 코드 전문이 아니라, 세션 방식에서 DB 방식으로 넘어가며 겪은 시행착오와 최종 방어 흐름을 남기는 기록이다.

시행착오 요약:

| 단계 | 시도한 생각 | 문제/깨달음 | 다음 방향 |
|---|---|---|---|
| 1 | 일반 변수 `$fail_count`로 실패 횟수를 세려고 함 | PHP 파일은 요청마다 다시 실행되므로 일반 변수는 다음 로그인 시도까지 유지되지 않음 | 요청 사이에 유지되는 저장소 필요 |
| 2 | `$_SESSION['fail_count']`로 실패 횟수를 저장하려고 함 | 같은 브라우저 세션 안에서는 유지되지만, 브라우저 종료/쿠키 삭제/새 세션으로 초기화 가능 | 세션은 Brute Force 방어의 최종 저장소로 부족 |
| 3 | `$_SESSION['rest_time']`로 잠금 해제 시간을 저장하려고 함 | 잠금 시간도 세션이 사라지면 같이 사라짐 | 잠금 상태도 서버 쪽에 저장해야 함 |
| 4 | `login_fail(id, ip, fail_count, locked_count, rest_time)` 테이블을 설계함 | ID + IP 조합으로 실패 횟수와 잠금 상태를 서버가 기억할 수 있음 | DB 기반 방어로 전환 |
| 5 | `SELECT * FROM login_fail WHERE id='$id' AND ip='$ip'`로 row 조회를 시작함 | 처음에는 테이블이 비어 있을 수 있으므로 조회 결과가 없을 수 있음 | row 없으면 기본 row를 `INSERT`해야 함 |
| 6 | `$row_fail['fail_count']` 등으로 DB 값을 PHP 변수에 꺼냄 | PHP 변수만 바꾸면 DB 값은 변하지 않음 | 실패/잠금/초기화 시점마다 `UPDATE` 필요 |

구현 성숙도 기록:

| 점수대 | 구현 상태 |
|---|---|
| 60점 | DB에서 실패 기록을 읽는 방향은 잡았지만, row 생성과 DB 저장이 아직 빠짐 |
| 70점 | `login_fail`에 해당 ID/IP row가 없으면 기본 row를 `INSERT`할 수 있음 |
| 80점 | 로그인 실패 시 `fail_count` 증가 결과를 DB에 `UPDATE`할 수 있음 |
| 90점 | 5회 이상 실패 시 `rest_time`, `locked_count`를 DB에 저장할 수 있음 |
| 95점 | 로그인 성공 또는 잠금 만료 시 `fail_count`, `rest_time`을 DB에서 초기화할 수 있음 |

`login_fail` 테이블 설계:

```sql
CREATE TABLE login_fail (
    id varchar(50) not null,
    ip varchar(45) not null,
    fail_count int unsigned not null default 0,
    locked_count int unsigned not null default 0,
    rest_time int unsigned default null,
    primary key(id, ip)
);
```

컬럼 의미:

| 컬럼 | 의미 |
|---|---|
| `id` | 로그인 시도한 ID |
| `ip` | 로그인 시도한 클라이언트 IP |
| `fail_count` | 현재 연속 로그인 실패 횟수 |
| `locked_count` | 해당 ID/IP 조합이 잠긴 횟수 |
| `rest_time` | 잠금 해제 예정 시간. PHP `time()` 값 저장 |
| `primary key(id, ip)` | 같은 ID + 같은 IP 조합을 한 줄로 관리 |

최종 구현 흐름:

1. 클라이언트 IP를 구한다.
2. `login_fail`에서 `id + ip` row를 조회한다.
3. row가 없으면 기본 row를 생성한다.
4. DB에서 `fail_count`, `locked_count`, `rest_time`을 읽는다.
5. `rest_time`이 있고 아직 잠금 중이면 로그인 판정 전에 차단한다.
6. 잠금 시간이 지났으면 DB에서 `fail_count=0`, `rest_time=NULL`로 초기화한다.
7. 로그인 성공 시 DB에서 실패 상태를 초기화한다.
8. 로그인 실패 시 DB에서 `fail_count`를 증가시킨다.
9. 증가한 실패 횟수가 기준 이상이면 DB에 `rest_time`, `locked_count`를 저장한다.

핵심 문장:

```text
SELECT로 읽는 것만으로는 방어가 완성되지 않는다.
상태가 바뀌는 모든 지점에서 INSERT 또는 UPDATE로 DB에 다시 저장해야 한다.
```

#### 구현 성공 기록

- 세션 기반 `fail_count` 방식은 브라우저 종료/새 세션으로 우회 가능하다는 한계를 확인했다.
- `login_fail` 테이블을 이용해 ID + IP 조합 기준으로 실패 횟수와 잠금 시간을 DB에 저장하는 방향으로 전환했다.
- 로그인 실패 시 `fail_count`를 증가시키고, 기준 횟수 이상이면 `rest_time`과 `locked_count`를 DB에 저장하는 흐름까지 구현했다.
- 로그인 성공 시 `fail_count`, `rest_time`을 초기화하는 흐름도 구현했다.
- 성공 분기에서 500 에러가 났던 원인은 `UPDATE login_fail SET ...` SQL의 오타였다. `SET` 누락과 `fail_count` 오타를 고친 뒤 로그인 성공을 확인했다.

Hydra 실행 비교:

| 시점 | 실행 | 관찰 |
|---|---|---|
| 방어 적용 전 | Hydra를 1회 실행 | 숫자 3자리 후보를 반복 시도해 유효한 로그인 조합을 찾음 |
| 방어 적용 후 | 같은 흐름으로 Hydra를 1회 재실행 | 반복 로그인 시도가 DB 기반 실패 횟수/잠금 상태 관리 대상이 됨 |

#### 방어 후 증거 보강 슬롯

방어 전 Hydra 실행 결과는 성공 조합을 찾은 출력이 이미 남아 있다. 반면 방어 후 실행은 "DB 기반 실패 횟수/잠금 상태 관리 대상이 됨"까지는 정리했지만, 실제 출력과 DB 변화 증거는 나중에 보강해야 한다.

나중에 추가할 증거:

| 증거 | 확인할 내용 | 스샷 위치 |
|---|---|---|
| 방어 후 Hydra 출력 | 성공 조합을 찾았는지, 잠금 때문에 더 진행하지 못했는지 | 여기에 Hydra 터미널 스샷 추가 |
| 잠금 시 서버 응답 | 잠금 alert 또는 실패 판별 문자열이 어떻게 보이는지 | 여기에 브라우저/Paros 응답 스샷 추가 |
| `login_fail` 잠금 전 DB 상태 | `fail_count`, `locked_count`, `rest_time`의 초기값 | 여기에 MySQL 조회 스샷 추가 |
| `login_fail` 잠금 후 DB 상태 | 실패 횟수 증가, 잠금 횟수 증가, `rest_time` 저장 여부 | 여기에 MySQL 조회 스샷 추가 |

확인용 SQL:

```sql
SELECT id, ip, fail_count, locked_count, rest_time
FROM login_fail
WHERE id='unoh03';
```

이 증거가 들어오면 "방어 후 Hydra가 실제로 어떻게 실패했는가"까지 닫힌다. 지금 노트는 구현 사고 과정과 구조 설명은 충분하지만, 방어 후 재현 증거는 이 슬롯에 추가해야 완성도가 올라간다.

### 강사님 코드 분석 - `member(atime, count, flag)` 기반 고속 요청 차단

원본 파일:

```text
C:\Users\Unoh\Downloads\loginModel(계정잠금정책).php
```

강사님 코드는 별도 `login_fail` 테이블을 만들지 않고, 기존 `member` 테이블에 로그인 시도 상태를 저장한다.

추가하는 컬럼:

```sql
ALTER TABLE member
ADD atime INT NOT NULL DEFAULT 0,
ADD count INT NOT NULL DEFAULT 0,
ADD flag TINYINT(1) NOT NULL DEFAULT 0;
```

| 컬럼 | 의미 | 해설 |
|---|---|---|
| `atime` | 기준 시간 | 마지막으로 요청 시간 창을 갱신한 시각. PHP `time()` 값이 들어간다. |
| `count` | 짧은 시간 안의 요청 횟수 | 같은 ID로 1초 안에 몇 번 요청했는지 센다. |
| `flag` | 계정 잠금 여부 | `1`이면 잠김, `0`이면 정상으로 해석한다. |

흐름 분해:

1. `id`, `pw`를 `POST`에서 받는다.
2. 서버에서 ID 길이를 6~10자로 다시 검증한다.
3. DB에 연결한다.
4. `member` 테이블에서 해당 ID의 `atime`, `count`, `flag`를 읽는다.
5. `flag == 1`이면 이미 잠긴 계정으로 보고 로그인 판정 전에 차단한다.
6. 현재 시간과 `atime`의 차이가 1초 이하이면 같은 시간 창 안의 요청으로 보고 `count`를 증가시킨다.
7. 증가한 `count`가 5를 초과하면 `flag=1`로 바꿔 계정을 잠근다.
8. 1초를 초과했으면 새로운 시간 창으로 보고 `count=1`, `atime=현재 시간`으로 초기화한다.
9. 그 다음에야 `id`와 `pw`로 실제 로그인 성공 여부를 조회한다.

의도 추론:

```text
Hydra는 짧은 시간 안에 같은 로그인 요청을 빠르게 반복한다.
강사님 코드는 비밀번호 실패 횟수 자체보다,
같은 ID에 대해 1초 안에 요청이 몰리는지를 보고 자동화 공격을 감지하려는 구조다.
```

정확한 표현:

```text
강사님 코드는 "비밀번호를 5번 틀리면 5분 잠금" 정책이 아니다.
"같은 ID로 1초 안에 5회를 초과해 로그인 요청이 들어오면 계정을 잠그는" 요청 빈도 기반 차단에 가깝다.
```

### 두 구현 비교 - 막는 공격 모델이 다르다

네 코드와 강사님 코드의 차이:

| 항목 | 내 구현 | 강사님 코드 |
|---|---|---|
| 저장 위치 | 별도 `login_fail` 테이블 | 기존 `member` 테이블 |
| 식별 기준 | `id + ip` | `id` |
| 증가 기준 | 로그인 실패 시 증가 | 로그인 성공/실패 판정 전에 요청 횟수 증가 |
| 차단 기준 | 실패 횟수 기준 | 1초 안의 요청 횟수 기준 |
| 해제 방식 | `rest_time`이 지나면 자동 해제 가능 | `flag=1`이 되면 별도 해제 로직이 필요 |
| 막는 공격 모델 | 같은 ID/IP 조합의 반복 실패 | 같은 ID로 들어오는 고속 자동 요청 |

강사님 코드의 좋은 점:

- Hydra처럼 빠르게 반복 요청하는 도구를 실습에서 바로 관찰하기 좋다.
- `member` 테이블 안에 잠금 상태가 들어가므로 구조가 단순하다.
- 로그인 성공/실패 판정 전에 차단하므로 DB 인증 쿼리까지 가기 전에 막을 수 있다.

강사님 코드의 주의할 점:

- `id`만 기준으로 잠그므로, 누군가 특정 ID에 요청을 몰아 계정을 잠그는 공격이 가능하다.
- `flag=1`이 된 뒤 자동 해제 로직이 없다. 운영하려면 관리자 해제 또는 시간 기반 해제가 추가로 필요하다.
- 로그인 성공 여부와 무관하게 요청 횟수를 세므로, 정확히는 "로그인 실패 횟수 제한"이 아니라 "로그인 요청 빈도 제한"이다.
- 존재하지 않는 ID로 요청하면 `member` 조회 결과가 없을 수 있으므로 별도 예외 처리가 필요하다.
- SQL Injection 구조와 평문 비밀번호 비교 구조는 그대로 남아 있다. 이번 실습의 초점은 Brute Force 방어 흐름이지, 완성된 인증 보안 구현이 아니다.
- DB 접속 비밀번호가 코드에 평문으로 들어가 있다. 노트에는 값을 남기지 않는다.

비교 정리:

```text
내 구현은 "실패 상태를 별도 테이블에 기록하는 계정 잠금 정책"에 가깝고,
강사님 코드는 "member 테이블에 요청 빈도와 잠금 플래그를 추가한 고속 요청 차단"에 가깝다.
둘 다 Hydra 실습 방어로 쓸 수 있지만, 막으려는 기준이 다르다.
```

## 정리 및 남은 확인

### 진행 로그

#### 2026-05-21

- Hydra로 `POST /member/loginModel.php`에 3자리 숫자 비밀번호 후보를 반복 대입해 유효한 조합을 찾는 흐름을 확인.
- 세션 기반 실패 횟수 제한의 한계를 확인하고, DB 기반 `login_fail` 테이블 방식으로 전환했다.
- 방어 전/후 Hydra를 각각 1회 실행해 계정 잠금 방어 흐름을 비교할 기준을 남겼다.
- 강사님 `member(atime, count, flag)` 방식의 계정잠금정책 코드를 분석했다.

### 향후 별도 개선

- 방어 후 Hydra 출력과 `login_fail` 테이블 변화 스샷 추가
- 최종 방어 코드의 핵심 diff 또는 민감값을 제거한 snippet 정리
- 실무적으로 안전한 구현으로 리팩터링

### 다음 정리 때 할 일

1. 현재 내가 완성한 코드 흐름을 그대로 보존한다.
2. 방어 후 Hydra 재실행 결과와 `login_fail` DB 상태 변화를 스샷으로 보강한다.
3. 강사님 코드 분석을 실제 실행 결과와 다시 대조한다.
4. 실습용 구현과 실무형 구현의 차이를 따로 정리한다.

### 보안 관찰 포인트

- 스크립트에 DB 비밀번호가 평문으로 포함되어 있다. 실습 노트에는 값 자체를 남기지 않는다.
- "패스워드 크랙"은 노트에서는 무작위 악용 절차가 아니라 로그인 Brute Force와 계정 보호 대책을 이해하기 위한 실습으로 정리한다.

### 실습용 구현과 실무형 구현의 차이

- 이번 실습 구현은 Hydra 방어 흐름을 이해하기 위한 것이다.
- SQL Injection, 평문 비밀번호 비교, DB 접속 정보 평문 보관 문제는 별도로 해결해야 한다.
- 실무에서는 rate limiting, MFA, 로그/알림, 안전한 비밀번호 해싱, Prepared Statement를 함께 고려해야 한다.
