# 이중화

상태: 초안  
목적: 이중화 개념을 수업/팀프로젝트 구조와 연결해서 설명한다.

## 1. 이중화란

이중화는 서버, 네트워크, 저장소, 서비스 중 하나가 장애가 나도 전체 서비스가 바로 죽지 않게 예비 경로 또는 예비 장비를 두는 구조다.

중요한 점은 이중화가 항상 무중단이나 무손실을 뜻하지는 않는다는 것이다.

- 장비가 두 대여도 자동 전환이 없으면 단순 예비 장비다.
- 자동 전환이 있어도 데이터가 복제되지 않으면 장애 후 데이터가 빠질 수 있다.
- 데이터가 복제되어도 split-brain 방어가 없으면 양쪽 데이터가 서로 달라질 수 있다.
- 로드밸런싱은 트래픽 분산이 목적이고, 저장소 이중화는 데이터 일관성이 핵심이다.

## 2. 핵심 용어

| 용어 | 의미 |
|---|---|
| HA | High Availability. 장애가 나도 서비스를 계속 제공하려는 설계 |
| SPOF | Single Point of Failure. 하나만 죽어도 전체가 죽는 단일 장애점 |
| Failover | 장애 발생 시 예비 노드로 서비스가 넘어가는 것 |
| Failback | 장애 복구 후 원래 노드로 서비스를 되돌리는 것 |
| VIP | Virtual IP. 클라이언트가 바라보는 고정 서비스 IP |
| Health Check | 서비스가 살아 있는지 확인하는 검사 |
| RTO | 장애 후 서비스 복구까지 걸리는 시간 |
| RPO | 장애 시 잃어도 되는 데이터 범위 |
| Split-Brain | 두 노드가 동시에 자신이 주인이라고 판단하는 상태 |
| Fencing | 이상 노드를 강제로 격리해서 split-brain을 막는 장치 |

## 3. 이중화 종류

### 3.1 Active-Passive

한 대가 서비스하고 다른 한 대는 대기한다. 장애가 나면 대기 노드가 VIP나 서비스를 넘겨받는다.

장점:

- 구조가 비교적 단순하다.
- 상태가 있는 서비스에 적용하기 쉽다.
- 장애 시 어느 노드가 주인인지 이해하기 쉽다.

단점:

- 평소에는 대기 노드 자원을 적극적으로 쓰지 못한다.
- failover 시간 동안 짧은 중단이 생길 수 있다.
- 데이터 복제나 공유 저장소가 없으면 최신 데이터가 빠질 수 있다.

예시:

- keepalived 기반 LB VIP
- keepalived 기반 NFS VIP
- Pacemaker 기반 DB/NFS active-passive cluster

### 3.2 Active-Active

여러 노드가 동시에 서비스를 처리한다.

장점:

- 자원을 동시에 활용한다.
- 트래픽 분산이 가능하다.
- 한 노드 장애 시 남은 노드가 계속 처리할 수 있다.

단점:

- 상태 공유가 어렵다.
- 세션, 파일, DB write가 섞이면 설계 난도가 올라간다.
- 저장소에서 잘못 구성하면 데이터 충돌이 생긴다.

예시:

- WEB1/WEB2가 동시에 Tomcat 요청을 처리하는 구조
- nginx upstream으로 여러 WEB 서버에 round-robin 분산

### 3.3 Load Balancing

여러 서버에 요청을 나눠 보내는 방식이다. 이중화 효과도 있지만, 핵심 목적은 트래픽 분산이다.

장점:

- WEB 서버 확장이 쉽다.
- 장애 서버를 제외하면 서비스 지속성이 올라간다.
- HTTP 같은 stateless 요청에 잘 맞는다.

단점:

- LB 자체가 죽으면 전체가 죽을 수 있다.
- 세션 저장 위치를 잘못 잡으면 로그인 상태가 흔들린다.
- 백엔드 서버가 모두 죽으면 LB가 살아 있어도 서비스는 죽는다.

### 3.4 Primary-Secondary

주 서버가 원본 데이터를 가지고, 보조 서버가 복제본을 가진다. DNS에서 흔히 사용한다.

장점:

- primary가 죽어도 secondary가 질의에 응답할 수 있다.
- 읽기 질의 중심 서비스에 잘 맞는다.

단점:

- 원본 변경은 primary 기준이다.
- zone transfer 실패 시 secondary가 오래된 정보를 줄 수 있다.
- 외부 DNS 위임이 ns1/ns2 모두를 가리켜야 효과가 있다.

예시:

- DNS1 primary/master
- DNS2 secondary/slave

### 3.5 Storage Replication

데이터를 다른 서버나 저장소에 복제하는 방식이다.

종류:

- 파일 단위 비동기 복제: rsync
- 블록 단위 복제: DRBD
- 공유 저장소: SAN, NAS, clustered storage
- 애플리케이션 레벨 복제: DB replication

rsync는 단순하고 구현이 쉽지만, 무손실 저장소 HA는 아니다. 복제 주기 사이에 장애가 나면 최신 파일이 반대편에 없을 수 있다.

## 4. 우리 팀프로젝트의 이중화 구조

### 4.1 전체 흐름

```text
사용자
  -> DNS: www.zzaphub.com 조회
  -> Public IP: 1.2.2.10
  -> NAT: 1.2.2.10 -> 192.168.2.10
  -> LB VIP: 192.168.2.10
  -> nginx
  -> WEB1 192.168.2.3:8080 또는 WEB2 192.168.2.4:8080
  -> upload 경로
  -> NFS VIP: 192.168.2.50:/share_directory
  -> NFS1 또는 NFS2
```

### 4.2 DNS 이중화

구조:

- DNS1: primary/master
- DNS2: secondary/slave
- 도메인: `zzaphub.com`

이 방식은 Active-Passive라기보다 Primary-Secondary DNS 이중화다. DNS1이 원본 zone을 관리하고, DNS2는 zone transfer로 복제본을 받아 질의에 응답한다.

장점:

- DNS1 장애 시 DNS2가 응답할 수 있다.
- DNS는 읽기 질의가 많아서 secondary 구조와 잘 맞는다.

단점:

- DNS2가 실제로 최신 zone을 받았는지 확인해야 한다.
- `ns1`, `ns2`가 모두 클라이언트 또는 상위 DNS에 알려져 있어야 한다.
- secondary인데 zone 파일을 직접 작성하면 실제 동작과 문서가 헷갈릴 수 있다.

보안 관점:

- 외부 공개 권위 DNS라면 recursion을 제한해야 한다.
- zone transfer는 secondary DNS IP로만 제한해야 한다.
- 운영 환경이면 TSIG 같은 zone transfer 인증도 검토한다.

확인 명령:

```bash
dig @192.168.1.2 zzaphub.com
dig @192.168.1.3 zzaphub.com
dig @192.168.1.2 www.zzaphub.com
dig @192.168.1.3 www.zzaphub.com
```

### 4.3 LB 이중화

구조:

- LB1/LB2는 keepalived로 `192.168.2.10` VIP를 공유한다.
- VIP를 가진 LB만 실제 진입점이 된다.
- nginx upstream은 WEB1/WEB2로 요청을 분산한다.

정확한 분류:

```text
LB1/LB2: Active-Passive VIP failover
WEB1/WEB2: Active-Active HTTP load balancing
```

장점:

- WEB 서버 한 대가 죽어도 다른 WEB 서버로 처리 가능하다.
- LB 한 대가 죽어도 다른 LB가 VIP를 받을 수 있다.
- WEB 서버는 수평 확장하기 쉽다.

단점:

- keepalived가 VIP만 옮긴다. nginx 설정 오류나 백엔드 전체 장애까지 자동 해결하지는 않는다.
- health check가 nginx 프로세스만 보면 WEB1/WEB2 실제 상태를 놓칠 수 있다.
- 세션을 WEB 로컬에 저장하면 사용자가 다른 WEB으로 갈 때 로그인 상태가 깨질 수 있다.

보안 관점:

- 외부에는 LB VIP만 노출하고 WEB1/WEB2 직접 접근은 제한하는 것이 좋다.
- LB는 필요한 포트만 열어야 한다. HTTP면 80, HTTPS면 443.
- 운영 환경이면 TLS 종료, 보안 헤더, 접근 로그, rate limit을 고려한다.
- keepalived VRRP는 같은 L2 네트워크 안에서 동작하므로 VRRP 접근망을 통제해야 한다.

확인 명령:

```bash
ip a | grep 192.168.2.10
systemctl status nginx --no-pager
systemctl status keepalived --no-pager
curl http://192.168.2.10
```

### 4.4 WEB 이중화

구조:

- WEB1: Tomcat
- WEB2: Tomcat
- LB가 두 WEB으로 HTTP 요청을 분산한다.

분류:

```text
WEB 계층: Active-Active
```

장점:

- HTTP 요청을 두 서버가 나눠 처리한다.
- 한 WEB 서버 장애 시 나머지 WEB이 계속 처리할 수 있다.

단점:

- 세션 저장 방식이 중요하다.
- 업로드 파일이 WEB 로컬 디스크에만 있으면 서버마다 파일 상태가 달라진다.
- 그래서 upload 경로를 NFS로 공통화했다.

보안 관점:

- WEB 서버는 직접 외부 노출을 줄이고 LB 뒤에 둔다.
- 업로드 파일은 확장자, MIME type, 실행 권한, 저장 경로를 제한해야 한다.
- WEB 서버가 NFS에 쓰기 권한을 가지므로 침해 시 파일 저장소까지 영향을 줄 수 있다.

### 4.5 NFS 이중화

구조:

- NFS1/NFS2는 keepalived로 `192.168.2.50` VIP를 공유한다.
- WEB은 물리 IP가 아니라 `192.168.2.50:/share_directory`만 mount한다.
- VIP 소유 노드가 원본 역할을 하고, rsync로 반대편에 파일을 복제한다.

정확한 분류:

```text
NFS 계층: Active-Passive VIP failover
데이터 복제: rsync 기반 비동기 파일 복제
```

이 구조는 NFS 이중화는 맞지만, 무손실/완전 투명 NFS HA는 아니다.

장점:

- 구조가 단순해서 수업 환경에서 구현 가능하다.
- WEB은 항상 VIP만 보면 된다.
- NFS1 장애 시 NFS2가 VIP를 받을 수 있다.
- rsync로 반대편에 파일 복제본을 유지한다.

단점:

- 장애 직전 파일은 아직 반대편에 없을 수 있다.
- 기존 NFS client file handle이 failover 후 stale 상태가 될 수 있다.
- rsync는 파일 복제이지 클러스터 파일시스템이 아니다.
- split-brain 방어가 약하다.
- 자동 delete sync를 켜면 failover 후 최신 파일이 삭제될 수 있다.

보안 관점:

- 편의 우선 구성에서는 export 범위가 넓고 권한이 약하다.
- `/share_directory`가 `777`이면 침해된 WEB 서버가 파일 저장소를 마음대로 쓸 수 있다.
- NFS 인증이 강한 Kerberos 방식이 아니면 네트워크/IP 신뢰에 많이 의존한다.
- rsync용 SSH key가 탈취되면 peer NFS에 쓰기 가능하다.

확인 명령:

```bash
# WEB에서
findmnt -T /opt/tomcat/tomcat-10/webapps/upload

# NFS1/NFS2에서
ip a | grep 192.168.2.50
ls -la /share_directory
tail -n 50 /var/log/nfs-ha-sync.log
```

## 5. 계층별 비교

| 계층 | 우리 구조 | 이중화 종류 | 핵심 장점 | 핵심 위험 |
|---|---|---|---|---|
| DNS | DNS1 primary, DNS2 secondary | Primary-Secondary | DNS1 장애 시 DNS2 응답 가능 | zone transfer 실패, open resolver 위험 |
| LB | keepalived VIP `192.168.2.10` | Active-Passive | LB 장애 시 VIP 이동 | LB health check 범위 부족 |
| WEB | WEB1/WEB2 Tomcat | Active-Active | 요청 분산, WEB 장애 대응 | 세션/업로드 상태 관리 필요 |
| NFS | keepalived VIP `192.168.2.50` + rsync | Active-Passive + async replication | 단순한 파일 저장소 이중화 | 무손실 아님, stale handle, split-brain |

## 6. 보안 관점에서 보는 이중화

이중화는 보안을 자동으로 강화하지 않는다. 오히려 장비와 경로가 늘어나기 때문에 공격면도 늘어난다.

확인해야 할 것:

- VIP만 외부 또는 클라이언트에 노출되는가
- 실제 backend 서버 직접 접근이 막혀 있는가
- health check가 공격자에게 악용될 수 없는가
- 복제 계정 권한이 과도하지 않은가
- split-brain 발생 시 데이터 손상을 막을 수 있는가
- 로그 시간이 서로 맞는가
- failover/failback 절차가 문서화되어 있는가

계층별 보안 포인트:

| 계층 | 보안 체크 |
|---|---|
| DNS | recursion 제한, zone transfer 제한, 53/tcp와 53/udp 허용 범위 |
| LB | 80/443 최소 개방, backend 직접 접근 차단, TLS, 보안 헤더 |
| WEB | 업로드 검증, 세션 저장소, Tomcat 권한, 로그 |
| NFS | export 범위, 권한, root squash, SSH key 제한, delete sync 정책 |
| Router/Firewall | NAT/ACL이 실제 서비스 흐름과 일치하는지 확인 |

## 7. 수업에서 말할 수 있는 한 줄 정리

우리 팀프로젝트의 이중화는 다음처럼 설명하면 된다.

```text
DNS는 Primary-Secondary,
LB는 Active-Passive VIP,
WEB은 Active-Active 부하분산,
NFS는 Active-Passive VIP와 rsync 비동기 복제를 사용한다.
```

더 짧게 말하면 다음과 같다.

```text
트래픽은 LB와 WEB에서 분산하고,
서비스 진입점은 VIP로 이중화하며,
파일 데이터는 NFS VIP와 rsync로 복제한다.
단, NFS는 무손실/완전 투명 HA가 아니다.
```
