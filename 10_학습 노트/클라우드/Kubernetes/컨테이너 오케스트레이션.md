# 컨테이너 오케스트레이션

> [!summary]
> Container는 Application과 Dependency를 실행 가능한 단위로 격리한다. Container Orchestration은 여러 Node에서 많은 Container의 배치·복제·갱신·복구를 원하는 상태에 맞춰 지속적으로 조정한다.

## 이 노트가 답하는 질문

- Container와 Virtual Machine은 무엇이 다른가?
- Container를 실행하는 것만으로는 왜 운영 자동화가 완성되지 않는가?
- Kubernetes가 자동화하는 범위와 사용자가 여전히 설계해야 하는 범위는 어디까지인가?

## Virtual Machine, Container, Orchestration의 경계

| 구분 | 주된 격리·관리 단위 | Kernel 관계 | 해결하는 문제 |
|---|---|---|---|
| Virtual Machine | Virtual Hardware 위의 Guest OS와 Application | VM마다 Guest OS Kernel 사용 | Machine 수준의 강한 격리와 서로 다른 OS 환경 제공 |
| Container | Image에서 실행된 격리 Process | 일반적으로 Host Kernel 공유 | Application과 Dependency를 일관된 실행 단위로 Packaging |
| Container Orchestration | 여러 Node의 Container Workload | Container Runtime 위에서 동작 | 배치·복제·Network 연결·갱신·장애 복구를 Cluster 수준에서 조정 |

Container는 작은 VM이 아니다. VM은 Guest OS와 Kernel을 포함하지만, Container는 Host Kernel의 Namespace·cgroup 같은 기능을 이용해 Process와 Resource를 격리한다. 따라서 Container는 보통 VM보다 가볍지만 VM과 동일한 보안 경계라고 단정할 수 없다.

VM과 Container는 경쟁 관계만은 아니다. Cloud에서는 VM을 Worker Node로 만들고 그 안에서 여러 Container를 실행하는 구성이 일반적이다.

> [!important] Portable은 모든 환경에서 무조건 실행된다는 뜻이 아니다
> Container Image가 Dependency를 Packaging해 환경 차이를 줄여도 Host OS와 CPU Architecture 호환성은 남는다. 예를 들어 `linux/amd64` Image를 `arm64`에서 실행하려면 해당 Platform Variant나 Emulation이 필요하다.

## Container만으로 부족한 이유

강의 원자료는 Developer가 Image를 Build·Push한 뒤 Test와 Production Server에서 각각 Image를 Pull하고 `docker run`을 20회 반복하는 흐름을 보여준다. Container는 실행 단위를 표준화하지만 다음 운영 판단까지 자동으로 해결하지는 않는다.

- 어느 Node에 실행할 것인가?
- 몇 개의 복제본을 유지할 것인가?
- Instance가 죽으면 어디에서 다시 실행할 것인가?
- 새 Version으로 어떤 순서로 교체하고, 실패하면 어떻게 되돌릴 것인가?
- 변하는 Pod 주소 앞에 안정적인 Service Endpoint를 어떻게 제공할 것인가?
- CPU·Memory·Storage와 설정 정보를 어떻게 배정할 것인가?

Container Orchestration은 이 문제를 개별 명령의 반복이 아니라 Cluster의 상태 관리 문제로 다룬다.

## Kubernetes의 핵심은 Desired State 조정이다

Kubernetes를 단순히 정해진 순서대로 Script를 실행하는 도구로 보면 핵심을 놓친다.

```text
Manifest에 Desired State 선언
→ Kubernetes API에 저장
→ Controller와 Scheduler가 현재 상태를 관찰
→ 필요한 Pod를 생성·교체·배치
→ 실제 상태가 달라지면 Desired State에 맞게 다시 조정
```

예를 들어 `replicas: 3`은 Pod 생성 명령을 세 번 실행하라는 절차가 아니라, 해당 Workload의 복제본 3개가 유지되어야 한다는 상태 선언이다. Pod 하나가 사라지면 Controller는 현재 상태와 원하는 상태의 차이를 감지해 대체 Pod를 만든다.

이 반복 조정을 **Reconciliation**이라고 이해하면 이후의 Deployment, ReplicaSet, StatefulSet, DaemonSet을 같은 관점으로 연결할 수 있다.

## Kubernetes가 제공하는 것과 제공하지 않는 것

| Kubernetes가 제공하는 기반 | 자동으로 해결되지 않는 것 |
|---|---|
| Service Discovery와 Load Balancing | Application 내부 오류와 잘못된 Business Logic 수정 |
| Replica 유지와 실패한 Container·Pod 교체 | Data 정합성, Backup, 복구 절차 설계 |
| Controlled Rollout과 Rollback | 안전한 Release 기준과 실제 성공 판정 |
| Resource Request 기반 Scheduling | 적절한 CPU·Memory 값의 자동 결정 |
| Config·Secret 전달 수단 | Secret의 생성·회전·최소 권한 정책 완성 |
| Storage 연결을 위한 추상화 | Storage 자체의 가용성과 Data 보호 보장 |

Kubernetes는 Application을 담는 Image를 대신 Build하지 않으며, 완성된 CI/CD·Monitoring·Logging Platform 전체를 기본 제공하지도 않는다. 필요한 구성 요소를 연결할 수 있는 기반을 제공한다.

## 실무에서 먼저 기억할 점

### 실행 중인 Container를 고치지 않는다

Kubernetes 공식 문서는 Container를 Stateless·Immutable하게 다루고, 변경이 필요하면 새 Image를 Build한 뒤 Container를 다시 생성하는 방식을 권장한다. 실행 중인 Container에 접속해 직접 Package나 Code를 수정하면 재생성 시 변경이 사라지고 환경 재현도 어려워진다.

### Production에서는 Image Version을 식별 가능하게 한다

`latest` Tag는 같은 이름이 다른 Image를 가리킬 수 있어 실행 Version 추적과 Rollback을 어렵게 한다. Version Tag 또는 Content Digest를 사용해 어떤 Image가 배포됐는지 재현 가능하게 만드는 편이 안전하다.

### Self-healing은 Application을 자동으로 이해한다는 뜻이 아니다

- Process 종료는 `restartPolicy`와 Controller가 대응할 수 있다.
- 응답 불능 상태를 재시작하려면 적절한 Liveness Probe가 필요하다.
- 요청을 받을 준비가 안 된 Pod를 Service에서 제외하려면 Readiness Probe가 필요하다.
- 시작 시간이 긴 Application은 Startup Probe 없이 Liveness Probe부터 적용하면 정상 시작 중인 Container를 반복 종료할 수 있다.

즉 Kubernetes의 Self-healing은 사용자가 선언한 상태와 Health Signal을 기준으로 동작한다.

### Kubernetes 도입 자체가 목적은 아니다

> [!note] Parametric judgment
> 소수의 Container를 단일 Host에서 운영하고 복잡한 배포·확장·복구 요구가 없다면 Kubernetes Cluster의 운영비용이 자동화 이득보다 클 수 있다. Node·권한·Network·Storage·Upgrade까지 관리할 필요가 생기므로, 실제 조정 문제가 충분한지 먼저 판단해야 한다.

## 강의자료와 현재 공식 기준의 차이

- 강의자료는 Docker를 대표 Container 기술로 설명한다. 현재 Kubernetes는 CRI(Container Runtime Interface)를 통해 `containerd`, `CRI-O` 같은 Runtime을 지원하므로 Kubernetes 실행환경이 Docker Engine에 종속된다고 이해하면 안 된다.
- 강의자료의 `사실상의 표준`이라는 표현은 당시 자료의 평가다. Concept Note에서는 특정 도구의 순위보다 Desired State와 Reconciliation이라는 동작 원리를 기준으로 삼는다.
- 강의자료는 오케스트레이션의 장점을 개괄한다. 실제 Production 환경에서는 Control Plane 가용성, Worker Node 수, 접근 제어, Resource 정책, Storage와 운영 절차를 별도로 설계해야 한다.
