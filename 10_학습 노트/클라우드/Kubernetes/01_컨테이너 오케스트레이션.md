# 컨테이너 오케스트레이션

> [!summary]
> 컨테이너를 몇 개 직접 실행하는 것은 어렵지 않다. 하지만 서버 여러 대에서 수십·수백 개를 운영하려면 어디에 몇 개를 실행하고, 고장 난 것은 어떻게 다시 살릴지 계속 관리해야 한다. 이 일을 자동화하는 것이 **컨테이너 오케스트레이션**이다.

## 먼저 컨테이너가 무엇인가

애플리케이션을 실행하려면 프로그램뿐 아니라 Library, 설정, 실행환경도 필요하다. 컨테이너는 이 실행에 필요한 것들을 Image로 묶고, 다른 프로그램과 분리된 공간에서 실행하는 방법이다.

덕분에 개발 PC에서 실행한 프로그램을 Test Server와 Production Server에서도 비슷한 환경으로 실행하기 쉬워진다.

### Virtual Machine과 무엇이 다른가

| Virtual Machine | Container |
|---|---|
| 각 VM에 운영체제를 따로 설치한다 | Host의 운영체제 핵심 부분을 함께 사용한다 |
| Machine 한 대를 통째로 나눈 모습에 가깝다 | 실행 중인 프로그램을 분리한 모습에 가깝다 |
| 비교적 무겁지만 강한 격리와 서로 다른 OS 환경을 제공한다 | 비교적 가볍고 빠르게 만들고 없앨 수 있다 |

Container와 VM 중 하나만 선택해야 하는 것은 아니다. Cloud에서는 VM을 Server로 만들고 그 안에서 여러 Container를 실행하는 경우가 많다.

> [!important] 어디서나 무조건 실행되는 것은 아니다
> Container가 환경 차이를 줄여주는 것은 맞지만 운영체제와 CPU 종류가 호환되어야 한다. 예를 들어 x86용 Image를 ARM Server에서 실행하려면 ARM용 Image나 별도 변환 기능이 필요하다.

## 컨테이너만으로는 왜 부족한가

강의자료에는 다음과 같은 수동 배포 과정이 나온다.

```text
개발자가 Image를 Build하고 저장소에 Push
→ Test Server에서 Pull하고 여러 번 실행
→ Production Server에서도 Pull하고 여러 번 실행
```

Container 1~2개라면 사람이 직접 관리할 수 있다. 하지만 개수가 많아지면 다음 질문이 생긴다.

- 어느 Server에 실행해야 할까?
- 같은 프로그램을 몇 개 실행해야 할까?
- 하나가 고장 나면 누가 다시 실행할까?
- 사용자가 늘어나면 어떻게 개수를 늘릴까?
- 새 Version은 어떤 순서로 교체할까?
- 실행 위치가 바뀌어도 사용자는 어떻게 같은 주소로 접속할까?

이 문제들을 사람이 명령어로 하나씩 처리하면 느리고 실수하기 쉽다. 컨테이너 오케스트레이션 도구는 여러 Server와 Container를 한 묶음으로 보고 관리한다.

## Kubernetes는 어떻게 관리하는가

Kubernetes에서는 보통 “이 명령을 세 번 실행해라”보다 “이 프로그램이 항상 세 개 실행되어 있어야 한다”처럼 **원하는 상태**를 적는다.

```text
사용자가 원하는 상태를 작성
→ Kubernetes가 현재 상태와 비교
→ 부족하면 만들고, 너무 많으면 줄임
→ 고장이나 변경이 생기면 다시 비교
```

예를 들어 원하는 개수를 `3`으로 정했는데 하나가 사라져 현재 두 개만 남았다면, Kubernetes는 하나를 새로 만들어 다시 세 개로 맞춘다.

이처럼 현재 상태를 원하는 상태에 계속 맞추는 동작을 **Reconciliation**이라고 한다. 지금은 용어를 외우기보다 “Kubernetes는 원하는 상태를 계속 유지하려 한다”라고 이해하면 충분하다.

## Kubernetes가 도와주는 일

- Container를 어느 Server에 배치할지 결정한다.
- 정해둔 개수를 유지하고 고장 난 Container를 교체한다.
- 새 Version으로 순서대로 교체하거나 이전 Version으로 되돌릴 기반을 제공한다.
- 여러 Container 앞에 일정한 접속 주소를 제공하고 요청을 나눠준다.
- 저장공간, 설정, 비밀정보를 연결할 방법을 제공한다.
- 사용량에 따라 개수를 늘리거나 줄일 수 있다.

## Kubernetes가 대신하지 않는 일

- 애플리케이션의 Bug를 고쳐주지 않는다.
- Image를 대신 Build해주지 않는다.
- Backup과 Data 복구 방법을 자동으로 설계해주지 않는다.
- 적절한 Server 크기와 Container 개수를 알아서 정해주지 않는다.
- Kubernetes를 설치했다는 이유만으로 보안과 고가용성이 완성되지는 않는다.

즉 Kubernetes는 운영 자동화를 위한 기반이지, 애플리케이션과 운영에 관한 모든 판단을 대신하는 도구는 아니다.

## 처음부터 알아두면 좋은 실무 팁

### 실행 중인 Container를 직접 고치지 않는다

실행 중인 Container에 접속해 프로그램을 수정하면 그 Container가 다시 만들어질 때 변경 내용이 사라진다. 수정이 필요하면 새 Image를 만들고 그 Image로 Container를 교체하는 방식이 재현하기 쉽다.

### `latest`만 사용하지 않는다

같은 `latest`라는 이름이 나중에 다른 Image를 가리킬 수 있다. 실제 운영에서는 Version이나 변경되지 않는 Image Digest를 사용해야 어떤 코드가 실행 중인지 확인하고 되돌리기 쉽다.

### 고장 복구에도 판단 기준이 필요하다

Kubernetes는 프로그램이 정상인지 저절로 이해하지 못한다. 나중에 배우는 Health Check를 올바르게 설정해야 “실행 중이지만 요청을 처리하지 못하는 상태”도 구분할 수 있다.

### 작은 서비스에 항상 Kubernetes가 필요한 것은 아니다

> [!note] AI의 실무적 판단
> Container가 적고 Server 한 대로 충분하다면 Kubernetes 자체를 관리하는 비용이 더 클 수 있다. 배포·확장·복구 자동화가 실제로 필요한 규모인지 먼저 판단하는 편이 좋다.

## 강의자료와 현재 기준의 차이

- 강의자료는 Docker를 대표적인 Container 기술로 설명한다.
- 현재 Kubernetes는 Docker Engine만 사용하는 것이 아니라 `containerd`, `CRI-O` 같은 여러 Container Runtime을 지원한다.
- 따라서 **Container = Docker**, **Kubernetes = Docker 관리 도구**라고만 이해하면 범위가 너무 좁다.
- 강의자료의 “사실상의 표준”이라는 평가는 원자료의 표현이다. 이 노트에서는 도구 순위보다 원하는 상태를 자동으로 유지하는 원리를 핵심으로 본다.

## 지금은 이것만 기억한다

```text
Container
= 애플리케이션을 필요한 실행환경과 함께 분리해 실행하는 단위

Container Orchestration
= 많은 Container의 배치·개수·갱신·복구를 자동으로 관리하는 것

Kubernetes
= 사용자가 적어둔 원하는 상태를 실제 상태에 계속 맞추는 도구
```
