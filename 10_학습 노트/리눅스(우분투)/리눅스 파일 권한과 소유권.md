# 리눅스 파일 권한과 소유권

> `chmod`는 접근 mode를, `chown`은 owner와 group을 바꾸며, 재귀 적용과 symbolic link 순회는 별도의 위험 경계다.

## 범위

- 설명하는 것: `rwx`, 파일과 디렉토리 차이, 숫자 mode, `chmod`, `chown`, `umask`, 재귀·symbolic link 주의
- 다루지 않는 것: POSIX ACL 전체, SELinux/AppArmor 정책, NFS identity mapping

## 권한 구조

```text
파일 종류 | owner 권한 | group 권한 | other 권한
-          rwx          rwx           rwx
```

| 권한 | 일반 파일 | 디렉토리 |
|---|---|---|
| `r` | 내용 읽기 | entry 이름 목록 읽기 |
| `w` | 내용 수정 | entry 생성·삭제·이름 변경 |
| `x` | 실행 | 경로 통과와 entry 접근 |

디렉토리의 파일을 삭제할 수 있는지는 파일 자체의 `w`보다 부모 디렉토리의 `w`와 `x`, sticky bit 등의 영향을 받는다.

## 숫자 mode

| 값 | 권한 |
|---:|---|
| `4` | read |
| `2` | write |
| `1` | execute |

```bash
chmod 0640 app.conf
chmod 0750 /srv/app
```

- `0640`: owner `rw`, group `r`, other 없음
- `0750`: owner `rwx`, group `rx`, other 없음
- 앞자리 `0`은 setuid, setgid, sticky bit 같은 special bit가 없음을 명시하는 전통적 표기다.

`chmod`는 firewall rule을 바꾸는 명령이 아니라 파일 mode bit를 바꾸는 명령이다.

## 소유권

```bash
chown appuser:appgroup /srv/app
```

- `appuser`: 새 owner
- `appgroup`: 새 group
- owner와 실제 쓰기 가능 여부는 동일한 말이 아니다. mode, ACL, 부모 디렉토리 권한과 편집기의 저장 방식도 함께 본다.

## 재귀 변경 전 확인

`-R`은 대상 아래 전체에 영향을 준다. 편의를 위한 기본 옵션으로 사용하지 않는다.

```bash
ls -ld /srv/app
stat -c '%A %a %U:%G %n' /srv/app
```

확인할 것:

1. 대상의 절대 경로와 mount 경계
2. 바뀌어야 할 owner/group과 mode
3. 파일과 디렉토리에 같은 mode를 적용해도 되는지
4. symbolic link가 있는지
5. 변경 뒤 서비스가 실제로 필요한 파일을 읽고 쓸 수 있는지

`chmod -R 777`이나 목적이 불명확한 `chown -R`은 문제 원인을 숨기고 영향 범위를 넓힌다.

## `chown -R`과 symbolic link

GNU Coreutils 기준:

| 옵션 | 재귀 순회 방식 |
|---|---|
| `-P` | symbolic link를 따라가지 않는다. `chown -R`의 기본값이다. |
| `-H` | 명령줄 인자로 준 경로 자체가 디렉토리 symbolic link일 때만 그 대상을 순회한다. |
| `-L` | 순회 중 발견하는 모든 디렉토리 symbolic link를 따라간다. |

`-H`를 “하위에서 발견되는 모든 symbolic link를 따라간다”라고 설명하면 `-L`과 혼동한 것이다. 공격자가 경로를 바꿀 수 있는 환경에서 dereference와 재귀를 결합하면 별도의 보안 위험이 생긴다.

## `umask`

`umask`는 새 객체를 만들 때 요청된 mode에서 허용하지 않을 bit를 가린다. 일반적인 시작값은 파일 `0666`, 디렉토리 `0777`이지만 프로그램이 다른 mode를 요청할 수도 있으므로 단순 뺄셈 공식으로 일반화하지 않는다.

```bash
umask 0027
```

일반적인 결과:

- 디렉토리: `0777`에서 mask 적용 → `0750`
- 일반 파일: `0666`에서 mask 적용 → `0640`

현재 shell의 `umask`와 계정·서비스 생성 정책은 별도로 확인한다.

## 적용 뒤 검증

```bash
stat -c '%A %a %U:%G %n' /srv/app
namei -l /srv/app/config/app.conf
```

- `stat`: 최종 mode와 owner/group 확인
- `namei -l`: 경로 각 구성 요소의 통과 권한 확인
- 서비스 계정으로 실제 read/write가 되는지도 필요한 범위에서 검증

## 오판하기 쉬운 지점

- 오해: owner만 파일을 저장할 수 있다.
  - 정확한 설명: 실제 권한과 저장 방식, 부모 디렉토리 권한을 봐야 한다.
- 오해: `-v`는 대량 변경 시 필수 안전장치다.
  - 정확한 설명: 출력 옵션일 뿐이며 대상 검증과 rollback을 대신하지 않는다.
- 오해: 재귀 `chown` 실수는 곧바로 kernel panic을 일으킨다.
  - 정확한 설명: 시스템 파일 소유권을 망가뜨려 서비스와 부팅을 손상시킬 수 있지만 결과는 변경 대상과 범위에 따라 다르다.
