# 암호화 기초

## 한 줄 요약

암호화는 평문을 제3자가 읽기 어려운 암호문으로 바꾸는 기술이고, 키를 어떻게 쓰느냐에 따라 대칭키 방식과 비대칭키 방식으로 나뉜다.

---

## 기본 용어

| 용어 | 의미 |
| --- | --- |
| 평문, plaintext | 아직 암호화하지 않은 원래 내용 |
| 암호문, ciphertext | 암호화한 뒤의 내용 |
| 암호화, encryption | 평문을 암호문으로 바꾸는 과정 |
| 복호화, decryption | 암호문을 다시 평문으로 되돌리는 과정 |
| 암호화 알고리즘 | 암호화와 복호화에 쓰는 수학적 절차 |
| 키, key | 알고리즘이 실제 결과를 만들 때 사용하는 값 |

암호화 알고리즘만 알아도 충분하지 않다. 같은 알고리즘을 쓰더라도 어떤 키를 쓰느냐에 따라 결과가 달라진다.

```text
평문 + 알고리즘 + 키
→ 암호문
```

---

## 대칭키 암호

대칭키 암호는 **암호화와 복호화에 같은 키를 사용하는 방식**이다.

```text
송신자: 평문 + 비밀키 → 암호문
수신자: 암호문 + 같은 비밀키 → 평문
```

### 장점

- 암호화와 복호화 속도가 빠르다.
- 많은 양의 데이터를 처리하기 좋다.
- 다양한 알고리즘이 존재한다.

### 단점

- 송신자와 수신자가 같은 키를 안전하게 공유해야 한다.
- 중간 공격자가 키를 얻으면 암호문을 해독할 수 있다.
- 참여자가 늘어날수록 키 관리가 복잡해진다.

### 대표 예시

- `DES`
- `3DES`
- `AES`
- `SEED`

---

## 비대칭키 암호

비대칭키 암호는 **서로 다른 공개키와 개인키를 한 쌍으로 사용하는 방식**이다.

```text
공개키로 암호화
→ 개인키로 복호화
```

공개키는 다른 사람에게 알려도 되지만, 개인키는 소유자만 보관해야 한다.

### 장점

- 비밀키를 미리 안전하게 전달하지 않아도 된다.
- 기밀성뿐 아니라 인증, 무결성, 부인방지 같은 목적과 연결할 수 있다.

### 단점

- 대칭키 방식보다 느리다.
- 대량 데이터 전체를 직접 암호화하는 용도로는 비효율적이다.

### 대표 예시

- `RSA`
- `Diffie-Hellman` 계열 키 교환

> [!note] 용어 주의
> `Diffie-Hellman`은 이 강의자료에서 비대칭형 방식 예시로 함께 나오지만, 핵심 역할은 데이터를 직접 암호화하는 것이 아니라 **공유 비밀을 합의하는 키 교환**에 가깝다.

---

## 대칭키와 비대칭키 비교

| 구분 | 대칭키 | 비대칭키 |
| --- | --- | --- |
| 사용하는 키 | 같은 비밀키 | 공개키와 개인키 한 쌍 |
| 속도 | 빠름 | 느림 |
| 키 전달 문제 | 큼 | 상대적으로 작음 |
| 주 용도 | 대량 데이터 암호화 | 키 교환, 인증, 서명과의 연결 |
| 예시 | AES, DES, SEED | RSA, Diffie-Hellman 계열 키 교환 |

실제 보안 프로토콜은 둘 중 하나만 쓰기보다, 각 방식의 장점을 조합하는 경우가 많다.

```text
비대칭키 / 키 교환
→ 안전하게 공유 비밀을 만들거나 키를 전달

대칭키
→ 이후 실제 데이터 통신을 빠르게 암호화
```

---

## 암호화가 제공하는 보안 목표

- **기밀성, Confidentiality**
  - 허가받은 수신자 외에는 내용을 읽지 못하게 한다.
- **무결성, Integrity**
  - 전송 중 내용이 바뀌지 않았는지 확인한다.
- **인증, Authentication**
  - 통신 상대나 문서 주체가 누구인지 확인한다.

자세한 무결성 및 인증 구조는 [[해시와 인증]]에서 다룬다.

## 실무 연결

암호화는 한 기술로 끝나는 주제가 아니라, 여러 보안 기능의 바탕이 된다.

| 연결 주제 | 암호화가 필요한 이유 |
| --- | --- |
| `SSH` | 원격 접속 중 계정 정보와 명령 내용을 보호 |
| `HTTPS / TLS` | 웹 통신 내용을 보호 |
| `VPN / IPsec` | 네트워크 구간의 데이터 보호 |
| `전자서명` | 발신자 확인과 무결성 검증 |
