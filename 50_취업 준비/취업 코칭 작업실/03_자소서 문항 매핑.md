# 자기소개서 문항 매핑

## 작성 원칙

- 희망 직무가 하는 일, 요구 기술, 선호하는 성격·기술 인재상을 먼저 조사한다.
- 직무 설명과 인재상을 답변에 직접 나열할 필요는 없다. 조사 결과는 경험과 강점 선택의 기준으로 사용한다.
- 관심 분야의 유행을 나열하지 않고 실제 프로젝트 판단과 연결한다.
- 배운 기술 목록보다 역할, 문제, 행동, 검증, 변화가 드러나게 쓴다.
- 각 경험은 `상황 → 역할 수행 → 결과` 순서로 준비한다.
- 같은 사례를 여러 문항에 재사용할 때는 강조점을 달리한다.
- 면접에서 설명할 수 없는 문장은 넣지 않는다.

## 과제 목적과 범위

- 목적: 자기소개서 완성이 아니라 세 질문에 사용할 경험·사례를 준비한다.
- 현재 단계: 경험의 사실, 상황, 역할 수행, 결과를 적는다.
- 다음 단계: 전체 글의 구조와 완성 문장은 다음 주 코칭 수업에서 다룬다.
- 정정 사항: `직무가 하는 일과 인재상을 직접 작성하는 문항`으로 이해하지 않는다.

## 공통 사전 조사

> 아래 내용은 세 문항의 사례를 고르는 기준이며, 답변 본문에 직접 넣을 필요는 없다.

- 희망 직무: 미확정
- 직무가 하는 일: 2026년 신입·현재 공고 기반 1차 조사 완료
- 직무가 요구하는 기술: 직무군별 1차 비교 완료
- 직무가 선호하는 성격·인재상: 일부 공고에서 확인, 희망 직무 확정 후 추가 조사 필요
- 조사 노트: [[50_취업 준비/취업 코칭 작업실/04_직무 시장 조사|직무 시장 조사]]
- 조사 후 결정할 것: 문항 1의 강점 키워드와 최종 사례

## 문항 1

> 본인만의 강점·장점과 이를 설명할 수 있는 경험

- 사전 조사 가설: 웹 취약점 진단·모의해킹·보안 컨설팅 계열은 취약점 재현, 원인 분석, 증거 수집, 오탐 구분, 조치·재검증 역량을 요구할 가능성이 높다. 실제 공고로 검증한다.
- 강점 키워드 가안: 결과를 그대로 믿지 않고, 증거가 무엇을 증명하는지 끝까지 구조화하는 태도
- 사용할 경험 가안: A-01 CARE 웹 애플리케이션 취약점 집중 진단
- 상황: 21개 웹 진단 항목의 절차·증거·진행 상태가 한 문서에서 뒤섞일 수 있었다.
- 역할 수행: 점검자로서 라우터와 항목별 문서를 분리하고, SQLi 등 우선 항목에 화면·요청/응답·로그·DB·코드·재검증 기준을 연결했다.
- 결과: 웹 진단 작업을 추적 가능한 구조로 만들고 SQLi의 절차·증거·조치안을 남겼다.
- 대안: 원인 분석 성향을 강점으로 선택하면 A-05 이중 PAT 트러블슈팅을 사용한다.
- 근거 ID: E-001~E-005, E-015
- 미확정 사항: 실제 직무 조사 후 강점 키워드와 A-01/A-05 중 하나를 확정

## 문항 2

> 관심 분야와 관련된 이슈, 프로젝트 적용 경험, 관심 분야를 발전시킨 과정

- 관심 분야: 웹 취약점 진단 자동화와 AI 보조 분석
- 관련 이슈: 웹 진단은 인증 상태·업무 흐름·상태 변경 때문에 고정 규칙만으로 오탐 없이 최종 판정하기 어렵다.
- 상황: KISA Web Application 21개 항목을 자동화하려 했지만 인증·권한·업무 흐름이 필요한 항목이 `manual_required`로 남고, HTTP 200만으로 파일 다운로드를 취약 판정하는 오탐도 발생했다.
- 역할 수행: manual의 원인을 재분류하고 checker를 최종 판정기가 아니라 요청·응답·source·ledger를 보존하는 evidence collector로 재설계했다. AI는 reviewer, 사람은 승인·최종 판정자로 제한했다.
- 결과: checker의 workflow 회귀 테스트와 설정 검증은 통과했고, AI review layer는 미구현 후속 설계로 명확히 분리했다.
- 발전 과정: 완전 자동화 구상 → 다수 manual과 HTTP 200 오탐 발견 → evidence collector로 역할 재정의 → AI reviewer와 human approval 구조 설계
- 사용할 실제 사례: A-02. 기반 경험은 A-01.
- 근거 ID: E-006~E-010
- 미확정 사항: 지원 회사가 AI 보안 자동화를 실제 직무 범위로 보는지 여부

## 문항 3

> 학원에서 배운 것과 수행한 일, 맡은 역할, 트러블슈팅

- 배운 것: Linux/Tomcat, 네트워크/NAT, 웹 취약점, AWS/IAM, KISA 진단, 증거와 보고서 관리
- 수행한 일: WEB 통합 구축 자동화, CARE 웹 취약점 점검, GNS 패킷 흐름 분석
- 맡은 역할: 1차 WEB 자동화 구현, 2차 `172.168.10.0/24`와 LB·WEB·NFS 구간 및 웹 애플리케이션 점검
- 상황: 장운호 R2에서 상대 R2로 보낸 ping의 reply가 장운호 R1까지 돌아왔지만 내부 R2로 전달되지 않았다.
- 역할 수행: Wireshark 관찰과 두 R1의 NAT ACL을 대조해 request와 reply에 PAT가 각각 걸리면서 최초 NAT session의 reverse matching이 깨지는 흐름을 단계별로 분석했다.
- 결과: 단순 routing 장애가 아니라 팀 간 대역을 다시 PAT한 정책 문제로 원인을 좁히고, 중간 R1은 팀 간 대역을 routing만 해야 한다는 수정 방향을 제시했다.
- 핵심 트러블슈팅: A-05 GNS 이중 PAT 분석
- 사용할 실제 사례: A-05와 역할 근거 A-06
- 대안: 학원에서 수행한 기술의 폭과 자동화를 강조할 경우 A-03을 사용한다.
- 근거 ID: E-011~E-016
- 미확정 사항: A-05 수정 후 성공 ping 증적 확인 여부, A-03/A-05 최종 선택

## 중복 점검

| 사례 | 문항 1 강조점 | 문항 2 강조점 | 문항 3 강조점 |
|---|---|---|---|
| A-01 | 취약점 재현·증거 관리 | 자동화가 필요해진 출발점 | 웹 점검 담당 역할 |
| A-02 | 오탐을 숨기지 않는 판단 | **우선 사례: 관심 분야의 문제·전환·발전** | 개인 확장 구현과 중단 판단 |
| A-03 | 재현 가능한 구축과 안전장치 | DevSecOps 관심의 실제 적용 | 대안: WEB 자동화 역할과 gateway 제약 해결 |
| A-05 | 대안: 패킷 기반 원인 분석 | 해당 없음 | **우선 사례: 가장 선명한 트러블슈팅** |

> [!note]
> 이 문서는 문항별 경험을 `상황 → 역할 수행 → 결과`로 배치하는 단계까지만 한다. 직무 조사와 코칭 전에는 최종 자기소개서 문장을 작성하지 않는다.
