# 자기소개서 문항 매핑

## 작성 원칙

- 직무명과 인재상을 먼저 확정한 뒤 강점을 연결한다.
- 관심 분야의 유행을 나열하지 않고 실제 프로젝트 판단과 연결한다.
- 배운 기술 목록보다 역할, 문제, 행동, 검증, 변화가 드러나게 쓴다.
- 같은 사례를 여러 문항에 재사용할 때는 강조점을 달리한다.
- 면접에서 설명할 수 없는 문장은 넣지 않는다.

## 문항 1

> 지원할 직무, 그 직무가 요구하는 인재상, 본인의 강점

- 지원 직무: 웹 취약점 진단·모의해킹·보안 컨설팅 계열을 우선 후보로 둔다. 최종 확정은 지원 공고 분석 뒤 한다.
- 요구 역량 / 인재상: 취약점 재현, 원인 분석, 증거 수집, 오탐 구분, 조치·재검증, 기술 내용을 설명하는 능력
- 본인의 강점: 결과를 그대로 믿지 않고 증거가 무엇을 증명하는지 제한하며, 실패 원인을 구조와 절차의 개선으로 연결하는 태도
- 사용할 실제 사례: A-01을 중심으로 A-05를 보조한다.
- 근거 ID: E-001~E-005, E-015
- 미확정 사항: 실제 지원 회사의 직무 기술서와 인재상 문구

## 문항 2

> 관심 분야와 관련된 이슈, 프로젝트 적용 경험, 관심 분야를 발전시킨 과정

- 관심 분야: 웹 취약점 진단 자동화와 AI 보조 분석
- 관련 이슈: 웹 진단은 인증 상태·업무 흐름·상태 변경 때문에 고정 규칙만으로 오탐 없이 최종 판정하기 어렵다.
- 프로젝트 적용: checker는 요청·응답·source·ledger를 수집하고, `manual_required`와 `inconclusive`를 숨기지 않도록 설계했다.
- 발전 과정: 완전 자동화 구상 → 다수 manual과 HTTP 200 오탐 발견 → evidence collector로 역할 재정의 → AI reviewer와 human approval 구조 설계
- 사용할 실제 사례: A-02. 기반 경험은 A-01.
- 근거 ID: E-006~E-010
- 미확정 사항: 지원 회사가 AI 보안 자동화를 실제 직무 범위로 보는지 여부

## 문항 3

> 학원에서 배운 것과 수행한 일, 맡은 역할, 트러블슈팅

- 배운 것: Linux/Tomcat, 네트워크/NAT, 웹 취약점, AWS/IAM, KISA 진단, 증거와 보고서 관리
- 수행한 일: WEB 통합 구축 자동화, CARE 웹 취약점 점검, GNS 패킷 흐름 분석
- 맡은 역할: 1차 WEB 자동화 구현, 2차 LB·WEB·NFS 구간과 웹 애플리케이션 점검
- 핵심 트러블슈팅: 인터넷/내부망 gateway 제약을 실행 순서에 반영한 A-03, 양쪽 R1의 이중 PAT를 규명한 A-05
- 사용할 실제 사례: A-03과 A-05, 보안 직무 연결은 A-01
- 근거 ID: E-011~E-016
- 미확정 사항: 문항 글자 수에 따라 A-03 또는 A-05 중 하나만 본문에 사용

## 중복 점검

| 사례 | 문항 1 강조점 | 문항 2 강조점 | 문항 3 강조점 |
|---|---|---|---|
| A-01 | 취약점 재현·증거 관리 | 자동화가 필요해진 출발점 | 웹 점검 담당 역할 |
| A-02 | 오탐을 숨기지 않는 판단 | 관심 분야의 문제·전환·발전 | 개인 확장 구현과 중단 판단 |
| A-03 | 재현 가능한 구축과 안전장치 | DevSecOps 관심의 실제 적용 | WEB 자동화 역할과 gateway 제약 해결 |
| A-05 | 패킷 기반 원인 분석 | 해당 없음 | 가장 선명한 트러블슈팅 |

> [!note]
> 이 문서는 문항별 근거 배치까지만 한다. 지원 직무와 공고가 확정되기 전에는 최종 자기소개서 문장을 작성하지 않는다.
