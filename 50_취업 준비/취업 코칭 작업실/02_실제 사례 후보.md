## 사례 카드

### A-01. CARE 웹 애플리케이션 취약점 집중 진단

- 프로젝트와 시기: 2차 프로젝트, 2026년 6월
- 상황: KISA 기준으로 온프레미스와 AWS를 진단하는 팀 프로젝트에서 CARE 웹 애플리케이션 점검을 맡았다.
- 문제: 21개 항목을 한 문서에서 관리하면 공격 절차·증거·조치·진행 상태가 뒤섞였다.
- 맡은 역할: 보고서에 명시된 웹 애플리케이션 점검자. LB·WEB·NFS 구간 담당과 연결됐다.
- 직접 한 행동: 라우터와 항목별 노트로 분리하고, 화면·요청/응답·서버 로그·DB 결과·코드 diff·재검증을 증거 기준으로 정했다. SQLi에서는 검색 참·거짓 조건과 로그인 인증 우회를 나눠 정리했다.
- 판단과 이유: 항목 수보다 재현 가능성과 증거 추적성을 우선했다.
- 결과: 21개 항목의 작업 구조와 우선순위를 만들었고, SQLi에는 테스트 계정·payload·화면 증거·조치안이 남았다.
- 검증 증거: E-001~E-005
- 한계와 정직한 표기: 라우터의 다수 항목은 `점검 예정` 또는 진행 중이다. 21개 전체 완료로 표현하지 않는다.
- 지원 직무 연결: 웹 취약점 진단, 모의해킹, 보안 컨설팅
- 예상 꼬리질문: SQLi 유형과 payload는 무엇이었나? 참·거짓 응답을 어떻게 구분했나? Prepared Statement가 왜 막는가? 실제 조치·재검증 범위는 어디까지인가?

### A-02. `Manual`에서 evidence collector로 방향 전환

- 프로젝트와 시기: 2차 프로젝트 연장 개인 작업, 2026년 6월
- 상황: KISA Web Application 21개 항목을 선언형 check와 profile로 자동화하려 했다.
- 문제: 인증·권한·업무 흐름·상태 변경이 필요한 항목이 `manual_required`로 남았고, 파일 다운로드에서는 HTTP 200만 보고 취약으로 오판했다.
- 맡은 역할: 전용 저장소의 checker·profile·check·payload·mock·test 구현 및 설계 기록.
- 직접 한 행동: manual을 fixture 부족, 조건부 manual, 정상 manual로 재분류했다. HTTP status 외 header·body·redirect·login state·source·DB state를 판정 근거로 추가했다. mode gate, evidence 저장, workflow fixture/cleanup ledger와 회귀 테스트를 구현했다.
- 판단과 이유: 고정 규칙이 최종 판정을 독점하면 오탐을 감춘다고 보고, checker를 증거 수집기로 낮추고 AI는 reviewer, 사람은 승인·최종 판정자로 재설계했다.
- 결과: 현재 checker는 설정 검증과 workflow 회귀 테스트를 통과한다. 완전 자동 판정의 한계를 오탐 사례로 설명할 수 있게 됐다.
- 검증 증거: E-006~E-010
- 한계와 정직한 표기: AI review 자동화와 전체 R4 runtime은 구현하지 않았다. 프로젝트는 중단됐으며 완성된 AI 진단기로 표현하지 않는다.
- 지원 직무 연결: 보안 자동화, 취약점 진단, AI 보조 분석, DevSecOps
- 예상 꼬리질문: 어떤 항목이 왜 manual이었나? 15번 오탐은 왜 났나? AI가 무엇을 입력받고 어디까지 판단하나? 테스트가 증명하지 못하는 것은 무엇인가?

### A-03. WEB 통합 구축·재구축 자동화

- 프로젝트와 시기: 1차 프로젝트 및 후속 정리
- 상황: WEB 서버마다 Tomcat 설치, source build, WAR 배포, DB/Gmail 설정, NFS, 로그 수집을 반복해야 했다.
- 문제: 인터넷 gateway와 내부망 gateway를 동시에 안정적으로 쓰기 어려웠고, 평문 secret이 repo와 WAR에 남을 수 있었다.
- 맡은 역할: WEB 통합 설치·재구축 스크립트 구현.
- 직접 한 행동: 97개 함수 기반으로 설치·build·deploy·backup·secret 외부화·NFS·Promtail·검증을 묶었다. 인터넷 필요 작업을 먼저 끝내고 사람이 network를 전환한 뒤 내부망 작업을 수행하도록 순서를 나눴다.
- 판단과 이유: 단순 한 줄 자동화보다 환경 제약, 재실행 안전성, 백업, 실패 확인점을 실행 흐름에 포함했다.
- 결과: `webCompZzinFinal.sh`와 설명서가 남았고 현재 Bash 구문 검사를 통과한다.
- 검증 증거: E-011~E-014
- 한계와 정직한 표기: VM 생성이나 LB upstream 변경은 자동화하지 않았고, 환경변수·env 파일은 서버가 침해되면 노출될 수 있다. 이번 조사에서 실제 서버 재배포는 재실행하지 않았다.
- 지원 직무 연결: 시스템·클라우드 보안, DevSecOps, 보안 운영 자동화
- 예상 꼬리질문: env 파일 권한은? Secrets Manager와 차이는? 왜 완전 무인화하지 않았나? 실패 시 롤백은? 새 서버에서도 재현 가능한가?

### A-04. SQL Injection 참·거짓 비교와 인증 우회

- 프로젝트와 시기: 기존 CARE 실습과 2차 프로젝트 재정리
- 상황: 멘토에게 SQLi 방식을 질문받았을 때 경험을 유형과 과정으로 즉시 설명하지 못했다.
- 문제: 수행 경험과 설명 가능한 이해 사이에 간극이 있었다.
- 직접 한 행동: 게시판 검색의 true/false payload와 로그인 `victim'--`를 구분하고, boolean comparison·authentication bypass·error exposure·prepared statement 조치를 문서로 연결했다.
- 결과: A-01의 기술적 깊이를 보여주는 대표 에피소드가 됐다.
- 검증 증거: E-003, E-004
- 한계와 정직한 표기: time-based blind나 stacked query까지 수행했다고 말하지 않는다. `UNION SELECT`는 checker payload 후보와 실제 데이터 추출 성공을 구분한다.

### A-05. GNS 이중 PAT 트러블슈팅

- 프로젝트와 시기: 2차 프로젝트, 2026-06-08
- 상황: 장운호 R2에서 상대 R2 `1.1.2.2`로 ping이 0/5 실패했다.
- 문제: reply가 장운호 R1까지 돌아오는데 내부 R2로 전달되지 않았다.
- 맡은 역할: 본인 네트워크 구간의 패킷 흐름 분석.
- 직접 한 행동: Wireshark 관찰과 두 R1의 NAT ACL을 대조했다. request는 장운호 R1에서 PAT되고 reply는 김용준 R1에서 다시 PAT돼 source가 바뀌므로, 최초 NAT session의 reverse matching이 깨진다고 단계별로 정리했다.
- 결과: 단순 routing 장애가 아니라 팀 간 대역까지 PAT한 정책 문제라는 원인을 규명하고, 팀 간 대역은 routing만 해야 한다는 정상 흐름을 제시했다.
- 검증 증거: E-015
- 한계와 정직한 표기: 최종 설정 변경과 성공 ping은 현재 확인한 기록에 없다. `원인 규명과 수정 방향 제시`까지로 표현한다.
- 지원 직무 연결: 보안관제 원인 분석, 네트워크 보안, 침해 경로 분석
- 예상 꼬리질문: PAT가 왜 두 번 걸렸나? reply source 변화가 reverse NAT에 왜 영향을 주나? NAT exemption은 어느 ACL에 어떻게 넣어야 하나?

### A-08. 온프레미스/AWS 비교 관점의 전환
