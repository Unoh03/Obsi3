# 취업 코칭 사전 과제 초안

> [!note]
> 자기소개서 완성본이 아니라 코칭 시간에 보여드릴 경험 정리 초안이다. 직무 조사 결과와 세 문항의 경험을 `상황 → 역할 수행 → 결과` 순서로 작성했다.

## 0. 희망 직무 사전 조사

### 희망 직무 가안

**기술 취약점 진단·정보보호 컨설팅**

순수 공격 연구만을 의미하는 모의해킹보다는 웹·서버·네트워크·DB·클라우드 환경을 점검하고, 취약점의 재현 근거와 개선안을 전달하는 직무를 우선 가정했다.

### 실제로 하는 일

- 웹·모바일·서버·네트워크·DB·클라우드의 취약점을 점검하고 재현한다.
- 자동 진단 결과를 수동으로 검증해 오탐과 실제 취약점을 구분한다.
- 발견한 문제의 영향과 원인을 정리하고 조치 가이드를 제시한다.
- 진단 결과와 증적을 보고서로 작성하고 조치·재점검·이행점검을 수행한다.
- 프로젝트에 따라 고객 담당자와 범위, 일정, 증적, 개선안을 조율한다.

### 요구되는 기술

- 웹 애플리케이션과 HTTP, 인증·인가, 주요 웹 취약점에 대한 이해
- Linux, 서버, 네트워크, DB와 보안 장비에 대한 기초 지식
- 취약점 진단 기준과 도구를 사용하고 결과를 수동 검증하는 능력
- 클라우드 환경에서는 IAM, VPC, EC2, Security Group 등 기본 구조에 대한 이해
- Python·Shell 등 스크립트 활용과 증적·보고서 작성 능력

### 선호하는 성격·인재상

- 작은 차이와 누락을 지나치지 않는 꼼꼼함
- 결과를 그대로 받아들이지 않고 근거를 확인하는 분석적 태도
- 새로운 공격 기법과 환경을 계속 학습하는 자세
- 고객과 팀원에게 기술 내용을 설명할 수 있는 의사소통 능력
- 진단 범위와 결과에 책임을 지고 끝까지 추적하는 태도

## 1. 본인만의 강점·장점과 이를 설명할 수 있는 경험

### 강점 키워드

**애매한 결과를 그대로 넘기지 않고, 판단 근거를 구조화하여 끝까지 확인하는 태도**

### 상황

2차 프로젝트에서 KISA 기준으로 온프레미스와 AWS 환경을 진단하면서 CARE 웹 애플리케이션 점검을 맡았다. 웹 진단 항목은 21개였고, 한 문서에 공격 절차, 증거, 조치 방법, 진행 상태를 모두 기록하면 무엇을 실제로 확인했고 무엇이 예정 상태인지 구분하기 어려웠다.

### 역할 수행

웹 애플리케이션 점검자로서 전체 항목을 관리하는 라우터 문서와 항목별 점검 문서를 분리했다. 증거도 화면만 남기는 것이 아니라 요청·응답, 서버 로그, DB 결과, 관련 코드와 재검증 결과로 나누어 연결하도록 기준을 세웠다. SQL Injection은 게시판 검색의 참·거짓 조건 비교와 로그인 인증 우회를 구분하고, 문자열 결합이라는 원인과 Prepared Statement 등의 조치안을 함께 정리했다.

### 결과

웹 진단 작업의 진행 상태와 판정 근거를 추적할 수 있는 구조를 만들었고, 우선 점검한 SQL Injection에는 테스트 절차, payload, 화면 증거와 조치안을 남겼다. 다만 21개 항목 전체를 완료한 것은 아니므로, 전수 진단을 완료했다고 표현하지 않고 점검 구조를 만들고 일부 우선 항목을 깊게 확인한 경험으로 정리했다.

## 2. 관심 분야와 관련 이슈, 프로젝트 적용과 발전 과정

### 관심 분야

**웹 취약점 진단 자동화와 AI를 이용한 증거 해석 보조**

### 관련 이슈와 상황

CARE 웹 진단 경험을 바탕으로 KISA Web Application 21개 항목을 자동화하는 개인 확장 프로젝트를 진행했다. 그러나 인증 상태, 사용자 권한, 업무 흐름과 상태 변경이 필요한 항목은 고정된 요청과 응답 규칙만으로 판정하기 어려워 `manual_required`로 남는 경우가 많았다. 파일 다운로드 항목에서는 HTTP 200 응답만 보고 취약하다고 판단했지만, 실제로는 로그인 화면이나 차단 응답일 수 있어 오탐이 발생했다.

### 역할 수행과 프로젝트 적용

처음에는 `Manual` 항목을 줄여 자동 판정 비율을 높이는 것이 목표였다. 진행하면서 Manual의 원인을 fixture 부족, 조건부 수동 검증, 자동화하기 부적절한 정상적인 수동 검토로 다시 분류했다. 또한 status code뿐 아니라 header, body, redirect, login state, source와 DB state를 함께 수집하도록 구조를 보완했다. 최종적으로 스크립트의 역할을 취약·양호를 확정하는 진단기에서 요청·응답과 실행 상태를 재현 가능하게 보존하는 evidence collector로 바꾸고, AI는 증거 검토와 후속 점검 제안을 보조하며 사람은 승인과 최종 판정을 맡는 구조를 설계했다.

### 결과와 관심 분야의 발전

상태 변경형 workflow, mode gate, fixture ledger, secret redaction과 cleanup 처리를 구현하고 회귀 테스트를 통과시켰다. 반면 AI review layer는 구현하지 못한 상태에서 프로젝트를 중단했으므로 AI 진단 시스템을 완성했다고 표현할 수는 없다. 이 경험을 통해 관심이 `웹 진단을 완전히 자동화하는 방법`에서 `자동화가 수집한 증거의 한계를 표시하고, AI와 사람이 판정을 분담하는 방법`으로 발전했다.

## 3. 학원에서 배운 것과 수행한 역할, 인상 깊었던 트러블슈팅

### 배운 것과 수행한 일

학원에서 Linux와 Tomcat 기반 웹 서버, 네트워크와 NAT, 웹 취약점, AWS와 IAM, KISA 기준 취약점 진단을 학습했다. 1차 프로젝트에서는 Tomcat 설치부터 Git/Maven build, WAR 배포, DB·메일 접속정보 분리, NFS와 로그 설정, 최종 검증을 한 흐름으로 수행하는 WEB 통합 스크립트를 만들었다. 2차 프로젝트에서는 `172.168.10.0/24` 대역과 LB·WEB·NFS·KALI 구간을 담당하고 CARE 웹 애플리케이션 점검을 맡았다.

### 상황

GNS 네트워크를 구성하던 중 내 R2에서 상대 팀 R2로 보낸 ping이 모두 실패했다. 패킷을 추적하니 요청은 상대편까지 도착했고 reply도 내 R1까지 돌아왔지만, 내부 R2로 전달되지 않았다.

### 역할 수행

단순 routing 장애라고 가정하지 않고 Wireshark에서 request와 reply의 주소가 각 구간에서 어떻게 변하는지 확인했다. 양쪽 R1의 NAT ACL을 비교한 결과, 팀 간 통신 패킷에도 PAT가 각각 적용되면서 reply의 source가 다시 변환됐고, 최초 NAT session의 reverse matching이 깨지는 흐름을 확인했다.

### 결과

문제의 원인을 경로 누락이 아니라 팀 간 대역을 다시 PAT한 정책으로 좁혔다. 중간 R1에서는 상대 팀 대역을 NAT 대상에서 제외하고 routing만 해야 한다는 수정 방향을 제시했다. 현재 기록에는 수정 후 성공 ping 증적이 남아 있지 않으므로, 해결 완료가 아니라 원인 규명과 수정 방향 제시까지 수행한 트러블슈팅으로 작성했다.
