# 취업 코칭 RAW 수집함

> 두 채팅, Vault, 관련 저장소에서 들어오는 사실·기억·파일 경로·후보 사례를 판단 전에 모으는 작업용 수집함이다.

## 과제 원문

### 실제 사례 3가지

- 내가 프로젝트에서 실제로 한 것
- 예시: 2차 프로젝트에서 웹 애플리케이션 분야를 집중적으로 파고든 경험

### 자기소개서 3문항

1. 지원할 직무, 그 직무가 요구하는 인재상, 본인의 강점
2. 관심 분야와 관련 이슈, 프로젝트 적용 경험, 관심 분야를 발전시킨 과정
3. 학원에서 배운 것과 수행한 일, 맡은 역할, 트러블슈팅

## 수집 원칙

- 학원 공식 프로젝트와 프로젝트의 개인 확장 작업을 모두 후보로 받는다.
- 공식 산출물, 개인 확장, 완성, 중단 상태를 구분한다.
- 두 참조 채팅의 사용자 진술과 정리 내용은 사실 기반 자료로 취급한다.
- 최종 자기소개서 문장보다 실제 행동과 판단 근거를 먼저 기록한다.
- 팀이 한 일과 장운호가 직접 한 일을 분리한다.

### 2026-07-22 참조 채팅 2개 1차 추출

#### 인물·학습 맥락

- 이름은 장운호다. 앞선 대화에서 다른 조원인 정원의 개인 질문이 장운호의 것으로 잘못 연결된 적이 있었고, 사용자가 직접 이를 정정했다.
- 건축학과를 다니다가 2026년 2월 말부터 클라우드·보안 분야 학습을 시작했다.
- 노량진에서 받는 교육 자체는 재미있지만, 구체적으로 원하는 보안 직무는 아직 확정하지 못했다.
- 단순히 취업이 잘된다는 이유로 직무를 선택하기보다 실제로 흥미를 느끼는 문제 유형을 찾고 있다.

#### 1차 프로젝트와 이전 관심

- 교육 초기에는 DevSecOps에 관심이 있었고, JSP 환경의 `application.properties`에 담긴 DB 애플리케이션 계정 보호와 침해 서버 대응 문제를 고민했다.
- DB 계정 정보를 다른 디렉터리나 별도 서버에 두는 방안을 떠올렸지만, 설정 파일을 탈취할 수준의 공격자라면 그 위치도 접근할 수 있다고 스스로 반박했다.
- 서버가 망가지거나 침해되면 고쳐 쓰기보다 없애고 새로 만든다는 발상을 했다. 이후 이 생각은 재현 가능한 구축과 재배포 관점으로 연결됐다.
- 1차 프로젝트에서는 하나의 스크립트로 웹 서버 구축을 완료할 수 있게 했고, 스크립트 안에서 환경 변수를 처리할 수 있게 구성했다.
- 이 경험은 단순 아이디어가 아니라 실제 스크립트가 존재하는 작업으로 진술됐다. 정확한 파일과 실행 범위는 Vault에서 확인한다.

#### 2차 프로젝트

- 조의 발표 주제는 온프레미스 환경과 AWS 환경에서 보안 진단의 축이 어떻게 달라지는지를 비교하는 것이었다.
- 2차 프로젝트에서 KISA 기준 온프레미스·AWS 보안 진단, 조치, 보고서화를 수행했다.
- 장운호는 특히 웹 애플리케이션 취약점 분야를 집중적으로 파고들었다.
- SQL Injection은 로그인 창과 게시글 검색 입력 지점을 대상으로 참·거짓 조건을 넣어 비정상적인 결과, 인증 우회 가능성, 검색 결과 차이 또는 정보 노출을 확인하는 방식으로 수행했다고 진술했다.
- 멘토에게 SQL Injection 방식을 질문받았을 때 즉시 구조화해 설명하지 못한 경험이 남았다. 이는 수행 경험과 설명 가능한 이해 사이의 차이를 인식한 계기다.
- AWS IAM에서는 평상시 ReadOnly 권한을 사용하고, 문제 해결이나 변경 작업 시 특정 서비스·행동만 가능한 관리자 성격의 권한을 세분화했으며, root는 일상적으로 사용하지 않는 구조를 적용했다.
- 멘토링을 통해 2차 프로젝트의 약점은 실제 공격 수행과 그 공격이 클라우드 환경 어디에서 탐지·확인되는지 매핑하는 부분이 부족했다는 점으로 정리했다.

#### 웹 애플리케이션 진단 checker 확장

- 2차 프로젝트의 연장선에서 KISA Web Application 진단 스크립트를 별도로 만들려 했다.
- 서버·인프라의 정적 설정 진단과 달리 웹 애플리케이션 진단은 인증 상태, 사용자 역할, 업무 흐름, 요청 순서, 응답 의미 등 변수가 많아 결과가 `Manual`로 남는 항목이 많았다.
- 처음에는 21개 항목을 자동화하는 진단기를 구상했지만, 고정 규칙만으로 취약·양호를 확정하면 오탐이나 과장된 판정이 생길 수 있다고 판단했다.
- `Manual` 판정을 줄이기 위해 스크립트는 요청·응답·소스·로그 등의 evidence를 수집하고, AI가 문맥 해석과 후속 검증 설계를 보조하며, 사람이 최종 판정하는 구조로 방향을 전환했다.
- 프로젝트는 중단됐으므로 완성된 자동 진단기로 표현하지 않는다. 핵심 경험은 자동화 한계를 발견하고 설계 목표를 수정한 과정이다.

#### 멘토링에서 바뀐 관점

- 온프레미스와 AWS의 경계가 명확하다고 생각했으나, 실제 환경은 VPN 등으로 연결된 하이브리드 형태가 많다는 설명을 들었다.
- 두 환경의 보안 통제 목적은 이어지지만 책임 범위, 증적 위치, 조치 방식, 권한 모델의 결이 다르고, 실제 진단은 환경 이름보다 서비스 흐름과 공격 경로로 통합해 봐야 한다는 관점을 받아들였다.
- 멘토는 단순 서비스 개념 질문보다 실제 공격을 수행하고 그 공격이 어디에서 탐지되는지를 확인하는 프로젝트를 강조했다.
- 이 경험을 통해 보안 진단의 실무적 가치는 체크리스트 수행량보다 공격·장애·침해 사건의 증거를 해석하고 조치·재점검하는 데 있다는 관점이 강화됐다.

#### 역할 선호와 강점 후보

- 대시보드 시각화나 단순 조사보다 공격 시나리오 설계, 취약점 재현, 조치 전후 검증 같은 역할에 더 끌린다.
- 결과를 그대로 믿기보다 무엇을 증명했고 무엇을 증명하지 못했는지 구분하는 데 관심이 크다.
- AI를 많이 활용하지만 AI가 만든 구현·판정·설명을 그대로 본인 능력으로 간주하지 않고, 실제 이해와 검증의 경계를 문제 삼는다.
- 문제를 구조화하고 증거와 결론 사이의 경계를 따지는 성향, 실패 원인을 설계 변경으로 연결하는 태도, 재현 가능하게 문서화하려는 습관이 강점 후보로 반복해서 나타난다.

#### 충돌 방지 메모

- 정원의 `전수 유발 폐루프`, 보고서 분석·해석 비중, 진단 설계·문서화 개인 질문은 장운호의 개인 질문으로 사용하지 않는다.
- 과거 assistant가 장운호에게 귀속한 역할이라도 사용자 직접 진술이나 로컬 증거로 연결되지 않으면 개인 성과로 확정하지 않는다.
- 미래의 3차 프로젝트 관심 방향은 관심·계획 증거이지, 이미 수행한 실제 사례가 아니다.
