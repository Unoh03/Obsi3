## 토론 기록

### 19:05 코드 인젝션 문서 구조 토론

`웹 취약점 - 01 코드 인젝션` 문서는 단순한 첫 취약점 문서가 아니라, 이후 취약점별 보고서 작성 방식의 기준 문서로 삼기로 했다.

핵심 전제는 다음과 같다.

| 논점 | 최종 판단 |
|---|---|
| 전체 보고서 구조 | `웹 애플리케이션 취약점 점검 보고서.md`는 라우터 역할만 맡고, 절차·증거·판정·조치는 취약점별 문서로 분리한다. |
| 코드 인젝션의 기존 CARE 적용성 | 기존 CARE 기능에는 코드 인젝션 6개 세부 유형을 직접 재현할 기능이 명확하지 않다. |
| 웹쉘 업로드의 분류 | PHP 웹쉘 업로드는 서버 코드 실행으로 이어질 수 있지만, 발생 경로 기준으로는 `코드 인젝션`이 아니라 `악성 파일 업로드`에서 다룬다. |
| SQL Injection과 Code Injection 구분 | 둘 다 Injection 계열이지만, SQL Injection은 DBMS가 SQL로 해석하고 Code Injection은 OS shell, LDAP, XML parser, XPath, SSI, template engine 등이 입력값을 실행·해석한다. |
| 구현 방향 | 기존 CARE 기능과 섞지 않고 `/vuln/code-injection/` 같은 별도 실습 경로에 구현한다. |
| 정직성 기준 | 원래 앱에 있던 취약점인지, 실습을 위해 추가한 취약 기능인지 반드시 구분한다. |
| 결과 표현 | 각 유형은 실제 재현, 모의 재현, 조건부 재현, 비교 학습, 해당 없음으로 구분한다. |

PDF p.679~692의 코드 인젝션 세부 유형은 다음 6개로 보고, 모두 문서에 담되 구현 성격을 다르게 처리하기로 했다.

| 세부 유형 | 처리 방향 |
|---|---|
| LDAP Injection | 실제 LDAP 서버가 없으므로 구현보다 SQL Injection과의 비교 학습 중심 |
| OS Command Injection | 가장 명확한 실제 재현 대상으로 우선 구현 |
| SSI Injection | Apache SSI 설정이 필요하므로 조건부 재현으로 관리 |
| XPath Injection | XML 데이터와 XPath 질의를 이용해 실제 재현 가능 |
| XXE Injection | XML 외부 엔티티 처리를 이용해 실제 재현 가능 |
| SSTI | 실제 템플릿 엔진이 없으므로 교육용 렌더러 기반 모의 재현 |

문서 구조는 처음 제안된 감사 체크리스트형 구조보다, 사용자가 제안한 흐름 중심 구조를 채택했다.

```markdown
# 웹 취약점 - 01 코드 인젝션

## 1. 항목 개요
## 2. PDF 요약
### SQL Injection과의 차이
## 3. PDF 기준 세부 유형 및 요약
## 4. 기존 기능 적용 가능성 분석
## 5. 실습 구현 범위
## 6. 유형별 실습 — 취약점 파악 및 조치 방안
### 6-1. LDAP Injection
### 6-2. OS Command Injection
### 6-3. SSI Injection
### 6-4. XPath Injection
### 6-5. XXE Injection
### 6-6. SSTI
## 7. 종합 판단
```

`종합 판단`은 감상문식 결론이 아니라 판정 요약으로 둔다. 각 유형이 실제 재현인지, 모의 재현인지, 조건부 재현인지, 비교 학습인지 표로 정리하는 역할이다.

`실습 환경 특이사항`은 독립 대제목으로 두지 않고 `실습 구현 범위`와 `종합 판단` 안에 흡수한다. CARE 자체를 설명하는 문서가 아니라 주요정보통신기반시설 웹 취약점 항목을 학습·시연하는 문서이기 때문이다.

### 19:45 속성 처리와 링크 표기 원칙 토론

Obsidian 문서 작성 방식에 대해서도 다음 기준을 정했다.

| 논점 | 최종 판단 |
|---|---|
| frontmatter 위치 | 속성은 파일 맨 첫 줄의 `---`로 시작해야 하며, 코드블록 안에 넣으면 Obsidian 속성으로 인식되지 않는다. |
| frontmatter 코드블록 금지 | ` ```yaml ... ``` `로 감싸면 속성이 아니라 본문 코드블록이 된다. |
| 속성 타입 충돌 | `source_pages`, `aliases`, `tags` 등은 vault에 이미 잡힌 속성 타입과 충돌할 수 있으므로 실제 Obsidian UI에서 확인한다. |
| 라우터 문서 링크 | `웹 애플리케이션 취약점 점검 보고서.md`처럼 하위 문서를 안내하는 라우터에서는 앞으로 만들 취약점 문서도 wiki link로 미리 연결한다. |
| 백틱 사용 대상 | 실제 코드 파일, 서버 경로, 명령어, 파라미터는 백틱으로 표기한다. |
| wiki link 사용 대상 | 보고서, 취약점별 문서, MOC처럼 이동해야 하는 문서 경로는 wiki link로 표기한다. |

이 토론에서 정리한 보고서 작성 가치관은 다음과 같다.

- 읽는 흐름을 우선한다. 체크리스트 완결성을 위해 대제목을 과하게 쪼개지 않는다.
- 세부 내용은 필요한 위치에 흡수한다. 증거, 조치, 판단을 매번 독립 대제목으로 만들 필요는 없다.
- 취약점은 결과가 아니라 발생 경로로 분류한다.
- 기존 기능 취약점과 실습용 추가 취약점을 구분한다.
- 결론은 감상문이 아니라 판정 요약으로 쓴다.
- 처음부터 최종 제출본처럼 압축하지 말고, 나중에 자르거나 부록으로 옮길 수 있게 독립적으로 작성한다.
- 라우터는 전체 구조를 보여주고, 세부 절차와 증거는 개별 취약점 문서가 담당한다.

현재 `웹 취약점 - 01 코드 인젝션.md`는 이 원칙을 반영해 작성 중이며, 6-2 OS Command Injection까지 완성되고 6-3 SSI Injection은 진행 중이다.
