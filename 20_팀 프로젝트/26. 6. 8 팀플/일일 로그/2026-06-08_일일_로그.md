## 결정한 것

| 결정 | 이유 | 영향 |
|---|---|---|
| AWS와 GNS/VM을 병행한다 | AWS VPC만으로 ARP spoofing, DHCP spoofing, 라우터 ACL 미흡 같은 네트워크 취약점을 자연스럽게 시연하기 어렵다 | AWS는 클라우드/서비스 보안, GNS/VM은 네트워크 취약점 시연 담당 |
| 초반에는 AWS와 GNS/VM을 억지로 연결하지 않는다 | VPN/하이브리드 연결까지 넣으면 초기 범위가 커진다 | 먼저 분리된 실습 환경으로 만들고, 시간이 남으면 확장 고려 |
| 취약점 후보는 많이 모은다 | 초반에 줄이면 회의와 보고서 설계 재료가 부족해진다 | 후보표에서 `상/중/하`, 기존 경험, 난이도, 시연성, 환경을 함께 기록 |
| PDF 등급 `상`만 보지 않는다 | `중/하` 항목도 시연성이 좋거나 프로젝트 구조와 맞으면 가치가 있을 수 있다 | 후보표와 일일 로그에 등급을 속성으로 남기고, 필터로 쓰지 않음 |
| 이번에는 LB/NFS 이중화를 하지 않는다 | 현재 요구는 단일 구성 검증과 프로젝트 실습 재사용이 우선이다 | HA/VIP/keepalived 계열은 제외하고 단일 LB/NFS 검증에 집중 |
| WEB 스크립트는 통합 스크립트를 그대로 쓰지 않는다 | 이전 통합 스크립트는 설치, 배포, secret, NFS, 로그, SMTP 정책이 한 파일에 얽혀 있다 | `web-base`, `web-deploy`, `web-secret`, `web-nfs-client`, `web-log-agent`, `web-verify`로 분해 후보를 둠 |

## 남은 문제 / 막힌 점

| 문제 | 현재 증상 | 추정 원인 | 다음 확인 |
|---|---|---|---|
| Cisco NAT 구조 | R2에서 상대 R2 공개 주소로 ping 실패 | 양쪽 R1에서 PAT가 걸려 reply 패킷의 source가 다시 바뀌고 NAT reverse 매칭이 깨짐 | R1은 팀 간 `1.1.x.0/24` 통신을 라우팅만 하게 하고, 서버 공개는 R2 static NAT로 처리하는 구조 확인 |
| NFS export 범위 | `EXPORT_NET="172.168.10.10"` 입력 후보가 나옴 | 단일 호스트면 `/32`가 필요하고, `172.168.10.10`은 RFC1918 사설 대역이 아님 | 실제 VM 내부망 IP와 export CIDR 확인 |
| 취약점 후보 등급 | 후보표에 아직 PDF 등급 컬럼이 전부 채워진 상태는 아님 | 오늘은 후보 풀과 방향을 먼저 잡음 | PDF 세부 항목을 읽으면서 `상/중/하/확인 필요`를 후보별로 보강 |
| LB 502 | `curl -i http://127.0.0.1/`에서 `502 Bad Gateway` 발생 | nginx upstream/backend에 `:8080`이 붙어 현재 WEB 구조와 맞지 않음 | 해결됨. upstream에서 `8080` 제거 후 접속 성공 |
