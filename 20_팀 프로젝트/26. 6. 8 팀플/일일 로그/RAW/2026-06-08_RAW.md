### 12:43

- 프로젝트 범위 보정: AWS만으로는 네트워크 쪽 취약점 구현/방어가 부족하므로 GNS/VM도 사용하기로 함.
- 이유: AWS VPC는 L2 네트워크 취약점(ARP spoofing, DHCP spoofing, VLAN/스위치/라우터 ACL 미흡 등)을 자연스럽게 시연하기 어렵다. 이런 취약점은 GNS/VM이 적합.
- 역할 분리안:
  - AWS: 실제 서비스 운영 환경. VPC, Public/Private Subnet, ALB, ASG, EC2, DB, S3/EFS, CloudWatch, Secret 분리.
  - GNS/VM: 네트워크 취약점 실습 환경. 공격자 VM, 사용자 PC, 내부 서버, 라우터/방화벽, ACL 미흡, 관리 포트 노출, 평문 통신, 내부망 분리 미흡 시연.
- AWS와 GNS/VM을 처음부터 VPN 등으로 억지 연결하는 것은 범위가 커질 수 있음. 우선은 "온프레미스/GNS 내부망 + AWS 클라우드 서비스망"으로 분리하고, 시간이 남으면 하이브리드 연결을 확장으로 고려.
- 보고서 스토리 후보: 온프레미스 내부망의 네트워크 취약점과 AWS 클라우드 서비스 보안 구성을 나란히 보여주고, 같은 보안 원칙이 GNS/VM에서는 ACL/망분리/관리망 분리로, AWS에서는 SG/NACL/Public-Private Subnet/Bastion/Secret/Logging으로 대응됨을 설명.

- 기본 상태에서 장운호 R1과 김용준 R1 모두 자기 `1.1.x.0/24` 대역이 outside로 나가면 목적지와 관계없이 PAT하도록 설정되어 있었음.

```cisco
! 장운호 R1
ip nat inside source list 99 int f0/0 overload
access-list 99 permit 1.1.1.0 0.0.0.255

! 김용준 R1
ip nat inside source list 99 int f0/0 overload
access-list 99 permit 1.1.2.0 0.0.0.255
```

- 장운호 R2에서 실행한 테스트:

```text
R2#ping 1.1.2.2

Type escape sequence to abort.
Sending 5, 100-byte ICMP Echos to 1.1.2.2, timeout is 2 seconds:
.....
Success rate is 0 percent (0/5)
R2#
```

- NAT 흐름 분석:

```text
[1] 장운호 R2가 ICMP request 생성

src 1.1.1.2
dst 1.1.2.2
```

```text
[2] 장운호 R1에서 첫 번째 PAT

장운호 R1 기준:
inside f0/1 -> outside f0/0

변환 전:
src 1.1.1.2
dst 1.1.2.2

변환 후:
src 192.168.31.110
dst 1.1.2.2
```

- 와이어샤크에서 reply의 목적지가 `192.168.31.110`으로 보인 이유: 장운호 R1이 request의 출발지를 이미 `1.1.1.2`에서 `192.168.31.110`으로 PAT했기 때문에, 김용준 R2는 응답 목적지를 `192.168.31.110`으로 설정함.

```text
[3] 김용준 R2가 ICMP reply 생성

src 1.1.2.2
dst 192.168.31.110
```

- 문제 지점: 이 reply가 김용준 R1을 지나갈 때, 김용준 R1도 자기 ACL 99에 의해 출발지 `1.1.2.2`를 PAT 대상으로 봄.

```text
[4] 김용준 R1에서 두 번째 PAT

김용준 R1 기준:
inside f0/1 -> outside f0/0

변환 전:
src 1.1.2.2
dst 192.168.31.110

변환 후:
src 192.168.31.111
dst 192.168.31.110
```

- 장운호 R1이 원래 기대하는 NAT reverse 대상 reply:

```text
src 1.1.2.2
dst 192.168.31.110
```

- 실제로 장운호 R1에 돌아오는 reply는 김용준 R1에서 한 번 더 PAT되어 다음처럼 바뀔 수 있음.

```text
src 192.168.31.111
dst 192.168.31.110
```

- 그래서 장운호 R1은 이 패킷을 기존 `1.1.1.2 -> 1.1.2.2` ICMP NAT 세션의 응답으로 되돌리지 못하고, 목적지가 자기 outside IP `192.168.31.110`인 패킷처럼 처리하여 R2 쪽으로 넘기지 않는 현상이 발생.
- 와이어샤크 관찰과 맞는 현상: reply가 장운호 R1까지 돌아오고 목적지는 `192.168.31.110`인데, 장운호 R1이 R2로 전달하지 않음.
- 결론: `R2에서 보낸 ping이 양쪽 R1 NAT를 타는 구조라서 문제`이며, 정확히는 `장운호 R1에서 request를 PAT하고 김용준 R1에서 reply를 다시 PAT해서 장운호 R1의 NAT reverse 매칭이 깨지는 구조`로 정리.
- 서버 통신까지 고려한 의미: R2 뒤 서버끼리 통신하려면 R2의 static NAT로 서버를 `1.1.x.254`로 공개하되, 팀 간 `1.1.1.0/24`, `1.1.2.0/24`, `1.1.3.0/24` 통신은 R1에서 다시 `192.168.31.x`로 PAT하지 않아야 함.
- 정상 의도 흐름:

```text
장운호 서버 192.168.0.1
-> 장운호 R2 static NAT
-> 1.1.1.254
-> 장운호 R1은 NAT하지 않고 라우팅만
-> 김용준 R1도 NAT하지 않고 라우팅만
-> 김용준 R2 static NAT
-> 김용준 서버 192.168.0.1
```

- 정리: 서버끼리는 `1.1.1.254 <-> 1.1.2.254 <-> 1.1.3.254` 공개 주소로 통신해야 하고, 중간 R1들이 팀 간 대역을 다시 PAT하면 이중 NAT로 세션 추적이 깨질 수 있음.
