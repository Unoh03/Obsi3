## 2. 코드 인젝션 항목의 1차 판단

PDF p.679~692는 코드 인젝션 항목에 해당한다.

처음 판단은 다음과 같았다.

- 기존 CARE 기능에는 직접적인 코드 인젝션 지점이 명확하지 않다.
    
- CARE의 기존 취약 기능은 SQL Injection, XSS, 악성 파일 업로드, 파일 다운로드 등에 더 가깝다.
    
- PHP 웹쉘 업로드는 서버에서 코드 실행으로 이어질 수 있지만, PDF 항목 분류상 코드 인젝션이 아니라 `악성 파일 업로드` 항목에서 다루는 것이 더 정확하다.
    
- 코드 인젝션으로 보려면 사용자 입력값이 `shell_exec`, `system`, `exec`, `eval`, 동적 `include`, XML 파서, XPath 질의, LDAP 질의, 템플릿 엔진 등 실행·해석 문맥에 직접 들어가는 구조가 필요하다.
    

따라서 기존 CARE 기능만 기준으로는 코드 인젝션을 “취약”으로 단정하지 않고, 필요하면 실습용 취약 기능을 별도로 추가하는 방향을 검토했다.

## 3. SQL Injection과 Code Injection의 차이 논의

질문: SQL Injection하고 코드 인젝션이 뭐가 다른가?

정리한 차이는 다음과 같다.

|구분|SQL Injection|Code Injection|
|---|---|---|
|주입 대상|SQL 쿼리|OS 명령어, PHP 코드, LDAP 필터, XPath 질의, XML 파서, 템플릿 엔진 등|
|해석 주체|DBMS|OS shell, PHP runtime, LDAP 서버, XML parser, template engine 등|
|CARE 기존 예시|로그인 SQL, 게시판 검색 SQL, 게시글 조회 SQL|기존 기능에는 직접 지점 없음|
|대표 영향|인증 우회, DB 조회·변조·삭제|서버 명령 실행, 파일 열람, 내부 정보 탈취, 원격 코드 실행|
|핵심 차이|DB에 명령을 주입|서버나 다른 해석 환경에 명령을 주입|

결론은 둘 다 Injection 계열이지만, 입력값을 해석하는 주체가 다르다는 것이다.

- SQL Injection: 입력값이 SQL 문맥에 삽입되어 DBMS가 명령으로 해석한다.
    
- Code Injection: 입력값이 운영체제 명령어, 코드, XML, XPath, LDAP, 템플릿 등 다른 실행 문맥에 삽입되어 해당 해석기가 명령처럼 처리한다.
    

이 비교는 이후 다른 취약점 문서에서도 유용하다. 유사한 취약점들을 구분할 때는 “무엇이 입력값을 해석하는가?”를 기준으로 삼으면 된다.

## 4. 6개 세부 유형을 모두 다룰지에 대한 논의

그러나 조원들의 분위기는 “일단 많이 만들고, 나중에 잘라내면서 정리하자”에 가까웠다. 이에 따라 6개 세부 유형을 모두 문서에 담는 방향으로 수정했다.

- 나중에 보고서 분량이나 완성도 문제로 일부를 잘라낼 수 있도록 각 유형을 독립적으로 작성해야 한다.

- 결론은 필요하다.
    
- 다만 거창한 감상문식 결론은 필요 없다.
    
- 코드 인젝션 6개 유형을 다룬 뒤, 마지막에 각 유형이 어떤 방식으로 실습되었는지 정리하는 판정 요약은 필요하다.
    

종합 판단은 다음 질문에 답하는 역할을 한다.

- LDAP Injection은 실제 구현인가, 모의 구현인가?
    
- OS Command Injection은 실제로 서버 명령 실행까지 확인했는가?
    
- SSI Injection은 Apache 설정까지 포함했는가, 아니면 원리 확인 수준인가?
    
- XPath Injection과 XXE Injection은 어떤 방식으로 재현했는가?
    
- SSTI는 실제 템플릿 엔진 기반인가, 교육용 렌더러 기반인가?
    
- 결국 코드 인젝션 항목은 전체적으로 어떻게 처리되었는가?
    

따라서 `종합 판단`은 남긴다.  
단, 길게 쓰지 않고 표와 짧은 문단으로 처리한다.

## 7. `실습 환경 특이사항` 섹션에 대한 논의

사용자는 `## 8. 실습 환경 특이사항`도 굳이 필요한지 의문을 제기했다.

핵심 문제의식은 다음과 같다.

- 우리는 CARE 자체를 공부하는 것이 아니다.
    
- 목적은 주요정보통신기반시설 웹 취약점 항목을 학습하고 실습하는 것이다.
    
- 그러므로 CARE 앱 중심의 설명이 문서 뒤에 크게 붙으면 보고서 흐름이 흐려질 수 있다.
    

이에 대해 최종 판단은 다음과 같다.

- `## 8. 실습 환경 특이사항`은 별도 대제목으로 두지 않는 쪽이 낫다.
    
- 다만 “이 취약점들이 원래 CARE에 있던 것인지, 실습용으로 추가한 것인지”는 반드시 어딘가에 명시해야 한다.
    
- 그 내용은 `## 5. 실습 구현 범위`와 `## 7. 종합 판단`에 흡수하는 것이 낫다.
    

즉, 실습 환경 특이사항이라는 제목을 별도로 두는 대신 다음 문장을 `## 5. 실습 구현 범위`에 넣는다.

```markdown
CARE 기본 기능에는 코드 인젝션 6개 세부 유형을 모두 직접 확인할 수 있는 기능이 존재하지 않았다. 따라서 본 실습에서는 PDF p.679~692의 코드 인젝션 항목을 학습하고 재현하기 위해 `/vuln/code-injection/` 경로에 실습용 취약 기능을 분리 구현했다. 이 구현은 운영 기능이 아니라 취약점 동작 원리를 확인하기 위한 실습용 기능이다.
```

이렇게 하면 문서 목적은 취약점 실습에 유지하면서도, 보고서의 정직성을 지킬 수 있다.

## 8. `CARE 기존 기능 분석` 제목 수정

사용자는 CARE 자체를 공부하는 것이 아니라 취약점을 공부하는 것이라고 지적했다.  
이에 따라 `## 4. CARE 기존 기능 분석`이라는 제목은 앱 중심으로 보일 수 있다고 판단했다.

최종 제목 후보는 다음과 같았다.

- `## 4. 실습 대상 기능 분석`
    
- `## 4. 기존 기능 적용 가능성 분석`
    

최종적으로는 `## 4. 기존 기능 적용 가능성 분석`이 더 적절하다고 판단했다.

이 제목은 다음 의미를 가진다.

- CARE를 분석하는 것이 목적이 아니다.
    
- 현재 앱 기능으로 이 취약점을 바로 실습할 수 있는지 판단하는 섹션이다.
    
- 기존 기능으로 불가능하면 실습용 기능을 추가할 근거를 제시한다.

## 10. 이번 토론에서 정리된 노트 작성 원칙

이번 토론을 통해 다음 원칙을 세웠다.

### 10.1 보고서 구조는 읽는 흐름을 우선한다

감사용 체크리스트처럼 모든 항목을 잘게 쪼개면 빠진 내용은 줄어들지만, 보고서로 읽히는 흐름은 약해질 수 있다.

특히 하나의 취약점 안에 여러 세부 유형이 들어가는 경우에는 대분류를 과도하게 늘리지 않고, 큰 흐름을 단순하게 유지하는 것이 낫다.

### 10.2 세부 내용은 필요한 위치에 흡수한다

증거, 조치 방안, 판단을 무조건 독립 대제목으로 만들 필요는 없다.

이번 코드 인젝션 문서에서는 각 유형별로 취약점 파악과 조치 방안을 함께 다루는 것이 더 자연스럽다고 판단했다.

즉, 구조는 다음처럼 생각한다.

- 대제목: 문서의 큰 흐름
    
- 소제목: 실제 실습 단위
    
- 세부 내용: 각 실습 단위 안에서 필요한 만큼 배치
    

### 10.3 취약점 분류를 섞지 않는다

PHP 웹쉘 업로드는 서버에서 코드 실행으로 이어질 수 있지만, PDF 항목상 코드 인젝션이 아니라 악성 파일 업로드 항목으로 분리한다.

이 원칙은 이후 문서에도 중요하다.

- 코드 인젝션: 입력값이 실행·해석 문맥에 직접 삽입되는 경우
    
- 악성 파일 업로드: 공격자가 실행 가능한 파일을 업로드하고 실행하는 경우
    
- SQL Injection: 입력값이 SQL 문맥에 삽입되는 경우
    
- XSS: 입력값이 브라우저에서 스크립트로 실행되는 경우
    

결과가 비슷해 보여도 취약점 발생 경로가 다르면 항목을 분리한다.

### 10.4 기존 기능과 실습용 추가 기능을 구분한다

보고서에서 가장 중요한 정직성 기준은 “원래 앱에 있던 취약점인지, 실습을 위해 만든 취약점인지”를 구분하는 것이다.

이번 코드 인젝션의 경우, 기존 CARE 기능에는 6개 세부 유형을 모두 직접 재현할 기능이 없으므로 `/vuln/code-injection/` 경로에 실습용 취약 기능을 추가하는 방향으로 간다.

이때 문서에는 반드시 다음을 명시한다.

- 기존 기능 적용 가능성
    
- 실습용 기능 추가 이유
    
- 실제 재현인지, 모의 재현인지, 조건부 재현인지
    
- 운영 기능이 아니라 교육용 취약 기능이라는 점
    

### 10.5 “결론”은 감상이 아니라 판정 요약이다

사용자는 결론이라는 말에 부담을 느꼈다.  
최종적으로는 `종합 판단`을 감상문처럼 쓰지 않고, 유형별 결과를 정리하는 판정 표로 작성하기로 했다.

좋은 종합 판단은 다음 질문에 답해야 한다.

- 무엇을 실습했는가?
    
- 어떤 방식으로 실습했는가?
    
- 실제 재현과 모의 재현은 무엇인가?
    
- 전체 항목의 학습 결론은 무엇인가?
    
- 다음 취약점으로 넘어가기 전에 남은 애매함은 무엇인가?
    

### 10.6 노트는 나중에 잘라낼 수 있게 만든다

조원들의 방향은 “일단 많이 만들고, 나중에 잘라내며 정리하자”였다.

따라서 처음부터 최종 제출본처럼 압축하기보다, 각 유형을 독립적으로 작성해 나중에 다음 선택이 가능하도록 한다.

- 그대로 유지
    
- 일부 축약
    
- 일부 삭제
    
- 부록으로 이동
    
- 라우터 문서에서는 요약만 유지
    

이를 위해 각 유형별 실습은 가능한 독립적으로 작성한다.

- LDAP Injection은 실제 LDAP 서버 없이 구현하면 모의 재현 성격이 강하다.
    
- SSI Injection은 Apache 설정 영향이 있어 장애 가능성이 크므로 마지막에 다루는 것이 낫다.
