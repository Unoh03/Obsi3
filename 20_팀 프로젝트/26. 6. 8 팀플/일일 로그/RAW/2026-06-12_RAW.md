  - “기존 CARE 서비스의 실제 취약점 발견”으로 쓰지 않는다.
- 문제:
  - `/var/www/html/care`에서 `sudo composer install --no-dev --optimize-autoloader` 실행 시 Composer root/super user 경고가 발생했다.
  - 운영 환경에서는 root Composer 실행을 피해야 하지만, 본 환경은 통제된 실습 서버이고 “SSTI 하기 직전” 스냅샷을 생성했으므로 일회성 진행 가능하다고 판단했다.
  - 더 원칙적인 방식으로는 `sudo -u webuser composer install --no-dev --optimize-autoloader`를 사용한다.
- SSTI 점검 구조:
  - PDF p.687~688 흐름에 맞춰 `취약점 파악`을 Step 1 / Step 2로 정리하기로 했다.
  - Step 1: `{{ 7 * 7 }}` 입력 시 `49`가 출력되는지 확인하여 사용자 입력이 Twig 표현식으로 평가되는지 판단.
  - Step 2: `_context`, `current_user`, `lab_secret` 접근으로 context 접근 가능성을 확인하고, 필요 시 통제된 RCE sink를 통해 서버 측 함수 호출과 OS 명령 실행 가능성을 확인.
- RCE 판단:
  - `lab_secret`은 실습용 고정 proof value이지만, PHP가 특정 문자열을 감지해 가짜 결과를 출력하는 하드코딩은 아니다.
  - `{{ lab_secret }}` 표현식을 Twig가 실제 평가하고 context에서 값을 조회하여 출력하는 구조다.
  - RCE는 기본 Twig 설정만으로 무조건 가능하다고 쓰지 않는다.
  - 취약 모드에 실습용 `lab_exec()` Twig 함수를 의도적으로 노출한 경우, SSTI가 서버 측 함수 호출 및 OS 명령 실행으로 확장될 수 있음을 통제된 방식으로 확인하는 것으로 정리한다.
  - 허용 payload는 `id`, `whoami`, `hostname`, `pwd` 정도로 제한하고 reverse shell, 웹쉘 생성, 파일 삭제, 파일 변조, 권한 상승, 외부 서버 전송은 수행하지 않는다.
- 실습 중 확인:
  - safe 모드에서 `<pre>{{ lab_exec('whoami') }}</pre>` 입력 시 실행되지 않고 문자열 그대로 출력됨을 확인했다.
  - 이 결과는 safe 모드에서 사용자 입력이 템플릿 구조로 해석되지 않고 변수 값으로만 출력된다는 조치 확인 증거로 사용한다.

  - PDF p.693~699의 SQL Injection 점검 흐름 중 우리가 직접 수행한 실습은 특히 Step 2, 즉 인증 페이지에 참이 되는 SQL 조건을 삽입해 인증 우회 여부를 확인하는 흐름에 대응한다.

  - 세션에는 입력 payload가 아니라 DB에서 반환된 실제 row의 `id`를 저장하도록 조정했으며, 이는 SQL Injection으로 반환된 DB row가 실제 로그인 사용자처럼 서버 세션에 반영되는지 확인하기 위한 것이다.
