### 09:42

- 한 일: `웹 취약점 - 01 코드 인젝션.md`의 `6-2. OS Command Injection` 실습 정리를 마무리했다.
- 문제:
  - OS Command Injection 섹션에서 취약 모드와 조치 모드의 코드 차이가 충분히 설명되지 않았다.
  - `취약점 파악` 표의 판단 기준 행이 비어 있었고, 명령 연결 문자 표에서 `|`, `||` 문자가 Markdown 표 문법과 충돌해 깨졌다.
  - 증거 섹션에 브라우저 화면, Apache access log, 관련 코드 설명, Paros 관련 내용이 한 덩어리로 섞여 있었다.
- 조치:
  - `정상 기능과 취약 원리` 흐름을 유지하기로 했다.
  - 취약 모드는 `target` 값이 `shell_exec()` 명령 문자열에 그대로 연결되어 `;id` 같은 추가 명령이 실행되는 구조로 설명한다.
  - safe 모드는 화이트리스트 검증과 `escapeshellarg()` 적용으로 명령 연결 문자를 차단하는 구조로 설명한다.
  - 명령 연결 문자 표는 `|`, `||`가 깨지지 않도록 `&#124;`, `&#124;&#124;` 또는 안전한 코드 표기로 수정하기로 했다.
  - 증거 섹션은 브라우저 화면, Apache access log, 관련 코드 설명으로 나누는 방향이 적절하다고 판단했다.
- 검토:
  - Apache access log는 `/var/log/apache2/access.log`가 아니라 CARE 가상 호스트용 `/var/log/apache2/care_access.log`에 기록됨을 확인했다.
  - `target=...%3Bid` 요청은 실제 공격 요청이며, `%3B`는 세미콜론 `;`로 해석된다.
  - `target=...%253Bid`는 `%`가 다시 인코딩된 이중 인코딩 사례라서 실패 사례로 설명할 수 있다.
  - access log는 공격 요청이 서버에 도달했다는 증거이고, `id` 명령 실행 여부는 브라우저 응답 화면에서 확인하는 증거로 분리하는 것이 맞다.
  - Paros 요청/응답 화면은 필수 증거가 아니라 선택 증거로 본다. 로그의 User-Agent에 `Paros/3.2.13`이 남아 있어 Paros 경유 요청이라는 보조 증거도 확보된다.
- 다음:
  - `6-3. SSI Injection`으로 넘어간다.
  - SSI는 PHP 코드만으로 끝나는 취약점이 아니라 Apache SSI 설정 여부가 핵심이므로, 실제 구현 전에 “조건부 재현”으로 둘지, Apache 설정까지 건드릴지 먼저 판단한다.
