## PART 1 · 프로젝트 개요·위협 시나리오

### 슬라이드 3 — Divider · 01 개요  `⏱ 0:10`

먼저 첫 번째, 프로젝트 개요와 위협 시나리오입니다. 우리가 왜 이 진단을 했는지부터 말씀드리겠습니다.

### 슬라이드 4 — 위협이 현실이 되는 순간  `⏱ 0:50`

보안 진단이 왜 필요한가. 추상적인 이야기가 아니라, 실제로 뚫릴 수 있는 세 가지 경로를 먼저 보여드리겠습니다. 첫째, 평문 DNS 서버 노출입니다. DNS 서버에서 서비스 구성과 내부 호스트 정보가 평문으로 드러나면 공격자에게 내부 지도를 그대로 넘겨주는 셈입니다. 둘째, Zone Transfer 탈취. 'allow-transfer any' 설정 하나 때문에 dig AXFR 명령 한 줄로 내부 호스트와 IP 전 구간이 통째로 복제됩니다. 셋째, root 무차별 대입입니다. root 원격 접속이 허용되어 있으면 Kali에서 hydra로 직접 접속을 시도할 수 있습니다. 이 세 가지가 막연한 가정이 아니라, 잘못된 설정 하나에서 실제로 일어난다는 것 — 그래서 진단이 필요합니다.

### 슬라이드 5 — 진단 4대 원칙  `⏱ 0:40`

진단은 네 가지 원칙 위에서 진행했습니다. 정보보안의 고전적인 3요소인 기밀성, 무결성, 가용성에 더해서, 우리는 네 번째로 '실증성'을 추가했습니다. 단순히 체크리스트에 양호라고 표시하는 게 아니라, 실제로 취약하게 만들어 보고 공격으로 검증한 다음 조치하는 방식입니다. '양호'라는 판정에 근거를 만드는 것, 그게 이번 진단의 차별점입니다.

### 슬라이드 6 — 진단 범위 매트릭스  `⏱ 0:40`

진단 범위를 한눈에 정리한 표입니다. 공통 네 개 카테고리 — Unix, 웹 서비스, DBMS, Web Application은 양 환경에 동일하게 적용했습니다. 여기서 핵심은 진단 깊이가 다르다는 점입니다. Unix는 67개 항목 전수를 심화 진단했고, 나머지는 표준 진단으로 폭을 확보했습니다. 그리고 네트워크 장비는 온프레미스 전용, 클라우드는 AWS 전용입니다. 환경마다 고유한 진단 영역이 따로 있다는 점, 기억해 주십시오.


## PART 2 · 진단 기획 및 방법론

### 슬라이드 7 — Divider · 02 방법론  `⏱ 0:10`

두 번째, 진단 기획과 방법론입니다. 어떻게 진단했는지 전략을 설명드리겠습니다.

### 슬라이드 8 — 왜 Unix만 전수 진단했나  `⏱ 0:50`

먼저 가장 많이 받는 질문에 답하겠습니다 — 왜 Unix만 전수 진단했나? 이건 시간이 부족해서가 아니라 전략적 선택입니다. 깊이 한 곳, 넓이 여러 곳 전략입니다. Unix 67개 항목은 직접 취약하게 만들고, 공격으로 검증하고, 조치하고, 재점검까지 전 과정을 심화했습니다. 나머지는 표준 진단으로 폭을 확보했고요. Unix를 고른 데는 네 가지 근거가 있습니다. 첫째, 모든 서버가 Ubuntu 위에서 돌아가기 때문에 OS 보안이 상위 전 계층으로 전이됩니다. 둘째, 67개 항목이 계정·파일·서비스·패치·로그 보안 전 영역을 포괄하는 표준 커리큘럼입니다. 셋째, 설정 파일 단위라 안전하게 유발하고 복구할 수 있습니다. 넷째, 직접 취약화를 통해 공격자와 방어자 양쪽 시점을 모두 체득할 수 있습니다.

### 슬라이드 9 — 카테고리 → 대상 서버 매핑  `⏱ 0:30`

각 진단 카테고리를 실제로 어느 서버에 적용했는지 매핑한 그림입니다. 왼쪽이 진단 카테고리, 오른쪽이 실제 대상 서버입니다. Unix는 DNS·LOG·MON 서버와 EC2 웹 서버에, DBMS는 온프 DB 서버와 AWS RDS에 적용했습니다. 보시면 공통 네 카테고리는 양쪽에 다 적용되지만, 네트워크 장비는 온프 전용, 클라우드는 AWS 전용으로 갈라집니다. Web과 Web App은 온프에서 도출한 조치를 스크립트로 AWS에 그대로 이식했습니다.


## PART 3 · 보안 아키텍처

### 슬라이드 10 — Divider · 03 아키텍처  `⏱ 0:10`

세 번째, 보안 아키텍처입니다. 두 환경이 실제로 어떻게 구성되어 있는지 보겠습니다.

### 슬라이드 11 — 온프레미스 네트워크 토폴로지 (GNS3)  `⏱ 0:40`

먼저 온프레미스 환경입니다. GNS3로 구성한 네트워크 토폴로지인데요, 중앙의 Core Transit를 기준으로 Branch A, B, C 세 개 망으로 분리했습니다. 라우터 R1, R2가 핵심 진단 대상이고, DNS·LOG·MON·DB·WEB 서버가 각 망에 배치되어 있습니다. 이 환경에서 앞서 본 위협 시나리오들 — Zone Transfer라든지 root 접속 같은 걸 실제로 재현하고 조치했습니다.

### 슬라이드 12 — AWS 아키텍처 — Multi-AZ 이중화  `⏱ 0:50`

AWS는 완전히 다른 그림입니다. 서울 리전 두 개 가용 영역에 걸쳐 7레이어로 격리 설계했습니다. 트래픽 흐름을 따라가 보면, 사용자 요청이 Route 53, CloudFront와 WAF, ALB를 거쳐서 — 여기까지가 글로벌 엣지와 공개 구간입니다 — 그다음 공인 IP가 없는 Private 웹 서버로, 마지막으로 완전히 격리된 RDS 데이터 계층으로 들어갑니다. Public, Private, Isolated 3단 격리죠. 그리고 이 모든 걸 가로지르는 세 축이 있습니다 — IAM 최소권한과 KMS 암호화 같은 보안·접근제어, CloudWatch와 CloudTrail 기반 관측·감사, 그리고 Terraform IaC와 무중단 배포·일본 리전 DR로 이어지는 자동화입니다.

### 슬라이드 13 — 다층 방어 (Defense in Depth)  `⏱ 0:40`

방금 본 구성을 방어 관점에서 다시 정리하면, 다층 방어입니다. 1차로 글로벌 엣지에서 WAF가 SQL 인젝션, XSS, Rate Limit을 걸러내고, 2차로 네트워크 경계에서 Security Group과 NACL이 서브넷·포트 단위로 통제하고, 3차로 데이터 계층을 완전히 격리합니다. 외부에서 데이터까지 들어오려면 세 겹의 문을 다 통과해야 합니다. 여기에 Multi-AZ 고가용성과 IaC 기반 자동화가 받쳐줍니다.


## PART 4 · 진단 수행 (카테고리별 시연)

### 슬라이드 14 — Divider · 04 진단 수행  `⏱ 0:15`

네 번째, 진단 수행입니다. 지금부터는 각 카테고리에서 대표 항목을 골라, 어떻게 취약점을 유발하고, 공격으로 검증하고, 조치했는지를 차례로 보여드리겠습니다. 여섯 개 카테고리 — Unix, DBMS, 네트워크, 웹, Web App, 클라우드 순으로 갑니다. Unix부터 시작합니다.

### 슬라이드 15 — Unix 대표 3종 시연  `⏱ 1:00`

먼저 Unix입니다. 67개 항목 중에서 가장 대표적인 세 가지를, 유발부터 공격 검증, 조치까지 보여드리겠습니다. 첫째, U-01 root 원격 접속입니다. PermitRootLogin이 yes로 열려 있으면 Kali에서 hydra로 무차별 대입이 가능합니다. 조치는 이걸 no로 바꾸는 겁니다. 둘째, U-18 패스워드 파일 보호입니다. /etc/shadow의 해시를 john으로 크랙해 봤고, 조치 후에는 강력한 SHA-512 정책으로 막았습니다. 셋째, U-50 DNS Zone Transfer입니다. 앞서 위협 시나리오에서 봤던 그 항목인데요, dig AXFR로 전송이 되던 걸 'allow-transfer none'으로 차단했습니다. 핵심은, 모든 항목에서 '뚫리는 걸 먼저 보여주고 그다음 막았다'는 겁니다.

### 슬라이드 16 — DBMS 시연 — 계정·암호화·감사  `⏱ 0:45`

이어서 DBMS입니다. 온프레미스 DB 서버와 AWS RDS를 같은 기준으로 진단했는데, 세 가지 빈틈을 보여드리겠습니다. 첫째, 계정과 비밀번호 정책입니다. 불필요한 계정이 남아 있거나 복잡도가 약하면 무차별 대입이나 계정 도용 위험이 있죠. 불필요 계정을 제거하고 복잡도·잠금 정책을 걸었습니다. 둘째, 암호화입니다. 약한 알고리즘이나 평문 하드코딩이 문제였는데 — 특히 AWS RDS는 초기 테라폼에 패스워드가 password123!으로 그대로 박혀 있었습니다. 이걸 SSM SecureString으로 옮기고 KMS 저장 암호화와 TLS를 적용했습니다. 셋째, 감사 기록입니다. 접근·변경·삭제가 기록되지 않으면 침해가 나도 추적이 안 됩니다. 감사 로깅을 켰고요. 온프 9건, AWS 6건 취약을 모두 양호로 전환했습니다.

### 슬라이드 17 — 네트워크 장비 시연 — 평문·기본값  `⏱ 0:45`

네트워크 장비는 온프레미스 전용입니다. GNS3 라우터 R1, R2를 진단했는데, 취약이 26건으로 가장 많았던 영역이죠. 핵심 세 가지를 보겠습니다. 첫째, VTY 프로토콜입니다. Telnet 같은 평문 원격 접속이 열려 있으면 세션을 스니핑해서 관리자 자격증명을 그대로 탈취할 수 있습니다. SSH 전용으로 바꿨습니다. 둘째, SNMP입니다. community가 public이고 ACL이 없으면 snmpwalk 한 번으로 장비 구성 정보가 다 수집됩니다. 복잡한 community와 ACL로 제한했고, v3를 권고했습니다. 셋째, 비밀번호 암호화입니다. enable password가 평문으로 저장돼 있으면 설정 파일만 노출돼도 끝이죠. 암호화를 적용했습니다. 26건 모두 조치했고, 잔여 2건은 구형 IOS가 지원하지 않는 기능이라 장비 교체 시 해소됩니다.

### 슬라이드 18 — 웹 서비스 시연 — 노출·전송·정보유출  `⏱ 0:35`

웹 서비스입니다. 여기부터 두 슬라이드는 환경 독립 영역이라, 온프와 AWS 결과가 동일합니다. 세 가지인데요. 첫째, 디렉터리 리스팅입니다. Indexes 옵션이 켜져 있으면 디렉터리를 훑어서 파일 구조가 노출됩니다. 옵션을 껐고요. 둘째, SSL/TLS입니다. 평문 HTTP나 약한 프로토콜은 전송 구간 스니핑과 중간자 공격에 취약하니 TLS를 강제하고 약한 cipher를 비활성화했습니다. 셋째, 에러 페이지입니다. 기본 에러페이지는 스택이나 서버 버전 정보를 흘리기 때문에 커스텀 페이지로 바꿨습니다. 4건 조치했고, 애플리케이션 계층이라 이 조치가 AWS에 그대로 적용됩니다.

### 슬라이드 19 — Web Application 시연 — 코드 레벨 공격  `⏱ 0:45`

Web Application은 가장 공격적인 영역입니다. Web과 Kali로 직접 공격해서 13건을 도출했는데, 대표 세 가지가 인상적입니다. 첫째, SQL 인젝션입니다. 입력을 검증하지 않으면 쿼리를 조작할 수 있고, sqlmap으로 DB를 통째로 덤프하는 데 성공했습니다. Prepared Statement와 입력 검증으로 막았습니다. 둘째, XSS입니다. 출력을 이스케이프하지 않으면 스크립트를 심어서 세션을 탈취할 수 있죠. 출력 인코딩과 CSP를 적용했습니다. 셋째, 파일 업로드입니다. 확장자나 MIME를 검증하지 않으면 웹쉘을 올려서 원격 코드 실행까지 갑니다. 검증과 실행 차단을 걸었습니다. 이 13건의 조치를 스크립트로 정리해서 AWS에 그대로 이식했습니다 — '환경 독립, 조치 이식'의 실제 사례입니다.

### 슬라이드 20 — 클라우드 시연 — 새로운 위협축  `⏱ 0:45`

남은 개선 과제는, 위협의 무게중심을 그대로 따라갑니다. 위협이 클라우드 거버넌스로 옮겨갔으니 개선도 거기에 집중합니다. 즉시 조치 사항은 노출된 자격증명 폐기, IAM 최소권한화, OIDC 전환, root MFA와 Security Group 정비입니다. 단기로는 IaC를 완성합니다 — 남은 수동 구성을 Terraform으로 코드화하고, 강화 설정을 베이스라인에 역반영하는 거죠. 중기로는 운영을 고도화합니다 — EventBridge 기반 탐지 자동화, Athena 로그 분석, 권한 세분화, Shield Advanced와 DR 강화입니다. 모두 클라우드 거버넌스를 단단히 하는 방향입니다.

### 슬라이드 40 — 마무리 — 동작하는 것과 안전한 것은 다르다  `⏱ 0:40`

정리하겠습니다. 오늘 발표를 관통한 한 문장, '동작하는 것과 안전한 것은 다르다.' 서비스가 동작한다고 보안이 끝난 게 아닙니다. 우리는 동일한 기준으로 두 환경을 진단해서, 조치는 분명히 효과가 있었다는 것 — 축 A, 그리고 위협의 무게중심은 서버 설정에서 클라우드 거버넌스로 이동했다는 것 — 축 B, 이 두 가지를 데이터로 증명했습니다. 보안은 한 번의 조치로 끝나는 게 아니라, 무게중심을 따라 계속 옮겨가는 과정입니다. 이상으로 발표를 마치겠습니다. 감사합니다.
