# 웹 애플리케이션 취약점 점검 보고서

> [!summary]
> 이 문서는 CARE 웹 애플리케이션의 취약점 점검 라우터다. 전체 환경, 점검 기준, 취약점별 상태와 개별 문서 위치만 관리하고, 실제 공격 절차·증거·조치 내용은 취약점별 문서에 분리한다.

## 1. 점검 개요

- 점검 대상: CARE 웹 애플리케이션
- 점검 URL: `http://1.1.1.10/` 또는 LB 경유 URL
- 점검 환경: WEB 서버 Apache/PHP, DB 서버 MariaDB/MySQL, LB, Log/Monitor 서버
- 점검 기준: [[40_자료/주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드.pdf|주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드]] `10. 웹(Web)`
- 기준 범위: PDF p.676 이후 Web Application 점검 항목
- 준비 문서: [[20_팀 프로젝트/26. 6. 8 팀플/준비 과정 보고서|CARE 웹 취약점 실습 준비 과정 보고서]]
- 후보 원장: [[20_팀 프로젝트/26. 6. 8 팀플/취약점 후보표|취약점 후보표]]
- 점검자: 장운호
- 점검 기간: 2026. 06. 10. ~

## 2. 문서 분리 원칙

보고서 전체를 하나의 긴 문서로 작성하지 않고, 다음처럼 분리한다.

```text
웹 애플리케이션 취약점 점검 보고서.md
  -> 전체 개요, 환경, 기준, 요약표, 취약점별 문서 라우팅

웹 취약점 - NN 항목명.md
  -> 항목별 점검 대상, 절차, 재현 결과, 증거, 영향, 조치 방안
```

이렇게 나누는 이유는 다음과 같다.

| 기준 | 이유 |
|---|---|
| 보고서 가독성 | 라우터 문서는 전체 흐름과 결과만 보여준다. |
| 증거 관리 | 스크린샷, 요청/응답, 로그, DB 결과는 취약점별 문서에 모은다. |
| 분류 안정성 | 코드 인젝션, 파일 업로드, 인증·인가처럼 섞이기 쉬운 항목을 분리한다. |
| 진행 관리 | 점검 예정, 재현 완료, 조치 완료 상태를 항목별로 추적한다. |
| 최종 제출 | 제출 직전 필요한 항목만 병합하거나 PDF화할 수 있다. |

> [!important]
> 라우터 문서의 `문서` 칸은 이동 대상이므로 wiki link로 관리한다. 링크가 끊기지 않도록 취약점별 문서 파일을 먼저 만들고, 세부 절차·증거·판정은 각 문서에서 채운다.

### 3.2 배포 구조

```text
D:\care
  -> GitHub: Unoh03/care
  -> WEB 서버 /opt/care-src
```

## 5. 우선 점검 후보

처음부터 21개 문서를 모두 상세 작성하지 않는다. 먼저 실제 시연성이 높고 `care` 기능과 바로 연결되는 항목부터 진행한다.

| 우선순위 | 항목 | 이유 |
|---|---|---|
| P0 | SQL 인젝션 | 로그인/게시판/조회 기능과 직접 연결된다. |
| P0 | XSS | 게시글 입력/출력 구조와 연결된다. |
| P0 | CSRF | 회원정보 수정이나 게시글 작성 기능으로 시연 가능하다. |
| P0 | 악성 파일 업로드 | 게시판 첨부파일 기능과 연결하기 쉽다. |
| P0 | 데이터 평문 전송 | 현재 HTTP 기반이면 네트워크 캡처로 증거 확보가 쉽다. |
| P1 | 불충분한 권한 검증 | 사용자 A/B 또는 관리자 기능 시나리오가 필요하다. |
| P1 | 파일 다운로드 | 별도 다운로드 기능 또는 경로 조작 기능이 필요하다. |
| P1 | 디렉터리 인덱싱 | Apache 설정 전후 비교로 증거를 만들 수 있다. |

## 7. 증거 수집 기준

| 증거 | 목적 |
|---|---|
| 화면 캡처 | 사용자가 보는 취약 동작 확인 |
| 요청/응답 캡처 | 파라미터 변조, payload, 서버 응답 확인 |
| 서버 로그 | Apache access/error log, PHP 오류, 인증 시도 확인 |
| DB 결과 | 데이터 변조, 조회 성공, 권한 우회 결과 확인 |
| 코드 캡처 또는 diff | 취약 원인과 조치 방안 연결 |
| 조치 후 재검증 | 취약 상태와 양호 상태 비교 |

## 8. 최종 종합 의견 자리

최종 종합 의견은 모든 항목 점검이 끝난 뒤 작성한다.

작성할 내용:

- 전체 점검 항목 수
- 취약 / 양호 / 해당 없음 / 보류 개수
- 가장 위험한 취약점
- 실제 침해 시나리오로 연결 가능한 취약점 조합
- 조치 우선순위
- 실습 환경이라 의도적으로 취약하게 둔 부분
- 운영 환경에 적용해야 할 보안 기준
