# 웹 취약점 - 18 쿠키 변조

source: [[40_자료/주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드.pdf|주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드]], p.769-774.

## 1. 개요

**쿠키 변조**는 브라우저에 저장된 쿠키 값을 사용자가 직접 바꾸었을 때, 서버가 그 값을 그대로 신뢰해 다른 사용자나 높은 권한으로 처리되는지 확인하는 항목이다.

PDF p.769의 판단 기준은 다음처럼 정리할 수 있다.

| 판단 | 기준 |
|---|---|
| 양호 | 쿠키를 사용하지 않고 서버 사이드 세션을 사용하거나, 쿠키를 사용하는 경우 안전한 알고리즘이 적용되어 있음 |
| 취약 | 안전한 알고리즘이 적용되지 않은 쿠키를 사용하거나, 쿠키로만 인증 및 권한 부여를 적용함 |

핵심은 단순히 “쿠키가 있다”가 아니다. 브라우저가 들고 있는 쿠키 안에 `id=user1`, `role=user`, `isAdmin=false` 같은 사용자 식별값이나 권한값이 들어 있고, 그 값을 바꾸면 서버의 판단이 바뀌는지가 핵심이다.

```text
취약한 쿠키 기반 권한 구조 예시

Cookie: id=user1; role=user
-> 사용자가 role=admin으로 변조
-> 서버가 변조된 role 값을 그대로 신뢰
-> 관리자 권한으로 처리
```

CARE는 이런 커스텀 인증 쿠키를 사용하지 않고, PHP 서버 사이드 세션을 사용한다. 따라서 18번은 실제 침투 재현형이라기보다, **쿠키 기반 권한 구조가 없음을 확인하는 B 유형 항목**으로 본다.

## 2. CARE 적용 가능성 분석

CARE의 로그인 상태와 사용자 식별은 클라이언트 쿠키의 `id`, `role` 값을 신뢰하는 방식이 아니라 서버의 `$_SESSION` 값을 기준으로 처리된다.

확인한 주요 지점은 다음과 같다.

| 파일 | 확인 내용 | 18번 관련 판단 |
|---|---|---|
| `member/loginModel.php` | 로그인 성공 후 `$_SESSION['id']`, `$_SESSION['name']` 등에 사용자 정보 저장 | 서버 사이드 세션 사용 |
| `header.php` | `$_SESSION['id']` 존재 여부로 로그인 상태 표시 | 쿠키의 사용자 ID를 직접 신뢰하지 않음 |
| `member/modifyModel.php` | 회원정보 수정 대상 ID를 `$_SESSION['id']`에서 가져옴 | 요청 쿠키나 POST의 임의 ID가 아니라 세션 사용자 기준 |
| 전체 소스 | `$_COOKIE` 기반 인증/권한 판단 없음 | PDF 18번의 핵심 취약 구조가 확인되지 않음 |
| `member/logout.php` | `setcookie(session_name(), ...)`는 세션 쿠키 삭제 용도 | 커스텀 인증 쿠키 발급 아님 |

CARE에서 실제로 브라우저가 들고 있는 쿠키는 주로 `PHPSESSID`다. `PHPSESSID`는 서버 세션 저장소를 가리키는 식별자일 뿐, 그 안에 사용자 ID나 권한값이 직접 들어 있는 것은 아니다.

```text
PHPSESSID=abc123
-> 서버가 abc123에 해당하는 서버 세션을 조회
-> 서버 세션 안의 $_SESSION['id'] 값을 사용
```

따라서 `PHPSESSID` 탈취와 재사용은 중요한 문제지만, 이것은 쿠키 안의 권한값을 바꾸는 **쿠키 변조**라기보다 **세션 토큰 탈취와 세션 하이재킹** 문제로 분리하는 편이 정확하다.

## 3. 문제가 되는 부분과 되지 않는 부분

18번에서 구분해야 할 부분은 다음과 같다.

| 구분 | CARE 판단 |
|---|---|
| 쿠키에 사용자 ID나 권한값을 저장하는가 | 확인되지 않음 |
| `$_COOKIE` 값을 기준으로 인증하거나 권한을 부여하는가 | 확인되지 않음 |
| `PHPSESSID`가 노출될 경우 세션 탈취가 가능한가 | 가능할 수 있음. 단, 18번이 아니라 06번 XSS와 16번 세션 관리 범위 |
| `PHPSESSID`에 `HttpOnly`, `Secure`, `SameSite`가 명시 적용되어 있는가 | 별도 확인 필요. 16번 세션 관리 및 17번 HTTPS와 인접 |

즉, CARE는 PDF 18번의 본류인 “쿠키값 변조로 사용자나 권한 변경” 구조는 보이지 않는다. 다만 세션 쿠키 자체가 탈취되면 서버 사이드 세션 방식이어도 세션 재사용 공격은 가능할 수 있다.

## 4. 경계 사례: XSS를 이용한 Session Token 탈취

[[10_학습 노트/시스템보안/웹보안/XSS를 이용한 Session Token 탈취 실습|XSS를 이용한 Session Token 탈취 실습]]에서는 Stored XSS로 Victim 브라우저의 `PHPSESSID`를 Kali로 전송하고, Burp에서 해당 값을 요청 Cookie에 주입해 로그인 상태를 재사용했다.

이 사례는 쿠키를 공격에 이용했다는 점에서는 18번과 가까워 보인다. 하지만 본질은 다르다.

| 항목 | 본질 |
|---|---|
| XSS Session Token 탈취 | XSS로 `document.cookie`를 읽어 세션 ID를 훔치고 재사용 |
| 18번 쿠키 변조 | 쿠키 안의 사용자/권한 값을 사용자가 직접 바꾸어 권한 상승 |
| 16번 세션 관리 | 세션 ID가 고정·탈취·재사용에 안전한지 점검 |

따라서 XSS 기반 `PHPSESSID` 탈취는 18번의 직접 취약 증거가 아니라, 18번에서 제외해야 할 인접 사례로 둔다.

```text
XSS 세션 탈취
= PHPSESSID라는 세션 키를 훔쳐서 재사용

쿠키 변조
= 쿠키 안의 id, role, 권한값을 바꿔 서버 판단을 바꿈
```

> [!important] 헷갈리면 여기부터 본다
> 서버 사이드 세션은 `role=admin` 같은 쿠키값 직접 변조를 막는 구조다. 하지만 `PHPSESSID` 자체가 털리면 공격자가 그 세션 키를 재사용할 수 있으므로, 세션 탈취 문제까지 자동으로 사라지는 것은 아니다.

## 5. 점검 방법

PDF p.769-770의 점검 흐름은 두 단계다.

| PDF 단계 | CARE에서의 처리 |
|---|---|
| Step 1. 일반 사용자로 로그인 후 쿠키에 ID, 권한 구분자 등 중요 정보가 노출되는지 확인 | 브라우저 개발자도구 또는 프록시에서 쿠키 확인. `PHPSESSID` 외 커스텀 인증/권한 쿠키가 있는지 본다. |
| Step 2. 쿠키 내 중요 정보를 변조하여 다른 사용자나 권한으로 이용 가능한지 확인 | 변조할 `id`, `role`, `auth` 같은 쿠키가 없으면 직접 재현 대상이 없다. |

실제 확인 기준은 다음과 같다.

```text
확인 대상 쿠키 예시
- PHPSESSID만 존재하는가?
- id, user, role, auth, admin 같은 쿠키가 있는가?
- 쿠키 값을 바꿨을 때 서버가 그 값을 권한 판단에 쓰는가?
```

소스 기준 확인도 함께 한다.

```bash
rg -n -F '$_COOKIE' D:\care
rg -n -F 'setcookie(' D:\care
rg -n -F '$_SESSION' D:\care\member D:\care\center D:\care\header.php
```

현재 확인 결과, 인증/권한 판단에 `$_COOKIE`를 사용하는 구조는 보이지 않고, 사용자 상태는 `$_SESSION` 중심으로 처리된다.

## 6. 조치 방안

PDF p.770-774의 조치 방향을 CARE 기준으로 정리하면 다음과 같다.

| PDF 조치 | CARE 적용 판단 |
|---|---|
| PDF 조치 1. 쿠키 대신 서버 사이드 세션 방식 사용 | 초기 CARE부터 이미 적용된 구조 |
| PDF 조치 2. 쿠키에 중요 정보나 인증을 구현해야 할 경우 안전한 알고리즘 적용 | 적용 대상 없음. 중요 인증/권한 정보를 담은 커스텀 쿠키가 없음 |
| PDF 조치 3-1. 쿠키 서명 HMAC으로 변조 여부 검증 | 적용 대상 없음. 변조 검증할 커스텀 인증 쿠키가 없음 |
| PDF 조치 3-2. `HttpOnly`, `Secure`, `SameSite` 속성 적용 | PHPSESSID 속성 강화는 보강 가능. 단, 16번 세션 관리와 17번 HTTPS 범위와 겹침 |

### PDF 조치 1. 서버 사이드 세션 사용

CARE는 로그인 성공 후 사용자 정보를 쿠키에 직접 담지 않고 서버 세션에 저장한다.

`member/loginModel.php`:

```php
session_start();

if($num == 1){
    $_SESSION['id'] = $id;
    $row = mysqli_fetch_assoc($result);
    $_SESSION['name'] = $row['name'];
    $_SESSION['mobile'] = $row['mobile'];
    $_SESSION['address'] = $row['address'];
    $_SESSION['email'] = $row['email'];
    $_SESSION['num'] = $row['num'];
}
```

로그인 상태 표시도 쿠키의 `id`나 `role`이 아니라 서버 세션의 `id`를 기준으로 한다.

`header.php`:

```php
if(empty($_SESSION['id'])){
    // Login / Register 표시
}else{
    // Logout / Modify 표시
}
```

회원정보 수정 대상도 클라이언트 쿠키가 아니라 서버 세션에서 가져온다.

`member/modifyModel.php`:

```php
session_start();
if($_SESSION['id'] == ""){
    echo "<script>location.href='login.php';</script>";
    exit;
}
$id = $_SESSION['id'];
```

이 구조는 PDF 18번 조치 1의 “쿠키 대신 서버 사이드 세션 방식 사용”에 해당한다.

### PDF 조치 2. 쿠키 중요 정보 암호화

PDF 조치 2는 쿠키에 인증 정보나 권한 정보를 넣어야 할 때 안전한 암호화 알고리즘을 적용하라는 내용이다.

하지만 CARE에는 다음과 같은 커스텀 인증 쿠키가 보이지 않는다.

```text
id=user1
role=user
isAdmin=false
auth=...
```

따라서 이 조치는 “미조치”라기보다 **적용 대상 없음**으로 보는 편이 정확하다. 암호화할 인증/권한 쿠키 자체가 없기 때문이다.

### PDF 조치 3-1. HMAC 변조 검증

PDF 조치 3의 HMAC은 쿠키 payload가 변조되었는지 확인하기 위한 무결성 검증이다.

```text
쿠키에 role=user 같은 권한값을 넣음
-> HMAC을 함께 붙임
-> 사용자가 role=admin으로 바꾸면 HMAC 검증 실패
```

CARE는 인증/권한 정보를 담은 커스텀 쿠키가 없으므로, HMAC으로 검증할 대상도 없다. 따라서 이 부분도 적용 대상 없음으로 처리한다.

### PDF 조치 3-2. 쿠키 속성 강화

`HttpOnly`, `Secure`, `SameSite`는 쿠키 변조 자체보다는 세션 쿠키 탈취와 교차 사이트 요청 위험을 줄이는 보강 조치다.

| 속성 | 의미 | CARE 판단 |
|---|---|---|
| `HttpOnly` | JavaScript에서 쿠키를 읽기 어렵게 함 | XSS 기반 `document.cookie` 탈취 완화 |
| `Secure` | HTTPS에서만 쿠키 전송 | HTTP 환경에서는 적용 제한 |
| `SameSite` | 교차 사이트 요청에서 쿠키 전송 범위 제한 | CSRF 완화에 도움 |

이 속성들은 중요하지만, 18번의 본류인 “쿠키값 변조로 권한 상승”과는 약간 다르다. 이 노트에서는 보강 가능 항목으로만 기록하고, 세션 쿠키 속성 자체는 16번 세션 관리와 17번 HTTPS 쪽에서 함께 다루는 편이 맞다.

## 7. 조치 후 점검

18번의 조치 후 점검은 “새 방어코드를 넣고 막히는지 확인”보다 “CARE가 애초에 위험한 쿠키 구조를 사용하지 않는지 확인”에 가깝다.

| 점검 | 기대 결과 |
|---|---|
| 로그인 후 브라우저 쿠키 확인 | `PHPSESSID` 외 사용자 ID/권한 쿠키가 없음 |
| 소스에서 `$_COOKIE` 검색 | 인증/권한 판단에 사용하는 코드 없음 |
| 소스에서 `$_SESSION` 검색 | 사용자 식별과 로그인 상태가 서버 세션 기반으로 처리됨 |
| 임의 쿠키 추가 실험 | `id=admin`, `role=admin` 등을 추가해도 권한 변화 없음 |

이번 노트에서는 임의 쿠키 추가 실험은 수행하지 않는다. 코드 구조상 변조할 커스텀 인증 쿠키가 없고, 소스 근거만으로 B 판정이 가능하기 때문이다.

## 8. 증거

### 서버 사이드 세션 사용 근거

`member/loginModel.php`는 로그인 성공 후 사용자 정보를 `$_SESSION`에 저장한다.

```php
$_SESSION['id'] = $id;
$_SESSION['name'] = $row['name'];
$_SESSION['mobile'] = $row['mobile'];
$_SESSION['address'] = $row['address'];
$_SESSION['email'] = $row['email'];
$_SESSION['num'] = $row['num'];
```

`header.php`는 로그인 상태를 `$_SESSION['id']` 기준으로 판단한다.

```php
if(empty($_SESSION['id'])){
    // 비로그인 상태
}else{
    // 로그인 상태
}
```

`member/modifyModel.php`는 수정 대상 ID를 클라이언트 입력이나 쿠키가 아니라 세션에서 가져온다.

```php
$id = $_SESSION['id'];
```

### 쿠키 기반 권한 구조 부재

소스 검색 기준으로 `$_COOKIE`를 사용해 인증이나 권한을 판단하는 흐름은 확인되지 않았다.

`setcookie()`는 `member/logout.php`에서 PHP 세션 쿠키를 삭제할 때만 확인된다.

```php
setcookie(session_name(), '', time() - 42000,
    $params["path"],
    $params["domain"],
    $params["secure"],
    $params["httponly"]
);
```

이 코드는 커스텀 인증 쿠키를 발급하는 코드가 아니라, 로그아웃 시 기존 세션 쿠키를 만료시키는 코드다.

### 인접 사례: 세션 쿠키 탈취

[[10_학습 노트/시스템보안/웹보안/XSS를 이용한 Session Token 탈취 실습|XSS를 이용한 Session Token 탈취 실습]]에서는 `PHPSESSID`가 탈취되면 세션 재사용이 가능할 수 있음을 확인했다.

하지만 이 증거는 18번 쿠키 변조의 직접 증거가 아니다. `PHPSESSID`를 훔쳐 재사용하는 것은 세션 토큰 탈취이며, 쿠키 안의 `role` 값을 바꾸는 쿠키 변조와 다르다.

따라서 이 사례는 18번의 취약 증거가 아니라, 18번과 06번/16번을 구분하기 위한 경계 사례로 사용한다.

| 판정 상태 | 확정 |
| 조치 상태 | PDF 조치 1은 초기 CARE 구조에서 이미 충족. PDF 조치 2와 3-1은 적용 대상 없음. PDF 조치 3-2는 세션 쿠키 속성 보강 항목으로 분리 |
| 증거 상태 | 소스 근거 확보. 임의 쿠키 추가 실험은 생략 |

CARE는 인증과 권한 판단을 `id=admin`, `role=admin` 같은 클라이언트 쿠키에 맡기지 않고, 서버 사이드 `$_SESSION`을 사용한다. 따라서 PDF 18번의 핵심 취약 구조인 “쿠키 값 변조를 통한 타 사용자 권한 탈취”는 현재 CARE 원본 구조에서 확인되지 않는다.

조치 방법 중 **PDF 조치 1. 쿠키 대신 서버 사이드 세션 사용**은 초기 CARE부터 이미 적용되어 있다. **PDF 조치 2. 쿠키 중요 정보 암호화**와 **PDF 조치 3-1. HMAC 변조 검증**은 중요 인증/권한 쿠키가 없으므로 적용 대상이 없다.

다만 `PHPSESSID` 탈취와 재사용은 별개의 보안 문제다. 이것은 쿠키 변조가 아니라 XSS 기반 세션 토큰 탈취와 세션 관리 문제이므로 06번 XSS 및 16번 세션 관리 범위에서 다루는 것이 더 정확하다.

최종적으로 18번은 **CARE가 쿠키 기반 권한 구조를 사용하지 않음을 확인하는 B 유형 항목**으로 본다.
