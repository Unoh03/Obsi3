# 웹 취약점 - 14 악성 파일 업로드

source: [[40_자료/주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드.pdf|주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드]], p.744-752.

## 1. 개요

**악성 파일 업로드**는 게시판, 자료실, 프로필 이미지, 첨부파일 기능에서 사용자가 업로드한 파일을 서버가 충분히 검증하지 않아 서버 사이드 스크립트, 악성 파일, 비정상 파일이 저장되거나 실행될 수 있는 취약점이다.

핵심은 단순히 “파일 업로드 기능이 있다”가 아니다. 다음 조건이 결합될 때 위험해진다.

| 확인 지점 | 의미 |
|---|---|
| 확장자 검증 | `.php`, `.phtml`, `.phar` 등 서버 실행 가능 확장자를 차단하는가 |
| MIME Type 검증 | 사용자가 조작한 확장자만 믿지 않고 실제 파일 형식을 확인하는가 |
| 파일명 처리 | 원본 파일명, 경로 구분자, 널바이트, 특수문자를 그대로 신뢰하지 않는가 |
| 저장 위치 | 웹에서 직접 접근 가능한 위치에 저장되는가 |
| 실행 제한 | 업로드 디렉터리에서 PHP 등 서버 사이드 스크립트 실행이 차단되는가 |
| 크기 제한 | 과도한 대용량 업로드를 제한하는가 |

CARE는 게시글 작성 기능에 파일 업로드 입력칸이 있고, 업로드 파일을 `../data/` 아래에 저장한다. 이 기능은 CARE 원본 기능이므로 14번은 실습용 새 기능을 만들 필요가 없는 **A-2 기존 기능 악용형**으로 판단한다.

## 2. 문제가 되는 부분

CARE의 게시글 작성 화면은 파일 업로드 input을 제공한다.

```php
<form action="writeModel.php" method="post" enctype="multipart/form-data">
...
<input type="file" name="upfile">
```

서버 처리 파일인 `center/writeModel.php`는 업로드된 원본 파일명과 임시 파일 경로를 그대로 받는다.

```php
$upfile = $_FILES['upfile']['name'];
$tmp_file = $_FILES['upfile']['tmp_name'];
```

그 후 게시글 정보를 DB에 저장하면서 파일명도 함께 저장한다.

```php
$query = "INSERT INTO center(id, subject, content, date, hit, filename) ";
$query = $query . "VALUES('$id','$subject', '$content', '$date', 0, '$upfile')";
```

마지막으로 업로드 파일이 있으면 원본 파일명을 붙여 `../data/`에 저장한다.

```php
if(is_uploaded_file($tmp_file)){
    $destination = "../data/" . $upfile;
    move_uploaded_file($tmp_file, $destination);
}
```

현재 원본 흐름에서 확인되는 문제는 다음과 같다.

| 문제 | 설명 |
|---|---|
| 확장자 검증 없음 | 서버 실행 가능 확장자나 이중 확장자 차단 로직이 없음 |
| MIME Type 검증 없음 | 파일 내용 기반 형식 검사가 없음 |
| 파일 크기 제한 없음 | 대용량 업로드 제한이 없음 |
| 원본 파일명 사용 | 사용자가 보낸 파일명을 그대로 저장 파일명으로 사용함 |
| 저장 경로 직접 결합 | `../data/`와 원본 파일명을 문자열로 연결함 |
| 실행 차단 설정 부재 | 조치 전 `/data/`에서 PHP proof 파일이 실행됨 |

따라서 소스상으로는 업로드 검증 부재가 명확하다. 이번 실습에서는 실제로 업로드한 PHP 계열 파일이 `/data/` 경로에서 실행되는 것도 확인했다.

## 3. 악용 흐름

이 항목의 일반적인 악용 흐름은 다음과 같다.

```text
로그인
-> 게시글 작성 화면 접근
-> 서버에서 실행 가능한 스크립트 파일 또는 위장 파일 업로드
-> 서버가 확장자, MIME, 파일명, 크기, 저장 위치를 검증하지 않고 저장
-> 업로드 경로로 접근
-> 서버 설정에 따라 스크립트 실행 또는 파일 노출 발생
```

CARE에서 직접 확인해야 할 핵심은 다음 두 가지다.

| 구분 | 확인 내용 |
|---|---|
| 업로드 검증 부재 | PHP 계열 확장자나 비허용 형식이 업로드 단계에서 차단되는가 |
| 실행 가능 여부 | `/data/업로드파일`에 접근했을 때 서버 사이드 스크립트가 실행되는가, 다운로드/원문 출력/차단되는가 |

즉, 이 취약점은 두 단계로 나눠 판단한다.

```text
1. 파일 업로드 검증 부재 = 소스상 확인 가능
2. 웹쉘 실행 가능성 = 서버 실행 검증으로 확인됨
```

## 4. 점검 방법

PDF의 공격 절차는 크게 두 단계다. 이 문서에서는 PDF의 부모 번호를 유지하고, CARE에서 필요한 세부 확인은 `1-1`, `1-2`처럼 하위 번호로 적는다.

### PDF Step 1. 악성 스크립트 업로드 가능 여부 확인

| 하위 점검 | CARE에서의 확인 |
|---|---|
| 1-1 | `center/write.php`에 파일 업로드 input이 있는지 확인 |
| 1-2 | `center/writeModel.php`가 업로드 파일의 확장자, MIME Type, 크기, 파일명을 검증하는지 확인 |
| 1-3 | 업로드 파일이 어느 경로에 어떤 이름으로 저장되는지 확인 |
| 1-4 | PHP 계열 파일이나 위장 파일 업로드가 차단되는지 확인 |

확인 대상은 다음이다.

```text
/center/write.php
/center/writeModel.php
/data/
```

현재 코드 기준으로는 `write.php`가 파일 업로드 입력칸을 제공하고, `writeModel.php`가 원본 파일명을 그대로 `../data/`에 저장한다. 실제 화면에서도 PHP proof 파일 업로드가 성공했으므로 PDF Step 1은 취약 상태로 확인했다.

### PDF Step 2. 업로드한 악성 스크립트 요청 및 실행 여부 확인

| 하위 점검 | CARE에서의 확인 |
|---|---|
| 2-1 | 업로드된 파일의 웹 접근 경로를 확인 |
| 2-2 | `/data/파일명` 접근 시 실행, 원문 출력, 다운로드, 차단 중 어떤 결과가 나오는지 확인 |

실행 검증에서 결과는 다음처럼 해석한다.

| 결과 | 판단 |
|---|---|
| PHP 계열 파일 업로드 자체가 차단됨 | 업로드 검증 조치가 적용된 상태 |
| 업로드는 되지만 실행되지 않고 다운로드/원문 출력됨 | 업로드 검증은 미흡하나 실행 차단은 일부 존재 |
| 업로드 후 웹 접근 시 서버 사이드 코드가 실행됨 | 악성 파일 업로드 취약점 영향이 큼 |
| `/data/` 접근이 차단됨 | 저장 검증 미흡과 별개로 웹 접근 차단 방어 존재 |

## 5. 조치 방안

PDF 조치 방안은 6가지로 정리된다. 이번 CARE 프로젝트에서는 그중 **1~4번만 실제 조치 코드 후보로 반영**하고, 5번은 보류, 6번은 기술력·운영 범위 부족으로 포기한다.

| 번호 | 조치 방안 | 이번 프로젝트 처리 |
|---:|---|---|
| 1 | 파일명 정규화/필터링 | 주석 방어 코드로 반영 |
| 2 | 확장자, MIME Type, 파일 크기 화이트리스트 검증 | 주석 방어 코드로 반영 |
| 3 | 저장 파일명 난수화 | 주석 방어 코드로 반영 |
| 4 | 업로드 디렉터리 실행 권한 제한 | Apache 설정 예시를 주석으로 반영 |
| 5 | 업로드 경로를 웹 디렉터리와 격리 | 보류. 구조 변경과 다운로드 흐름 재설계가 필요함 |
| 6 | 주기적 바이러스 검사 | 포기. ClamAV 등 백신, 업데이트, 스케줄러, 격리 정책이 필요해 현재 기술력과 실습 범위를 초과함 |

`center/writeModel.php`에는 1~4번 조치 후보가 주석으로 들어가 있다. 현재는 취약 상태 캡처를 위해 비활성 상태다.

반영 커밋:

```text
59c86a2c6860a6ef2b98ac85fbd1c079f834b782
```

### PDF 조치 1. 파일명 정규화와 위험 문자 제거

파일명은 널바이트 제거, `basename()` 처리, 허용 문자 정규식으로 정규화한다.

```php
// PDF 조치 1: 파일명 정규화와 위험 문자 제거
$originalName = $_FILES['upfile']['name'];
$normalizedName = basename(str_replace("\0", '', $originalName));
$normalizedName = preg_replace('/[^A-Za-z0-9._-]/', '_', $normalizedName);
$extension = strtolower(pathinfo($normalizedName, PATHINFO_EXTENSION));
```

### PDF 조치 2. 확장자, MIME Type, 파일 크기 화이트리스트 검증

허용 확장자, MIME Type, 최대 크기를 먼저 정의한다.

```php
// PDF 조치 2: 확장자, MIME Type, 파일 크기 화이트리스트 검증
$allowedExtensions = array('jpg', 'jpeg', 'png', 'gif', 'pdf');
$allowedMimeTypes = array(
    'image/jpeg',
    'image/png',
    'image/gif',
    'application/pdf'
);
$maxUploadSize = 2 * 1024 * 1024;
```

파일 크기는 2MB 이하로 제한한다.

```php
// PDF 조치 2: 파일 크기 제한
if($_FILES['upfile']['size'] > $maxUploadSize){
    echo "<script>alert('파일 크기는 2MB 이하만 허용됩니다.'); history.go(-1); </script>";
    exit;
}
```

확장자는 화이트리스트로 제한한다.

```php
// PDF 조치 2: 확장자 화이트리스트 검증
if(! in_array($extension, $allowedExtensions, true)){
    echo "<script>alert('허용되지 않는 파일 확장자입니다.'); history.go(-1); </script>";
    exit;
}
```

MIME Type은 `finfo(FILEINFO_MIME_TYPE)`로 확인한다.

```php
// PDF 조치 2: MIME Type 검증
$finfo = new finfo(FILEINFO_MIME_TYPE);
$mimeType = $finfo->file($tmp_file);

if(! in_array($mimeType, $allowedMimeTypes, true)){
    echo "<script>alert('허용되지 않는 파일 형식입니다.'); history.go(-1); </script>";
    exit;
}
```

### PDF 조치 3. 서버 저장 파일명 난수화

저장 파일명은 원본 파일명이 아니라 난수 기반 이름을 사용한다.

```php
// PDF 조치 3: 서버 저장 파일명 난수화
$upfile = bin2hex(random_bytes(16)) . '.' . $extension;
```

저장 경로는 `realpath()`로 업로드 디렉터리를 고정한 뒤 `DIRECTORY_SEPARATOR`로 결합한다. 이 부분은 파일명이 경로 조작으로 바뀌지 않도록 1번 조치와 함께 동작한다.

```php
// PDF 조치 1/3: 업로드 디렉터리 고정 후 난수 파일명으로 저장
$uploadDir = realpath(__DIR__ . '/../data');
$destination = $uploadDir . DIRECTORY_SEPARATOR . $upfile;
```

### PDF 조치 4. 업로드 디렉터리 실행 권한 제한

4번 조치는 PHP 코드만으로 완결되지 않는다. 업로드 디렉터리에서 서버 사이드 스크립트가 실행되지 않도록 Apache/PHP 설정에서 제한해야 한다.

이번 커밋에는 `/data/.htaccess` 또는 Apache VirtualHost/Directory 설정에 넣을 수 있는 예시를 `center/writeModel.php` 하단 주석으로 추가했다.
```apache
# PDF 조치 4: 업로드 디렉터리에서 PHP 계열 스크립트 실행 차단
<FilesMatch "\.(php|phtml|phar|php[0-9]?)$">
    Require all denied
</FilesMatch>
RemoveHandler .php .phtml .phar .php3 .php4 .php5 .php7 .php8
RemoveType .php .phtml .phar .php3 .php4 .php5 .php7 .php8
php_flag engine off
```
```bash
sudo tee /var/www/html/care/data/.htaccess >/dev/null <<'EOF'
<FilesMatch "\.(php|phtml|phar|php[0-9]?)$">
    Require all denied
</FilesMatch>
RemoveHandler .php .phtml .phar .php3 .php4 .php5 .php7 .php8
RemoveType .php .phtml .phar .php3 .php4 .php5 .php7 .php8
php_flag engine off
EOF

sudo apachectl configtest
sudo systemctl reload apache2
```
주의할 점은 이 코드가 Apache 설정 예시라는 것이다. 실제 적용 위치는 서버 구성에 따라 달라질 수 있다.

| 방식 | 설명 |
|---|---|
| `.htaccess` | Apache에서 AllowOverride가 허용되어야 동작함 |
| VirtualHost/Directory 설정 | 서버 설정 파일에 직접 넣는 방식. 더 안정적이나 서버 권한 필요 |
| PHP-FPM/nginx 구성 | 위 Apache 지시어가 그대로 적용되지 않을 수 있어 별도 location 설정 필요 |

### PDF 조치 5. 업로드 경로를 웹 디렉터리와 격리

업로드 경로를 웹 디렉터리와 격리하는 방식은 보안상 좋지만, 현재 프로젝트에서는 보류한다.

예상 변경은 다음과 같다.

```text
현재: /var/www/html/care/data
변경 후보: /var/lib/care/uploads
```

이렇게 바꾸면 사용자가 업로드 파일에 직접 접근하지 못하므로, 파일 제공은 `download.php` 같은 서버 로직을 통해서만 해야 한다. 즉 파일 업로드뿐 아니라 다운로드 구조, DB 저장값, 권한 확인 흐름까지 같이 바뀐다.

따라서 5번은 불가능한 조치는 아니지만 이번 14번 단독 조치 범위를 넘어선다고 보고 보류한다.

### PDF 조치 6. 주기적 바이러스 검사

주기적 바이러스 검사는 현재 프로젝트 범위에서 포기한다.

필요한 구성은 다음과 같다.

| 필요 요소 | 설명 |
|---|---|
| 백신 엔진 | 예: ClamAV |
| 시그니처 업데이트 | freshclam 등 업데이트 체계 |
| 검사 시점 | 업로드 직후 검사 또는 cron 기반 주기 검사 |
| 격리 정책 | 탐지 파일 삭제, 격리, 관리자 알림 등 |
| 로그 관리 | 검사 결과와 처리 결과 보관 |

이 조치는 웹 애플리케이션 코드 수정이라기보다 운영 보안 체계에 가깝다. 현재 팀 프로젝트의 시간, 기술력, 실습 범위를 고려해 실제 구현 대상에서 제외한다.

## 6. 조치 후 점검

PDF 조치 1~4번을 실제로 활성화한 뒤에는 다음을 확인한다.

| 점검 | 기대 결과 |
|---|---|
| PDF 조치 1 | 파일명에 경로 구분자, 널바이트, 특수문자를 넣어도 안전한 이름으로 정규화 |
| PDF 조치 2 | PHP 계열 확장자, MIME 위조 파일, 2MB 초과 파일 업로드 차단 |
| PDF 조치 3 | 업로드 파일이 원본 파일명이 아니라 난수 파일명으로 저장 |
| PDF 조치 4 | `/data/` 안의 PHP 계열 파일이 실행되지 않거나 접근 차단 |
| 정상 파일 회귀 확인 | 허용 확장자 이미지와 PDF는 정상 업로드 |

실습 순서는 조치 전 증거를 먼저 확보한 뒤, 방어 코드를 해제하거나 서버 설정을 적용하고 다시 확인한다. 현재 로컬 `care` 저장소의 방어 코드는 취약 상태 재현을 위해 주석 상태로 보존되어 있고, 실습 서버에서는 필요한 조치만 적용해 조치 후 증거를 수집했다.

## 7. 증거

### 7-1. PDF Step 1: 악성 스크립트 업로드 가능 여부

게시글 작성 화면에서 `.php` 확장자인 `14_fake_webshell_proof.php`를 첨부할 수 있었다.

![[Pasted image 20260616143532.png]]

업로드 후 게시글 상세 화면에서 첨부파일명이 그대로 노출되었다. 이 시점에서 CARE는 확장자 기준으로 업로드를 차단하지 않았다.

![[Pasted image 20260616143749.png]]

첨부파일 링크를 누르면 브라우저가 `14_fake_webshell_proof.php`를 다운로드했다. 즉 업로드된 파일은 웹에서 접근 가능한 첨부파일로 취급되었다.

![[Pasted image 20260616143803.png]]

### 7-2. PDF Step 2: 업로드한 스크립트 요청과 실행 여부

업로드된 파일을 `/data/14_fake_webshell_proof.php`로 직접 요청하자 PHP 코드가 서버에서 실행되었다.

![[Pasted image 20260616143940.png]]

화면에 출력된 proof 문자열은 다음 흐름을 보여준다.

```text
사용자 업로드
-> /var/www/html/care/data/14_fake_webshell_proof.php 저장
-> http://172.168.10.10/data/14_fake_webshell_proof.php 직접 요청
-> Apache/PHP가 업로드 파일을 실행
-> proof 문자열 응답
```

따라서 조치 전 CARE는 단순히 비정상 파일 업로드만 가능한 상태가 아니라, 업로드 디렉터리에서 PHP 파일 실행까지 가능한 상태였다.

### 7-3. 조치 코드 증거

조치 코드는 `center/writeModel.php`에 PDF 조치 번호가 달린 주석 형태로 보존했다. 핵심 대응은 다음과 같다.

| PDF 조치 | 코드 반영 내용 |
|---:|---|
| 1 | 파일명에서 널바이트를 제거하고 `basename()`과 정규식으로 위험 문자를 정리 |
| 2 | 허용 확장자, MIME Type, 업로드 오류, 파일 크기를 화이트리스트 방식으로 검증 |
| 3 | 서버 저장 파일명을 `random_bytes()` 기반 난수명으로 변경 |
| 4 | `/data/.htaccess`에서 PHP 계열 확장자 접근과 실행을 차단 |

상세 코드는 5절의 조치 방안에 PDF 번호별로 정리했다.

### 7-4. 조치 후 증거

방어 코드를 적용한 뒤 `.php` 파일 업로드를 시도하면 확장자 검증에서 차단되었다.

![[Pasted image 20260616150340.png]]

이미 업로드되어 있던 `/data/14_fake_webshell_proof.php`에 직접 접근하면 403 Forbidden이 반환되었다. 이는 PDF 조치 4의 업로드 디렉터리 실행 제한이 적용된 결과다.

## 8. 판단

| 항목 | 판단 |
|---|---|
| 1차 분류 | A-2 |
| 판정 상태 | 확정 |
| 조치 상태 | PDF 조치 1~4 반영, 5 보류, 6 제외 |
| 증거 상태 | 조치 전 업로드·다운로드·PHP 실행 증거 확보, 조치 후 확장자 차단·403 차단 증거 확보 |

CARE 원본에는 게시글 파일 업로드 기능이 있고, 서버가 업로드 파일의 확장자, MIME Type, 크기, 파일명, 저장 경로를 충분히 검증하지 않는다. 실습에서는 `.php` proof 파일이 업로드되고 `/data/` 경로에서 PHP로 실행되는 것까지 확인했다. 따라서 14번은 원래 CARE 기능을 이용해 재현 가능한 **A-2 기존 기능 악용형 취약점**으로 확정한다.

이번 조치안에서는 PDF의 6가지 조치 중 1~4번을 실제 적용 후보로 반영했다. 적용 후에는 `.php` 업로드가 확장자 검증에서 차단되고, 기존 업로드 파일에 직접 접근해도 403 Forbidden이 반환되었다. 5번 웹 디렉터리 격리는 다운로드 구조까지 바뀌므로 보류하고, 6번 바이러스 검사는 별도 운영 보안 체계가 필요해 이번 프로젝트 범위에서는 제외한다.
