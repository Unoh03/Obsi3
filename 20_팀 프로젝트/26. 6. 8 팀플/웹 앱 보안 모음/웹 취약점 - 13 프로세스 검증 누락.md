# 웹 취약점 - 13 프로세스 검증 누락

source: [[40_자료/주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드.pdf|주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드]], p.739-743.

> [!summary] 통합 처리
> 13번은 별도 실습 기능으로 분리하지 않고 [[20_팀 프로젝트/26. 6. 8 팀플/웹 앱 보안 모음/웹 취약점 - 12·13 비밀번호 복구 절차와 프로세스 검증 누락|12번 비밀번호 복구 절차 문서]] 안에서 함께 다룬다. 같은 `/vuln/password-recovery/` 흐름에서 12번은 인증번호 생성·전달·검증을 보고, 13번은 인증번호 검증 단계를 건너뛰고 `reset.php`를 직접 호출할 수 있는지를 본다.

## 1. 통합한 이유

처음에는 CARE 원본의 `center/writeModel.php`를 13번 후보로 볼 수 있었다. 게시글 작성 처리에서 로그인 여부는 확인하지만, 정상 작성 화면 경유 여부나 업무 단계 검증은 강하지 않기 때문이다.

하지만 이 방식은 피해와 설명력이 약하다.

| 후보 | 판단 |
|---|---|
| `center/writeModel.php` 직접 POST | 가능은 하지만 피해가 빈 글/비정상 글 등록 수준이라 13번의 핵심이 약하게 보임 |
| 비밀번호 복구 절차의 `reset.php` 직접 호출 | 이전 인증번호 검증 단계를 생략하고 계정 비밀번호를 바꾸는 흐름이라 13번의 핵심이 명확함 |

따라서 이번 프로젝트에서는 13번을 12번 비밀번호 복구 절차와 통합한다.

## 2. 13번의 핵심

13번의 질문은 다음이다.

```text
이 요청이 정상적인 이전 단계를 거쳐 도달한 요청인지
서버가 세션, DB 상태, 토큰 같은 서버 측 상태로 확인하는가?
```

비밀번호 복구 실습에서는 다음 흐름으로 확인한다.

```text
request.php 또는 verify.php를 정상적으로 거치지 않음
-> reset.php에 직접 접근
-> 서버가 인증 완료 상태를 확인하지 않음
-> 새 비밀번호 설정 처리
```

이것은 단순한 필수값 누락이 아니라, **업무 단계 완료 여부를 서버가 검증하지 않은 문제**다.

## 3. 현재 증거 위치

실행 증거는 12번 통합 문서에 둔다.

| 증거 | 위치 |
|---|---|
| 취약 모드에서 `reset.php` 직접 접근 | 12번 문서 `## 8. 13번 프로세스 검증 누락 통합` |
| 인증번호 검증 없이 비밀번호 변경 성공 | 12번 문서 `### 8.1 취약 모드 직접 호출` |
| 조치 모드에서 인증 완료 상태 없을 때 차단 | 12번 문서 `### 8.2 조치 모드 직접 호출 차단` |

## 4. 판단

| 항목 | 판단 |
|---|---|
| 1차 분류 | C-연계 |
| 판정 상태 | 12번 문서에서 실행 검증 일부 완료 |
| 조치 상태 | 12번 문서에 통합 |
| 증거 상태 | 12번 문서에 통합 |

CARE 원본에는 비밀번호 복구 기능이 없으므로 13번을 원래 CARE의 자연 취약점으로 쓰지 않는다. 이번 프로젝트에서는 `/vuln/password-recovery/` 아래에 만든 실습용 복구 기능을 통해 12번과 13번을 한 흐름으로 재현한다.
