# 웹 취약점 - 20 자동화 공격

source: [[40_자료/주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드.pdf|주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드]], p.777-778.

## 1. 개요

**자동화 공격**은 로그인 시도, 게시글 등록, SMS 발송, 본인인증 요청처럼 반복 실행될 수 있는 기능에 대해 서버가 횟수 제한, 지연, CAPTCHA, 잠금, WAF/IDS/IPS 같은 통제를 두는지 확인하는 항목이다.

PDF p.777의 판단 기준은 다음처럼 정리할 수 있다.

| 판단 | 기준 |
|---|---|
| 양호 | 특정 프로세스의 반복적인 요청이 적절하게 통제됨 |
| 취약 | 특정 프로세스의 반복적인 요청 통제가 미흡함 |

이 항목은 단순히 “비밀번호가 약한가”만 보는 것이 아니다. 핵심은 **서버가 짧은 시간에 반복되는 요청을 구분하고 제한하는가**다.

```text
자동화 공격 예시

로그인 요청 반복 -> 계정 탈취 시도
게시글 등록 반복 -> 게시판 도배 / 자원 고갈
SMS 인증 요청 반복 -> 인증번호 탈취 시도 / 비용 유발
```

## 2. CARE 적용 가능성 분석

CARE에는 PDF가 예시로 든 반복 요청 대상 중 최소 두 가지가 존재한다.

| PDF 예시 | CARE 기능 | 관련 파일 | 판단 |
|---|---|---|---|
| 로그인 시도 | 로그인 | `D:\care\member\loginModel.php` | 실패 횟수 제한, 계정 잠금, CAPTCHA 없음 |
| 게시글 등록 | 게시글 작성 | `D:\care\center\write.php`, `writeModel.php` | 작성 횟수 제한, 재전송 방지, CAPTCHA 없음 |
| SMS 발송 / 본인인증 | CARE 원본 기능 없음 | - | 원본 CARE 기준 해당 없음 |

따라서 20번은 CARE 원본에 공격면이 있는 **A-2 기존 기능 악용형**으로 본다.

다만 로그인 반복 시도만 다루면 기존 [[10_학습 노트/시스템보안/웹보안/Hydra 로그인 Brute Force 실습|Hydra 로그인 Brute Force 실습]]과 거의 겹친다. 그래서 이 노트에서는 로그인 반복은 보조 근거로 두고, **게시글 반복 등록 통제 부재**를 중심 증거로 삼는다.

## 3. 문제가 되는 부분

### 3.1 로그인 반복 시도 통제 부재

`loginModel.php`는 사용자가 보낸 `id`, `pw`를 SQL에 넣고 결과 행 수로 성공/실패를 판단한다.

```php
$id = $_POST['id'];
$pw = $_POST['pw'];

$query = "SELECT * FROM member WHERE id='$id' and pw='$pw'";
$result = mysqli_query($link, $query);
$num = mysqli_num_rows($result);
```

실패했을 때는 alert를 띄우고 이전 페이지로 돌려보낸다.

```php
if($num == 1){
    // 로그인 성공
}else{
    // 로그인 실패
?>
    <script>
        alert('로그인 실패');
        history.go(-1);
    </script>
<?php
    exit;
}
```

여기에는 다음 통제가 없다.

| 통제 항목 | 현재 상태 |
|---|---|
| 실패 횟수 저장 | 없음 |
| 계정 잠금 | 없음 |
| IP / 계정 기준 요청 제한 | 없음 |
| CAPTCHA | 없음 |
| 로그인 처리 지연 | 없음 |

이 구조는 Hydra 같은 도구로 반복 요청을 보내기 쉽다. 다만 이 부분은 이미 별도 Hydra 노트에서 깊게 다뤘으므로, 20번의 중심 증거로는 게시글 등록 반복을 사용한다.

### 3.2 게시글 반복 등록 통제 부재

`write.php`는 로그인한 사용자에게 게시글 작성 form을 제공한다.

```php
<form action="writeModel.php" method="post" enctype="multipart/form-data">
    <input type="text" name="subject">
    <textarea name="content"></textarea>
    <input type="file" name="upfile">
</form>
```

`writeModel.php`는 로그인 여부만 확인한 뒤, 요청이 들어올 때마다 바로 `center` 테이블에 글을 추가한다.

```php
session_start();
$id = $_SESSION['id'];
if($id == ""){
    echo "<script>location.href='/member/login.php'; </script>";
    exit;
}

$subject = $_POST['subject'];
$content = $_POST['content'];

$query = "INSERT INTO center(id, subject, content, date, hit, filename) ";
$query = $query . "VALUES('$id','$subject', '$content', '$date', 0, '$upfile')";
mysqli_query($link, $query);
```

여기에는 다음 통제가 없다.

| 통제 항목 | 현재 상태 |
|---|---|
| 같은 사용자의 연속 게시 제한 | 없음 |
| 짧은 시간 내 반복 등록 제한 | 없음 |
| 중복 제출 토큰 | 없음 |
| CAPTCHA | 없음 |
| IP / 사용자 기준 rate limit | 없음 |

따라서 로그인한 사용자가 같은 요청을 반복 전송하면 게시글이 계속 등록될 수 있다.

## 4. 점검 방법

### 4.1 로그인 반복 시도 확인

이 방법은 기존 Hydra 노트와 거의 겹치므로, 20번에서는 보조 확인으로만 둔다.

확인할 내용은 다음과 같다.

| 확인 | 기대 결과 |
|---|---|
| 틀린 비밀번호로 여러 번 로그인 시도 | 계속 `로그인 실패`만 출력 |
| 5회 이상 실패 | 계정 잠금 없음 |
| 빠른 반복 요청 | CAPTCHA, 지연, 차단 없음 |

### 4.2 게시글 반복 등록 확인

20번의 주 증거는 게시글 등록 반복으로 잡는다.

1. 일반 사용자로 로그인한다.
2. 게시글 작성 화면에서 제목을 `AUTO-20-반복등록-1`처럼 눈에 띄게 입력한다.
3. 같은 POST 요청을 여러 번 반복한다.
4. 게시글 목록 또는 DB에서 같은 패턴의 글이 여러 개 등록되는지 확인한다.

증거로 남길 화면은 다음이 적절하다.

| 증거 | 의미 |
|---|---|
| 게시글 작성 화면 | 반복 요청 대상 프로세스가 존재함 |
| 같은 요청 반복 전송 화면 | 자동화 또는 반복 요청 수행 |
| 게시글 목록에 같은 패턴 글이 여러 개 생성된 화면 | 반복 등록 통제 부재 |
| DB `center` 테이블 조회 | 서버 측으로 실제 INSERT가 반복됨 |

DB 확인 예시는 다음과 같다.

```sql
SELECT num, id, subject, date
FROM center
WHERE subject LIKE 'AUTO-20-%'
ORDER BY num DESC
LIMIT 10;
```

### 4.3 빠른 새로고침은 증거로 충분한가

단순히 페이지를 빠르게 새로고침하는 것만으로는 자동화 공격 증거가 약하다.

| 행위 | 판단 |
|---|---|
| 목록 페이지 `GET` 새로고침 반복 | 부하 유발 가능성은 있지만 20번의 핵심 증거로는 약함 |
| 작성 완료 후 form 재전송 반복 | 게시글이 실제로 여러 개 생성되면 20번 증거로 사용 가능 |
| Paros / Burp / curl로 같은 POST 요청 반복 | 가장 명확한 반복 요청 증거 |

즉, “빠른 새로고침” 자체보다 **서버 상태가 바뀌는 요청이 반복 실행되는가**가 중요하다.

## 5. 조치 방안

PDF p.778의 조치 방법은 크게 두 가지다.

| PDF 조치 | 내용 | CARE 적용 |
|---|---|---|
| 조치 1 | 로그인 시도, 게시글 등록, 본인인증/SMS 발송 요청에 횟수 제한 또는 CAPTCHA 구현 | PHP 코드에서 적용 가능 |
| 조치 2 | 짧은 시간의 다량 패킷을 감지/차단할 IDS/IPS 구축 | 인프라 조치. 현재 CARE 코드만으로는 한계 |

이 노트에서는 PDF 조치 1을 CARE 코드 수준에서 다룬다. PDF 조치 2는 운영 환경 조치로 기록만 남긴다.

### 5.1 PDF 조치 1-1: 로그인 시도 횟수 제한

로그인 반복 시도는 기존 Hydra 실습에서 다룬 방식처럼 실패 횟수와 잠금 시간을 서버에 저장해야 한다.

세션만 사용하는 방식은 브라우저를 바꾸거나 쿠키를 지우면 우회될 수 있으므로, 실무형으로는 DB 기준 저장이 더 적절하다.

```text
권장 흐름

login_fail 테이블 생성
-> id + ip 기준으로 실패 횟수 저장
-> 기준 횟수 초과 시 잠금 해제 시간 저장
-> 로그인 시도 전에 잠금 상태 확인
-> 성공하면 실패 횟수 초기화
```

이 부분은 [[10_학습 노트/시스템보안/웹보안/Hydra 로그인 Brute Force 실습|Hydra 로그인 Brute Force 실습]]의 DB 기반 계정 잠금 구현을 참고한다.

### 5.2 PDF 조치 1-2: 게시글 등록 횟수 제한

게시글 등록 반복은 `writeModel.php`에서 INSERT 전에 제한할 수 있다.

세션 기준 cooldown은 구현이 쉽지만, 쿠키 삭제, 새 브라우저, 시크릿 모드, 새 세션으로 우회될 수 있다. 따라서 이번 실습에서는 **DB에 사용자 ID와 IP별 최근 게시글 등록 시간을 저장하는 방식**으로 조치한다.

먼저 DB에 반복 등록 상태를 저장할 테이블을 만든다.

```sql
-- 웹 취약점 20번 PDF 조치 1-2:
-- 게시글 등록 요청에 대해 사용자 ID와 IP 기준 횟수 제한을 적용하기 위한 테이블
USE care;

CREATE TABLE post_rate_limit (
    id varchar(50) NOT NULL,
    ip varchar(45) NOT NULL,
    last_post_at int unsigned NOT NULL,
    updated_at timestamp NOT NULL DEFAULT current_timestamp() ON UPDATE current_timestamp(),
    PRIMARY KEY (id, ip)
);
```

그 다음 `writeModel.php`에서 DB 연결 직후, 게시글 `INSERT` 전에 아래 검사를 넣는다.

이 방식은 세션 기준보다 강하다. 브라우저를 바꾸거나 세션 쿠키를 삭제해도, 같은 사용자 ID와 같은 IP에서 짧은 시간 안에 반복 등록하면 DB에 저장된 `last_post_at` 기준으로 차단된다.

| 방식 | 장점 | 한계 |
|---|---|---|
| 세션 기준 cooldown | 구현이 가장 쉬움 | 새 세션, 다른 브라우저, 쿠키 삭제로 우회 가능 |
| ID + IP 기준 DB 제한 | 같은 계정과 출발지의 반복 등록을 안정적으로 제한 | DB 테이블 필요 |
| 사용자 ID 기준 DB 제한 | IP가 바뀌어도 같은 계정 반복 등록을 제한 가능 | 여러 정상 환경에서 같은 계정 사용 시 영향 가능 |
| IP 기준 제한 | 같은 출발지의 대량 요청을 제한 가능 | NAT 환경에서 정상 사용자까지 영향 가능 |
| CAPTCHA | 자동화 도구를 어렵게 함 | 사용자 불편, 우회 가능성, 외부 연동 필요 |
| WAF / IDS / IPS | 앱 바깥에서 대량 요청 감지 가능 | 별도 인프라 구성 필요 |

이번 조치는 `ID + IP 기준 DB 제한`이다. 다중 계정, 프록시 분산, 여러 IP를 사용하는 공격까지 완전히 막지는 못하지만, 세션 기준보다 보고서용 근거가 훨씬 강하다.

### 5.3 PDF 조치 1-3: 본인인증 / SMS 요청 제한

CARE 원본에는 SMS 발송이나 본인인증 요청 기능이 없다. 따라서 이 조치는 원본 CARE에는 직접 적용하지 않는다.

다만 `/vuln/password-recovery/` 같은 실습용 복구 기능을 사용할 경우에는 다음 제한이 필요하다.

```text
- 인증번호 요청 횟수 제한
- 인증번호 만료 시간
- 인증 실패 횟수 제한
- 새 인증번호 요청 시 이전 인증번호 폐기
```

### 5.4 PDF 조치 2: IDS / IPS / WAF 기반 차단

PDF 조치 2는 짧은 시간에 다량의 패킷이 전송되는 자동화 공격을 감지하고 차단할 수 있는 IDS/IPS 구축이다.

현재 CARE PHP 코드만으로는 IDS/IPS를 구현할 수 없다. 실습 환경에서 다룬다면 Apache, nginx, WAF, 네트워크 보안 장비, Suricata/Snort 같은 별도 계층에서 처리해야 한다.

이 노트에서는 코드 수준 조치와 구분해 운영 조치로만 기록한다.

## 6. AWS 구축 후 이어갈 작업

이 항목은 AWS에서 CARE 배포와 DB 연결이 안정화된 뒤 이어서 진행한다. 핵심은 **취약 상태 증거를 먼저 확보한 뒤, DB 기반 제한을 적용하고, 같은 요청이 차단되는지 다시 확인하는 것**이다.

### 6.1 AWS 환경 확인

실습 시작 전에 다음 값들을 먼저 확정한다.

| 항목 | 확인할 값 |
|---|---|
| CARE 접속 URL | `http://<WEB_PUBLIC_IP>/` 또는 `http://<ALB_DNS>/` |
| WEB 서버의 CARE 경로 | 보통 `/var/www/html/care` |
| DB 서버 | EC2 MariaDB private IP 또는 RDS endpoint |
| DB 이름 | `care` |
| 게시글 테이블 | `center` |
| 방어 테이블 | `post_rate_limit` |
| 반복 등록 제한 시간 | 실습 기본값 `10초` |

> [!important]
> AWS에서 ALB나 reverse proxy를 경유하면 PHP의 `$_SERVER['REMOTE_ADDR']`가 실제 브라우저 IP가 아니라 ALB 또는 proxy의 IP로 보일 수 있다.
> WEB 서버에 직접 접속하는 실습이면 `REMOTE_ADDR` 기준으로 충분하다. ALB 경유 구조라면 `X-Forwarded-For`를 신뢰할지 별도로 판단해야 한다.

### 6.2 조치 전 증거 수집

조치 코드를 적용하기 전에 먼저 취약 상태를 캡처한다.

1. 일반 사용자로 CARE에 로그인한다.
2. 게시글 작성 화면에 들어간다.
3. 제목을 `AUTO-20-before-1`, `AUTO-20-before-2`, `AUTO-20-before-3`처럼 구분되게 입력한다.
4. 짧은 시간 안에 게시글을 여러 번 등록한다.
5. 게시글 목록에서 같은 패턴의 글이 여러 개 생성된 것을 캡처한다.
6. DB에서 `center` 테이블을 조회해 실제 INSERT가 반복된 것을 확인한다.

DB 확인 예시는 다음과 같다.

```sql
USE care;

SELECT num, id, subject, date
FROM center
WHERE subject LIKE 'AUTO-20-before-%'
ORDER BY num DESC
LIMIT 10;
```

이 단계의 목적은 “반복 요청을 서버가 통제하지 않는다”는 조치 전 상태를 남기는 것이다.

### 6.3 DB 테이블 생성

조치 전 증거를 확보한 뒤 DB 서버에서 `post_rate_limit` 테이블을 생성한다.

```sql
USE care;

CREATE TABLE IF NOT EXISTS post_rate_limit (
    id varchar(50) NOT NULL,
    ip varchar(45) NOT NULL,
    last_post_at int unsigned NOT NULL,
    updated_at timestamp NOT NULL DEFAULT current_timestamp() ON UPDATE current_timestamp(),
    PRIMARY KEY (id, ip)
);

SHOW TABLES LIKE 'post_rate_limit';
DESC post_rate_limit;
```

캡처할 증거는 다음 두 가지다.

| 증거 | 의미 |
|---|---|
| `SHOW TABLES LIKE 'post_rate_limit';` | 방어용 테이블 생성 확인 |
| `DESC post_rate_limit;` | `id`, `ip`, `last_post_at` 저장 구조 확인 |

### 6.4 `writeModel.php` 조치 적용

WEB 서버의 `writeModel.php`에서 DB 연결 직후, `center` 테이블 `INSERT` 전에 5.2의 방어 코드를 적용한다.

적용 위치는 이 흐름이다.

```text
로그인 세션 확인
-> 입력값 수신
-> DB 연결
-> [20번 자동화 공격 방어 코드]
-> center 테이블 INSERT
-> 파일 업로드 처리
-> list.php로 이동
```

PHP 파일만 바꾸는 경우 보통 Apache reload는 필요 없다. 다만 GitHub 배포 스크립트를 사용 중이면 `care-web-deploy.sh`로 다시 배포하고, OPcache를 쓰는 환경이면 Apache reload가 필요할 수 있다.

```bash
# 배포 스크립트를 쓰는 경우 예시
sudo ./care-web-deploy.sh

# OPcache나 Apache 상태가 의심될 때만
sudo systemctl reload apache2
```

### 6.5 조치 후 검증

조치 후에는 같은 사용자로 다시 게시글을 연속 등록한다.

| 순서 | 기대 결과 |
|---|---|
| 첫 번째 게시글 등록 | 성공 |
| 10초 안에 두 번째 게시글 등록 | `게시글은 10초에 한 번만 등록할 수 있습니다.` alert |
| 10초 이상 기다린 뒤 다시 등록 | 성공 |
| DB `post_rate_limit` 조회 | 해당 `id`, `ip`, `last_post_at` 기록 확인 |

DB 확인 예시는 다음과 같다.

```sql
USE care;

SELECT id, ip, last_post_at, FROM_UNIXTIME(last_post_at) AS last_post_time, updated_at
FROM post_rate_limit
ORDER BY updated_at DESC
LIMIT 10;
```

조치 후 증거로 남길 화면은 다음과 같다.

| 증거 | 의미 |
|---|---|
| 첫 번째 게시글 등록 성공 | 정상 사용은 허용됨 |
| 두 번째 즉시 등록 차단 alert | 반복 요청 통제 적용 |
| `post_rate_limit` 조회 결과 | 서버가 사용자/IP별 상태를 DB에 저장함 |
| 10초 후 등록 성공 | 영구 차단이 아니라 횟수/시간 제한임 |

### 6.6 AWS에서 주의할 점

| 상황 | 주의점 |
|---|---|
| WEB 서버 직접 접속 | `REMOTE_ADDR`가 클라이언트 IP에 가깝게 잡히므로 현재 코드로 실습하기 쉬움 |
| ALB 경유 접속 | `REMOTE_ADDR`가 ALB IP로 보일 수 있어 IP 기준 제한의 의미가 약해질 수 있음 |
| NAT, 프록시, 공동망 | 여러 사용자가 같은 IP로 묶일 수 있음 |
| 여러 계정으로 공격 | `id + ip` 기준이므로 다른 계정으로 분산하면 우회 가능 |
| 여러 IP로 공격 | IP 분산 공격까지 막으려면 WAF/IDS/IPS 또는 더 넓은 정책 필요 |

따라서 이번 실습의 결론은 “완벽한 자동화 공격 방어”가 아니라, **PDF 조치 1의 게시글 등록 횟수 제한을 CARE 코드와 DB로 구현하고 검증했다**로 잡는다.

## 8. 판단

CARE 원본은 로그인과 게시글 작성이라는 반복 요청 대상 프로세스를 가지고 있다. 그러나 로그인 실패 횟수 제한, 계정 잠금, CAPTCHA, 게시글 등록 cooldown, 중복 제출 방지, WAF/rate-limit 설정이 확인되지 않는다.

따라서 20번 자동화 공격은 **A-2 기존 기능 악용형**으로 판단한다.

다만 기존 Hydra 실습과 중복을 줄이기 위해, 이 노트의 중심 증거는 로그인 brute force가 아니라 **게시글 반복 등록 통제 부재**로 잡는다.

```text
최종 판단:
CARE에는 반복 요청 대상 프로세스가 있고,
반복 요청 통제가 없으므로 취약하다.

주 증거:
게시글 반복 등록

보조 증거:
로그인 반복 시도 통제 부재
```
