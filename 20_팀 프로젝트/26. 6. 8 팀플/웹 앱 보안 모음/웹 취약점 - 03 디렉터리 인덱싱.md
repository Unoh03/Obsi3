# 웹 취약점 - 03 디렉터리 인덱싱

source: [[40_자료/주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드.pdf|주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드]], p.700-702.

## 1. 개요

**디렉터리 인덱싱**은 웹 서버의 특정 디렉터리에 기본 페이지가 없을 때, 브라우저에 해당 디렉터리의 파일 목록이 그대로 출력되는 취약점이다.

파일 목록이 노출되면 웹 애플리케이션의 디렉터리 구조, 백업 파일, 설정 파일, 업로드 파일, 테스트 파일을 공격자가 직접 탐색할 수 있다.

## 2. 문제가 되는 부분

CARE 자체 PHP 코드보다 **Apache 디렉터리 설정**이 핵심이다.

취약한 설정은 다음처럼 `Indexes` 옵션이 켜져 있는 경우다.

```apache
<Directory /var/www/html/care>
    Options Indexes FollowSymLinks
</Directory>
```

이 상태에서 특정 디렉터리에 `index.php`, `index.html` 같은 기본 페이지가 없으면, Apache가 파일 목록을 자동으로 보여줄 수 있다.

실습에서 확인하기 쉬운 후보 경로는 다음과 같다.

| 후보 경로 | 확인 이유 |
|---|---|
| `/uploads/` | 업로드 파일이 노출될 수 있는지 확인 |
| `/data/` | CARE에서 파일 저장 용도로 사용할 수 있는 디렉터리 후보 |
| `/vuln/` | 실습용 취약 페이지 목록이 노출될 수 있는지 확인 |
| `/vuln/code-injection/` | 개별 취약점 실습 파일명이 노출되는지 확인 |

## 3. 악용 흐름

공격자는 별도 로그인 없이 디렉터리 URL을 직접 요청한다.

```text
http://1.1.1.10/uploads/
http://1.1.1.10/vuln/
http://1.1.1.10/vuln/code-injection/
```

서버가 디렉터리 목록을 반환하면 다음 정보를 얻을 수 있다.

| 노출 정보 | 악용 가능성 |
|---|---|
| 파일명 | 백업 파일, 테스트 파일, 업로드 파일 추측 |
| 디렉터리 구조 | 공격 대상 기능과 내부 경로 파악 |
| 확장자 | PHP, txt, zip, bak 등 민감 파일 후보 식별 |
| 수정 시간 / 크기 | 최근 업로드 파일이나 중요한 파일 추정 |

즉, 디렉터리 인덱싱은 단독으로 시스템을 장악하는 취약점이라기보다, **다른 취약점을 찾기 쉽게 만드는 정보 노출 취약점**이다.

## 4. 점검 방법

PDF p.700의 판단 기준은 단순하다.

| 판단 | 기준 |
|---|---|
| 양호 | 디렉터리 파일 리스트가 노출되지 않음 |
| 취약 | 디렉터리 파일 리스트가 노출됨 |

점검 절차는 다음처럼 잡는다.

1. 기본 페이지가 없을 가능성이 있는 디렉터리 URL에 직접 접근한다.
2. `Index of /...` 형태의 파일 목록이 뜨는지 확인한다.
3. 파일명, 디렉터리명, 수정 시간, 크기 등이 노출되는지 확인한다.
4. 조치 후 같은 URL에 다시 접근하여 목록이 더 이상 보이지 않는지 확인한다.

## 5. 조치 방안

본 프로젝트는 Apache/PHP 기반이므로 Apache 설정을 우선 적용한다.

### 5.1 현재 설정 확인

먼저 Apache가 어떤 설정 파일을 읽고 있는지 확인한다.

```bash
sudo apache2ctl -S
```

CARE용 site 설정 파일이 있다면 보통 다음 경로 중 하나다.

```text
/etc/apache2/sites-available/care.conf
/etc/apache2/sites-enabled/care.conf
/etc/apache2/sites-available/000-default.conf
```

실제 care에선 이렇게 나왔다.

```
VirtualHost configuration:
*:80                   care.local (/etc/apache2/sites-enabled/care.conf:1)
```

`Indexes` 옵션이 켜진 곳을 찾는다.

```bash
sudo grep -R "Options .*Indexes" /etc/apache2/sites-available /etc/apache2/sites-enabled /etc/apache2/apache2.conf
```

결과는 이렇게 나왔다.

```bash
/etc/apache2/sites-available/care.conf:        Options -Indexes +FollowSymLinks
/etc/apache2/sites-enabled/care.conf:        Options -Indexes +FollowSymLinks
/etc/apache2/apache2.conf:      Options Indexes FollowSymLinks
/etc/apache2/apache2.conf:#     Options Indexes FollowSymLinks
```

`/etc/apache2/sites-enabled/care.conf` 을 사용중인 것을 알고 있으니 그 옆에 있는걸 보면 된다.
Options -Indexes +FollowSymLinks 로 되어 있어 이미 인덱스 기능은 꺼져있다.
### 5.3 `Indexes` 옵션 켜보기

설정 파일을 연다.

```bash
sudo nano /etc/apache2/sites-available/care.conf
```

`/var/www/html/care`에 대한 `<Directory>` 블록을 찾는다.

```apache
<Directory /var/www/html/care>
    Options -Indexes +FollowSymLinks
    AllowOverride All
    Require all granted
</Directory>
```

-와 +를 지워 활성화한다.

```apache
<Directory /var/www/html/care>
    Options Indexes FollowSymLinks
    AllowOverride All
    Require all granted
</Directory>
```

비활성화할 땐 다시 -와 +를 붙이면 된다.

### 5.4 Apache 설정 검사 및 반영

수정 후 바로 재시작하지 말고 문법 검사를 먼저 한다.

```bash
sudo apache2ctl configtest
```

정상이라면 다음처럼 나온다.

```text
Syntax OK
```

`Syntax OK`가 나온 뒤에만 설정을 반영한다.

```bash
sudo systemctl reload apache2
```

### 5.5 조치 후 확인

서버 안에서 먼저 확인한다.

```bash
curl -i http://127.0.0.1/vuln/
curl -i http://127.0.0.1/vuln/code-injection/
curl -i http://127.0.0.1/uploads/
```

브라우저에서도 같은 경로를 확인한다.

```text
http://1.1.1.10/vuln/
http://1.1.1.10/vuln/code-injection/
http://1.1.1.10/uploads/
```

조치가 적용되면 `Index of /...` 형태의 파일 목록이 보이지 않아야 한다. `403 Forbidden`, `404 Not Found`, 또는 애플리케이션에서 의도한 에러 페이지가 나오면 디렉터리 인덱싱은 차단된 것으로 본다.

반대로 화면에 파일명 목록이 계속 보이면 다음을 다시 확인한다.

| 확인 항목                                  | 의미                                    |
| -------------------------------------- | ------------------------------------- |
| 수정한 파일이 실제 활성화된 site 설정인지              | `apache2ctl -S` 결과와 비교                |
| `<Directory>` 경로가 실제 DocumentRoot와 같은지 | `/var/www/html/care` 경로 확인            |
| `Options Indexes`가 다른 설정 파일에 남아 있는지    | `grep -R "Options .*Indexes" ...` 재확인 |
| Apache reload가 성공했는지                   | `systemctl status apache2` 확인         |

### 진단 스크립트에서 걸릴 시

## 7. 판단

디렉터리 URL 접근 시 파일 리스트가 출력되면 취약으로 판단한다.

본 항목의 핵심은 PHP 코드 수정이 아니라 **웹 서버가 디렉터리 목록을 자동으로 공개하지 않도록 설정하는 것**이다.

CARE 환경에서는 Apache 설정 전후를 비교하면 짧고 명확하게 재현할 수 있다.
