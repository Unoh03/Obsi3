# 웹 취약점 - 05 정보 누출

source: [[40_자료/주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드.pdf|주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드]], p.708-710.

## 1. 개요

**정보 누출**은 웹 애플리케이션이나 웹 서버가 서비스에 필요하지 않은 내부 정보, 테스트 파일, 샘플 계정, 개인정보, DB 구조 등을 외부에 보여주는 취약점이다.

PDF p.708의 판단 기준은 다음처럼 정리할 수 있다.

| 판단 | 기준 |
|---|---|
| 양호 | 중요 정보가 마스킹되어 있고, 서비스 운영이나 서버 구조를 파악할 수 있는 불필요한 정보가 노출되지 않음 |
| 취약 | 중요 정보가 평문으로 노출되거나, 서버 사이드 구조를 파악할 수 있는 과도한 정보가 노출됨 |

이 항목은 단순히 DB 비밀번호 파일 하나만 보는 것이 아니라, 웹루트에 남은 테스트 파일, 백업 파일, 샘플 페이지, 주석, 디버그 출력까지 같이 확인해야 한다.

## 2. 문제가 되는 부분

CARE는 DB 접속정보 분리 자체는 이미 어느 정도 조치되어 있다.

| 파일 | 상태 | 판단 |
|---|---|---|
| `config.php` | 기본 더미값을 두고 `config.local.php`를 병합 | 실제 DB 접속정보를 코드에 직접 박아두는 구조는 아님 |
| `config.local.php` | 실제 DB 접속정보를 담는 로컬 파일 | 값은 보고서에 노출하지 않음 |
| `.gitignore` | `config.local.php`, `.env`, `*.local.php` 제외 | Git 노출 방지 측면에서는 적절함 |

하지만 정보 누출 항목 전체로 보면 아직 문제가 남아 있다.

| 문제 지점 | 노출 내용 | 영향 |
|---|---|---|
| `/dbconn.php` | 조회된 계정의 `id`, `pw`, `name` 출력 | 샘플 계정과 비밀번호가 브라우저에 직접 노출됨 |
| `/member/테이블 생성.txt` | DB 생성 명령, 샘플 계정, 비밀번호, 전화번호, 이메일 출력 | DB 구조와 개인정보성 샘플 데이터가 그대로 노출됨 |
| `/center/center(table).txt` | 게시판 테이블 구조 출력 | 게시판 DB 컬럼과 내부 구조 파악 가능 |

즉, **DB secret 분리 조치와 별개로 웹에서 직접 접근 가능한 테스트/샘플 파일이 남아 있는 상태**가 핵심 문제다.

## 3. 악용 흐름

공격자는 별도 로그인 없이 웹루트에 남은 파일을 직접 요청할 수 있다.

```text
http://172.168.10.10/dbconn.php
http://172.168.10.10/member/테이블%20생성.txt
http://172.168.10.10/center/center(table).txt
```

이때 파일이 그대로 노출되면 다음 정보를 얻을 수 있다.

| 요청 | 얻을 수 있는 정보 |
|---|---|
| `/dbconn.php` | 테스트 계정, 비밀번호, 계정명 |
| `/member/테이블 생성.txt` | DB명, 테이블명, 컬럼명, 샘플 계정, 샘플 개인정보 |
| `/center/center(table).txt` | 게시판 테이블 구조, 파일 업로드 관련 컬럼명 |

이 정보는 단독 공격보다 후속 공격에 더 위험하다. 예를 들어 샘플 계정으로 로그인하거나, SQL Injection 공격 시 테이블명과 컬럼명을 추측하는 시간을 줄일 수 있다.

## 4. 점검 방법

PDF p.709-710의 점검 흐름은 다음 세 가지다.

1. 페이지 안에 중요 정보가 평문으로 노출되는지 확인한다.
2. 서버나 WAS의 샘플 페이지, 백업 파일, 텍스트 파일 등 불필요한 파일이 노출되는지 확인한다.
3. HTML 소스코드나 주석 안에 중요 정보가 남아 있는지 확인한다.

CARE 환경에서는 우선 웹루트에서 삭제 권고 확장자와 테스트 파일을 찾는다.

```bash
find /var/www/html/care -type f \( \
  -name "*.bak" -o -name "*.backup" -o -name "*.old" -o -name "*.zip" -o \
  -name "*.log" -o -name "*.sql" -o -name "*.txt" -o -name "*.tmp" \
\)
```

그 다음 브라우저에서 실제 접근 가능 여부를 확인한다.

```text
http://172.168.10.10/dbconn.php
http://172.168.10.10/member/테이블%20생성.txt
http://172.168.10.10/center/center(table).txt
```

## 5. 조치 방안

### 5.1 테스트/샘플 파일 웹 접근 차단

`dbconn.php`는 DB 연결 확인과 계정 조회 테스트용 파일이다. `테이블 생성.txt`와 `center(table).txt`도 DB 구조 확인을 위해 만든 작업 파일에 가깝다.

가장 깔끔한 조치는 삭제지만, 실습 증거와 비교를 위해 파일을 남겨두고 싶다면 Apache에서 웹 접근만 차단할 수 있다.

CARE의 Apache site 설정 파일을 연다.

```bash
sudo nano /etc/apache2/sites-available/care.conf
```

`<VirtualHost *:80>` 블록 안에 다음 설정을 추가한다.

```apache
<Files "dbconn.php">
    Require all denied
</Files>

<Files "테이블 생성.txt">
    Require all denied
</Files>

<Files "center(table).txt">
    Require all denied
</Files>
```

이 설정은 파일을 서버에 남겨두되, 브라우저에서 해당 파일명을 직접 요청했을 때 접근을 거부한다.

설정 문법을 검사한 뒤 Apache에 반영한다.

```bash
sudo apache2ctl configtest
sudo systemctl reload apache2
```

조치 후 같은 URL에 다시 접근해 계정 정보와 DB 생성 기록이 더 이상 출력되지 않는지 확인한다.

```bash
curl -i http://127.0.0.1/dbconn.php
curl -i "http://127.0.0.1/member/테이블%20생성.txt"
curl -i "http://127.0.0.1/center/center(table).txt"
```

정상적으로 차단되면 `403 Forbidden` 또는 사용자 정의 에러 페이지가 출력된다.

### 5.2 불필요 파일 삭제 또는 웹루트 밖 이동

PDF p.710은 `*.txt`, `*.sql`, `*.log`, `*.bak`, `*.zip` 같은 작업 중 생성된 파일을 삭제 권고 대상으로 본다.

최종 운영 기준에서는 테스트 파일과 DB 생성 기록 파일을 웹루트에 남기지 않는 것이 더 안전하다.

```bash
sudo rm -f /var/www/html/care/dbconn.php
sudo rm -f "/var/www/html/care/member/테이블 생성.txt"
sudo rm -f "/var/www/html/care/center/center(table).txt"
```

파일을 보존해야 한다면 웹루트 밖으로 옮긴다.

```bash
sudo mkdir -p /opt/care-evidence
sudo mv /var/www/html/care/dbconn.php /opt/care-evidence/
sudo mv "/var/www/html/care/member/테이블 생성.txt" /opt/care-evidence/
sudo mv "/var/www/html/care/center/center(table).txt" /opt/care-evidence/
```

이번 실습에서는 파일을 바로 삭제하지 않고, Apache 접근 차단으로 조치 후 상태를 확인했다.

### 5.3 DB 접속정보 파일 관리

현재 구조처럼 실제 접속정보는 `config.local.php`로 분리하고 Git에는 올리지 않는다.

```text
config.php          # 공통 로더와 기본 더미값
config.example.php  # 예시 설정
config.local.php    # 실제 서버 접속정보, Git 추적 제외
```

단, `config.local.php`는 실제 비밀값이 들어가는 파일이므로 스크린샷이나 보고서에 값을 넣지 않는다.

가능하면 더 안전하게는 실제 접속정보 파일을 DocumentRoot 밖으로 옮긴다. 당장 구조 변경이 부담되면 최소한 Apache에서 직접 접근을 차단한다.

```apache
<FilesMatch "^config.*\.php$">
    Require all denied
</FilesMatch>
```

AWS 환경까지 고려하면 비밀번호, DB 접속정보, API Key 같은 값은 서버 파일에 직접 보관하지 않고 비밀값 저장소에서 런타임에 불러오도록 구성할 수 있다.

| 방식 | 용도 |
|---|---|
| AWS Secrets Manager | DB 계정, API Key처럼 회전과 수명주기 관리가 필요한 secret 저장 |
| AWS Systems Manager Parameter Store `SecureString` | 비교적 단순한 암호화 설정값이나 환경별 설정값 저장 |
| HashiCorp Vault | 클라우드와 온프레미스가 섞인 환경에서 중앙 secret 관리 |

이 프로젝트에서는 우선 `config.local.php`를 Git에서 제외하고, 최종 보고서에는 실제 값을 쓰지 않는 방식으로 처리한다. 실무형 개선안으로는 `config.local.php` 대신 Secrets Manager나 Parameter Store에서 값을 가져오게 바꾸는 방향을 제시할 수 있다.

### 5.4 검색엔진 인덱싱 차단

PDF p.710은 `robots.txt`, `web.config`, `nginx.conf`, HTML `META` 태그를 이용해 검색 차단할 디렉터리, 확장자, 페이지를 지정할 수 있다고 설명한다.

CARE는 Apache/PHP 환경이므로 우선 `robots.txt`를 사용할 수 있다.

```bash
sudo nano /var/www/html/care/robots.txt
```

예시는 다음과 같다.

```text
User-agent: *
Disallow: /member/
Disallow: /center/
Disallow: /data/
Disallow: /vuln/
Disallow: /*.txt$
Disallow: /*.sql$
Disallow: /*.bak$
Disallow: /*.old$
Disallow: /*.zip$
```

페이지 단위로 검색엔진 인덱싱을 막고 싶다면 HTML `<head>` 안에 다음 meta 태그를 넣을 수 있다.

```html
<meta name="robots" content="noindex, nofollow">
```

다만 `robots.txt`와 `meta robots`는 검색엔진 노출을 줄이는 보조 조치일 뿐이다. 공격자가 URL을 직접 요청하는 것을 막지는 못하므로, 민감 파일 삭제와 웹 서버 접근 차단이 우선이다.

### 5.5 주석과 디버그 출력 정리

개발 중 남긴 주석, `echo`, `var_dump`, `print_r`, 테스트 계정 출력 코드는 운영 웹루트에서 제거한다.

특히 다음 유형을 확인한다.

| 유형 | 예시 |
|---|---|
| 테스트 출력 | `echo "pw : ..."`, `echo "name : ..."` |
| 샘플 계정 | `admin / 123123` |
| DB 구조 파일 | `테이블 생성.txt`, `center(table).txt` |
| 임시 파일 | `*.txt`, `*.sql`, `*.bak`, `*.old`, `*.zip`, `*.log` |

## 7. 판단

현재 CARE는 `config.php`와 `config.local.php`를 통해 DB 접속정보를 분리하고 있으므로, **DB 접속정보를 코드 저장소에 직접 박아두는 문제는 상당 부분 조치된 상태**다.

하지만 웹에서 직접 접근 가능한 `dbconn.php`와 `member/테이블 생성.txt`가 샘플 계정, 비밀번호, 개인정보성 데이터, DB 구조를 그대로 노출하고 있다.

따라서 PDF p.708-710의 `정보 누출` 기준으로 보면 현재 상태는 취약으로 판단한다.

조치 후에는 Apache 설정으로 해당 파일명에 대한 직접 접근을 거부하여 `/dbconn.php` 요청이 사용자 정의 에러 페이지로 바뀌는 것을 확인했다.

본 항목의 핵심은 비밀값 파일 하나만 숨기는 것이 아니라, **웹루트에 남은 테스트 파일과 디버그 출력까지 제거하거나 접근 차단하여 공격자가 내부 구조를 추측할 단서를 줄이는 것**이다.
