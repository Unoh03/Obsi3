# 웹 취약점 - 16 불충분한 세션 관리

source: [[40_자료/주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드.pdf|주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드]], p.759-765.

## 1. 개요

**불충분한 세션 관리**는 로그인 후 사용자 식별에 쓰이는 세션 ID가 안전하게 발급·갱신·만료되는지 확인하는 항목이다.

PDF p.759의 판단 기준은 다음처럼 정리할 수 있다.

|판단|기준|
|---|---|
|양호|추측하기 어려운 세션 ID가 발급되고, 로그인 시 세션 ID가 갱신되며, 세션 종료 시간이 설정되어 있음|
|취약|세션 ID가 일정한 패턴이 있거나 고정되어 있고, 세션 종료 시간이 설정되지 않아 세션 재사용이 가능함|

세션 관리에서 중요한 점은 단순히 “로그인 기능이 있다”가 아니다. 로그인 이후 브라우저와 서버가 사용자를 구분할 때 사용하는 **세션 ID가 탈취·고정·재사용에 강한지**가 핵심이다.

```text
브라우저
-> 로그인 요청
-> 서버가 세션 ID 발급 또는 기존 세션 ID 사용
-> 이후 요청마다 PHPSESSID 쿠키로 로그인 상태 유지
```

CARE는 PHP 기본 세션을 사용한다. 따라서 이번 항목은 JWT가 아니라 **PHPSESSID 기반 세션 관리**를 중심으로 점검한다.

## 2. CARE 적용 가능성 분석

CARE에는 로그인, 로그아웃, 회원정보 수정, 게시판 작성 등 세션을 사용하는 기능이 존재한다.

확인한 주요 지점은 다음과 같다.

|파일|확인 내용|16번 관련 판단|
|---|---|---|
|`member/loginModel.php`|로그인 성공 시 `session_start()` 후 `$_SESSION` 값 저장|로그인 성공 시 세션 ID 재발급 여부 확인 필요|
|`header.php`|`session_start()` 후 `$_SESSION['id']` 존재 여부로 로그인 상태 표시|앱 전반이 PHP 세션 상태에 의존함|
|`member/logout.php`|`session_destroy()` 수행|쿠키 제거, 재로그인 시 세션 ID 변경 여부 확인 필요|
|전체 소스|조치 전에는 `session_regenerate_id`, `session_set_cookie_params` 검색 결과 없음|세션 고정 방지와 쿠키 보안 속성은 별도 확인 필요|
|전체 소스|JWT 관련 코드 없음|JWT 세부 점검은 해당 없음|

CARE 로그인 처리의 핵심 구조는 다음과 같다.

```php
session_start();

if($num == 1){
    $_SESSION['id'] = $id;
    $_SESSION['name'] = $row['name'];
    $_SESSION['mobile'] = $row['mobile'];
    $_SESSION['address'] = $row['address'];
    $_SESSION['email'] = $row['email'];
    $_SESSION['num'] = $row['num'];
}
```

여기서 확인해야 할 핵심은 로그인 성공 시 `session_regenerate_id(true)`로 기존 세션 ID를 폐기하고 새 세션 ID를 발급하는지다.

조치 전 코드 기준으로는 명시적인 세션 ID 재발급 로직이 보이지 않았다. 따라서 CARE의 16번은 원래 기능을 기준으로 점검 가능한 **A-2 기존 기능 악용형**으로 본다.

## 3. 문제가 되는 부분

조치 전 CARE 세션 관리에서 의심되는 부분은 다음 세 가지다.

|문제|설명|
|---|---|
|로그인 성공 시 세션 ID 재발급 없음|로그인 전 발급된 PHPSESSID가 로그인 후에도 유지될 가능성이 있음|
|앱 레벨 세션 만료 검사 없음|일정 시간 미사용 후 자동 로그아웃되는지 코드상 확인되지 않음|
|쿠키 보안 속성 설정 없음|`HttpOnly`, `Secure`, `SameSite` 같은 쿠키 속성은 PHP/Apache 기본값에 의존할 가능성이 있음|

다만 이 항목은 소스만 보고 최종 확정하면 안 된다. PHP 세션은 서버의 `php.ini`, Apache/PHP 모듈 설정, 브라우저 쿠키 동작에 영향을 받기 때문이다.

따라서 현재 판단은 다음처럼 둔다.

```text
조치 전 소스 기준:
- 세션 ID 재발급 코드 없음
- 앱 레벨 세션 만료 코드 없음
- JWT 사용 없음

조치 후 소스 기준:
- `member/loginModel.php`에 `session_regenerate_id(true)` 추가
- `member/loginModel.php`, `header.php`에 `last_activity` 기반 세션 만료 코드 추가
- `member/logout.php`에 세션 쿠키 제거 코드 추가

실행 검증 필요:
- 로그인 전/후 PHPSESSID가 실제로 바뀌는지
- 로그아웃 후 재로그인 시 PHPSESSID가 바뀌는지
- 일정 시간 방치 후 세션이 유지되는지
- 쿠키 속성이 어떻게 내려오는지
```

## 4. 악용 흐름

### 4.1 세션 고정 가능성

세션 고정은 공격자가 미리 알고 있는 세션 ID를 피해자가 로그인한 뒤에도 유지시키는 문제다.

일반적인 흐름은 다음과 같다.

```text
공격자 또는 테스트 사용자가 로그인 전 PHPSESSID를 확보
-> 같은 브라우저 세션으로 로그인
-> 로그인 후에도 PHPSESSID가 바뀌지 않음
-> 로그인 전 세션 ID가 인증 세션으로 승격됨
```

CARE에서 로그인 성공 후 세션 ID가 재발급되지 않으면, 로그인 전 세션과 로그인 후 인증 세션이 같은 PHPSESSID를 사용할 수 있다.

이 경우 세션 ID를 추측할 수 없더라도, **이미 정해진 세션 ID가 로그인 후에도 유지되는 구조**가 문제가 된다.

### 4.2 세션 만료 미흡

세션 만료가 없거나 너무 길면 사용자가 로그아웃하지 않고 자리를 비웠을 때, 같은 브라우저를 사용하는 다른 사람이 기존 로그인 상태를 그대로 사용할 수 있다.

```text
사용자 로그인
-> 일정 시간 사용하지 않음
-> 다시 보호 페이지 접근
-> 세션이 계속 유지됨
-> 자동 로그아웃이 적용되지 않음
```

PDF p.760은 일정 시간 경과 후 세션이 유지되는지 확인하라고 한다. CARE에는 별도 idle timeout 검사 코드가 확인되지 않으므로, 실제 브라우저에서 일정 시간 방치 후 확인해야 한다.

## 5. 점검 방법

PDF p.760-761의 점검 흐름은 크게 세션과 JWT로 나뉜다. CARE는 JWT를 사용하지 않으므로 세션 항목만 점검한다.

|PDF 항목|CARE에서의 처리|
|---|---|
|세션 ID 패턴 확인|PHPSESSID 값을 여러 번 비교|
|로그아웃 후 재로그인 시 세션 ID 변경 확인|로그인 전/후, 로그아웃 후 재로그인 시 쿠키 비교|
|일정 시간 경과 후 세션 만료 확인|방치 후 보호 페이지 접근|
|JWT 사용 여부 확인|CARE는 JWT 미사용, 해당 없음|
|JWT none/서명/만료 검증|CARE는 JWT 미사용, 해당 없음|

### 5.1 로그인 전 PHPSESSID 확인

브라우저 개발자도구 또는 Burp에서 로그인 전 쿠키를 확인한다.

확인 대상:

```text
Cookie: PHPSESSID=...
```

또는 서버/터미널에서 다음처럼 확인할 수 있다.

```bash
curl -i -c /tmp/care-cookie.txt http://172.168.10.10/member/login.php
cat /tmp/care-cookie.txt
```

### 5.2 로그인 후 PHPSESSID 변경 여부 확인

같은 쿠키 jar로 로그인 요청을 보낸다.

```bash
curl -i -b /tmp/care-cookie.txt -c /tmp/care-cookie.txt \
  -X POST \
  -d "id=테스트계정&pw=테스트비밀번호" \
  http://172.168.10.10/member/loginModel.php
```

그 다음 쿠키 값을 다시 확인한다.

```bash
cat /tmp/care-cookie.txt
```

판단 기준은 다음이다.

|결과|판단|
|---|---|
|로그인 전 PHPSESSID와 로그인 후 PHPSESSID가 다름|세션 고정 방지 측면에서 양호|
|로그인 전 PHPSESSID와 로그인 후 PHPSESSID가 같음|로그인 시 세션 ID 재발급이 없어 취약 가능성 있음|

브라우저 캡처를 할 경우에는 다음 두 화면을 남긴다.

- 로그인 전 `PHPSESSID`
    
- 로그인 성공 후 `PHPSESSID`
    

### 5.3 로그아웃 후 재로그인 시 세션 ID 변경 확인

로그아웃한다.

```bash
curl -i -b /tmp/care-cookie.txt -c /tmp/care-cookie.txt \
  http://172.168.10.10/member/logout.php
```

그 뒤 다시 로그인하여 PHPSESSID가 새로 바뀌는지 확인한다.

```bash
curl -i -b /tmp/care-cookie.txt -c /tmp/care-cookie.txt \
  -X POST \
  -d "id=테스트계정&pw=테스트비밀번호" \
  http://172.168.10.10/member/loginModel.php

cat /tmp/care-cookie.txt
```

이 단계는 PDF p.760의 “로그아웃 후 재로그인 시 각각 발급받은 세션 ID 비교”에 해당한다.

### 5.4 일정 시간 경과 후 세션 만료 여부 확인

로그인 후 일정 시간 동안 브라우저를 사용하지 않는다. 이번 실습에서는 `header.php`의 `$sessionTimeout = 300`에 맞춰 5분 방치 후 세션이 만료되는지 확인한다.

확인 대상 URL 예시:

```text
http://172.168.10.10/member/modify.php
http://172.168.10.10/center/write.php
```

판단 기준은 다음이다.

|결과|판단|
|---|---|
|일정 시간 후 로그인 페이지로 이동|세션 만료 또는 접근 차단 확인|
|일정 시간 후에도 보호 페이지 접근 가능|앱 레벨 세션 만료가 없거나 서버 기본 세션 시간이 길 가능성 있음|

### 5.5 쿠키 속성 확인

응답 헤더의 `Set-Cookie`를 확인한다.

```bash
curl -i http://172.168.10.10/member/login.php | grep -i set-cookie
```

확인할 속성은 다음이다.

|속성|의미|
|---|---|
|`HttpOnly`|JavaScript에서 세션 쿠키 접근을 어렵게 함|
|`Secure`|HTTPS에서만 쿠키 전송|
|`SameSite`|CSRF성 교차 사이트 요청에서 쿠키 전송 제한|
|`Expires` 또는 `Max-Age`|쿠키 만료 시간|

현재 프로젝트가 HTTP 기반이면 `Secure` 적용은 제한될 수 있다. 이 경우 “HTTPS 미구성으로 Secure 쿠키 적용은 보류”라고 분리해서 쓴다.

## 6. 조치 방안

PDF p.762-765의 조치 방향을 CARE에 맞추면 다음과 같다. 이번 프로젝트에서는 전체 구조 개편이 아니라 **최소 조치**만 적용한다.

| PDF 조치 | CARE 적용 방식 | 적용 파일 |
|---|---|
| PDF 조치 1. 세션 고정 및 예측 방지 | 로그인 성공 시 기존 세션 ID 폐기 후 새 세션 ID 발급 | `member/loginModel.php` |
| PDF 조치 2. 세션 만료 | 마지막 활동 시간을 저장하고 일정 시간 초과 시 자동 로그아웃 | `member/loginModel.php`, `header.php`, `member/logout.php` |
| PDF 조치 3. JWT 안전성 | CARE는 JWT를 사용하지 않으므로 해당 없음 | 해당 없음 |

### PDF 조치 1. 세션 고정 및 예측 방지

PDF p.762-763은 새 로그인 시 기존 세션 ID를 폐기하고 새로운 세션 ID를 발급하라고 한다. CARE에서는 로그인 성공 지점이 `member/loginModel.php`이므로, 로그인 성공 직후 `session_regenerate_id(true)`를 넣는다.

```php
session_start();

if($num == 1){
    // 웹 취약점 16번 PDF 조치 1: 새 로그인 시 기존 세션 ID를 폐기하고 새로운 세션 ID를 발급한다.
    session_regenerate_id(true);

    $_SESSION['id'] = $id;
    $_SESSION['name'] = $row['name'];
    $_SESSION['mobile'] = $row['mobile'];
    $_SESSION['address'] = $row['address'];
    $_SESSION['email'] = $row['email'];
    $_SESSION['num'] = $row['num'];
}
```

이 조치는 로그인 전 세션 ID가 로그인 후 인증 세션으로 그대로 승격되는 것을 막는다.

> [!note] PDF의 PHP 예시 범위
> PDF p.763에는 `session.entropy_length`, `session.entropy_file` 예시도 나온다. 이번 CARE 최소 조치에서는 PHP 전체 세션 설정을 건드리지 않고, 앱 코드에서 직접 확인 가능한 `session_regenerate_id(true)`만 적용한다.

### PDF 조치 2. 세션 만료

PDF p.763-764는 세션 타임아웃을 설정하고 자동 로그아웃을 구현하라고 한다. CARE에서는 별도 공통 세션 유틸 파일을 새로 만들지 않고, 현재 대부분의 화면에서 포함되는 `header.php`에 최소 timeout 검사를 넣는다.

먼저 `member/loginModel.php`에서 로그인 성공 시 마지막 활동 시간을 저장한다.

```php
if($num == 1){
    // 웹 취약점 16번 PDF 조치 1: 새 로그인 시 기존 세션 ID를 폐기하고 새로운 세션 ID를 발급한다.
    session_regenerate_id(true);

    $_SESSION['id'] = $id;
    $_SESSION['name'] = $row['name'];
    $_SESSION['mobile'] = $row['mobile'];
    $_SESSION['address'] = $row['address'];
    $_SESSION['email'] = $row['email'];
    $_SESSION['num'] = $row['num'];
    // 웹 취약점 16번 PDF 조치 2: 세션 만료 판단을 위해 마지막 활동 시간을 기록한다.
    $_SESSION['last_activity'] = time();
}
```

그 다음 `header.php` 맨 위에서 세션 만료를 검사한다.

```php
<?php
// 웹 취약점 16번 PDF 조치 2: 세션 종료 시간 설정 및 자동 로그아웃 구현.
session_start();

$sessionTimeout = 300; // 실습용 5분

if (!empty($_SESSION['id'])) {
    if (isset($_SESSION['last_activity']) && time() - $_SESSION['last_activity'] > $sessionTimeout) {
        $_SESSION = [];
        session_destroy();

        echo "<script>alert('세션이 만료되었습니다. 다시 로그인하세요.'); location.href='/member/login.php';</script>";
        exit;
    }

    $_SESSION['last_activity'] = time();
}
?>
```

기존 로그인 상태 확인 코드도 `$_SESSION['id']`를 바로 읽지 않고 `empty()`로 바꾼다.

```php
if(empty($_SESSION['id'])){
```

마지막으로 `member/logout.php`에서 서버 세션뿐 아니라 브라우저의 세션 쿠키도 함께 폐기한다.

```php
<?php
session_start();

// 웹 취약점 16번 PDF 조치 2: 로그아웃 시 서버 세션과 브라우저의 세션 쿠키를 함께 폐기한다.
$_SESSION = [];

if (ini_get("session.use_cookies")) {
    $params = session_get_cookie_params();
    setcookie(session_name(), '', time() - 42000,
        $params["path"],
        $params["domain"],
        $params["secure"],
        $params["httponly"]
    );
}

session_destroy();
?>
```

이번 최소 조치에서는 `session_set_cookie_params()`로 `Secure`, `HttpOnly`, `SameSite`를 전역 설정하는 방식은 적용하지 않는다. CARE는 여러 파일에서 직접 `session_start()`를 호출하고 있어서, 쿠키 속성까지 제대로 적용하려면 공통 세션 초기화 파일로 구조를 정리하는 편이 안전하기 때문이다.

### PDF 조치 3. JWT 안전성

PDF p.765는 JWT 서명 알고리즘, `none` 알고리즘, 만료 시간, 비밀 키 관리를 다룬다. CARE는 JWT를 사용하지 않고 PHP `PHPSESSID` 기반 세션을 사용하므로 이번 프로젝트에서는 해당 없음으로 처리한다.

## 7. 조치 후 점검

조치 후에는 다음을 확인한다.

|점검|기대 결과|
|---|---|
|로그인 전/후 PHPSESSID 비교|로그인 성공 후 PHPSESSID가 변경됨|
|로그아웃 후 재로그인|이전 인증 세션 ID가 재사용되지 않음|
|일정 시간 미사용 후 보호 페이지 접근|로그인 페이지로 이동하거나 세션 만료 메시지 출력|
|쿠키 속성 확인|가능한 범위에서 `HttpOnly`, `SameSite`, HTTPS 환경에서는 `Secure` 확인|
|정상 로그인 회귀 확인|세션 보안 조치 후에도 정상 로그인/로그아웃 가능|

### 세션 만료

-  `header.php`에 `$sessionTimeout = 300`을 적용하여 5분 세션 만료 기준을 설정했다.
-  5분 방치 후 보호 페이지 접근 캡처는 미수집 상태로 남긴다.
-  따라서 PDF 조치 2는 코드 적용까지 확인했고, 실행 화면 증거는 후속 보강 대상으로 둔다.

> [!note] 세션 만료 타이머
> 현재 `header.php`의 `$sessionTimeout` 값은 `300`초다. 즉 실습용 세션 만료 기준은 **5분**이다.
    

### 쿠키 속성

-  쿠키 속성 캡처는 이번 증거 묶음에 포함하지 않았다.
-  현재 VM 실습 환경은 HTTP 기반이므로 `Secure` 속성 검증은 HTTPS 구성 이후 별도 항목에서 다루는 편이 맞다.
-  16번의 핵심 증거는 세션 ID 재발급과 세션 만료 정책 적용 여부로 정리한다.
    

### 조치 코드

-  `member/loginModel.php`의 `PDF 조치 1` 주석과 `session_regenerate_id(true)` 적용 코드
    
-  `member/loginModel.php`, `header.php`의 `PDF 조치 2` 주석과 `last_activity` 기반 timeout 코드
    
-  `member/logout.php`의 `PDF 조치 2` 주석과 세션 쿠키 제거 코드
    

## 9. 판단

|항목|판단|
|---|---|
|1차 분류|A-2|
|판정 상태|부분 검증 완료|
|조치 상태|최소 조치 코드 적용. PDF 조치 1은 `member/loginModel.php`, PDF 조치 2는 `member/loginModel.php`, `header.php`, `member/logout.php`에 반영|
|증거 상태|로그인 전/후 세션 ID 변경과 로그아웃 후 재로그인 세션 ID 변경 증거 확보. 세션 만료 5분 대기 증거는 미수집|

CARE는 원래 로그인 기능과 PHP 세션 기반 인증 상태 유지 기능을 사용한다. 따라서 16번은 실습용 기능을 새로 만들 필요가 없는 항목이다.

조치 전 소스에는 로그인 성공 시 세션 ID를 재발급하는 코드가 확인되지 않았고, 앱 레벨 세션 만료 검사도 확인되지 않았다. 조치 후에는 로그인 성공 시 `session_regenerate_id(true)`가 실행되고, 실제 캡처에서도 로그인 전 `PHPSESSID`와 로그인 후 `PHPSESSID`가 달라진 것을 확인했다.

또한 로그아웃 후 재로그인 시에도 이전 로그인 세션 ID와 다른 `PHPSESSID`가 확인되었다. 따라서 PDF 조치 1의 “새 로그인 시 기존 세션 ID 폐기 및 새 세션 ID 발급”은 현재 증거로 확인된다.

다만 `header.php`의 `$sessionTimeout = 300`에 따른 5분 세션 만료 동작은 아직 별도 캡처가 없다. 따라서 PDF 조치 2는 코드 적용까지 확인했고, 실제 만료 화면 증거는 추가 수집이 필요하다.

JWT 관련 점검은 CARE가 JWT를 사용하지 않으므로 PDF 조치 3은 해당 없음으로 처리한다.

최종적으로 이 항목은 **PHP 세션 ID 재발급과 세션 만료 정책을 중심으로 검증하는 A-2 기존 기능 악용형 항목**으로 본다.

## AWS 후속 조치

현재 VM/GNS 환경에서는 HTTPS 기반 `Secure` 쿠키 검증까지 완전히 닫지 않았다. `HttpOnly`와 `SameSite`는 HTTP 상태에서도 확인 가능하지만, `Secure`는 HTTPS 전송 경로가 정상 구성되어야 의미 있게 검증할 수 있다.

따라서 16번의 쿠키 보안 속성 최종 검증은 [[AWS 후속 조치와 재검증|AWS 후속 조치와 재검증]]에서 ALB/ACM 기반 HTTPS 구성 후 이어서 진행한다.
