# 웹 취약점 - 15 파일 다운로드

source: [[40_자료/주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드.pdf|주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드]], p.753-758.

## 1. 개요

**파일 다운로드 취약점**은 다운로드 기능의 파일명 또는 경로 파라미터를 조작해, 원래 허용된 다운로드 디렉터리 밖의 파일을 열람하거나 다운로드할 수 있는 취약점이다.

PDF p.753의 판단 기준은 다음처럼 정리할 수 있다.

| 판단 | 기준 |
|---|---|
| 양호 | 입력값 검증으로 허용된 경로와 파일만 접근 가능하고, 상위 디렉터리 접근이나 임의 파일 다운로드가 차단됨 |
| 취약 | 다운로드 파라미터가 검증 없이 처리되어 임의 경로 접근 또는 비인가 파일 다운로드가 가능함 |

CARE에는 게시글 첨부파일 다운로드 기능이 있다. `download.php`는 로그인 여부만 확인한 뒤 `filename` 파라미터를 그대로 `../data/` 뒤에 붙여 파일을 연다. 따라서 15번은 실습용 새 기능을 만들 필요가 없는 **A-1 자연 취약**으로 본다.

## 2. 문제가 되는 부분

게시글 보기 화면은 DB에 저장된 첨부파일명을 다운로드 링크로 출력한다.

```php
<a href="download.php?filename=<?=$filename?>"><?=$filename?> </a>
```

다운로드 처리 파일인 `center/download.php`는 다음 흐름으로 동작한다.

```php
$filename = $_GET['filename'];
header("content-disposition: attachment; filename = " . $filename);

$filename = "../data/" . $filename;
$handle = fopen($filename, "r");
fpassthru($handle);
fclose($handle);
```

현재 코드에서 확인되는 문제는 다음과 같다.

| 문제 | 설명 |
|---|---|
| 사용자 입력 직접 사용 | `$_GET['filename']`을 검증 없이 사용함 |
| 경로 이동 문자열 차단 없음 | `../`, `..\`, URL 인코딩 우회 등을 차단하지 않음 |
| 정규화 검증 없음 | `realpath()`로 최종 경로가 다운로드 디렉터리 안인지 확인하지 않음 |
| DB 등록 파일 대조 없음 | 요청 파일명이 실제 게시글 첨부파일로 등록된 값인지 확인하지 않음 |
| 헤더 파일명 검증 없음 | `Content-Disposition`에도 사용자 입력값을 그대로 넣음 |

즉, 원래 의도는 `../data/첨부파일명`만 다운로드하는 것이지만, 실제 코드는 `filename=../다른파일` 같은 입력으로 `data` 밖의 파일을 열 가능성이 있다.

## 3. 악용 흐름

일반적인 공격 흐름은 다음과 같다.

```text
로그인
-> 첨부파일이 있는 게시글 조회
-> 다운로드 링크의 filename 파라미터 확인
-> filename 값을 ../fd-proof.txt 같은 상대 경로로 변조
-> 서버가 ../data/../fd-proof.txt를 열어 data 밖 파일을 반환
```

원칙적으로는 실제 비밀 파일을 증거로 사용하지 않고, WEB 서버에 무해한 proof 파일을 하나 만들어 다운로드 디렉터리 밖의 파일을 읽을 수 있는지만 확인하는 편이 안전하다.

```bash
echo "fd-proof-$(date +%s)" | sudo tee /var/www/html/care/fd-proof.txt
sudo chmod 644 /var/www/html/care/fd-proof.txt
```

취약 상태에서는 다음 URL로 proof 파일이 다운로드되거나 화면에 출력될 수 있다.

```text
http://172.168.10.10/center/download.php?filename=../fd-proof.txt
```

이 방식은 `/etc/passwd`, `config.local.php` 같은 민감 파일을 직접 열지 않고도, 허용된 다운로드 디렉터리 밖으로 빠져나갈 수 있음을 증명한다.

이번 실습 캡처에서는 PDF가 예시로 언급한 DB 연결 파일 유형에 맞춰 `../dbconn.php` 다운로드를 확인했다. 단, 보고서에는 파일 내용이나 실제 접속 정보는 싣지 않고, 다운로드가 발생했다는 사실만 증거로 사용한다.

## 4. 점검 방법

PDF p.754-756의 점검 방법은 크게 세 단계다.

| PDF 단계 | CARE에서의 확인 |
|---|---|
| Step 1 | 파일 다운로드 기능을 식별하고 `filename` 파라미터가 노출되는지 확인 |
| Step 2 | `../`, `..\` 같은 상대 경로로 파일 경로를 변조해 다운로드되는지 확인 |
| Step 3 | 단순 경로 이동이 막힐 경우 URL 인코딩, 더블 인코딩, 중첩 문자, 종단 문자 등 우회 가능성을 확인 |

CARE에서는 다음 순서로 확인한다.

### 4.1 정상 다운로드 확인

1. 로그인한다.
2. 첨부파일이 있는 게시글을 연다.
3. 다운로드 링크를 클릭한다.
4. 정상 첨부파일이 다운로드되는지 확인한다.

확인 URL 예시는 다음과 같다.

```text
http://172.168.10.10/center/download.php?filename=업로드파일명
```

### 4.2 경로 조작 확인

WEB 서버에서 무해한 proof 파일을 만든다.

```bash
echo "fd-proof-$(date +%s)" | sudo tee /var/www/html/care/fd-proof.txt
sudo chmod 644 /var/www/html/care/fd-proof.txt
cat /var/www/html/care/fd-proof.txt
```

그 다음 브라우저에서 다음 URL을 요청한다.

```text
http://172.168.10.10/center/download.php?filename=../fd-proof.txt
```

proof 파일 내용이 다운로드되거나 응답에 포함되면, `../data/` 밖의 파일에 접근한 것이므로 취약하다.

### 4.3 우회 입력 확인

필요하면 다음 변형도 확인한다.

```text
../fd-proof.txt
..%2ffd-proof.txt
.%2e/fd-proof.txt
....//fd-proof.txt
```

다만 이번 프로젝트에서는 모든 우회 기법을 깊게 파기보다, **최소 하나의 경로 조작으로 다운로드 디렉터리 밖 파일을 읽을 수 있음**을 증명하는 것이 우선이다.

## 5. 조치 방안

PDF p.756-758의 조치 방향은 다음처럼 정리된다.

| 조치 | CARE 적용 방식 |
|---|---|
| DB에 저장된 파일명과 요청 파일명 비교 | `center.filename`에 존재하는 파일명만 다운로드 허용 |
| 다운로드 가능 디렉터리 제한 | `realpath()`로 최종 파일 경로가 `data` 디렉터리 안인지 확인 |
| URL에 실제 경로 노출 금지 | 가능하면 파일명 대신 게시글 번호나 파일 ID를 사용 |
| 파일명 화이트리스트 | 영문, 숫자, `.`, `_`, `-` 정도만 허용 |
| 경로 추적 문자 차단 | `/`, `\`, `%`, null byte, `..` 등을 차단 |

이번 CARE 프로젝트에서는 구조 변경이 큰 파일 ID 방식까지 바로 바꾸지는 않는다. 대신 `download.php`에 입력값 검증, 경로 정규화, DB 등록 파일 대조를 적용한다. 취약 상태 캡처 전에는 방어 코드를 비활성 상태로 두고, 조치 후에는 방어 블록을 활성화해 경로 이탈 요청이 차단되는지 확인한다.

### 5.1 입력값 검증

파일명이 비어 있거나 경로 조작 문자가 포함되면 차단한다.

```php
$requested = $_GET['filename'] ?? '';

if($requested === '' || preg_match('/[\/\\\\%\\x00]/', $requested) || strpos($requested, '..') !== false){
    http_response_code(400);
    echo 'invalid file name';
    exit;
}

if(! preg_match('/^[A-Za-z0-9._-]+$/', $requested)){
    http_response_code(400);
    echo 'invalid file name';
    exit;
}
```

### 5.2 경로 정규화와 디렉터리 고정

`realpath()`로 최종 경로를 계산한 뒤, 그 경로가 반드시 업로드 디렉터리 안에 있어야 한다.

```php
$downloadDir = realpath(__DIR__ . '/../data');
$filePath = realpath($downloadDir . DIRECTORY_SEPARATOR . $requested);

if($downloadDir === false || $filePath === false || strpos($filePath, $downloadDir . DIRECTORY_SEPARATOR) !== 0){
    http_response_code(404);
    echo 'file not found';
    exit;
}
```

### 5.3 DB 등록 파일 대조

요청한 파일명이 실제 게시글 첨부파일로 등록되어 있는지도 확인한다.

```php
require_once __DIR__ . '/../config.php';
$link = care_db_connect() or die('연결 실패');

$stmt = mysqli_prepare($link, 'SELECT COUNT(*) FROM center WHERE filename = ?');
mysqli_stmt_bind_param($stmt, 's', $requested);
mysqli_stmt_execute($stmt);
mysqli_stmt_bind_result($stmt, $fileCount);
mysqli_stmt_fetch($stmt);
mysqli_stmt_close($stmt);
mysqli_close($link);

if((int)$fileCount < 1){
    http_response_code(403);
    echo 'download denied';
    exit;
}
```

이 검사는 `data` 디렉터리 안에 파일이 있더라도, 게시글 첨부파일로 등록되지 않은 파일을 임의로 다운로드하지 못하게 하는 보조 방어다.

## 6. 조치 후 점검

조치 후에는 다음을 확인한다.

| 점검 | 기대 결과 |
|---|---|
| 정상 첨부파일 다운로드 | 정상 다운로드 |
| `filename=../fd-proof.txt` | 차단 |
| `filename=..%2ffd-proof.txt` | 차단 |
| `filename=....//fd-proof.txt` | 차단 |
| `filename=../config.php` | 차단 |
| DB에 등록되지 않은 `data` 내부 파일 | 차단 |

실습 순서는 조치 전 증거를 먼저 확보한 뒤 방어 코드를 활성화하고 다시 확인한다. 현재 `D:\care\center\download.php`는 방어 블록이 활성화되어 있고, 취약 원본 코드는 파일 하단에 주석으로 보존되어 있다.

### 7-1. PDF Step 1: 다운로드 기능과 파라미터 식별

### 7-2. PDF Step 2: 상대 경로 변조로 허용 경로 밖 파일 다운로드

다운로드 요청의 `filename` 파라미터에 `../dbconn.php`를 넣었다.

![[Pasted image 20260616152041.png]]

이 요청은 서버 내부에서 다음처럼 해석된다.

```text
../data/../dbconn.php
```

`../`가 상위 디렉터리로 이동시키기 때문에, 결과적으로 `data/` 밖의 `dbconn.php`를 가리킨다.

![[Pasted image 20260616152059.png]]

브라우저 다운로드 목록에 `_dbconn.php`가 생겼으므로, 조치 전에는 다운로드 기능을 통해 허용된 다운로드 디렉터리 밖의 파일을 내려받을 수 있었다.

> [!warning]
> `dbconn.php`는 PDF p.753에서 예시로 든 DB 연결 파일 유형에 해당한다. 증거로는 다운로드 발생 사실만 사용하고, 파일 내용이나 실제 접속 정보는 노트에 적지 않는다.

### 7-3. 조치 후: 경로 추적 입력 차단

방어 코드를 활성화한 뒤 같은 요청을 보내면 `invalid file name`이 출력되었다.

![[Pasted image 20260616152431.png]]

이는 `filename`에 포함된 `/`, `\`, `%`, 널바이트, `..` 같은 경로 추적 입력을 차단하는 조치가 적용된 결과다.

### 7-4. 조치 후: 정상 다운로드 회귀 확인

조치 후에도 DB에 등록되어 있고 `data/` 디렉터리 안에 존재하는 정상 첨부파일은 다운로드되었다.

![[Pasted image 20260616153231.png]]

여기서 중요한 점은 `download.php`가 파일을 PHP로 실행하는 것이 아니라, DB에 등록된 첨부파일을 `readfile()`로 내려준다는 것이다. 그래서 14번의 `/data/14_fake_webshell_proof.php` 직접 접근 차단과, 15번의 `download.php?filename=14_fake_webshell_proof.php` 다운로드 허용은 서로 모순이 아니다.

### 7-5. 증거 정리

| 확인 항목 | 결과 | 증거 |
|---|---|---|
| PDF Step 1. 다운로드 기능과 파일명 파라미터 식별 | 확인 | `download.php?filename=...` URL |
| PDF Step 2. 상대 경로 변조 | 조치 전 취약 | `filename=../dbconn.php` 요청 |
| PDF Step 2. 허용 경로 밖 파일 다운로드 | 조치 전 취약 | `_dbconn.php` 다운로드 발생 |
| PDF Step 3. 인코딩 우회 | 미수행 | Step 2가 이미 성공했으므로 필수 증거는 아님 |
| 조치 코드. 입력값 검증 | 조치 후 확인 | `invalid file name` 응답 |
| 조치 코드. 정상 파일 회귀 | 조치 후 확인 | DB 등록 첨부파일 정상 다운로드 |
| 잔여 한계 | 부분 보류 | URL에 실제 파일명이 노출되는 구조는 아직 남아 있음 |

## 8. 판단

| 항목 | 판단 |
|---|---|
| 1차 분류 | A-1 |
| 판정 상태 | 확정 |
| 조치 상태 | 입력값 검증, `realpath()` 경로 고정, DB 등록 파일 대조 적용 |
| 증거 상태 | 조치 전 경로 이탈 다운로드 증거 확보, 조치 후 차단 및 정상 다운로드 회귀 확인 |

CARE 원본에는 게시글 첨부파일 다운로드 기능이 있고, `center/download.php`는 로그인 여부만 확인한 뒤 `filename` 파라미터를 검증 없이 파일 경로에 결합한다.

실습에서는 `filename=../dbconn.php` 요청으로 `data/` 디렉터리 밖의 파일이 다운로드되는 것을 확인했다. 이는 PDF p.753의 취약 기준인 “입력값이 검증 없이 처리되어 임의 경로 접근 또는 비인가 파일 다운로드가 가능한 경우”에 해당한다. 따라서 15번은 **A-1 자연 취약**으로 확정한다.

조치 후에는 같은 경로 조작 요청이 `invalid file name`으로 차단되었고, DB에 등록된 정상 첨부파일은 계속 다운로드되었다. 다만 PDF 조치 3의 “파일 ID나 토큰을 사용하여 파일 경로 은닉”까지는 적용하지 않았으므로, 최종 구조 개선 여지는 남아 있다.
