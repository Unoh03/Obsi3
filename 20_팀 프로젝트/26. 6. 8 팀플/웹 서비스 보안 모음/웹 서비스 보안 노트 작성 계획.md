# 웹 서비스 보안 노트 작성 계획

## 1. 목적

이 문서는 PDF p.271부터 시작되는 **웹 서비스 보안 진단 항목**을 노트화하기 위한 작성 계획과 공통 템플릿을 정리한다.

기존 웹 애플리케이션 취약점 노트는 CARE 애플리케이션의 PHP 코드와 기능을 중심으로 분석했다. 반면 웹 서비스 보안 항목은 Apache 웹서버 설정, 파일 권한, 모듈, 헤더, 로그, SSL/TLS 등 서버 운영 설정을 중심으로 점검한다.

따라서 웹 서비스 보안 노트는 다음 기준으로 작성한다.

- PDF 기준을 먼저 정리한다.
- `AWS WEB 약점 진단 결과.md`의 실제 진단 결과를 기준 자료로 사용한다.
- N/A 항목을 제외한 모든 항목은 개별 노트로 작성한다.
- 양호 항목도 PDF 내용을 실습하기 위해 의도적으로 취약하게 만든 뒤, 다시 조치하고 복구한다.
- N/A 항목은 개별 노트로 만들지 않고, 하나의 묶음 노트에서 왜 N/A인지 설명한다.
- 웹 애플리케이션 취약점 노트에서 사용하던 A/B/C/D 분류는 사용하지 않는다.

## 2. 상태 표현 기준

웹 서비스 보안 노트에서는 A/B/C/D 분류를 사용하지 않는다. 팀 내 혼선을 줄이기 위해 진단 결과 중심으로 표현한다.

| 상태 | 의미 |
|---|---|
| 취약 | 최초 진단 또는 수동 확인에서 실제 취약 설정이 확인됨 |
| 양호 | 최초 진단에서 안전한 설정으로 확인됨 |
| 수동 | 자동 진단만으로 확정하기 어려워 직접 명령어, 요청, 설정 확인이 필요함 |
| N/A | 현재 Apache/PHP 기반 CARE 서버 환경에 직접 적용되지 않음 |
| 확인 필요 | 아직 증거가 부족해 최종 판단을 보류함 |
| 조치 완료 | 조치 후 재점검에서 기대 결과를 확인함 |

## 3. 공통 작성 원칙

### 3-1. 최초 상태와 실습 상태 분리

양호 항목을 일부러 취약하게 만들 경우, 다음 세 상태를 반드시 구분한다.

| 구분 | 설명 |
|---|---|
| 최초 진단 상태 | `AWS WEB 약점 진단 결과.md` 기준 상태 |
| 취약 재현 상태 | PDF 내용을 실습하기 위해 의도적으로 약화한 상태 |
| 조치 후 상태 | 취약 재현 후 PDF 조치 기준으로 복구한 상태 |

예시는 다음과 같다.

| 항목 | 예시 |
|---|---|
| 최초 진단 | WEB-05 CGI/ISAPI 실행 제한: 양호 |
| 취약 재현 | CGI 실행 범위를 의도적으로 넓힘 |
| 조치 후 | CGI 실행 디렉터리를 다시 제한함 |

### 3-2. Apache 설정 변경 안전 규칙

Apache 설정을 변경하는 항목은 다음 순서를 따른다.

1. 원본 설정 파일 백업
2. 취약 설정 적용 또는 기존 취약 상태 확인
3. `apachectl configtest` 수행
4. Apache reload
5. 취약 증거 수집
6. 조치 설정 적용
7. `apachectl configtest` 재수행
8. Apache reload
9. 조치 후 증거 수집

기본 명령 흐름은 다음과 같다.

```bash
sudo cp /etc/apache2/apache2.conf /etc/apache2/apache2.conf.bak.WEB-XX
sudo apachectl configtest
sudo systemctl reload apache2
```

설정 변경 후 `Syntax OK`를 확인하지 않고 reload하지 않는다.

### 3-3. GitHub에 올리는 것과 서버에 배치하는 것 구분

GitHub에는 노트, 실습용 소스 원본, 설정 예시, 배포 스크립트를 올린다.

서버의 실제 배치 결과는 GitHub에 그대로 올리지 않는다.

| GitHub에 올릴 수 있음 | GitHub에 그대로 올리지 않음 |
|---|---|
| 노트 | `/etc/apache2/...` 실제 설정 파일 |
| 실습용 소스 원본 | 실제 인증서와 개인키 |
| 설정 예시 | 실제 운영 secret |
| 배포 스크립트 | 서버 런타임 디렉터리 전체 |

```markdown
# Apache 환경에서 N/A 처리한 WEB 항목
```

### 5-2. 구성

1. 개요
2. N/A 항목 목록
3. 각 항목별 N/A 사유
4. 현재 Apache/PHP 환경 기준 판단
5. 실습 제외 근거

### 5-3. N/A 항목 표

| 항목 | 항목명 | N/A 사유 |
|---|---|---|
| WEB-01 | Default 관리자 계정명 변경 | Tomcat/IIS/JEUS 대상 항목으로 Apache/PHP CARE 환경에 직접 적용되지 않음 |
| WEB-02 | 취약한 비밀번호 사용 제한 | Tomcat/IIS/JEUS 관리자 계정 기준 항목으로 Apache/PHP CARE 환경에 직접 적용되지 않음 |
| WEB-03 | 비밀번호 파일 권한 관리 | Tomcat/IIS/JEUS 계정 파일 또는 제품별 설정 파일 기준 항목으로 Apache/PHP CARE 환경에 직접 적용되지 않음 |
| WEB-13 | 웹 서비스 설정 파일 노출 제한 | 진단 결과 기준 WEB-04와 중복 또는 Apache 환경 직접 적용 지점 없음으로 N/A 처리 |

### 5-4. N/A 판단 문장

N/A 항목은 취약하거나 양호한 것이 아니다. 현재 실습 환경이 Apache + PHP + CARE 웹 애플리케이션이므로, Tomcat Manager, JEUS 관리자 계정, IIS 전용 설정처럼 특정 제품군에 종속된 항목은 직접 실습하지 않는다.

## 6. 항목별 처리 계획

`AWS WEB 약점 진단 결과.md` 기준 26개 항목을 다음처럼 처리한다.

### 6-1. 취약 항목

취약 항목은 개별 노트에서 최초 취약 상태를 증거화하고, PDF 조치 방법에 따라 조치 후 재점검한다.

| 항목 | 처리 방향 |
|---|---|
| WEB-04 웹서비스 디렉터리 리스팅 방지 설정 | `Options Indexes` 확인 후 `-Indexes` 조치 |
| WEB-12 웹서비스 링크 사용 금지 | `FollowSymLinks` 확인 후 `SymLinksIfOwnerMatch` 또는 `-FollowSymLinks` 조치 |
| WEB-20 SSL/TLS 활성화 | 진단 스크립트 오탐 가능성 포함, 수동 확인 후 판단 |
| WEB-22 웹서비스 에러페이지 사용 | `ErrorDocument` 설정 추가 후 기본 에러 페이지 노출 제한 |

### 6-2. 양호 항목

양호 항목도 개별 노트로 작성한다. 최초 상태는 양호로 기록하되, PDF 내용을 실습하기 위해 의도적으로 취약 설정을 만든 뒤 다시 복구한다.

| 항목 | 처리 방향 |
|---|---|
| WEB-05 CGI/ISAPI 실행 제한 | CGI 실행 범위 약화 후 제한 복구 |
| WEB-07 불필요한 파일 제거 | 샘플/백업 파일 생성 후 제거 |
| WEB-08 .htaccess 오버라이드 제한 | `AllowOverride All`로 약화 후 `None` 복구 |
| WEB-09 웹서비스 프로세스 권한 제한 | 프로세스 사용자 확인, root 실행 위험 설명 중심 |
| WEB-10 불필요한 프록시 설정 제한 | 프록시 설정 예시로 위험 설명 후 미사용 상태 확인 |
| WEB-11 웹서비스 경로 설정 | 기본 경로와 분리 경로 비교, 현재 DocumentRoot 확인 |
| WEB-14 웹서비스 경로 내 파일 접근 통제 | `Require all granted/denied` 비교 실습 |
| WEB-15 불필요한 스크립트 매핑 제거 | 불필요 핸들러 매핑 예시와 제거 방식 정리 |
| WEB-16 웹서버 헤더 정보 노출 제한 | `ServerTokens`, `ServerSignature` 약화 후 복구 |
| WEB-17 웹서비스 가상 디렉터리 삭제 | Alias 예시와 미사용 상태 확인 |
| WEB-18 WebDAV 비활성화 | DAV 설정 활성/비활성 비교 |
| WEB-19 WebDAV 모듈 미로드 | `mod_dav` 로드 여부 확인, WebDAV와 묶어 설명 가능 |
| WEB-26 로그 디렉터리 및 파일 권한 설정 | 로그 권한 약화 예시 후 640/750 복구 |

### 6-3. 수동 진단 항목

수동 진단 항목은 자동 진단 결과만으로 확정하지 않는다. 각 항목별 수동 명령어와 HTTP 요청을 추가해 판단한다.

| 항목 | 처리 방향 |
|---|---|
| WEB-06 상위 디렉터리 접근 제한 | `../` 요청, Apache Directory 설정 확인 |
| WEB-21 동적 페이지 입력값 검증 | CARE 앱 입력값 검증과 기존 웹 앱 취약점 노트 연결 |
| WEB-23 HTTP 메서드 제한 | `OPTIONS`, `TRACE`, `PUT`, `DELETE` 요청 결과 확인 |
| WEB-24 X-Frame-Options 헤더 | HTTP 응답 헤더 확인 후 추가 설정 |
| WEB-25 X-XSS-Protection 헤더 | HTTP 응답 헤더 확인 후 추가 설정, 최신 브라우저 한계 설명 |

### 6-4. N/A 항목

N/A 항목은 개별 노트로 작성하지 않고 하나의 묶음 노트로 처리한다.

| 항목 | 처리 방향 |
|---|---|
| WEB-01 | Tomcat/IIS/JEUS 대상이므로 Apache 환경 N/A |
| WEB-02 | Tomcat/IIS/JEUS 대상이므로 Apache 환경 N/A |
| WEB-03 | Tomcat/IIS/JEUS 대상이므로 Apache 환경 N/A |
