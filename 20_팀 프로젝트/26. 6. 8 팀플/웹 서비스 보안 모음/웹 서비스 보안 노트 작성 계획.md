# 웹 서비스 보안 노트 작성 계획

## 1. 작성 범위

이 문서는 PDF p.271부터 시작되는 **웹 서비스 보안 진단 항목**을 노트화하기 위한 작성 계획과 공통 템플릿을 정리한다.

기존에 작성한 웹 애플리케이션 취약점 노트는 CARE 애플리케이션의 PHP 코드와 기능을 중심으로 분석했다. 반면 웹 서비스 보안 항목은 Apache 웹서버 설정, 파일 권한, 모듈, 헤더, SSL/TLS, 로그 권한 등 서버 환경을 중심으로 다룬다.

따라서 웹 서비스 보안 노트는 다음 자료를 함께 사용한다.

| 자료 | 용도 |
|---|---|
| 주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드 p.271 이후 | 공식 점검 기준과 조치 기준 |
| AWS WEB 약점 진단 결과.md | 현재 AWS Apache 서버의 실제 진단 결과 |
| 서버 명령어 실행 결과 | 실습 전후 증거 |
| Apache 설정 파일 | 취약 재현 및 조치 대상 |

## 2. 작성 원칙

웹 서비스 보안 파트에서는 웹 애플리케이션 파트에서 사용한 A/B/C/D 분류를 사용하지 않는다.

팀원들이 직관적으로 이해할 수 있도록 다음 상태 표현을 사용한다.

| 상태 | 의미 |
|---|---|
| 취약 | 최초 진단 또는 실습 결과 실제 취약 상태가 확인됨 |
| 양호 | 최초 진단 또는 조치 후 기준을 만족함 |
| 수동 | 자동 진단만으로 확정하기 어려워 직접 확인이 필요함 |
| N/A | 현재 Apache/PHP 기반 CARE 서버에 직접 적용되지 않음 |
| 확인 필요 | 명령어 실행 또는 캡처 증거가 아직 부족함 |

양호 항목도 PDF 내용을 실습하기 위해 의도적으로 취약 설정을 만든 뒤 다시 조치한다. 이 경우 최초 상태와 실습 상태를 반드시 구분한다.

```text
최초 진단 결과: 양호
취약 재현: 실습을 위해 의도적으로 설정 약화
조치 후 확인: PDF 기준으로 복구 후 양호 확인
```

취약 항목은 최초 취약 상태를 증거로 남긴 뒤 조치한다.

```text
최초 진단 결과: 취약
취약 재현: 최초 상태 자체가 취약 증거
조치 후 확인: 설정 수정 후 양호 확인
```

N/A 항목은 개별 노트를 만들지 않고, 별도 묶음 노트에서 왜 현재 환경에 해당하지 않는지 설명한다.

## 3. 전체 작성 계획

### 3-1. 개별 노트 작성 대상

N/A 항목을 제외한 모든 WEB 항목은 개별 노트로 작성한다.

| 구분 | 처리 방식 |
|---|---|
| 취약 항목 | 개별 노트 작성, 현재 취약 증거 확보, 조치 후 확인 |
| 양호 항목 | 개별 노트 작성, 의도적 취약 재현, 조치 후 확인 |
| 수동 항목 | 개별 노트 작성, 수동 진단 명령어와 결과로 판단 |
| N/A 항목 | 묶음 노트 1개로 정리 |

### 3-2. 우선순위

우선순위는 다음과 같이 진행한다.

```text
1. 최초 진단에서 취약으로 나온 항목
2. 수동 진단 항목
3. 최초 진단에서 양호로 나온 항목
4. N/A 묶음 노트
```

현재 `AWS WEB 약점 진단 결과.md` 기준 1차 상태는 다음과 같다.

| 구분 | 항목 수 | 처리 |

| 양호 | 13 | 개별 노트 작성 후 취약 재현·복구 실습 |
| 수동 | 5 | 수동 확인 후 개별 노트 작성 |
| N/A | 4 | 묶음 노트 작성 |
| 총계 | 26 | 전 항목 처리 |


## 6. 실습 시 운영 규칙

Apache 설정을 변경하는 항목은 다음 순서를 지킨다.

```bash
# 1. 원본 설정 백업
sudo cp /etc/apache2/apache2.conf /etc/apache2/apache2.conf.bak.WEB-XX

# 2. 취약 설정 적용 또는 조치 설정 적용
sudo nano /etc/apache2/apache2.conf

# 3. 문법 검사
sudo apachectl configtest

# 4. Apache reload
sudo systemctl reload apache2

# 5. 증거 수집
# curl, grep, apache2ctl, ss, ls, stat 등 사용

# 6. 조치 또는 원복
sudo apachectl configtest
sudo systemctl reload apache2

# 7. 조치 후 증거 수집
```

주의 사항은 다음과 같다.

- `apachectl configtest` 없이 reload하지 않는다.
- 설정을 망가뜨리기 전 반드시 백업한다.
- 양호 항목을 일부러 취약하게 만든 경우, 노트에 “최초 상태는 양호였고 실습을 위해 약화했다”고 명시한다.
- 실제 운영 secret, 인증서 개인키, DB 비밀번호는 노트나 GitHub에 기록하지 않는다.
- 취약 재현 증거와 조치 후 증거를 같은 명령어로 비교한다.

## 7. 항목별 처리 계획

### 7-1. 취약 항목

| 항목 | 항목명 | 처리 방향 |
|---|---|---|
| WEB-04 | 웹서비스 디렉터리 리스팅 방지 설정 | `Options Indexes` 취약 상태 증거화 후 `-Indexes` 적용 |
| WEB-12 | 웹서비스 링크 사용 금지 | `FollowSymLinks` 취약 상태 증거화 후 `SymLinksIfOwnerMatch` 또는 `-FollowSymLinks` 적용 |
| WEB-20 | SSL/TLS 활성화 | 진단 결과 모순 확인, 수동 확인 후 실제 상태 판단 |
| WEB-22 | 웹서비스 에러페이지 사용 | `ErrorDocument` 부재 증거화 후 커스텀 에러 페이지 적용 |

### 7-2. 수동 항목

| 항목 | 항목명 | 처리 방향 |
|---|---|---|
| WEB-06 | 상위 디렉터리 접근 제한 | `../` 요청, Directory/AllowOverride 설정 확인 |
| WEB-21 | 동적 페이지 입력값 검증 | CARE 소스 및 기존 웹 애플리케이션 취약점 노트와 연결 |
| WEB-23 | HTTP 메서드 제한 | `OPTIONS`, `TRACE`, `PUT`, `DELETE` 요청으로 확인 |
| WEB-24 | X-Frame-Options 헤더 | Header 설정과 curl 응답 헤더 확인 |
| WEB-25 | X-XSS-Protection 헤더 | Header 설정과 curl 응답 헤더 확인 |

### 7-3. 양호 항목

양호 항목은 최초 상태가 양호임을 먼저 기록한 뒤, PDF 실습을 위해 의도적으로 취약 설정을 만들고 다시 복구한다.

| 항목 | 항목명 | 처리 방향 |
|---|---|---|
| WEB-05 | CGI/ISAPI 실행 제한 | CGI 실행 범위 제한 확인, 필요 시 ScriptAlias 약화 실습 |
| WEB-07 | 불필요한 파일 제거 | 샘플/백업 파일 생성 후 제거 실습 |
| WEB-08 | .htaccess 오버라이드 제한 | `AllowOverride All` 약화 후 `None` 복구 |
| WEB-09 | 웹서비스 프로세스 권한 제한 | Apache 자식 프로세스 권한 확인, root 실행 금지 설명 |
| WEB-10 | 불필요한 프록시 설정 제한 | Proxy 설정 부재 확인, ProxyRequests 위험 설명 |
| WEB-11 | 웹서비스 경로 설정 | DocumentRoot 분리 확인, 기본 경로 사용 위험 설명 |
| WEB-14 | 웹서비스 경로 내 파일 접근 통제 | `Require` 지시어 확인, 접근 통제 약화/복구 실습 |
| WEB-15 | 불필요한 스크립트 매핑 제거 | 불필요 handler 부재 확인, AddHandler 위험 설명 |
| WEB-16 | 웹서버 헤더 정보 노출 제한 | ServerTokens/ServerSignature 약화 후 복구 |
| WEB-17 | 웹서비스 가상 디렉터리 삭제 | Alias 부재 확인, 불필요 Alias 추가/삭제 실습 |
| WEB-18 | WebDAV 비활성화 | DAV 비활성 확인, 활성화 위험 설명 후 비활성 유지 |
| WEB-19 | WebDAV 모듈 미로드 | mod_dav 미로드 확인, WebDAV와 연결해 설명 |
| WEB-26 | 로그 디렉터리 및 파일 권한 설정 | 로그 권한 확인, 권한 약화 후 640/750 복구 |
