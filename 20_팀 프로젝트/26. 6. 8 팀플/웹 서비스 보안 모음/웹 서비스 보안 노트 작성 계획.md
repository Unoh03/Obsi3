# 웹 서비스 보안 노트 작성 계획

## 1. 목적

이 문서는 PDF p.271부터 시작되는 **웹 서비스 보안 진단 항목**을 노트화하기 위한 작성 계획과 공통 템플릿을 정리한다.

기존 웹 애플리케이션 취약점 노트는 CARE 애플리케이션의 PHP 코드와 기능을 중심으로 분석했다. 반면 웹 서비스 보안 항목은 Apache 웹서버 설정, 파일 권한, 모듈, 헤더, 로그, SSL/TLS 등 서버 운영 설정을 중심으로 점검한다.

따라서 웹 서비스 보안 노트는 다음 기준으로 작성한다.

- PDF 기준을 먼저 정리한다.
- `AWS WEB 약점 진단 결과.md`의 실제 진단 결과를 기준 자료로 사용한다.
- N/A 항목을 제외한 모든 항목은 개별 노트로 작성한다.
- 양호 항목도 PDF 내용을 실습하기 위해 의도적으로 취약하게 만든 뒤, 다시 조치하고 복구한다.
- N/A 항목은 개별 노트로 만들지 않고, 하나의 묶음 노트에서 왜 N/A인지 설명한다.
- 웹 애플리케이션 취약점 노트에서 사용하던 A/B/C/D 분류는 사용하지 않는다.

## 2. 상태 표현 기준

웹 서비스 보안 노트에서는 A/B/C/D 분류를 사용하지 않는다. 팀 내 혼선을 줄이기 위해 진단 결과 중심으로 표현한다.

| 상태 | 의미 |
|---|---|
| 취약 | 최초 진단 또는 수동 확인에서 실제 취약 설정이 확인됨 |
| 양호 | 최초 진단에서 안전한 설정으로 확인됨 |
| 수동 | 자동 진단만으로 확정하기 어려워 직접 명령어, 요청, 설정 확인이 필요함 |
| N/A | 현재 Apache/PHP 기반 CARE 서버 환경에 직접 적용되지 않음 |
| 확인 필요 | 아직 증거가 부족해 최종 판단을 보류함 |
| 조치 완료 | 조치 후 재점검에서 기대 결과를 확인함 |

## 3. 공통 작성 원칙

### 3-1. 최초 상태와 실습 상태 분리

양호 항목을 일부러 취약하게 만들 경우, 다음 세 상태를 반드시 구분한다.

| 구분 | 설명 |
|---|---|
| 최초 진단 상태 | `AWS WEB 약점 진단 결과.md` 기준 상태 |
| 취약 재현 상태 | PDF 내용을 실습하기 위해 의도적으로 약화한 상태 |
| 조치 후 상태 | 취약 재현 후 PDF 조치 기준으로 복구한 상태 |

예시는 다음과 같다.

| 항목 | 예시 |
|---|---|
| 최초 진단 | WEB-05 CGI/ISAPI 실행 제한: 양호 |
| 취약 재현 | CGI 실행 범위를 의도적으로 넓힘 |
| 조치 후 | CGI 실행 디렉터리를 다시 제한함 |

### 3-2. Apache 설정 변경 안전 규칙

Apache 설정을 변경하는 항목은 다음 순서를 따른다.

1. 원본 설정 파일 백업
2. 취약 설정 적용 또는 기존 취약 상태 확인
3. `apachectl configtest` 수행
4. Apache reload
5. 취약 증거 수집
6. 조치 설정 적용
7. `apachectl configtest` 재수행
8. Apache reload
9. 조치 후 증거 수집

기본 명령 흐름은 다음과 같다.

```bash
sudo cp /etc/apache2/apache2.conf /etc/apache2/apache2.conf.bak.WEB-XX
sudo apachectl configtest
sudo systemctl reload apache2
```

설정 변경 후 `Syntax OK`를 확인하지 않고 reload하지 않는다.

### 3-3. GitHub에 올리는 것과 서버에 배치하는 것 구분

GitHub에는 노트, 실습용 소스 원본, 설정 예시, 배포 스크립트를 올린다.

서버의 실제 배치 결과는 GitHub에 그대로 올리지 않는다.

| GitHub에 올릴 수 있음 | GitHub에 그대로 올리지 않음          |
| --------------- | --------------------------- |
| 노트              | `/etc/apache2/...` 실제 설정 파일 |
| 실습용 소스 원본       | 실제 인증서와 개인키                 |
| 설정 예시           | 실제 운영 secret                |
| 배포 스크립트         | 서버 런타임 디렉터리 전체              |

### 3-4. ChatGPT 출력 및 복붙 규칙
- 노트 초안은 Obsidian에 바로 붙여넣을 수 있는 원문 Markdown 형태로 출력한다.
- 전체 노트를 하나의 코드블록으로 감싸지 않는다.
- frontmatter는 문서 맨 위에 그대로 출력한다.
- 코드블록에는 `id="..."` 같은 속성을 붙이지 않는다.
- 코드블록 언어는 `bash`, `apache`, `text`, `php`처럼 필요한 언어명만 사용한다.
- frontmatter와 본문 사이에 불필요한 설명문을 끼우지 않는다.
- GitHub는 읽기와 검토용으로만 사용하고, 사용자가 명시하기 전까지 직접 작성·수정·삭제하지 않는다.
- 사용자가 “노트 초안 출력”을 요청한 경우, YAML frontmatter도 코드블록 안이 아니라 문서 맨 위 raw markdown으로 출력한다.

## 4. 개별 노트 템플릿

아래 구조를 N/A를 제외한 WEB 항목 개별 노트에 적용한다.

개별 노트는 단순 설명용이 아니라, **노트를 보면서 명령어를 순서대로 따라 하면 실습 재현과 조치 후 확인까지 할 수 있는 수준**으로 작성한다.

### 4-1. Frontmatter

각 노트의 frontmatter는 다음 형식을 기본으로 한다.

```yaml
---
type: lab
topic: web-service-security
source: 주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드
source_pages:
  - PDF_PAGE

created: YYYY-MM-DD

  - 🏷️과목/웹서비스보안
  - 🏷️주제/WEB-XX
  - 🏷️주제/항목별-주제
  - 🏷️주제/Apache
  - 🏷️상태/draft
---
```

### 4-2. 제목

제목은 다음 형식을 사용한다.

```markdown
# WEB-XX 항목명
```

### 4-3. 1. PDF 기준

PDF에서 이 항목이 무엇을 점검하는지 정리한다.

작성 내용:

- 점검 대상
- 취약한 상태
- 양호한 상태
- Apache 환경에서 해석할 때의 기준
- 현재 CARE 서버 환경에서 해당 항목이 어떤 의미인지

### 4-4. 2. 최초 진단 결과

`AWS WEB 약점 진단 결과.md` 기준 최초 진단 상태를 적는다.

|항목|내용|
|---|---|
|진단 결과|취약 / 양호 / 수동|
|진단 근거|AWS WEB 약점 진단 결과 기준|
|관련 설정|Apache 설정 파일, 모듈, 헤더, 권한, 명령어 등|
|최초 판단|최초 진단 기준 판단|

진단 명령어는 별도 코드블록으로 적는다.  
코드블록에는 ChatGPT 내부 식별자나 `id="..."` 속성을 붙이지 않는다.

```bash
진단 명령어
```

진단 결과도 별도 코드블록으로 적는다.

```text
진단 결과
```

### 4-5. 3. 현재 서버 상태 해석

진단 결과가 의미하는 바를 설명한다.

작성 내용:

- 왜 취약인지
- 왜 양호인
- 자동 진단만으로 부족한 부분은 무엇인지
- Apache/PHP/CARE 환경에서는 어떤 의미인지
- 다른 WEB 항목 또는 웹 애플리케이션 취약점 항목과 연결되는 부분이 있는지

필요하면 표로 정리한다.

```markdown
| 확인 대상 | 의미 |
|---|---|
| Apache 설정 | 해석 내용 |
| 관련 모듈 | 해석 내용 |
| 관련 경로 | 해석 내용 |
```

### 4-6. 4. 실습 전 확인

실습을 시작하기 전에 현재 서버 환경과 기준 경로를 확인한다.

이 섹션은 사용자가 명령어를 그대로 따라 치기 전에, **경로·모듈·설정 파일 위치가 현재 서버와 맞는지 확인하는 단계**다.

작성 내용:

- Apache 가상호스트 확인
- DocumentRoot 확인
- 관련 모듈 로드 여부 확인
- 관련 설정 파일 위치 확인
- 실습에서 사용할 변수 또는 경로 확인


예시는 다음과 같다.

```bash
apache2ctl -S
```

```bash
grep -R "DocumentRoot" /etc/apache2/sites-enabled/ /etc/apache2/sites-available/ 2>/dev/null
```

필요하면 실습용 변수를 먼저 지정한다.

```bash
APP_ROOT=/var/www/care
```

현재 서버의 DocumentRoot가 다르면 이후 명령어의 경로를 실제 DocumentRoot에 맞게 바꾼다.

### 4-7. 5. 취약 재현

최초 진단에서 이미 취약한 항목은 굳이 다시 약화하지 않고, 현재 취약 설정을 증거화한다.

최초 진단에서 양호한 항목은 PDF 내용을 실습하기 위해 의도적으로 취약 상태를 재현한다.

수동 항목은 자동 진단 결과만으로 판단하지 않고, 직접 설정·명령어·HTTP 요청 결과를 통해 확인한다.

작성 내용:

- 취약 재현 목적
- 실습용 파일 또는 설정 생성
- 취약 설정 적용
- Apache 문법 검사
- Apache reload
- 취약 상태 확인 명령어
- 취약 상태에서 기대되는 결과

취약 설정 또는 재현 명령어는 다음처럼 적는다.

```bash
취약 설정 또는 재현 명령어
```

Apache 설정 예시는 다음처럼 적는다.

```apache
Apache 취약 설정 예시
```

취약 재현 후 확인 명령어도 반드시 적는다.

```bash
취약 상태 확인 명령어
```

```text
취약 상태 기대 결과
```

### 4-8. 6. 조치 방법

PDF 기준 조치와 Apache 환경에서의 실제 조치 방법을 정리한다.

작성 내용:

- 취약 재현에 사용한 설정 제거 또는 원복
- PDF 기준 보안 설정 적용
- 불필요한 파일·모듈·설정 제거
- Apache 문법 검사
- Apache reload

조치 명령어는 다음처럼 적는다.

```bash
조치 명령어
```

Apache 조치 설정 예시는 다음처럼 적는다.

```apache
Apache 조치 설정 예시
```

Apache 설정을 변경하는 경우 다음 순서를 유지한다.

1. 설정 백업 또는 실습용 설정 파일 분리
2. 설정 수정
3. `apachectl configtest`
4. `systemctl reload apache2`
5. 조치 후 확인

### 4-9. 7. 조치 후 확인

재진단 명령어와 기대 결과를 적는다.

조치 후 확인은 가능하면 두 단계로 한다.

1. 설정 파일 또는 모듈 상태 확인
2. 실제 HTTP 요청 또는 브라우저 접근 확인

재진단 명령어는 다음처럼 적는다.

```bash
재진단 명령어
```

기대 결과는 다음처럼 적는다.

```text
기대 결과
```

HTTP 요청으로 확인하는 항목은 다음처럼 적는다.

```bash
curl -i http://서버주소/확인경로
```

조치 후 확인에서 중요한 것은 단순히 설정 파일이 바뀐 것이 아니라, **실제 요청 결과가 기대한 보안 상태로 바뀌었는지** 확인하는 것이다.

### 4-11. 9. 증거 정리

보고서와 발표에 사용할 증거를 따로 정리한다.

### 4-12. 10. 판단

최종 판단 표는 다음 형식으로 작성한다.

|항목|판단|
|---|---|
|최초 진단|취약 / 양호 / 수동|
|진단 근거|자동 진단 결과 또는 수동 확인 결과|
|실습 처리|기존 취약 조치 / 취약 재현 후 조치 / 수동 확인|
|조치 전 판단|취약 / 양호 / 확인 필요|
|조치 후 판단|양호 / 취약 / 확인 필요|
|증거 상태|자동 진단 결과 확보 / 웹 접근 증거 필요 / 조치 후 증거 필요|

판단 문단에서는 다음을 명확히 쓴다.

- 최초 진단 기준 판단
    
- 취약 재현을 했다면 재현 상태의 판단
    
- 조치 후 기대 결과
    
- 최종적으로 양호로 볼 수 있는 조건
    
- 아직 추가 증거가 필요한 부분
    

예시는 다음과 같다.

```text
최초 진단 기준으로는 양호이다.
다만 PDF 취약 조건을 실습하기 위해 의도적으로 취약 상태를 재현했다.
조치 후 재진단에서 기대 결과가 확인되면 조치 후 양호로 판단한다.
```

## 6. 항목별 처리 계획

`AWS WEB 약점 진단 결과.md` 기준 26개 항목을 다음처럼 처리한다.

### 6-1. 취약 항목

취약 항목은 개별 노트에서 최초 취약 상태를 증거화하고, PDF 조치 방법에 따라 조치 후 재점검한다.

| 항목 | 처리 방향 |
|---|---|
| WEB-04 웹서비스 디렉터리 리스팅 방지 설정 | `Options Indexes` 확인 후 `-Indexes` 조치 |
| WEB-12 웹서비스 링크 사용 금지 | `FollowSymLinks` 확인 후 `SymLinksIfOwnerMatch` 또는 `-FollowSymLinks` 조치 |
| WEB-20 SSL/TLS 활성화 | 진단 스크립트 오탐 가능성 포함, 수동 확인 후 판단 |
| WEB-22 웹서비스 에러페이지 사용 | `ErrorDocument` 설정 추가 후 기본 에러 페이지 노출 제한 |

### 6-2. 양호 항목

양호 항목도 개별 노트로 작성한다. 최초 상태는 양호로 기록하되, PDF 내용을 실습하기 위해 의도적으로 취약 설정을 만든 뒤 다시 복구한다.
