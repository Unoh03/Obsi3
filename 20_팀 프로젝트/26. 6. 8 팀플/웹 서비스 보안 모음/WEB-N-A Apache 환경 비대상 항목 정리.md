# WEB-N/A Apache 환경 비대상 항목 정리

source: [[40_자료/주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드.pdf|주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드]], p.271-273, p.274-279, p.308-310.
진단 근거: [[AWS WEB 약점 진단 결과]]

## 1. 개요

이 노트는 PDF의 웹 서비스 취약점 분석·평가 항목 중, 현재 실습 환경인 **Apache + PHP + CARE** 서버에 직접 적용되지 않는 항목을 묶어 정리한다.

웹 서비스 파트의 나머지 항목은 개별 노트로 작성한다. 다만 이 노트에 포함된 항목은 현재 서버 제품군과 기능 범위에 직접 대응하는 점검 대상이 없으므로, 개별 실습 노트로 분리하지 않고 N/A 묶음으로 처리한다.

현재 자동 진단 결과에서 N/A로 분류된 항목은 다음 4개다.

| 항목 | 항목명 | 자동 진단 결과 | N/A 사유 요약 |
|---|---|---|---|
| WEB-01 | Default 관리자 계정명 변경 | N/A | Tomcat/IIS/JEUS 계열 관리자 계정 대상. Apache/PHP 환경에 직접 대응 없음 |
| WEB-02 | 취약한 비밀번호 사용 제한 | N/A | Tomcat/IIS/JEUS 관리자 계정 비밀번호 대상. Apache/PHP 환경에 직접 대응 없음 |
| WEB-03 | 비밀번호 파일 권한 관리 | N/A | Tomcat/IIS/JEUS 비밀번호 파일 대상. Apache/PHP 환경에 직접 대응 없음 |
| WEB-13 | 웹 서비스 설정 파일 노출 제한 | N/A | PDF 대상이 Tomcat/IIS/JEUS 중심이며, Apache 환경에서는 다른 항목과 중복 처리 |

## 2. 현재 실습 환경

현재 실습 환경은 다음과 같다.

| 구분 | 내용 |
|---|---|
| 웹 서버 | Apache |
| 애플리케이션 구조 | PHP 기반 CARE 웹 애플리케이션 |
| 관리자 콘솔 | Tomcat Manager, JEUS Admin Console, IIS Manager 같은 WAS 관리자 콘솔 없음 |
| 인증 저장소 | CARE MySQL `member` 테이블 중심 |
| 주요 설정 경로 | `/etc/apache2/`, `/var/www/care`, `/var/log/apache2` |

따라서 Tomcat, IIS, JEUS 같은 특정 웹서버 또는 WAS 제품의 관리자 계정·관리자 비밀번호·비밀번호 파일·제품별 설정 파일을 전제로 하는 항목은 현재 환경에서 직접 실습할 수 없다.

N/A는 “안전하다”는 뜻이 아니다. **점검 대상 제품 또는 기능이 현재 환경에 없다는 뜻**이다.

```text
N/A = 취약 아님
N/A = 양호 아님
N/A = 현재 Apache/PHP/CARE 환경에 직접 대응하는 점검 대상 없음
```

## 3. WEB-01 Default 관리자 계정명 변경

### 3.1 PDF 기준

WEB-01은 웹 서비스 설치 시 기본으로 생성되는 관리자 계정명을 변경했는지 점검하는 항목이다.

PDF는 대표적인 기본 관리자 계정 예시로 다음을 제시한다.

| 제품군 | 기본 관리자 계정 예시 |
|---|---|
| Tomcat | `tomcat`, `admin` |
| JEUS | `administrator` |
| IIS | `Administrator` |

이 항목의 취지는 기본 관리자 계정명이 널리 알려져 있으므로, 공격자가 관리자 계정명을 쉽게 추측하고 비밀번호 대입 공격을 시도할 수 있다는 점을 막는 것이다.

### 3.2 현재 환경 적용성

현재 CARE 서버는 Apache + PHP 구조다. Tomcat Manager, JEUS 관리자 콘솔, IIS 관리자 계정과 같은 제품 기본 관리자 계정이 존재하지 않는다.

즉 WEB-01의 점검 대상은 다음과 같다.

```text
Tomcat Manager 계정
JEUS 관리자 계정
IIS 관리자 계정
```

현재 실습 환경의 실제 구조는 다음과 같다.

```text
Apache 웹 서버
PHP 기반 CARE 애플리케이션
MySQL member 테이블 기반 일반 사용자 계정
```

CARE 애플리케이션에 `admin` 문자열이나 관리자 실습 페이지가 존재하더라도, 그것은 WEB-01에서 말하는 Tomcat/JEUS/IIS 제품 기본 관리자 계정과 다르다.

따라서 WEB-01은 현재 Apache/PHP/CARE 환경에서는 N/A로 처리한다.

### 3.3 판단

| 항목 | 판단 |
|---|---|
| 최초 진단 | N/A |
| 실습 처리 | 제외 |
| 최종 판단 | Apache 환경 비대상 |
| 사유 | Tomcat/JEUS/IIS 기본 관리자 계정 점검 항목이며, 현재 서버에는 해당 제품 관리자 계정이 없음 |

## 4. WEB-02 취약한 비밀번호 사용 제한

### 4.1 PDF 기준

WEB-02는 웹 서비스 관리자 계정의 비밀번호가 취약하게 설정되어 있는지 점검하는 항목이다.

PDF의 주요 판단 기준은 다음과 같다.

| 판단 | 기준 |
|---|---|
| 양호 | 관리자 비밀번호가 암호화되어 있거나, 유추하기 어려운 비밀번호로 설정된 경우 |
| 취약 | 관리자 비밀번호가 암호화되어 있지 않거나, 유추하기 쉬운 비밀번호로 설정된 경우 |

PDF의 점검 대상은 Tomcat, IIS, JEUS 관리자 계정 비밀번호다.

### 4.2 현재 환경 적용성

현재 서버에는 Tomcat Manager, JEUS Admin Console, IIS 관리자 계정이 없다. 따라서 PDF가 말하는 제품 관리자 계정 비밀번호를 확인할 대상이 없다.

CARE 자체의 회원 비밀번호 정책 문제는 이미 웹 애플리케이션 취약점 노트에서 다룬다.

```text
WEB-02: Tomcat/IIS/JEUS 관리자 계정 비밀번호
웹 앱 09: CARE 회원가입/회원 비밀번호 정책
```

두 항목은 비밀번호라는 공통어를 쓰지만 점검 대상이 다르다.

| 구분 | 점검 대상 |
|---|---|
| WEB-02 | 웹 서버/WAS 제품 관리자 계정 |
| 웹 앱 09 | CARE 애플리케이션 사용자 계정 |

따라서 WEB-02는 현재 Apache/PHP/CARE 환경에서는 N/A로 처리한다.

### 4.3 판단

| 항목 | 판단 |
|---|---|
| 최초 진단 | N/A |
| 실습 처리 | 제외 |
| 최종 판단 | Apache 환경 비대상 |
| 사유 | Tomcat/IIS/JEUS 관리자 비밀번호 점검 항목이며, 현재 서버에는 해당 제품 관리자 계정이 없음 |

## 5. WEB-03 비밀번호 파일 권한 관리

### 5.1 PDF 기준

WEB-03은 웹 서비스의 비밀번호 파일에 적절한 파일 권한이 설정되어 있는지 점검하는 항목이다.

PDF의 대표 예시는 다음과 같다.

| 제품군 | 점검 대상 파일 예시 |
|---|---|
| Tomcat | `tomcat-users.xml` |
| IIS | Windows `SAM` 파일 |
| JEUS | `accounts.xml`, `policies.xml` |

이 항목의 취지는 비밀번호 또는 계정 정보가 담긴 제품 설정 파일이 일반 사용자에게 노출되지 않도록 권한을 제한하는 것이다.

### 5.2 현재 환경 적용성

현재 서버는 Apache + PHP 기반이며, Tomcat의 `tomcat-users.xml`, IIS의 `SAM`, JEUS의 `accounts.xml` 또는 `policies.xml`을 사용하지 않는다.

CARE에는 DB 접속 설정 파일이나 PHP 설정 파일은 존재할 수 있다. 하지만 그것은 WEB-03에서 말하는 Tomcat/IIS/JEUS 제품 비밀번호 파일과는 다르다.

관련 문제는 다음 항목에서 다루는 것이 더 적절하다.

| 관련 항목 | 설명 |
|---|---|
| WEB-14 | 웹 서비스 경로 내 파일의 접근 통제 |
| WEB-16 | 웹 서비스 헤더 정보 노출 제한 |
| 웹 앱 05 | 정보 누출, DB 설정 파일 노출 가능성 |
| WEB-26 | 로그 디렉터리 및 파일 권한 설정 |

따라서 WEB-03은 현재 Apache/PHP/CARE 환경에서는 N/A로 처리한다.

### 5.3 판단

| 항목 | 판단 |
|---|---|
| 최초 진단 | N/A |
| 실습 처리 | 제외 |
| 최종 판단 | Apache 환경 비대상 |
| 사유 | Tomcat/IIS/JEUS 제품 비밀번호 파일 권한 점검 항목이며, 현재 서버에는 해당 파일 구조가 없음 |

## 6. WEB-13 웹 서비스 설정 파일 노출 제한

### 6.1 PDF 기준

WEB-13은 웹 서비스 설정 파일 또는 DB 연결 설정이 일반 사용자에게 노출되지 않도록 접근 권한과 스크립트 매핑을 제한하는 항목이다.

PDF의 주요 설명은 DB 연결 파일에 대한 접근 권한, 스크립트 매핑, DB 연결 정보 노출 위험을 점검하는 것이다.

PDF의 점검 대상은 다음 제품군 중심으로 제시되어 있다.

| 제품군 | 예시 |
|---|---|
| Tomcat | `server.xml` 내부 DB 연결 리소스, 설정 파일 권한 |
| IIS | `*.asa`, `*.asax` 처리기 매핑 및 요청 필터링 |
| JEUS | `domain.xml`, `jeus-web-dd.xml` 내부 DB 연결 리소스 |

### 6.2 현재 환경 적용성

현재 서버는 Apache + PHP 기반이다. PDF의 WEB-13에서 직접 다루는 Tomcat `server.xml`, IIS `*.asa`/`*.asax`, JEUS `domain.xml`·`jeus-web-dd.xml` 구조가 없다.

Apache/PHP 환경에서도 DB 접속 파일 노출 문제는 중요하다. 그러나 현재 웹 서비스 자동 진단 결과에서는 WEB-13을 독립 Apache 항목으로 다루지 않고, WEB-04와 중복 또는 Apache 환경 비대상으로 분류했다.

따라서 이 노트에서는 WEB-13 자체는 N/A로 두되, Apache/PHP 환경에서 대응되는 문제는 다른 항목으로 연결한다.

| Apache/PHP에서 대응되는 문제 | 다루는 노트 |
|---|---|
| 디렉터리 리스팅으로 설정 파일 목록 노출 | WEB-04 |
| 웹 경로 내 민감 파일 접근 통제 | WEB-14 |
| 불필요한 스크립트 매핑 | WEB-15 |
| DB 설정 파일 직접 노출 | 웹 앱 05 정보 누출 |

즉 WEB-13을 “위험이 없다”고 보는 것이 아니라, **PDF의 제품별 점검 대상이 현재 Apache/PHP 환경에 직접 대응하지 않으므로 N/A로 묶고, 관련 위험은 다른 Apache/PHP 항목에서 실습한다**고 정리한다.

### 6.3 판단

| 항목 | 판단 |
|---|---|
| 최초 진단 | N/A |
| 실습 처리 | 제외 |
| 최종 판단 | Apache 환경 비대상, 관련 위험은 다른 항목에서 처리 |
| 사유 | PDF의 직접 대상은 Tomcat/IIS/JEUS 설정 파일·스크립트 매핑이며, Apache/PHP 환경에서는 WEB-04, WEB-14, WEB-15, 웹 앱 05와 중복됨 |

## 7. 전체 판단

| 항목 | 최초 진단 | 실습 처리 | 최종 판단 |
|---|---|---|---|
| WEB-01 | N/A | 제외 | Apache 환경 비대상 |
| WEB-02 | N/A | 제외 | Apache 환경 비대상 |
| WEB-03 | N/A | 제외 | Apache 환경 비대상 |
| WEB-13 | N/A | 제외 | Apache 환경 비대상, 관련 위험은 다른 항목에서 처리 |

현재 실습 환경은 Apache + PHP 기반 CARE 웹 애플리케이션이다. WEB-01, WEB-02, WEB-03은 Tomcat/IIS/JEUS 같은 제품 관리자 계정과 비밀번호 파일을 전제로 하므로 현재 환경에서 직접 점검할 대상이 없다.

WEB-13은 DB 연결 파일과 설정 파일 노출이라는 보안 주제 자체는 Apache/PHP 환경에서도 중요하지만, PDF의 직접 점검 대상은 Tomcat/IIS/JEUS 설정 구조다. 따라서 WEB-13은 N/A로 처리하고, Apache/PHP에서 대응되는 설정 파일 노출 문제는 WEB-04, WEB-14, WEB-15, 웹 앱 05에서 다룬다.

## 8. 후속 작업

N/A 항목은 이 노트에서 묶어 종료한다.

이후 웹 서비스 보안 항목은 다음 원칙으로 진행한다.

```text
N/A 항목: 묶음 노트로 종료
취약 항목: 최초 취약 상태 증거화 후 조치
양호 항목: 의도적으로 취약 설정을 만든 뒤 복구
수동 항목: 직접 명령어와 요청으로 확인 후 판단
```
