# 웹 취약점 1차 분류표

## 기준

분류 기준 시점은 **최초 CARE/서버 환경**이다. 이후 조치, 실습용 약화, 복구 상태는 분류를 바꾸지 않고 `조치 상태`와 `증거 상태`로 따로 관리한다.

공격면은 CARE 앱 기능만 뜻하지 않는다. Apache/PHP 설정, HTTP 전송, 세션·쿠키, 배포 파일, 샘플 파일도 포함한다.

## 분류 코드

| 코드 | 의미 |
|---|---|
| A | 최초 상태에 공격면이 있고 취약함 |
| A-1 | 자연 취약. 원래 기능 그대로 공격 가능 |
| A-2 | 기존 기능 악용형. 계정, 게시글, 파일 같은 실습 조건만 준비하면 됨 |
| A-3 | 서버/환경 설정 취약 |
| A-4 | 배포 파일/정보 노출 취약 |
| B | 최초 상태에 공격면은 있지만 방어되어 있음 |
| B-L | B 원본을 실습을 위해 일부러 약화한 lab variant |
| C | 최초 상태에 공격면이 없어 `/vuln/...` 같은 실습용 기능을 새로 만들어 재현 |
| C-조건부 | 서버 모듈, 외부 구성, 환경 활성화 여부에 따라 재현 가능 |
| D | 공격면 또는 전제 환경이 없어 구현하지 않고 비교 학습/해당 없음 처리 |

## 21개 웹 취약점 1차 분류

| 번호 | PDF 항목 | 1차 분류 | 판정 상태 | 조치 상태 | 증거 상태 | 짧은 이유 / 다음 확인 |
|---:|---|---|---|---|---|---|
| 01 | 코드 인젝션 | 혼합 C/D | 확정에 가까움 | 항목별 관리 | 일부 있음 | CARE 기본 기능에는 LDAP/OS 명령/SSI/XPath/XML/Twig 공격면 없음. OS Command, XPath, XXE, SSTI는 `/vuln/code-injection/` 실습용. LDAP은 비교 학습. SSI는 조건부. |
| 02 | SQL 인젝션 | A-1 | 확정 | 조치안 작성 또는 일부 조치 | 조치 전 증거 있음 | 로그인, 게시판 검색 등 SQL에 사용자 입력이 결합되는 기본 기능 공격면 있음. |
| 03 | 디렉터리 인덱싱 | A-3 | 확정에 가까움 | 조치안 작성 | 조치 전 증거 있음 | CARE 코드 기능이 아니라 Apache 디렉터리 listing 설정 문제. |
| 04 | 에러 페이지 적용 미흡 | A-3 | 확정에 가까움 | 조치안 작성 | 조치 전 증거 있음 | Apache/PHP/DB 오류가 서버 버전, 경로, 내부 오류를 노출할 수 있음. |
| 05 | 정보 누출 | A-4 | 확정에 가까움 | 조치안 작성 | 조치 전 증거 있음 | DB 연결 파일, 테스트/스키마 파일, 샘플 파일 등 배포물 노출 문제. |
| 06 | 크로스사이트 스크립트 | A-1 | 확정 | 조치 적용 이력 있음 | 조치 전/시행착오 증거 있음 | 게시글 출력에서 사용자 입력 HTML/스크립트가 렌더링되는 공격면. 조치 후/재약화 이력은 별도 기록. |
| 07 | 크로스사이트 요청 위조(CSRF) | A-1 | 확정 | 조치안 작성 | 조치 전 증거 있음 | 회원정보 수정 기능이 있고 세션만 확인하며 CSRF token 검증 없음. |
| 08 | 서버사이드 요청 위조(SSRF) | C | 확정 | 미조치 | 조치 전 증거 있음 | CARE 기본 기능에는 URL을 서버가 대신 요청하는 기능 없음. `/vuln/ssrf/fetch.php`로 실습용 재현. PDF Step 2와 방어 검증은 미완료. |
| 09 | 약한 비밀번호 정책 | A-2 | 실행 검증 필요 | 미정 | 없음 | 회원가입/비밀번호 변경 기능이 있으므로 공격면 있음. 길이, 복잡도, 재사용, 계정 잠금 확인 필요. |
| 10 | 불충분한 인증 절차 | A-1 | 확정 | 미조치 | 없음 | PDF p.729-732는 개인정보 변경 등 중요 페이지 접근 시 추가 인증/서버 측 재인증을 요구함. CARE 회원정보 수정은 로그인 세션만 확인하고 현재 비밀번호나 최근 재인증 없이 비밀번호·개인정보를 변경함. 회원탈퇴는 현재 비밀번호 확인이 있어 반례이며, 실행 증거는 미수집. |
| 11 | 불충분한 권한 검증 | B | 확정 | 해당 없음 | 없음 | PDF p.733-734는 타 사용자 접근이 제한된 개인정보·비공개 리소스의 ID, 숫자, 일련번호 파라미터 변조를 본다. CARE 회원정보 수정/탈퇴는 서버에서 `$_SESSION['id']` 기준으로 자기 계정만 처리하고 POST id를 신뢰하지 않음. 게시글 조회는 공개 목록 기반이고, 게시글 수정/삭제 파일·관리자 endpoint는 현재 repo에서 없음. |
| 12 | 취약한 비밀번호 복구 절차 | C | 설계 확정, 실행 검증 필요 | 조치안 작성 | 없음 | CARE 원본에는 비밀번호 복구 기능이 없어 자연 공격면은 없다. PDF p.735-738 점검 흐름을 재현하기 위해 `/vuln/password-recovery/` 실습 기능을 만들었고, 취약 모드는 고정 인증번호와 화면 출력, 조치 모드는 `random_int()`, 만료 시간, 실패 횟수 제한으로 비교한다. 서버 실행 증거는 미수집. |
| 13 | 프로세스 검증 누락 | C-연계 | 설계 확정, 실행 검증 필요 | 조치안 작성 | 없음 | CARE 원본의 `writeModel.php`로 약한 A-2 사례를 만들 수 있으나 피해와 설명력이 낮다. 12번 비밀번호 복구 실습 기능에서 인증번호 검증 단계를 거치지 않고 reset endpoint를 직접 호출하는 방식으로 프로세스 검증 누락을 재현한다. 원래 CARE 취약점이 아니라 `/vuln/password-recovery/` 연계 실습으로 분리한다. |
| 14 | 악성 파일 업로드 | A-2 | 실행 검증 필요 | 조치안 작성 | 없음 | PDF p.744-752는 업로드 확장자·MIME 검증과 서버 사이드 스크립트 업로드 후 경로 접근 시 실행 여부를 본다. CARE 게시글 작성은 `$_FILES['upfile']['name']`을 그대로 `../data/`에 저장하고 확장자, MIME, 이중 확장자, 파일명 난수화 검증이 없음. 1~4번 조치안은 주석 코드/설정 예시로 반영했고, 5번 웹 디렉터리 격리는 보류, 6번 바이러스 검사는 기술력·범위 부족으로 포기. |
| 15 | 파일 다운로드 | A-1 | 실행 검증 필요 | 조치안 작성 | 없음 | PDF p.753-758은 다운로드 파라미터의 경로 조작으로 허용 경로 외 파일을 열람할 수 있는지 본다. CARE `download.php`는 로그인만 확인한 뒤 `filename`을 검증 없이 `../data/`에 붙여 `fopen()`하므로 `../` 기반 경로 이동 차단, realpath 검증, 허용 파일 대조가 없음. `fd-proof.txt`를 이용한 안전한 경로 이탈 검증 절차와 방어 주석 코드를 준비한다. |
| 16 | 불충분한 세션 관리 | A-2 | 실행 검증 필요 | 미조치 | 없음 | PDF p.759-765는 세션 ID 패턴·고정, 재로그인 시 세션 변경, 일정 시간 후 만료를 본다. CARE는 PHP session 기반이며 로그인 성공 후 `session_regenerate_id(true)`나 앱 레벨 timeout 설정이 없고, 소스 검색상 cookie HttpOnly/Secure/SameSite 설정도 없음. 실제 PHP 기본 세션 강도, 쿠키 속성, 만료 동작은 실행 검증 필요. |
| 17 | 데이터 평문 전송 | A-3 | 확정 | 미조치 | 없음 | PDF p.766-768은 로그인, 회원가입, 회원정보관리 등 중요 정보 전송구간의 암호화 통신 적용 여부를 본다. 현재 WEB/LB 스크립트는 Apache·nginx를 80/http로 구성하고 HTTPS/443 설정이 보이지 않으므로 로그인·회원정보가 평문 전송되는 구조임. Wireshark 등 패킷 증거는 미수집. |
| 18 | 쿠키 변조 | B | 확정 | 해당 없음 | 없음 | PDF p.769-774는 쿠키에 사용자 식별값·권한값이 노출되고 변조로 타 사용자 권한을 얻을 수 있는지 본다. CARE 원본은 커스텀 인증 쿠키나 `$_COOKIE` 기반 권한 부여 없이 PHP 서버 사이드 세션(`$_SESSION`)에 사용자 값을 저장함. PHPSESSID 탈취·재사용과 쿠키 속성 문제는 16번 세션 관리 범위로 분리. |
| 19 | 관리자 페이지 노출 | D | 확정 | 해당 없음 | 없음 | PDF p.775-776은 유추 가능한 관리자 URL·포트와 관리자 계정 로그인을 본다. CARE 원본 소스와 서버 스크립트에서 관리자 페이지, 관리자 메뉴, phpMyAdmin/Adminer 같은 관리 도구 endpoint가 확인되지 않음. `dbconn.php`의 DB 계정명과 실습용 vuln 샘플의 admin 문자열은 관리자 페이지 노출 근거가 아님. |
| 20 | 자동화 공격 | A-2 | 확정 | 미조치 | 없음 | PDF p.777-778은 로그인 시도, 게시글 등록, SMS 발송 같은 특정 프로세스의 반복 요청 통제 여부를 본다. CARE `loginModel.php`는 단순 조회 후 실패 alert만 출력하며 실패 횟수 제한, 계정 잠금, CAPTCHA, throttle 로직이 없고, 게시글 작성에도 반복 등록 제한이 없음. 서버 스크립트에서도 WAF/rate-limit 설정은 확인되지 않음. |
| 21 | 불필요한 Method 악용 | B 가능성 높음 | 실행 검증 필요 | 미조치 | 없음 | PDF p.779-784는 PUT, DELETE, TRACE, CONNECT 등 불필요한 HTTP Method로 임의 파일 생성·삭제·정보 노출이 가능한지 본다. 현재 Apache/nginx 스크립트에서 WebDAV, `dav_methods`, `allowTrace` 활성화는 확인되지 않지만 명시적 method 제한도 없음. OPTIONS/TRACE/PUT/DELETE 응답과 파일 생성 가능성은 실행 검증 후 A-3 또는 B로 확정. |

## 코드 인젝션 세부 분류

| 세부 유형 | 분류 | 판정 상태 | 이유 |
|---|---|---|---|
| LDAP Injection | D | 확정에 가까움 | CARE에 LDAP 서버/LDAP 질의 기능 없음. 실제 LDAP 없이 얕게 만들면 신뢰도 낮음. 비교 학습 처리. |
| OS Command Injection | C | 확정에 가까움 | CARE 기본 기능에 OS 명령 실행 입력점 없음. `/vuln/code-injection/` 실습용 재현. |
| SSI Injection | C-조건부 | 실행 검증 필요 | Apache SSI 설정/확장자 처리 여부에 따라 실제 재현 가능성 달라짐. |
| XPath Injection | C | 확정에 가까움 | CARE 기본 기능에 XML/XPath 질의 없음. 실습용 XML/XPath 기능으로 재현. |
| XXE Injection | C | 확정에 가까움 | CARE 기본 기능에 XML 파싱 기능 없음. 실습용 XML 파서로 재현. |
| SSTI | C | 확정에 가까움 | CARE 기본 기능은 PHP 단순 출력 구조. Twig 등 템플릿 엔진을 실습용으로 추가해 재현. |

## 작업 원칙

| 상황 | 처리 |
|---|---|
| A 항목 | 원래 CARE/서버 상태에서 조치 전 증거를 먼저 확보한다. 실습용 새 기능을 만들지 않는다. |
| B 항목 | 원본이 막혀 있음을 기록한다. 필요하면 B-L로 약화 변형을 만들되 원본과 분리한다. |
| C 항목 | `/vuln/...` 아래에 실습용 기능을 격리해 만든다. 원래 CARE 취약점이라고 쓰지 않는다. |
| D 항목 | 구현하지 않는다. 비교 학습, 기능 부재, 환경 부재로 정리한다. |
| 조치 | 분류를 바꾸지 않는다. `조치 상태`, `증거 상태`만 갱신한다. |
