# 웹 취약점 - 04 에러 페이지

source: [[40_자료/주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드.pdf|주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드]], p.703-707.

## 1. 개요

**에러 페이지 적용 미흡**은 웹 애플리케이션에서 오류가 발생했을 때 기본 서버 에러 페이지나 개발용 오류 메시지가 그대로 출력되어 불필요한 정보가 노출되는 취약점이다.

PDF p.703의 핵심 판단 기준은 단순하다.

| 판단 | 기준 |
|---|---|
| 양호 | 에러 발생 시 자체 정의 에러 페이지를 출력하고 과도한 정보가 노출되지 않음 |
| 취약 | 기본 에러 페이지가 출력되며 서버 버전, 시스템 경로, 스택 트레이스 등이 노출됨 |

## 2. 문제가 되는 부분

CARE는 Apache/PHP 기반이므로 우선 확인할 부분은 두 가지다.

| 구분 | 문제 지점 | 노출될 수 있는 정보 |
|---|---|---|
| Apache 기본 에러 페이지 | 없는 경로, 접근 차단 경로, 허용되지 않은 요청에서 기본 에러 화면 출력 | `Apache/2.4.66 (Ubuntu)`, 서버 IP, 포트 |
| PHP/DB 오류 출력 | PHP 경고, DB 연결 오류, SQL 오류가 브라우저에 직접 출력 | 파일 경로, 함수명, 쿼리 구조, DB 오류 메시지 |

이미 디렉터리 인덱싱 조치 후 `403 Forbidden` 화면에서 다음과 같은 Apache 정보가 노출되는 형태를 확인할 수 있다.
![[Pasted image 20260612160846.png]]
```text
Apache/2.4.66 (Ubuntu) Server at 172.168.10.10 Port 80
```

이런 정보는 단독으로 치명적인 공격은 아니지만, 공격자가 서버 종류와 버전을 파악하고 후속 공격 대상을 좁히는 데 사용할 수 있다.

## 3. 악용 흐름

공격자는 정상 기능을 쓰지 않고도 오류를 일부러 유도할 수 있다.

```text
http://172.168.10.10/not-exist
http://172.168.10.10/vuln/code-injection/
http://172.168.10.10/center/view.php?num='
```

에러 페이지가 제대로 처리되지 않으면 다음 정보가 노출될 수 있다.

| 유도 방식 | 확인할 정보 |
|---|---|
| 존재하지 않는 URL 요청 | 기본 404 페이지와 서버 버전 노출 여부 |
| 접근이 차단된 디렉터리 요청 | 기본 403 페이지와 서버 버전 노출 여부 |
| 비정상 파라미터 입력 | PHP warning, SQL error, 파일 경로 노출 여부 |
| 허용되지 않은 Method 요청 | 기본 405 페이지와 서버 버전 노출 여부 |

## 4. 점검 방법

PDF p.703의 점검 방법은 **에러를 유도한 뒤 에러 페이지에 불필요한 정보가 노출되는지 확인**하는 것이다.

CARE 환경에서는 다음 순서로 확인한다.

1. 존재하지 않는 URL에 접근한다.
2. 접근이 차단된 디렉터리에 접근한다.
3. 게시글 조회, 검색, 로그인 등 파라미터가 있는 기능에 비정상 값을 넣어본다.
4. 화면에 서버 버전, 시스템 경로, DB 오류, 스택 트레이스가 보이는지 확인한다.
5. 조치 후 같은 요청을 다시 보내 사용자 정의 에러 페이지가 출력되는지 확인한다.

서버 안에서는 다음처럼 확인할 수 있다.

```bash
curl -i http://127.0.0.1/not-exist
curl -i http://127.0.0.1/vuln/code-injection/
```

브라우저에서는 다음 경로를 확인한다.

```text
http://172.168.10.10/not-exist
http://172.168.10.10/vuln/code-injection/
```

## 5. 조치 방안

본 프로젝트는 Apache/PHP 기반이므로 Apache 기본 에러 정보와 PHP 오류 출력을 같이 줄인다.

### 5.1 Apache 서버 버전 정보 숨기기

PDF p.704의 Apache 조치 기준은 다음 두 설정이다.

```apache
ServerTokens Prod
ServerSignature Off
```

Ubuntu Apache에서는 보통 `/etc/apache2/conf-available/security.conf`에 설정한다.

```bash
sudo nano /etc/apache2/conf-available/security.conf
```

아래 값이 있는지 확인하고, 없으면 추가하거나 기존 값을 수정한다.

```apache
ServerTokens Prod
ServerSignature Off
```

의미는 다음과 같다.

| 설정 | 의미 |
|---|---|
| `ServerTokens Prod` | 응답 헤더의 서버 정보를 최소화 |
| `ServerSignature Off` | 기본 에러 페이지 하단의 서버 버전/호스트 정보 표시 차단 |

### 5.2 사용자 정의 에러 페이지 지정

기본 Apache 에러 페이지 대신 CARE용 에러 페이지를 지정한다.

예시 파일을 만든다.

```bash
sudo mkdir -p /var/www/html/care/errors
sudo nano /var/www/html/care/errors/error.html
```

내용은 상세 원인을 쓰지 않고 간단하게 둔다.

```html
<!doctype html>
<html lang="ko">
<head>
  <meta charset="utf-8">
  <title>오류</title>
</head>
<body>
  <h1>요청을 처리할 수 없습니다.</h1>
  <p>입력한 주소를 다시 확인하거나 관리자에게 문의하세요.</p>
</body>
</html>
```

Apache site 설정에 `ErrorDocument`를 추가한다.

```bash
sudo nano /etc/apache2/sites-available/care.conf
```

예시는 다음과 같다.

```apache
<VirtualHost *:80>
    ServerName care.local
    DocumentRoot /var/www/html/care

    ErrorDocument 400 /errors/error.html
    ErrorDocument 403 /errors/error.html
    ErrorDocument 404 /errors/error.html
    ErrorDocument 405 /errors/error.html
    ErrorDocument 500 /errors/error.html
</VirtualHost>
```

### 5.3 PHP 오류 직접 출력 끄기

PHP 오류가 브라우저에 직접 출력되면 파일 경로와 함수명이 노출될 수 있다. 실습 서버에서는 PHP 설정도 확인한다.

```bash
php -i | grep "display_errors"
```

운영 기준 조치는 다음과 같다.

```ini
display_errors = Off
log_errors = On
```

화면에는 일반 오류 메시지만 보여주고, 실제 오류는 서버 로그에 남기는 구조가 안전하다.

### 5.4 설정 검사 및 반영

Apache 설정을 검사한다.

```bash
sudo apache2ctl configtest
```

정상이라면 다음처럼 나온다.

```text
Syntax OK
```

그 뒤 Apache를 reload한다.

```bash
sudo systemctl reload apache2
```

## 7. 판단

에러 유도 시 기본 서버 에러 페이지가 출력되고 서버 버전, 시스템 경로, DB 오류, 스택 트레이스가 노출되면 취약으로 판단한다.

본 항목의 핵심은 오류 자체를 없애는 것이 아니라, **오류가 발생했을 때 공격자에게 내부 정보를 보여주지 않는 것**이다.

CARE 환경에서는 Apache 기본 에러 페이지의 서버 버전 노출과 PHP/DB 오류 직접 출력을 중심으로 확인하면 된다.
