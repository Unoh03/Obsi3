## SQL Injection과의 차이

SQL Injection과 Code Injection은 모두 사용자 입력값이 단순 데이터가 아니라 질의문, 명령어, 필터, 표현식 같은 실행·해석 문맥의 일부로 처리된다는 점에서 같은 Injection 계열이다.

공통 구조는 다음과 같다.

```text
사용자 입력값
→ 애플리케이션이 질의문 또는 명령 문자열에 삽입
→ 뒤쪽 해석기가 해당 문자열을 문법으로 해석
→ 입력값이 데이터가 아니라 조건식·연산자·명령으로 동작
→ 인증 우회, 정보 조회, 명령 실행 등으로 이어짐
```

다만 SQL Injection과 Code Injection은 입력값을 해석하는 대상이 다르다. SQL Injection은 DBMS의 SQL parser가 입력값을 SQL 문법으로 해석하는 취약점이다. 반면 Code Injection은 운영체제 쉘, LDAP 서버, XML parser, XPath 엔진, 웹 서버, 템플릿 엔진 등 다양한 서버 측 실행·해석 환경이 입력값을 문법으로 처리하는 취약점이다.

| 구분            | SQL Injection                          | Code Injection                                                        |
| ------------- | -------------------------------------- | --------------------------------------------------------------------- |
| 주입 대상         | SQL 쿼리                                 | OS 명령어, LDAP 필터, SSI 지시어, XPath 질의, XML 파서, 템플릿 엔진 등                  |
| 해석 주체         | DBMS                                   | 운영체제 쉘, LDAP 서버, XML 파서, 웹 서버, 템플릿 엔진 등                               |
| 대표 문맥         | `SELECT`, `WHERE`, `UNION`, `ORDER BY` | `shell`, LDAP search filter, XML DTD, XPath, SSI, template expression |
| 데이터 모델        | 테이블, 행, 컬럼                             | 파일 시스템, 디렉터리 엔트리, XML 노드, 템플릿 객체, 서버 환경 변수 등                          |
| CARE 기존 기능 예시 | 로그인, 게시판 검색, 게시글 조회 SQL                | 기존 기능에는 직접 지점 없음                                                      |
| 대표 영향         | 인증 우회, DB 조회, DB 변조, 데이터 삭제            | 서버 명령 실행, 파일 열람, 내부 정보 유출, 원격 코드 실행                                   |
| 대표 방어         | Prepared Statement, 파라미터 바인딩, 입력값 검증   | 실행 문맥 분리, 화이트리스트 검증, 위험 함수 제거, 안전한 API 사용, 문맥별 이스케이프                  |

특히 LDAP Injection은 SQL Injection과 매우 유사하다. 두 취약점 모두 사용자 입력값이 질의 문자열에 직접 삽입되고, 그 입력값이 단순 데이터가 아니라 질의 문법으로 해석되면서 조건식이 변조된다.

차이는 질의를 해석하는 대상이다. SQL Injection은 DBMS가 해석하는 SQL 문장을 조작하는 반면, LDAP Injection은 LDAP 서버가 해석하는 search filter를 조작한다. 따라서 SQL Injection에서는 테이블, 컬럼, `WHERE`, `UNION` 등이 주요 개념이 되고, LDAP Injection에서는 DN, 엔트리, 속성, `&`, `|`, `!`, `*` 같은 LDAP filter 문법이 주요 개념이 된다.

예를 들어 SQL Injection은 다음과 같은 SQL 조건식을 조작한다.

```sql
SELECT * FROM member
WHERE id = '$id' AND pw = '$pw'
```

반면 LDAP Injection은 다음과 같은 LDAP search filter를 조작한다.

```text
(&(uid={id})(userPassword={pw}))
```

따라서 LDAP Injection은 “SQL Injection과 완전히 다른 공격”이라기보다, 같은 Injection 원리가 SQL이 아닌 LDAP filter 문법에서 발생하는 경우로 이해하는 것이 적절하다.

정리하면 SQL Injection은 입력값이 SQL 문맥에서 실행되는 취약점이고, Code Injection은 입력값이 서버 측 실행·해석 문맥에서 실행되는 취약점이다. LDAP Injection은 그중에서도 SQL Injection과 가장 유사한 형태의 질의문 조작 취약점이다.

# 4. 기존 기능 적용 가능성 분석

CARE 기본 기능에는 로그인, 회원가입, 게시판 목록, 게시글 작성, 게시글 조회, 파일 업로드, 파일 다운로드 기능이 존재한다.

기존 기능을 기준으로 코드 인젝션 6개 유형을 적용할 수 있는지 검토하면 다음과 같다.

| 세부 유형 | 기존 CARE 기능 적용 가능성 | 판단 |
|---|---|---|
| LDAP Injection | CARE에 LDAP 인증 또는 디렉터리 조회 기능이 없음 | 기존 기능 직접 적용 어려움 |
| OS Command Injection | 사용자 입력값을 운영체제 명령어로 전달하는 기능이 없음 | 기존 기능 직접 적용 어려움 |
| SSI Injection | CARE 게시글 입력값이 SSI 지시어로 처리되는 구조가 아님 | 기존 기능 직접 적용 어려움 |
| XPath Injection | CARE에 XML 데이터 조회 또는 XPath 질의 기능이 없음 | 기존 기능 직접 적용 어려움 |
| XXE Injection | CARE에 XML 업로드 또는 XML 파싱 기능이 없음 | 기존 기능 직접 적용 어려움 |
| SSTI | CARE는 PHP 단순 출력 구조이며 별도 서버 템플릿 엔진을 사용하지 않음 | 기존 기능 직접 적용 어려움 |

기존 CARE 기능에는 코드 인젝션 6개 세부 유형을 모두 직접 확인할 수 있는 기능이 존재하지 않았다. 따라서 본 실습에서는 PDF p.679~692의 코드 인젝션 항목을 학습하고 재현하기 위해 별도 실습 기능을 분리 구현한다. 이 구현은 운영 기능이 아니라 취약점 동작 원리를 확인하기 위한 실습용 기능이다.

다만 LDAP Injection은 SQL Injection과 유사하게 별도 LDAP 서버가 질의 필터를 해석해야 의미 있는 취약점이 된다. 실제 LDAP 서버 없이 하드코딩 또는 얕은 모의 구현으로 결과를 만드는 방식은 학습 가치가 낮고 보고서 신뢰도를 떨어뜨릴 수 있으므로, 이 문서에서는 LDAP Injection을 구현하지 않고 SQL Injection과의 유사점·차이점을 정리하는 비교 학습 항목으로 처리한다.

또한 CARE의 게시판 파일 업로드 기능은 PHP 웹쉘 업로드로 이어질 가능성이 있으나, 이는 PDF 항목상 코드 인젝션이 아니라 `악성 파일 업로드` 항목에서 별도 점검한다.

# 5. 실습 구현 범위

코드 인젝션 실습 기능은 기존 CARE 핵심 기능과 섞지 않고 별도 경로에 구현한다. LDAP Injection은 구현에서 제외하고 비교 학습으로 처리한다.

```text
/vuln/code-injection/
├─ index.php
├─ os-command-injection.php
├─ ssi-injection.php
├─ xpath-injection.php
├─ xxe-injection.php
└─ ssti.php
```

| 세부 유형                | 구현 여부  | 구현 파일                      | 실습 성격  | 비고                                                      |
| -------------------- | ------ | -------------------------- | ------ | ------------------------------------------------------- |
| LDAP Injection       | 제외     | -                          | 비교 학습  | 실제 LDAP 서버가 필요하므로 SQL Injection과 유사한 질의 필터 조작형 취약점으로 정리 |
| OS Command Injection | 구현     | `os-command-injection.php` | 실제 재현  | PHP에서 OS 명령 실행 함수를 사용해 재현                               |
| SSI Injection        | 조건부 구현 | `ssi-injection.php`        | 조건부 재현 | Apache SSI 설정 여부에 따라 실제 재현 범위 결정                        |
| XPath Injection      | 구현     | `xpath-injection.php`      | 실제 재현  | XML 데이터와 XPath 질의를 이용해 인증 우회 원리 재현                      |
| XXE Injection        | 구현     | `xxe-injection.php`        | 실제 재현  | XML 외부 엔티티 처리를 의도적으로 허용하여 재현                            |
| SSTI                 | 구현     | `ssti.php`                 | 실제 재현  | 실제 Twig 템플릿 엔진 기반 재현                                    |

실습 결과를 기록할 때는 각 유형을 다음 기준으로 구분한다.

| 구분 | 의미 |
|---|---|
| 실제 재현 | 서버 기능 또는 파서 동작을 통해 취약점이 실제로 발생함 |
| 모의 재현 | 실제 외부 시스템 없이 취약점 원리를 교육용 코드로 재현함 |
| 조건부 재현 | 서버 설정 또는 모듈 활성화 여부에 따라 재현 가능 여부가 달라짐 |
| 비교 학습 | 구현하지 않고 유사 취약점과의 차이, 필요한 전제 조건, 방어 원리를 정리함 |
| 해당 없음 | 현재 실습 환경 또는 기능 구조상 재현하지 않음 |

# 6. 유형별 실습 — 취약점 파악 및 조치 방안

## 6-1. LDAP Injection

### 개요

LDAP는 Lightweight Directory Access Protocol의 약자로, 사용자 계정, 조직, 부서, 그룹, 메일 주소, 인증 정보 같은 디렉터리 데이터를 조회하거나 관리하기 위한 프로토콜이다. 일반적인 웹 서비스에서는 LDAP 서버를 중앙 사용자 저장소로 두고, 로그인이나 사용자 검색 기능에서 LDAP 서버에 질의를 보내는 방식으로 사용될 수 있다.

LDAP에서 사용자는 하나의 엔트리로 저장되고, 각 엔트리는 `uid`, `cn`, `mail`, `userPassword` 같은 속성을 가진다. 예시는 다음과 같다.

```text
dn: uid=admin,ou=people,dc=example,dc=local
uid: admin
cn: 관리자
mail: admin@example.local
userPassword: adminpass
```

웹 애플리케이션이 LDAP에서 사용자를 찾을 때는 검색 필터를 만든다. 예를 들어 사용자 ID와 비밀번호를 기준으로 사용자를 찾는 LDAP 검색 필터는 다음처럼 구성될 수 있다.

```text
(&(uid={id})(userPassword={pw}))
```

LDAP Injection은 사용자 입력값이 LDAP 검색 필터 또는 DN 문자열에 검증 없이 삽입될 때 발생한다. 공격자는 LDAP 필터 문법을 조작하여 인증 조건을 우회하거나 디렉터리 데이터를 비인가 조회할 수 있다.

### SQL Injection과의 유사성

LDAP Injection은 SQL Injection과 추상 구조가 매우 유사하다.

```text
사용자 입력값
→ 애플리케이션이 질의 문자열에 삽입
→ 뒤쪽 서버가 해당 문자열을 문법으로 해석
→ 입력값이 데이터가 아니라 조건식 또는 연산자로 동작
→ 인증 우회 또는 조회 범위 확장
```

차이는 질의를 해석하는 서버와 문법이다.

| 구분       | SQL Injection                     | LDAP Injection                   |
| -------- | --------------------------------- | -------------------------------- |
| 대상 서버    | DBMS                              | LDAP 디렉터리 서버                     |
| 조작 대상    | SQL 쿼리                            | LDAP search filter               |
| 데이터 모델   | 테이블, 행, 컬럼                        | DN, 엔트리, 속성                      |
| 대표 문법    | `SELECT ... WHERE id='...'`       | `(&(uid=...)(userPassword=...))` |
| 주요 공격 문법 | `' OR '1'='1`, `UNION SELECT`, 주석 | `*`, `)(`, `(                    |
| 대표 영향    | 인증 우회, DB 조회·변조·삭제                | LDAP 인증 우회, 사용자·조직·그룹 정보 조회      |
| 대표 방어    | Prepared Statement, 파라미터 바인딩      | LDAP filter escaping, 화이트리스트 검증  |

따라서 LDAP Injection은 “SQL Injection과 완전히 다른 공격”이라기보다, 같은 Injection 원리가 SQL이 아닌 LDAP search filter 문맥에서 발생하는 경우로 이해하는 것이 적절하다.

### 실습 처리 방식

| 구분 | 내용 |
|---|---|
| 구현 여부 | 구현 제외 |
| 처리 방식 | SQL Injection과 유사한 질의 필터 조작형 취약점으로 비교 학습 |
| 제외 이유 | 실제 LDAP 서버 없이 하드코딩 또는 얕은 모의 구현으로 결과를 만드는 방식은 취약점 원리 재현성이 낮음 |
| 문서상 판단 | `비교 학습` |

본 실습 환경에는 실제 LDAP 서버가 없다. LDAP Injection은 LDAP 서버가 search filter를 실제로 해석해야 정상적인 취약점 구현이 가능하다. 따라서 이 문서에서는 별도의 `ldap-injection.php`를 만들지 않고, SQL Injection과의 공통점과 차이점을 정리하는 방식으로 처리한다.

### 조치 방안

- 사용자 입력값을 LDAP 필터에 직접 연결하지 않는다.
- 입력값은 영문, 숫자, 일부 허용 문자만 통과시키는 화이트리스트 방식으로 검증한다.
- LDAP 필터에서 특수한 의미를 가지는 `*`, `(`, `)`, `\`, NUL 문자를 이스케이프 처리한다.
- DN에 사용되는 특수문자와 Search Filter에 사용되는 특수문자를 구분해서 처리한다.
- 인증용 LDAP 검색에서는 검색 결과 개수와 인증 조건을 엄격히 확인한다.
- 웹 방화벽에 LDAP Injection 관련 특수문자와 패턴을 탐지하도록 룰셋을 적용한다.

### 판단

LDAP Injection은 SQL Injection과 매우 유사한 질의문 조작형 인젝션이다. 현재 CARE 실습 환경에는 LDAP 서버가 없으므로 구현 실습은 진행하지 않고, SQL Injection과의 비교를 통해 같은 Injection 원리가 SQL 이외의 질의 언어에서도 반복된다는 점을 학습한다.

## 6-2. OS Command Injection

### 개요

**OS Command Injection**은 사용자 입력값이 운영체제 명령어 문자열에 검증 없이 삽입될 때 발생한다. 공격자는 명령 연결 문자를 이용해 원래 의도한 명령 외에 추가 명령을 실행할 수 있다.

이 취약점의 핵심은 사용자 입력값이 단순한 데이터가 아니라 운영체제 쉘이 해석하는 명령어 문맥의 일부가 된다는 점이다.

### 정상 기능과 취약 원리

본 실습에서는 서버 Ping 테스트 기능을 예시로 사용한다.

정상적인 의도는 다음과 같다.

```text
사용자 입력값: 127.0.0.1
서버 실행 명령: ping -c 2 127.0.0.1
결과: 지정한 대상에 대한 ping 결과 출력
```

하지만 입력값 검증 없이 사용자의 `target` 값을 명령어 문자열에 직접 연결하면, 공격자는 명령 연결 문자를 삽입할 수 있다.

```text
사용자 입력값: 127.0.0.1; id
서버 실행 명령: ping -c 2 127.0.0.1; id
결과: ping 실행 후 id 명령까지 실행
```

여기서 `;`는 앞 명령의 성공 여부와 관계없이 다음 명령을 실행하는 쉘 문법이다. 따라서 `target` 값은 IP 주소가 아니라 명령어 일부로 해석된다.

```php
else {
        /*
         * 취약 모드:
         * 사용자 입력값을 OS 명령어 문자열에 그대로 연결한다.
         */
        $cmd = 'ping -c 2 ' . $target;
        $output = shell_exec($cmd . ' 2>&1') ?? '';

```

위 구조에서는 `target` 값이 검증 없이 명령어 문자열에 연결된다. 따라서 공격자가 `;`, `&&`, `|` 같은 명령 연결 문자를 입력하면 운영체제 쉘이 이를 명령어 문법으로 해석할 수 있다.

취약 모드에서는 `target` 파라미터를 별도 검증 없이 `ping -c 2 ` 뒤에 그대로 연결한다. 이때 최종 명령어는 하나의 문자열로 만들어지고, `shell_exec()`가 해당 문자열을 운영체제 쉘에 전달한다.

따라서 사용자가 `127.0.0.1;id`를 입력하면 서버가 실행하는 명령은 다음처럼 변한다.

```php
ping -c 2 127.0.0.1;id
````

여기서 `;`는 쉘에서 명령을 분리하는 문자이므로, 서버는 `ping` 실행 후 `id` 명령까지 이어서 실행한다. 이 때문에 입력값이 단순한 IP 주소가 아니라 운영체제 명령어 문법의 일부로 처리된다.

### 취약점 파악

| 구분    | 요청                                                        | 예상 결과                        |
| ----- | --------------------------------------------------------- | ---------------------------- |
| 정상 요청 | `?mode=vuln&target=127.0.0.1`                             | ping 결과만 출력                  |
| 공격 요청 | `?mode=vuln&target=127.0.0.1%3Bid`                        | ping 결과와 함께 `id` 명령 실행 결과 출력 |
| 판단 기준 | 응답에 `uid=`, `gid=`, `www-data` 등 웹 서버 프로세스 권한 정보가 출력되면 취약 |                              |
공격 요청의 응답에 `uid=`, `gid=`, `www-data` 등 웹 서버 프로세스 권한 정보가 출력되면 OS Command Injection 취약점이 재현된 것으로 판단한다.

브라우저 주소창에서는 세미콜론을 직접 입력하면 처리 방식이 달라질 수 있으므로, 요청 예시는 URL 인코딩된 `%3B`를 사용한다.

```text
127.0.0.1%3Bid
```

이는 서버에서 다음 입력값으로 해석된다.

```text
127.0.0.1;id
```

Linux 환경에서 주의해야 할 대표 명령 연결 문자는 다음과 같다.

| 문자              | 의미                           |
| --------------- | ---------------------------- |
| `&`             | 첫 번째 명령을 백그라운드 실행 후 다음 명령 실행 |
| `&&`            | 앞 명령이 성공했을 때 다음 명령 실행        |
| `&#124;`        | 앞 명령의 출력을 다음 명령의 입력으로 전달     |
| `&#124;&#124;`  | 앞 명령이 실패했을 때 다음 명령 실행        |
| `;`             | 앞 명령 성공 여부와 관계없이 다음 명령 실행    |
| `` `command` `` | 명령 실행 결과를 치환                 |
| `$()`           | 괄호 안 명령 실행 결과를 치환            |
| `>` / `>>`      | 명령 실행 결과를 파일로 저장             |
| `<`             | 파일 내용을 명령 입력으로 전달            |
`&#124;` == `|`
### 조치 확인

**조치 모드**에서는 두 가지 방어를 적용한다.

**safe** 모드에서는 먼저 입력값이 허용된 형식인지 검사한다. 실습 코드에서는 IP 주소나 도메인에 필요한 영문, 숫자, 점, 하이픈만 허용하고, `;`, `&`, `|`, `` ` ``, `$`, `<`, `>` 같은 명령 연결 문자가 포함되면 실행을 차단한다.

또한 허용된 입력값이라도 `escapeshellarg()`로 감싸서 쉘이 입력값을 명령어 문법이 아니라 하나의 인자로 처리하도록 한다.
```php
if ($target !== '') {
    if ($mode === 'safe') {
        /*
         * 조치 모드:
         * IP 주소 또는 도메인처럼 필요한 형식만 허용한다.
         * 명령 연결 문자 ;, &, |, `, $, <, > 등은 허용하지 않는다.
         */
        if (!preg_match('/^[a-zA-Z0-9.-]+$/', $target)) {
            $error = '허용되지 않은 문자가 포함되어 실행을 차단했다.';
        } else {
            $cmd = 'ping -c 2 ' . escapeshellarg($target);
            $output = shell_exec($cmd . ' 2>&1') ?? '';
        }
```

| 조치                    | 설명                                      |
| --------------------- | --------------------------------------- |
| 화이트리스트 검증             | IP 주소나 도메인에 필요한 문자만 허용하고, 명령 연결 문자를 차단  |
| `escapeshellarg()` 적용 | 입력값을 쉘 인자로 안전하게 감싸 명령어 문법으로 해석되지 않도록 처리 |

조치 확인 요청은 다음과 같다.

```text
?mode=safe&target=127.0.0.1;id
```

예상 결과는 다음과 같다.

```text
허용되지 않은 문자가 포함되어 실행 차단
```

또는 입력값이 하나의 인자로 처리되어 `id` 명령이 별도로 실행되지 않아야 한다.

### 조치 방안

OS Command Injection을 방지하기 위한 조치 방안은 다음과 같다.

* OS 명령어를 직접 호출하는 기능을 가능한 제거한다.
* 명령 실행이 반드시 필요한 경우 허용된 값만 입력받도록 화이트리스트를 적용한다.
* PHP 내장 함수나 안전한 API로 대체 가능한 기능은 OS 명령 호출을 사용하지 않는다.
* 입력값을 쉘 명령어 문자열에 직접 연결하지 않는다.
* `escapeshellarg()`, `escapeshellcmd()` 등을 적용하되, 이것만으로 안전하다고 단정하지 않는다.
* 웹 서버 프로세스 권한을 최소화한다.
* 웹 방화벽에서 명령 연결 특수문자를 탐지하도록 룰셋을 적용한다.

## 6-3. SSI Injection

### 개요

**SSI Injection**은 사용자 입력값이 Server Side Includes 지시어로 처리될 때 발생하는 취약점이다. SSI가 활성화된 서버에서 공격자가 SSI 지시어를 삽입하면 서버 경로 노출, 파일 포함, 명령 실행 등이 가능할 수 있다.

**SSI**는 애플리케이션 코드만으로 동작하는 기능이 아니라, 웹 서버가 HTML 응답을 보내기 전에 문서 안의 특수 지시어를 먼저 해석하는 기능이다. Apache에서는 주로 `.shtml` 파일을 SSI 처리 대상으로 설정한다.

예를 들어 `.shtml` 파일 안에 다음 내용이 있으면, 브라우저에 주석 문자열이 그대로 출력되는 것이 아니라 Apache가 응답을 보내기 전에 해당 지시어를 처리한다.

```html
<!--#echo var="DOCUMENT_ROOT" -->
```

일반 HTML 주석과 SSI 지시어의 차이는 다음과 같다.

|구분|예시|처리 방식|
|---|---|---|
|일반 HTML 주석|`<!-- hello -->`|브라우저에서 주석으로 취급된다.|
|SSI 지시어|`<!--#echo var="DOCUMENT_ROOT" -->`|서버가 응답 전에 해석하여 결과값을 삽입한다.|

따라서 SSI Injection의 핵심은 사용자 입력값이 단순 문자열로 출력되는 것이 아니라, SSI 처리 대상 문서 안에 삽입되어 서버 측 지시어로 해석되는지 여부다.

### 정상 기능과 취약 원리

정상적인 의도는 사용자가 입력한 HTML 또는 문자열을 결과 페이지에 반영하는 것이다.

```text
사용자 입력값: 안녕하세요
서버 동작: 입력값을 결과 페이지에 출력
결과: 안녕하세요
```

하지만 SSI가 활성화된 `.shtml` 문서에 사용자 입력값이 검증 없이 반영되면, 공격자가 입력한 SSI 지시어가 단순 문자열이 아니라 서버 측 명령으로 처리될 수 있다.

```html
<!--#echo var="DOCUMENT_ROOT" -->
<!--#exec cmd="id" -->
```

취약한 흐름은 다음과 같다.

```text
사용자 입력: <!--#exec cmd="id" -->
↓
서버가 result.shtml에 입력값을 그대로 저장
↓
Apache가 result.shtml을 SSI 문서로 해석
↓
id 명령 실행
↓
브라우저 응답에 uid=33(www-data) 출력
```

즉 SSI Injection은 “사용자 입력값이 SSI 처리 대상 문서에 들어가는가”와 “웹 서버가 해당 문서를 SSI로 해석하는가”가 동시에 맞을 때 재현된다.

### PDF 점검 payload와 본 실습 적용 범위

PDF에서 제시하는 대표 점검 payload는 다음과 같다.

```html
<!--#echo var="DOCUMENT_ROOT" -->
```

심화 payload 예시는 다음과 같다.

```html
<!--#exec cmd="ls -al" -->
```

또한 요청 헤더에 SSI 지시어를 삽입하는 방식도 점검 대상이 될 수 있다.

```http
Referer: <!--#exec cmd="/bin/ps ax"-->
User-Agent: <!--#include virtual="/proc/version"-->
```

본 실습에서 확인한 항목은 다음과 같다.

|PDF 예시|본 실습 적용|상태|
|---|---|---|
|`<!--#echo var="DOCUMENT_ROOT" -->`|`ssi-smoke.shtml`에서 실행하여 `DOCUMENT_ROOT` 출력 확인|수행|
|`<!--#exec cmd="ls -al" -->`|같은 `#exec` 계열인 `<!--#exec cmd="id" -->`로 대체하여 명령 실행 확인|수행|
|`Referer`, `User-Agent` 헤더에 SSI 지시어 삽입|현재 실습 페이지는 헤더 값이 아니라 POST 본문 `content` 값을 `.shtml`에 반영함|미구현|

`#exec cmd` payload는 PDF 예시의 `ls -al` 대신 `id`를 사용했다. `id`는 웹 서버 프로세스 권한이 `www-data`로 표시되므로, SSI를 통해 서버 명령이 실행되었다는 증거로 더 명확하다.

요청 헤더 기반 SSI Injection은 애플리케이션이 `Referer` 또는 `User-Agent` 값을 수집하여 SSI 처리 대상 문서에 저장하거나 반영할 때 성립한다. 현재 실습은 헤더 기반이 아니라 본문 입력값 기반으로 재현한다. 따라서 헤더 payload는 이론적 점검 대상과 추가 구현 가능 항목으로 남긴다.

### 실습 구현

|구분|내용|
|---|---|
|구현 파일|`/vuln/code-injection/ssi/ssi-injection.php`|
|결과 파일|`/vuln/code-injection/ssi/generated/result.shtml`|
|구현 방식|사용자 입력값을 `.shtml` 문서에 반영하고 SSI 처리 결과 확인|
|입력값|`content`|
|취약 구조|입력값이 SSI 처리 대상 `.shtml` 문서에 그대로 반영됨|
|실습 성격|실제 재현 / Apache SSI 설정 필요|

취약 모드에서는 사용자가 입력한 값을 HTML 엔티티 처리하지 않고 `.shtml` 파일에 그대로 저장한다. Apache SSI 설정이 활성화되어 있으면, 해당 파일을 열 때 SSI 지시어가 서버 측에서 처리된다.

```php
if ($mode === 'safe') {
    $body = h($content);
} else {
    $body = $content;
}
```

이 구조에서 취약 모드는 `$content`를 그대로 사용하므로 `<!--#echo var="DOCUMENT_ROOT" -->` 또는 `<!--#exec cmd="id" -->` 같은 입력값이 `.shtml` 파일에 직접 삽입된다.

반대로 safe 모드는 `h($content)`를 사용해 `<`, `>`, `"`, `&` 등을 HTML 엔티티로 변환한다. 이 경우 SSI 지시어가 `<!--#... -->` 형태로 해석되지 못하고, 화면에 문자로 표시된다.

| `Require all granted`             | 해당 디렉터리의 파일을 웹에서 접근 가능하게 한다.             |

#### 실제 취약 재현

앞의 `ssi-smoke.shtml` 테스트는 하드코딩된 SSI 지시어가 서버에서 실행되는지 확인한 환경 검증 단계다. 이제 `ssi-injection.php`의 입력창에 SSI payload를 직접 입력하고, 해당 입력값이 `generated/result.shtml`에 저장된 뒤 Apache SSI 파서에 의해 실행되는지 확인한다.

먼저 다음 URL로 SSI Injection 실습 페이지에 접속했다.

```text
http://172.168.10.10/vuln/code-injection/ssi/ssi-injection.php
```

![[Pasted image 20260611142147.png]]

이 화면은 취약점 실행 결과가 아니라, 사용자 입력 기반 SSI Injection을 재현하기 위한 입력 화면이다. `mode`는 `vulnerable`로 선택되어 있고, `content` 입력창에는 다음 SSI payload가 들어 있다.

```html
<h2>사용자 입력 영역</h2>
<p>DOCUMENT_ROOT: <!--#echo var="DOCUMENT_ROOT" --></p>
<pre><!--#exec cmd="id" --></pre>
```

이 payload의 의미는 다음과 같다.

|payload|의미|
|---|---|
|`<!--#echo var="DOCUMENT_ROOT" -->`|Apache가 SSI 지시어를 해석하면 서버의 DocumentRoot 경로가 출력된다.|
|`<!--#exec cmd="id" -->`|Apache가 SSI `exec` 지시어를 허용하면 서버에서 `id` 명령이 실행된다.|

취약 모드에서는 입력값을 HTML 엔티티로 변환하지 않고 그대로 `.shtml` 파일에 저장한다.

```php
if ($mode === 'safe') {
    $body = h($content);
} else {
    $body = $content;
}
```

따라서 `vulnerable` 모드에서 `Generate .shtml` 버튼을 누르면, 입력창의 SSI 지시어가 `generated/result.shtml` 파일에 그대로 저장된다. 이후 생성된 `.shtml` 파일을 열면 Apache가 해당 파일을 SSI 문서로 처리하면서 `DOCUMENT_ROOT` 출력과 `id` 명령 실행이 발생할 수 있다.

재현 흐름은 다음과 같다.

```text
ssi-injection.php 접속
↓
vulnerable 모드 선택
↓
content 입력창에 SSI payload 입력
↓
Generate .shtml 클릭
↓
generated/result.shtml 생성
↓
result.shtml 요청
↓
Apache SSI 파서가 <!--#echo ... -->, <!--#exec ... --> 처리
↓
응답 화면에 DOCUMENT_ROOT와 uid=33(www-data) 출력
```
![[Pasted image 20260611143323.png]]
취약 모드의 판단 기준은 다음과 같다.

| 확인 항목                  | 판단                                     |
| ---------------------- | -------------------------------------- |
| `DOCUMENT_ROOT` 값 출력   | SSI `echo` 지시어가 서버에서 처리됨               |
| `uid=33(www-data)` 출력  | SSI `exec cmd="id"` 지시어가 웹 서버 권한으로 실행됨 |
| SSI 지시어가 문자 그대로 보이지 않음 | 서버가 SSI 지시어를 단순 문자열이 아니라 지시어로 해석함      |

따라서 취약 모드 결과 화면에서 `/var/www/html/care`와 `uid=33(www-data)`가 출력되면, 사용자 입력값이 SSI 처리 문서에 반영되어 서버 측에서 실행된 것으로 판단한다.

### 취약점 파악

이번 SSI Injection은 단순히 SSI 기능이 켜졌는지만 확인한 것이 아니라, **사용자 입력값과 HTTP 요청 헤더 값이 SSI 처리 대상 문서에 반영될 때 서버에서 실행되는지**까지 확인했다.

| 1 | SSI 스모크 테스트 | 하드코딩된 `ssi-smoke.shtml`에서 `DOCUMENT_ROOT`와 `id` 실행 결과 출력 | Apache가 `.shtml`을 SSI 처리 대상으로 해석함 |


### 조치 확인

safe 모드에서는 사용자 입력값을 HTML 엔티티로 변환하여 SSI 지시어가 `<!--#... -->` 형태로 해석되지 못하게 한다.


따라서 safe 모드에서 같은 payload를 입력하거나 헤더에 넣었을 때는 다음 문자열이 실행되지 않고 화면에 문자로 표시되어야 한다.

```html

<!--#exec cmd="id" -->
```

|구분|기대 결과|

### 조치 방안

- SSI 기능을 사용하지 않으면 비활성화한다.
    
- SSI가 필요하더라도 전체 DocumentRoot가 아니라 필요한 디렉터리에만 제한적으로 허용한다.
    
- 사용자 입력값을 SSI 처리 대상 문서에 직접 반영하지 않는다.
    
- `<`, `>`, `"`, `'`, `#`, `&` 등 SSI 지시어 구성에 사용되는 문자를 HTML 엔티티로 처리한다.
    
- GET, POST, 쿠키, URL, 요청 헤더 등 모든 입력 지점에 대해 검증을 적용한다.
    
- `#exec`가 필요하지 않다면 `IncludesNOEXEC` 등으로 명령 실행을 제한한다.
    
- 웹 방화벽에서 `<!--#echo`, `<!--#exec`, `<!--#include` 같은 SSI 지시어 패턴을 탐지하도록 룰셋을 적용한다.

## 6-4. XPath Injection

### 개요

**XPath Injection**은 사용자 입력값이 XPath 질의 문자열에 검증 없이 삽입될 때 발생하는 취약점이다. 공격자는 XPath 조건식을 조작하여 XML 기반 인증을 우회하거나, XML 문서 내부 데이터를 비인가 조회할 수 있다.

XPath는 XML 문서 안에서 특정 노드나 값을 찾기 위한 질의 언어다. SQL이 데이터베이스 테이블을 조회하는 언어라면, XPath는 XML 문서의 노드 구조를 조회하는 언어라고 볼 수 있다.

예를 들어 XML 사용자 데이터가 다음과 같다고 가정한다.

```xml
<users>
  <user>
    <username>admin</username>
    <password>admin123</password>
    <role>admin</role>
  </user>
</users>
```

정상적인 XPath 로그인 질의는 다음처럼 구성될 수 있다.

```text
//user[username/text()='admin' and password/text()='admin123']
```

이 질의는 `username`이 `admin`이고 `password`가 `admin123`인 `<user>` 노드를 찾는다.

하지만 사용자 입력값을 검증 없이 XPath 질의 문자열에 직접 연결하면, 공격자는 XPath 조건식 자체를 바꿀 수 있다. 이 점에서 XPath Injection은 SQL Injection과 구조가 매우 유사하다.

|구분|SQL Injection|XPath Injection|
|---|---|---|
|대상|DBMS SQL 질의|XML XPath 질의|
|입력 위치|SQL 문자열 내부|XPath 문자열 내부|
|공격 방식|조건식 조작, 인증 우회, 데이터 조회|조건식 조작, 인증 우회, XML 데이터 조회|
|대표 입력|`' OR '1'='1`|`' or '1'='1`|

핵심은 사용자 입력값이 단순 데이터가 아니라, XPath 파서가 해석하는 질의 문법의 일부가 된다는 점이다.

### 정상 기능과 취약 원리

본 실습에서는 XML 사용자 데이터를 조회하는 로그인 기능을 예시로 사용한다.

정상적인 의도는 다음과 같다.

```text
사용자 입력값:
username = admin
password = admin123

서버 XPath 질의:
//user[username/text()='admin' and password/text()='admin123']

결과:
admin 사용자 인증 성공
```

하지만 취약 모드에서 사용자의 `username`, `password` 값을 XPath 질의 문자열에 그대로 연결하면, 공격자는 비밀번호 입력값에 XPath 조건식을 삽입할 수 있다.

공격 입력 예시는 다음과 같다.

```text
' or '1'='1
```

예를 들어 `username=admin`, `password=' or '1'='1`을 입력하면 서버가 생성하는 XPath 질의는 다음처럼 변한다.

```text
//user[username/text()='admin' and password/text()='' or '1'='1']
```

여기서 `'1'='1'`은 항상 참인 조건이다. 따라서 원래는 올바른 비밀번호가 아니어도 XPath 조건식이 참으로 평가되어 사용자 노드가 조회될 수 있다.

즉 취약점의 본질은 다음과 같다.

```text
사용자 입력값
↓
XPath 질의 문자열에 직접 연결
↓
XPath 파서가 입력값을 문자열이 아니라 조건식 일부로 해석
↓
인증 조건 우회 또는 XML 데이터 조회 가능
```

### 실습 구현

|구분|내용|
|---|---|
|구현 파일|`/vuln/code-injection/xpath-injection.php`|
|구현 방식|XML 사용자 데이터와 XPath 로그인 질의 구현|
|입력값|`username`, `password`|
|취약 구조|사용자 입력값을 XPath 질의에 직접 연결|
|관련 함수|`DOMDocument`, `DOMXPath::query()`|
|실습 성격|실제 재현|

실습 페이지는 내부에 XML 사용자 데이터를 가지고 있고, 입력한 `username`, `password` 값으로 로그인 여부를 판단한다.

취약 모드에서는 다음처럼 사용자 입력값을 XPath 질의에 그대로 연결한다.

```php
$query = "//user[username/text()='$username' and password/text()='$password']";
$nodes = $xpath->query($query);
```

조치 모드에서는 사용자 입력값을 XPath 질의 문자열에 직접 넣지 않는다. 먼저 허용된 문자 형식인지 검사하고, 이후 XML의 `<user>` 노드를 조회한 뒤 PHP 문자열 비교로 인증 여부를 판단한다.

```php
if (!preg_match('/^[a-zA-Z0-9_-]{1,32}$/', $username)
    || !preg_match('/^[a-zA-Z0-9_-]{1,32}$/', $password)) {
    $error = '허용되지 않은 문자가 포함되어 요청을 차단했다.';
} else {
    $nodes = $xpath->query('//user');

    foreach ($nodes as $node) {
        if (node_text($node, 'username') === $username
            && node_text($node, 'password') === $password) {
            $matchedUsers[] = extract_user($node);
        }
    }
}
```

### 취약점 파악

이번 XPath Injection은 정상 인증, 실패 인증, 취약 모드 공격, safe 모드 차단을 순서대로 확인했다.

| 단계 | 입력 | 결과 | 의미 |
|---|---|---|---|
| 정상 로그인 | `admin` / `admin123` | `admin` 사용자 1명 조회 | 정상 인증 흐름 확인 |
| 로그인 실패 | `admin` / 임의의 틀린 비밀번호 | 조회 결과 없음 | 비밀번호 검증이 정상 동작함 |
| 취약 모드 공격 | `admin` / `' or '1'='1` | `admin`, `doctor`, `guest` 조회 | XPath 조건식 조작으로 인증 조건 우회 |
| safe 모드 공격 | `admin` / `' or '1'='1` | 요청 차단 | 화이트리스트 검증과 질의 분리 조치 확인 |

취약 모드에서 공격 입력을 넣으면 다음 XPath 질의가 생성된다.

```text
//user[username/text()='admin' and password/text()='' or '1'='1']
```

이 질의에는 항상 참인 조건인 `'1'='1'`이 포함된다. 그 결과 원래 비밀번호가 일치하지 않아도 XPath 조건식 전체가 참으로 평가될 수 있다.

실제 취약 모드에서는 `username=admin`, `password=' or '1'='1` 입력만으로 XML 문서 안의 사용자 데이터가 조회되었다.

```text
로그인 성공 또는 사용자 데이터 조회 성공
admin  / admin  / system
doctor / doctor / medical
guest  / viewer / public
```

따라서 이 실습에서는 단순히 `admin` 로그인만 우회된 것이 아니라, XPath 조건식 조작으로 XML 내부의 여러 `<user>` 노드가 함께 조회되는 결과까지 확인되었다.

심화 점검에서는 `count()`, `string-length()` 같은 XPath 함수를 이용해 XML 구조나 데이터 길이를 추론할 수 있다. 다만 본 실습에서는 인증 우회와 사용자 노드 조회를 중심으로 재현한다.

### 조치 확인

safe 모드에서는 두 가지 방어를 적용한다.

|조치|설명|
|---|---|
|화이트리스트 검증|`username`, `password`를 영문, 숫자, `_`, `-`로 제한한다.|
|XPath 질의 분리|사용자 입력값을 XPath 질의 문자열에 직접 연결하지 않고, XML 노드를 조회한 뒤 PHP 문자열 비교를 수행한다.|

safe 모드의 핵심은 단순히 특수문자를 몇 개 막는 것이 아니라, 사용자 입력값이 XPath 문법으로 해석될 기회를 제거하는 것이다.

```php
$query = '//user';
$nodes = $xpath->query($query);

foreach ($nodes as $node) {
    if (node_text($node, 'username') === $username
        && node_text($node, 'password') === $password) {
        $matchedUsers[] = extract_user($node);
    }
}
```

따라서 safe 모드에서 다음 공격 입력을 넣으면 차단되어야 한다.

```text
' or '1'='1
```

기대 결과는 다음과 같다.

```text
허용되지 않은 문자가 포함되어 요청을 차단했다.
```

또는 최소한 사용자 데이터가 조회되지 않아야 한다.

> [!important]  
> 본 실습의 safe 모드는 XPath Injection 조치 방안 중 “화이트리스트 방식으로 입력값을 제한”하고, “사용자 입력값을 XPath 질의에 직접 연결하지 않기”를 동시에 보여준다. 특정 특수문자만 블랙리스트로 막는 방식보다, 허용된 형식 이외의 모든 입력을 거부하는 방식이 더 안전하다.

### 조치 방안

XPath Injection을 방지하기 위한 조치 방안은 다음과 같다.

- XPath 질의에 사용자 입력값을 직접 연결하지 않는다.
    
- 입력값은 필요한 문자만 허용하는 화이트리스트 방식으로 검증한다.
    
- 로그인 ID, 검색 키워드 등 입력값별로 허용 가능한 형식을 좁게 정의한다.
    
- XPath 질의 오동작을 유발할 수 있는 `(`, `)`, `=`, `'`, `"`, `[`, `]`, `:`, `,`, `*`, `/` 등의 문자를 제한한다.
    
- 특정 문자만 블랙리스트로 막기보다 허용된 형식 이외의 모든 입력을 거부한다.
    
- XPath 질의 오류 메시지에 XML 구조, XPath 질의문, 서버 경로 등이 노출되지 않도록 한다.
    
- 웹 방화벽에 XPath Injection 관련 특수문자와 패턴 룰셋을 적용한다.

```php
            || !preg_match('/^[a-zA-Z0-9_-]{1,32}$/', $password)
```

### 판단

XPath Injection 재현은 완료된 것으로 판단한다.

근거는 세 가지다. 첫째, 정상 로그인에서는 올바른 `admin/admin123` 조합으로만 `admin` 사용자 1명이 조회되었다. 둘째, 틀린 비밀번호를 입력했을 때는 조회 결과가 없었다. 셋째, 취약 모드에서 `password=' or '1'='1`을 입력하자 생성된 XPath 질의에 항상 참인 조건이 삽입되었고, `admin`, `doctor`, `guest` 사용자 데이터가 함께 조회되었다.

따라서 취약 모드에서는 사용자 입력값이 단순 문자열이 아니라 XPath 조건식의 일부로 해석되어 인증 조건이 우회되었다고 볼 수 있다. 반대로 safe 모드에서는 동일한 payload가 `(blocked)` 처리되고 `허용되지 않은 문자가 포함되어 요청을 차단했다.`는 메시지가 출력되었으므로, 화이트리스트 검증과 XPath 질의 분리 방식이 조치로 동작한 것으로 판단한다.

정리하면, 이 실습의 핵심은 **XPath 질의에 사용자 입력값을 직접 문자열 연결하면 SQL Injection과 같은 방식으로 조건식 조작이 가능하다**는 점이다. 방어는 입력값을 XPath 문법으로 해석될 위치에 넣지 않고, 필요한 경우 허용 문자 검증 후 애플리케이션 코드에서 값 비교를 수행하는 방향으로 잡아야 한다.

## 6-5. XXE Injection

### 개요

**XXE Injection**은 XML External Entity Injection의 약자로, XML 파서가 외부 엔티티를 처리할 때 발생할 수 있는 취약점이다. 공격자는 XML 문서 안에 외부 엔티티를 선언하고, 서버가 해당 엔티티를 확장하도록 유도하여 서버 로컬 파일이나 내부 자원을 참조하게 만들 수 있다.

XML 문서에서는 `DOCTYPE` 선언 안에 엔티티를 정의할 수 있다. 예를 들어 다음 XML은 `xxe`라는 엔티티를 만들고, 그 값을 서버의 `/etc/hostname` 파일로 지정한다.

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE data [
  <!ENTITY xxe SYSTEM "file:///etc/hostname">
]>
<data>&xxe;</data>
```

만약 서버 XML 파서가 외부 엔티티를 허용하고 엔티티 값을 실제 내용으로 치환한다면, `<data>&xxe;</data>`는 단순 문자열이 아니라 서버 파일 내용을 읽어오는 동작으로 이어질 수 있다.

정상적인 XML 처리 흐름은 다음과 같다.

```text
사용자 XML 입력
↓
XML 파서가 문서 구조 파싱
↓
애플리케이션이 필요한 노드 값만 읽음
```

취약한 XML 처리 흐름은 다음과 같다.

```text
사용자 XML 입력
↓
DOCTYPE / ENTITY 선언 포함
↓
XML 파서가 외부 엔티티를 로드하고 치환
↓
서버 로컬 파일 또는 내부 자원 내용이 응답에 포함
```

따라서 **XXE Injection의 핵심**은 사용자 입력 XML이 단순 데이터가 아니라, XML 파서가 해석하는 선언부와 외부 참조 문맥까지 포함한다는 점이다.

### 정상 기능과 취약 원리

본 실습에서는 사용자가 입력한 XML 문서를 서버가 파싱한 뒤, 루트 노드의 텍스트 내용을 출력하는 기능을 예시로 사용한다.

정상 입력은 다음과 같다.

```xml
<?xml version="1.0" encoding="UTF-8"?>
<data>Hello XML</data>
```

정상 처리 결과는 다음과 같다.

```text
Root element: data
Text content: Hello XML
```

하지만 취약 모드에서 XML 파서가 외부 엔티티를 확장하도록 설정되어 있으면, 공격자는 다음과 같은 XML을 입력할 수 있다.

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE data [
  <!ENTITY xxe SYSTEM "file:///etc/hostname">
]>
<data>&xxe;</data>
```

이때 XML 파서가 `&xxe;`를 `/etc/hostname` 파일 내용으로 치환하면, 응답에는 사용자가 직접 입력하지 않은 서버 로컬 파일 내용이 출력된다.

본 실습에서는 `/etc/passwd` 대신 `/etc/hostname`을 사용한다. `/etc/hostname`은 서버 로컬 파일 읽기 가능성을 확인하기에 충분하면서, 계정 목록처럼 민감한 정보가 포함될 가능성이 낮아 증거로 남기기 적합하다.

### 실습 구현

|구분|내용|
|---|---|
|구현 파일|`/vuln/code-injection/xxe-injection.php`|
|구현 방식|사용자가 입력한 XML 본문을 `DOMDocument`로 파싱|
|입력값|XML 본문|
|취약 구조|외부 엔티티 확장을 허용하는 XML 파서 옵션 사용|
|관련 함수|`DOMDocument::loadXML()`|
|취약 모드 핵심 옵션|`LIBXML_NOENT`, `LIBXML_DTDLOAD`|
|조치 모드 핵심 옵션|`LIBXML_NONET`, `DOCTYPE`/`ENTITY` 차단, `LIBXML_NOENT` 미사용|
|실습 성격|실제 재현|

취약 모드의 핵심 구조는 다음과 같다.

```php
$dom->loadXML($xmlInput, LIBXML_NOENT | LIBXML_DTDLOAD);
```

`LIBXML_NOENT`는 엔티티 참조를 실제 값으로 치환하게 만들고, `LIBXML_DTDLOAD`는 DTD를 로드하게 만든다. 이 조합이 잘못 사용되면 외부 엔티티가 확장되어 XXE Injection이 발생할 수 있다.

조치 모드에서는 `DOCTYPE`, `ENTITY`, `SYSTEM`, `PUBLIC` 같은 외부 엔티티 관련 선언을 차단하고, `LIBXML_NOENT`를 사용하지 않는다.

```php
if (preg_match('/<!\s*DOCTYPE|<!\s*ENTITY|\bSYSTEM\b|\bPUBLIC\b/i', $xmlInput)) {
    $error = 'DTD 또는 외부 엔티티 선언이 포함되어 요청을 차단했다.';
} else {
    $dom->loadXML($xmlInput, LIBXML_NONET);
}
```

### 취약점 파악

이번 XXE Injection은 정상 XML 파싱, 취약 모드의 외부 엔티티 확장, safe 모드 차단을 순서대로 확인했다.

| 단계 | 입력 | 결과 | 의미 |
|---|---|---|---|
| 정상 XML 입력 | `<data>Hello XML</data>` | 입력 화면 정상 표시 | XML 입력 기능 확인 |
| 정상 XML 파싱 | `<data>Hello XML</data>` | `Text Content`에 `Hello XML` 출력 | 기본 XML 파싱 정상 |
| 취약 모드 공격 | `file:///etc/hostname` 외부 엔티티 선언 | `Text Content`에 `webserv` 출력 | 서버 로컬 파일 내용 노출 |
| safe 모드 공격 | 같은 XXE payload | `DTD 또는 외부 엔티티 선언이 포함되어 요청을 차단했다.` 출력 | 조치 효과 확인 |

정상 XML 입력은 다음과 같다.

```xml
<?xml version="1.0" encoding="UTF-8"?>
<data>Hello XML</data>
```

취약 모드에서 사용하는 XXE payload는 다음과 같다.

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE data [
  <!ENTITY xxe SYSTEM "file:///etc/hostname">
]>
<data>&xxe;</data>
```

공격 성공 시 `Text Content` 영역에는 사용자가 입력한 `&xxe;` 문자열이 아니라 서버의 `/etc/hostname` 파일 내용이 출력된다. 이번 실습에서는 다음 값이 출력되었다.

```text
webserv
```

또한 `Serialized Root Node`가 `<data>webserv</data>` 형태로 표시되었다. 이는 XML 파서가 `&xxe;` 엔티티 참조를 서버 로컬 파일 내용으로 치환한 뒤 DOM에 반영했음을 보여준다.

> [!warning]  
> `/etc/hostname`의 실제 출력값은 서버 설정에 따라 다를 수 있다. 중요한 것은 입력 XML에 직접 적지 않은 서버 로컬 파일 내용이 응답에 출력되는지 여부다.

### 조치 확인

safe 모드에서는 같은 XXE payload를 입력했을 때 요청이 차단되어야 한다.

이번 실습에서 safe 모드는 다음 메시지로 요청을 차단했다.

```text
DTD 또는 외부 엔티티 선언이 포함되어 요청을 차단했다.
```

**safe 모드의 핵심**은 다음 두 가지다.

|조치|설명|
|---|---|
|DTD/ENTITY 선언 차단|`DOCTYPE`, `ENTITY`, `SYSTEM`, `PUBLIC` 등 외부 엔티티 관련 선언을 입력 단계에서 거부|
|엔티티 치환 옵션 미사용|`LIBXML_NOENT`를 사용하지 않아 엔티티 참조가 파일 내용으로 치환되지 않게 함|
safe 모드에서는 먼저 `DOCTYPE`, `ENTITY`, `SYSTEM`, `PUBLIC` 포함 여부를 검사하고, 해당 선언이 있으면 파싱 전에 차단한다. 해당 선언이 없을 때만 `LIBXML_NONET` 옵션으로 XML을 파싱하며, 엔티티 치환을 유발하는 `LIBXML_NOENT`는 사용하지 않는다.

취약 모드와 조치 모드의 차이는 다음과 같다.

```php
// vulnerable
$dom->loadXML($xmlInput, LIBXML_NOENT | LIBXML_DTDLOAD);

// safe
$dom->loadXML($xmlInput, LIBXML_NONET);
```

다만 `LIBXML_NONET`은 네트워크 접근 제한용 보조 조치다. `file://` 같은 로컬 파일 참조까지 완전히 해결하는 핵심 조치는 아니다. 본 실습의 조치 모드에서는 외부 엔티티 관련 선언을 차단하고, `LIBXML_NOENT`를 사용하지 않는 방식으로 XXE 위험을 줄인다.

### 조치 방안

XXE Injection을 방지하기 위한 조치 방안은 다음과 같다.

- XML 파서에서 DTD와 외부 엔티티 처리를 비활성화한다.
    
- 사용자 입력 XML에서 `DOCTYPE`, `ENTITY`, `SYSTEM`, `PUBLIC` 선언을 허용하지 않는다.
    
- PHP 환경에서는 불필요한 `LIBXML_NOENT` 사용을 피한다.
    
- 외부 네트워크 참조를 제한하기 위해 `LIBXML_NONET` 같은 옵션을 보조적으로 적용한다.
    
- XML 입력이 꼭 필요한 경우 허용할 루트 태그, 하위 태그, 속성 구조를 명확히 제한한다.
    
- XML 파싱 오류 메시지에 서버 경로, 내부 파일명, 파서 설정 등이 노출되지 않도록 한다.
    
- 웹 방화벽에 XXE 관련 DTD, ENTITY, SYSTEM, PUBLIC 패턴 룰셋을 적용한다.
    
- 가능하면 민감한 서버 로컬 파일이나 내부 자원에 웹 서버 프로세스가 접근하지 못하도록 OS 권한을 최소화한다.

### 판단

XXE Injection 재현은 완료된 것으로 판단한다.

근거는 세 가지다. 첫째, 정상 XML 입력에서는 `<data>Hello XML</data>`가 정상 파싱되어 `Text Content`에 `Hello XML`이 출력되었다. 둘째, 취약 모드에서 `file:///etc/hostname`을 참조하는 외부 엔티티 payload를 입력하자 `Text Content`에 `webserv`가 출력되었다. 셋째, `Serialized Root Node`에도 `<data>webserv</data>`가 표시되어 `&xxe;` 참조가 서버 로컬 파일 내용으로 치환되었음을 확인할 수 있었다.

따라서 취약 모드에서는 XML 파서가 사용자가 선언한 외부 엔티티를 로드하고, 엔티티 참조를 서버 로컬 파일 내용으로 확장했다고 볼 수 있다. 반대로 safe 모드에서는 동일한 payload가 `DTD 또는 외부 엔티티 선언이 포함되어 요청을 차단했다.` 메시지로 차단되었으므로, DTD/외부 엔티티 선언 차단과 `LIBXML_NOENT` 미사용 조치가 동작한 것으로 판단한다.

본 실습 환경에서는 `/etc/hostname`을 이용해 서버 로컬 파일 참조 가능성을 확인했다. `/etc/passwd`처럼 더 민감한 파일을 사용할 수도 있지만, 보고서 증거 목적에서는 `/etc/hostname`만으로도 XXE 취약 원리를 입증하기에 충분하다.

## 6-6. SSTI

### 개요

**SSTI(Server-Side Template Injection)** 는 사용자 입력값이 서버 측 템플릿 엔진에서 단순 문자열이 아니라 템플릿 표현식 또는 템플릿 코드로 해석될 때 발생하는 취약점이다.

템플릿 엔진은 HTML 응답을 만들 때 서버 측 변수와 템플릿 문법을 결합해 최종 페이지를 생성한다. 예를 들어 Twig 템플릿에서는 다음과 같은 문법을 사용할 수 있다.

```twig
<p>{{ app_name }}</p>
<p>{{ 7 * 7 }}</p>
```

정상적인 사용에서는 개발자가 작성한 템플릿에 서버 측 변수를 전달한다.

```text
개발자가 작성한 템플릿
↓
서버가 안전한 변수 값을 전달
↓
템플릿 엔진이 HTML 응답 생성
```

하지만 사용자 입력값 자체를 템플릿으로 렌더링하면 공격자가 템플릿 표현식을 삽입할 수 있다.

```text
사용자 입력값
↓
서버가 입력값 자체를 템플릿으로 렌더링
↓
템플릿 엔진이 사용자 입력을 서버 측 표현식으로 평가
↓
계산 결과, 서버 context, 설정값 등이 응답에 노출될 수 있음
```

PDF에서 제시하는 대표 점검 payload는 다음과 같다.

```text
{{7*7}}
```

이 payload의 목적은 서버에서 계산기를 쓰는 것이 아니다. 응답에 `49`가 출력되면 사용자 입력값이 단순 문자열이 아니라 서버 템플릿 엔진의 표현식으로 평가되었다는 의미다.

즉 `{{7*7}} → 49`는 공격의 최종 목적이 아니라, **서버 측 템플릿 평가 여부를 확인하는 탐침 payload**다.

### 정상 기능과 취약 원리

정상적인 템플릿 사용 방식은 다음과 같다.

```text
고정된 서버 템플릿:
Hello {{ user_input }}

사용자 입력:
{{ 7 * 7 }}

서버 처리:
사용자 입력을 변수 값으로만 전달

결과:
Hello {{ 7 * 7 }}
```

이 경우 사용자 입력값은 템플릿 구조가 아니라 단순 변수 값으로 처리된다. 따라서 `{{ 7 * 7 }}`은 계산되지 않고 문자열로 출력된다.

반대로 취약한 방식은 다음과 같다.

```text
사용자 입력:
Hello {{ 7 * 7 }}

서버 처리:
사용자 입력값 자체를 Twig 템플릿으로 등록하고 렌더링

결과:
Hello 49
```

이 경우 사용자가 입력한 `{{ 7 * 7 }}`이 Twig 표현식으로 평가된다. 이는 사용자 입력값이 템플릿 엔진의 실행 문맥으로 들어갔다는 의미다.

SSTI가 위험한 이유는 단순 산술 계산 때문이 아니다. 실제 환경에서는 템플릿 엔진의 기능과 노출된 객체에 따라 다음 피해로 이어질 수 있다.

|가능 피해|설명|
|---|---|
|서버 context 조회|`request`, `session`, `config`, `user` 등 템플릿에 전달된 객체나 변수 확인|
|설정값 노출|secret key, DB 설정, API key 등 민감 정보 노출 가능|
|파일 접근|템플릿 로더, include 기능, 노출 객체에 따라 파일 읽기 가능성|
|내부 객체 접근|애플리케이션 객체, 사용자 객체, 환경 정보 접근 가능성|
|명령 실행|템플릿 엔진, sandbox 설정, 노출 객체 조건에 따라 RCE로 이어질 수 있음|
|DoS|반복문, 큰 연산, 재귀적 렌더링 등을 통한 자원 소모 가능성|

본 실습에서는 RCE payload까지 진행하지 않는다. 대신 `{{ 7 * 7 }}` 계산, 템플릿 context 접근, 실습용 가짜 secret 값 조회를 통해 SSTI가 단순 계산기가 아니라 서버 측 템플릿 context 접근으로 이어질 수 있음을 확인한다.

### 실습 환경 구성

SSTI 실습은 실제 PHP 템플릿 엔진인 Twig를 사용해 재현하기로 했다.

따라서 이번 실습은 단순한 교육용 미니 렌더러가 아니라, **CARE 실습 환경에 Twig 기반 취약 페이지를 의도적으로 추가하여 실제 템플릿 엔진 기반 SSTI를 재현하는 방식**으로 진행한다.

### 취약점 파악

PDF p.687~688의 SSTI 점검 흐름은 한 덩어리처럼 이어지지만, 역할은 두 단계로 나뉜다. p.687의 Step 1은 **SSTI 존재 여부 확인**이고, p.688의 Step 2는 **상위 context와 객체 접근을 통한 영향 범위 및 RCE 확장 가능성 판단**이다.

|PDF 위치|단계|확인 질문|본 실습에서의 판정 기준|
|---|---|---|---|
|p.687|Step 1|사용자 입력값이 서버 템플릿 엔진에서 표현식으로 해석되는가?|`{{ 7 * 7 }}` 입력 시 `49` 출력 여부 확인|
|p.688|Step 2|템플릿 내부 context, 객체, 함수 접근을 통해 영향이 확장될 수 있는가?|`_context`, `current_user`, `lab_secret` 접근 확인 및 통제된 RCE sink 확인|

따라서 본 실습도 Step 1과 Step 2를 분리해서 판단한다.

#### Step 1. 템플릿 엔진 해석 여부 확인

Step 1의 목적은 사용자 입력값이 단순 문자열이 아니라 서버 템플릿 엔진의 표현식으로 처리되는지 확인하는 것이다. 이 단계는 RCE 성공 여부를 증명하는 단계가 아니라, SSTI 취약점이 존재하는지 확인하는 최소 탐침이다.

취약 모드에서 다음 payload를 입력한다.

```twig
{{ 7 * 7 }}
```

취약 모드에서 결과는 다음과 같다.

```text
49
```

이는 사용자가 입력한 `{{ 7 * 7 }}`이 Twig 템플릿 표현식으로 평가되었음을 의미한다.

즉, 사용자 입력값이 단순 문자열로 출력된 것이 아니라 서버 측 Twig 템플릿 엔진의 해석 문맥으로 들어간 것이다. 이 결과를 통해 SSTI 취약점의 존재를 확인할 수 있다.

#### Step 2. context / 객체 접근과 RCE 확장 가능성 판단

Step 2의 목적은 단순 수식 평가를 넘어서, 템플릿 엔진 내부 context와 객체에 접근할 수 있는지 확인하고, 이 접근이 원격 코드 실행으로 확장될 수 있는지 판단하는 것이다.

다만 Step 2가 곧바로 “RCE 성공”을 의미하는 것은 아니다. 실제 RCE 가능 여부는 템플릿 엔진 설정, sandbox 적용 여부, context에 노출된 객체·함수, 프레임워크 구조에 따라 달라진다.

본 실습에서는 다음 순서로 Step 2를 확인한다.

```text
1. Twig context 접근 확인
2. 실습용 가짜 secret 값 조회
3. 통제된 RCE sink 확인
```

##### 2-1. Twig context key 접근 확인

먼저 템플릿 context 접근 가능성을 확인한다.

취약 모드에서 다음 payload를 입력한다.

```twig
Application: {{ app_name }}
Role: {{ current_user.role }}
User count: {{ users|length }}
Context keys: {{ _context|keys|join(', ') }}
```

결과는 다음과 같다.

```text
Application: CARE
Role: viewer
User count: 3
Context keys: app_name, environment, current_user, users, lab_secret
```

이 결과는 Twig 템플릿에서 서버가 전달한 context 값에 접근할 수 있음을 보여준다.

특히 `{{ _context|keys|join(', ') }}`는 템플릿에 전달된 context key 목록을 출력한다. 공격자가 context key를 확인할 수 있으면, 이후 어떤 객체나 값에 접근을 시도할지 추정할 수 있다.

##### 2-2. context 값 노출 확인

다음으로 context에 포함된 값을 직접 조회한다.

취약 모드에서 다음 payload를 입력한다.

```twig
{{ lab_secret }}
```

결과는 다음과 같다.

```text
demo-secret-value-do-not-use-in-production
```

이 값은 실제 운영 비밀값이 아니라 실습용 가짜 값이다. 하지만 템플릿 context에 민감 정보가 포함되어 있으면 SSTI를 통해 응답에 노출될 수 있음을 보여주는 증거로 사용할 수 있다.

`lab_secret`은 실습을 위해 context에 넣은 고정 proof value다. 그러나 PHP 코드가 특정 문자열을 감지해 결과를 가짜로 출력하는 구조는 아니다.

실제 처리 흐름은 다음과 같다.

```text
사용자 입력: {{ lab_secret }}
↓
Twig가 사용자 입력을 템플릿 표현식으로 평가
↓
Twig가 context에서 lab_secret 값을 조회
↓
렌더링 결과에 해당 값 출력
```

따라서 이는 하드코딩된 가짜 결과가 아니라, 통제된 실습 context를 이용한 SSTI 영향 확인으로 판단한다.

##### 2-3. 통제된 RCE sink 확인

PDF p.688의 Step 2는 상위 context 및 객체 접근을 통해 원격 코드 실행 가능 여부를 판단하는 단계다.

본 실습에서는 RCE 가능성을 확인하기 위해 취약 모드에 실습용 `lab_exec()` Twig 함수를 의도적으로 노출한다.

이 확인은 Twig 기본 설정만으로 무조건 임의 명령 실행이 가능하다는 뜻이 아니다. 서버 측에서 명령 실행 기능이 템플릿 함수 또는 객체 형태로 노출된 경우, SSTI가 RCE로 확장될 수 있음을 통제된 방식으로 확인하는 것이다.

취약 모드에서 다음 payload를 입력한다.

```twig
<pre>{{ lab_exec('id') }}</pre>
```

성공 시 결과는 다음과 같은 형태로 출력된다.

```text
uid=33(www-data) gid=33(www-data) groups=33(www-data)
```

또는 다음 payload를 사용할 수 있다.

```twig
<pre>{{ lab_exec('whoami') }}</pre>
```

성공 시 결과는 다음과 같다.

```text
www-data
```

이 결과가 출력되면 Twig 템플릿 표현식에서 서버 측 함수가 호출되었고, 해당 함수가 OS 명령을 실행한 뒤 결과를 렌더링 응답에 포함한 것이다.

즉, 이 경우에는 단순한 context 조회를 넘어서 **통제된 RCE sink를 통한 명령 실행 확인**으로 판단할 수 있다.

다만 이 실습의 RCE 확인은 다음 범위로 제한한다.

```text
허용 payload:
{{ lab_exec('id') }}
{{ lab_exec('whoami') }}
{{ lab_exec('hostname') }}
{{ lab_exec('pwd') }}
```

다음 행위는 수행하지 않는다.

```text
금지 범위:
reverse shell
웹쉘 생성
파일 삭제
파일 변조
권한 상승 시도
외부 서버로 데이터 전송
curl/wget을 이용한 추가 payload 다운로드
```

정리하면, 본 실습의 Step 2 결론은 다음과 같다.

```text
Twig context 접근이 가능하고,
취약 모드에 위험 함수가 노출된 경우,
SSTI가 서버 측 함수 호출 및 OS 명령 실행으로 확장될 수 있음을 확인한다.
```

### 조치 확인

PDF p.688의 조치 방법은 예약 문자 이스케이프, 템플릿 안전 모드, 엔진 버전과 보안 패치, 에러 메시지 제한, 웹 방화벽 룰셋 적용으로 정리된다.

본 실습의 핵심 조치는 사용자 입력값 자체를 Twig 템플릿으로 렌더링하지 않는 것이다. safe 모드에서는 고정된 안전 템플릿을 사용하고, 사용자 입력은 템플릿 코드가 아니라 변수 값으로만 전달한다.

|모드|처리 방식|의미|
|---|---|---|
|vulnerable|사용자 입력값 자체를 Twig 템플릿으로 렌더링|사용자 입력이 템플릿 표현식으로 평가될 수 있음|
|safe|고정 템플릿에 사용자 입력을 변수 값으로 전달|사용자 입력이 템플릿 구조로 해석되지 않음|

#### Step 1 차단 확인

Step 1 차단 여부는 수식 payload가 더 이상 계산되지 않는지로 확인한다.

safe 모드에서 다음 입력을 넣는다.

```twig
{{ 7 * 7 }}
```

기대 결과는 다음과 같다.

```text
Safe Render Result
사용자 입력은 템플릿 코드가 아니라 변수 값으로만 출력된다.
{{ 7 * 7 }}
```

즉 계산 결과인 `49`가 아니라 입력 문자열 자체가 출력되어야 한다. 이 결과가 나오면 Step 1에서 확인한 템플릿 표현식 평가가 차단된 것으로 판단한다.

#### Step 2 위험 완화 확인

Step 2 위험 완화 여부는 context 조회 payload와 RCE sink payload가 더 이상 평가되지 않는지로 확인한다.

safe 모드에서 다음 입력을 넣는다.

```twig
{{ lab_secret }}
```

기대 결과는 다음과 같다.

```text
Safe Render Result
사용자 입력은 템플릿 코드가 아니라 변수 값으로만 출력된다.
{{ lab_secret }}
```

즉 `demo-secret-value-do-not-use-in-production` 값이 출력되면 안 된다.

통제된 RCE sink payload도 safe 모드에서는 실행되지 않아야 한다.

safe 모드에서 다음 입력을 넣는다.

```twig
{{ lab_exec('id') }}
```

기대 결과는 다음과 같다.

```text
Safe Render Result
사용자 입력은 템플릿 코드가 아니라 변수 값으로만 출력된다.
{{ lab_exec('id') }}
```

즉 `uid=33(www-data)` 같은 OS 명령 실행 결과가 출력되면 안 된다.

#### PDF 조치 방법과 본 실습 대응

PDF p.688의 조치 항목을 본 실습에 대응시키면 다음과 같다.

|PDF 조치 방향|본 실습 대응|의미|
|---|---|---|
|예약 문자 이스케이프|safe 모드에서 사용자 입력을 변수 값으로만 출력|`{{ }}`, `{% %}` 같은 템플릿 문법이 실행 문맥으로 들어가지 않음|
|템플릿 안전 모드 / 명령 제한|사용자 입력을 템플릿 원문으로 등록하지 않음|사용자가 작성한 템플릿 표현식이 Twig에서 평가되지 않음|
|엔진 버전과 보안 패치 고려|`composer.lock`으로 Twig 버전 관리|사용 중인 템플릿 엔진 버전을 추적하고 패치 가능하게 유지|
|에러 메시지 제한|운영 환경에서는 템플릿 엔진명, 내부 경로, 객체 구조 노출 제한 필요|오류 기반 정보 수집을 줄임|
|웹 방화벽 룰셋 적용|`{{`, `{%`, `{#` 등 SSTI 패턴 탐지 룰 보조 적용 가능|애플리케이션 조치 이후의 추가 방어선|

여기서 safe 모드는 Twig Sandbox를 별도로 적용한 구성이 아니라, 사용자 입력값을 템플릿 원문으로 등록하지 않는 구조 변경이다. 따라서 본 실습의 핵심 조치는 “사용자 입력을 템플릿 코드로 다시 해석하지 않는 것”이다.

따라서 safe 모드에서 `{{ 7 * 7 }}`, `{{ lab_secret }}`, `{{ lab_exec('id') }}`가 모두 문자 그대로 출력되면, Step 1의 템플릿 표현식 평가와 Step 2의 context / RCE sink 접근이 함께 차단된 것으로 판단한다.

### 판단

Step 1에서 `{{ 7 * 7 }}`이 `49`로 평가되어 사용자 입력이 Twig 템플릿 표현식으로 해석되는 것을 확인했다.

Step 2에서는 context key 조회, 실습용 `lab_secret` 조회, 통제된 `lab_exec()` 호출을 통해 SSTI가 내부 context 접근과 서버 측 함수 호출로 확장될 수 있음을 확인했다. 단, `lab_exec()`는 실습을 위해 의도적으로 노출한 RCE sink이므로 Twig 기본 설정만으로 임의 명령 실행이 가능하다는 의미는 아니다.

safe 모드에서는 동일 payload가 실행되지 않고 문자열로 출력되어, 사용자 입력을 템플릿 코드가 아니라 변수 값으로만 처리하는 조치가 동작함을 확인했다.

정리하면, 이 실습의 핵심은 **사용자 입력을 출력하는 것 자체가 문제가 아니라, 사용자 입력을 서버 측 템플릿 코드로 다시 해석하는 것이 문제**라는 점이다.

# 7. 종합 판단

본 실습에서는 PDF p.679~692의 코드 인젝션 항목을 기준으로 6개 세부 유형을 분리하여 확인한다.

| 세부 유형 | 실습 방식 | 결과 | 비고 |
|---|---|---|---|
| LDAP Injection | SQL Injection과의 비교 학습 | 구현 제외 | 실제 LDAP 서버가 필요하므로 별도 기능 구현 없이 원리와 차이점 정리 |
| OS Command Injection | OS 명령 실행 기능 구현 | 실습 후 기입 | 명령 연결 문자 삽입으로 서버 명령 실행 확인 |
| SSI Injection | SSI 지시어 실행 구조 확인 | 실습 후 기입 | Apache SSI 설정 여부에 따라 재현 범위 달라짐 |
| XPath Injection | XPath 질의 조작 기능 구현 | 실습 후 기입 | 인증 조건 우회 가능성 확인 |
| XXE Injection | XML 외부 엔티티 처리 기능 구현 | 재현 완료 | `file:///etc/hostname` 참조로 서버 hostname 출력 확인 |
| SSTI | 실제 Twig 템플릿 엔진 기반 실습 | 재현 완료 | Step 1로 SSTI 존재 확인, Step 2로 context 접근 및 통제된 RCE sink 확인 |

종합하면 코드 인젝션은 SQL Injection처럼 입력값이 명령으로 해석되는 취약점이지만, SQL이 아니라 운영체제 명령어, LDAP 필터, SSI 지시어, XPath 질의, XML 파서, 템플릿 엔진 등 다양한 실행·해석 환경을 대상으로 한다.

따라서 방어의 핵심은 사용자 입력값을 실행 문맥에 직접 연결하지 않는 것이다. 실행 기능이 필요한 경우에도 화이트리스트 기반 입력값 검증, 특수문자 이스케이프, 안전한 API 사용, 위험 기능 비활성화, 웹 방화벽 룰셋 적용이 함께 이루어져야 한다.
