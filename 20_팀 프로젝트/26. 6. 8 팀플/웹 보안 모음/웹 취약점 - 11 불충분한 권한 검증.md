# 웹 취약점 - 11 불충분한 권한 검증

source: [[40_자료/주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드.pdf|주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드]], p.733-734.

## 1. 개요

**불충분한 권한 검증**은 사용자가 접근하면 안 되는 다른 사용자의 비공개 정보나 기능에 접근할 수 있는지 확인하는 항목이다.

PDF p.733의 판단 기준은 다음처럼 정리할 수 있다.

| 판단 | 기준 |
|---|---|
| 양호 | 중요 페이지에 사용자 검증 로직이 구현되어 있어 타 사용자의 권한 탈취가 제한됨 |
| 취약 | 중요 페이지의 사용자 검증 로직이 미흡하여 타 사용자의 권한 탈취가 가능함 |

이 항목은 단순히 “로그인 검사가 있느냐”만 보는 것이 아니다. 핵심은 `id`, `num`, 일련번호 같은 값을 바꿨을 때 **원래 접근하면 안 되는 다른 사용자의 비공개 리소스에 접근하거나 수정할 수 있는지**다.

CARE에서는 회원정보 수정/탈퇴 기능이 있지만, 서버가 POST로 전달된 사용자 ID를 신뢰하지 않고 `$_SESSION['id']`를 기준으로 처리한다. 그래서 현재 상태는 **B: 권한 검증을 의심할 만한 공격면은 있으나, 원본 코드 기준으로는 타 사용자 권한 탈취가 바로 성립하지 않는 상태**로 판단한다.

> [!note] ABCD 분류 풀어쓰기
> 이 문서의 `A/B/C/D`는 프로젝트 내부 분류 코드다. `A`는 원래 CARE 기능이나 서버 설정만으로 바로 취약한 경우, `B`는 공격면은 있지만 원본 코드가 막고 있어 그대로는 취약점 재현이 어려운 경우, `C`는 CARE 원본에 공격면이 없어 `/vuln/...` 같은 실습용 기능을 새로 만들어야 하는 경우, `D`는 현재 프로젝트 환경이나 전제 조건이 맞지 않아 구현하지 않고 비교 학습 또는 해당 없음으로 처리하는 경우를 뜻한다.

## 2. 문제가 되는 부분

처음 보면 CARE에도 권한 검증 문제가 있을 것처럼 보이는 지점이 있다.

| 기능 | 의심 지점 | 실제 판단 |
|---|---|---|
| 회원정보 수정 | 화면에 사용자 ID가 보이고 회원정보를 수정함 | 서버는 `$_SESSION['id']` 기준으로 수정하므로 타 사용자 ID 변조가 직접 통하지 않음 |
| 회원 탈퇴 | 회원 계정을 삭제함 | 서버는 `$_SESSION['id']` 기준으로 삭제하고 비밀번호도 다시 확인함 |
| 게시글 조회 | `view.php?num=1`처럼 숫자 파라미터 사용 | 공개 게시글 목록 기반 조회라 비공개 리소스 접근으로 보기 어려움 |
| 첨부파일 다운로드 | `download.php?filename=...` 사용 | 파일 다운로드 취약점 범위로 분리하는 것이 맞음 |

`modify.php`의 ID 입력칸은 disabled 상태다.

```php
<label>아이디</label><input type="text" name="id" id="id" value="<?=$_SESSION['id'] ?>" disabled>
```

중요한 점은 `modifyModel.php`가 이 값을 신뢰하지 않는다는 것이다. 서버는 클라이언트가 보낸 `id`가 아니라 세션의 ID를 사용한다.

```php
session_start();
if($_SESSION['id'] == ""){
    echo "<script>location.href='login.php';</script>";
    exit;
}
$id = $_SESSION['id'];
```

```php
$query = "UPDATE member SET pw='$pw', name='$name', mobile='$mobile',
email='$email', address='$address' WHERE id='$id'";
```

따라서 공격자가 요청에 `id=admin`이나 `id=victim` 같은 값을 추가해도, 현재 코드 흐름에서는 `WHERE id='$id'`의 `$id`가 세션에서 오므로 자기 계정만 수정된다.

회원 탈퇴도 같은 구조다.

```php
session_start();
$id = $_SESSION['id'];
```

```php
$query = "SELECT pw FROM member WHERE id='$id'";
```

```php
if($pw == $row['pw']){
    $query = "DELETE FROM member WHERE id='$id'";
    mysqli_query($link, $query);
}
```

여기서는 현재 비밀번호도 확인하므로, 11번 기준으로는 오히려 반례에 가깝다.

## 3. 악용 흐름

PDF p.733-734 기준의 공격 흐름은 다음과 같다.

```text
사용자 A 로그인
-> A의 개인정보/비공개 게시글/문의글 URL 확인
-> URL 또는 요청 파라미터의 id, num, serial 값을 B의 값으로 변경
-> 서버가 세션 사용자와 리소스 소유자를 비교하지 않음
-> B의 비공개 정보 조회, 수정, 삭제 성공
```

CARE에서 이 흐름을 적용하려면 다음 같은 시도가 필요하다.

```text
POST /member/modifyModel.php
id=victim&pw=1234&pwCheck=1234&name=changed...
```

하지만 현재 `modifyModel.php`는 POST의 `id`를 읽지 않는다. 세션 ID만 사용한다.

따라서 11번에서 기대하는 권한 탈취 흐름은 현재 CARE의 회원정보 수정 기능에서는 성립하지 않는다.

## 4. 점검 방법

PDF p.733-734의 점검 방법은 두 단계다.

| 단계 | PDF 기준 | CARE에서의 확인 |
|---|---|---|
| Step 1 | 타 사용자 접근이 제한된 페이지의 URL 구조와 파라미터 분석 | 회원 수정/탈퇴, 게시글 조회, 첨부파일 다운로드의 파라미터 확인 |
| Step 2 | URL 구조와 파라미터를 변조하여 타 사용자 비공개 정보나 권한 외 리소스 접근 가능 여부 확인 | `id`, `num`, `filename` 변조가 11번 권한 탈취에 해당하는지 분리 판단 |

CARE에서 우선 확인할 지점은 다음이다.

```text
/member/modify.php
/member/modifyModel.php
/member/delete.php
/member/deleteModel.php
/center/view.php?num=1
/center/download.php?filename=...
```

판단은 다음처럼 나뉜다.

| 확인 대상 | 11번 판단 |
|---|---|
| `modifyModel.php`에 `id=victim`을 추가해서 요청 | 서버가 POST `id`를 쓰지 않으므로 타 사용자 수정으로 이어지지 않음 |
| `deleteModel.php`에 `id=victim`을 추가해서 요청 | 서버가 POST `id`를 쓰지 않고 세션 ID와 비밀번호를 확인함 |
| `view.php?num=다른번호` | 게시글이 공개 목록에 노출되는 구조라 비공개 리소스 권한 우회로 보기 어려움 |
| `download.php?filename=...` | 권한 검증보다는 15번 파일 다운로드 취약점으로 분류 |

즉 CARE에는 권한 검증을 점검할 만한 사용자 기능은 있지만, 현재 자연 상태에서는 타 사용자 권한 탈취까지 이어지는 지점이 확인되지 않는다.

## 5. 조치 방안

현재 CARE 회원정보 수정/탈퇴 기능은 11번 기준으로 별도 조치가 필요한 상태는 아니다.

다만 앞으로 게시글 수정/삭제, 비공개 게시글, 문의글, 관리자 기능처럼 **소유자나 권한이 나뉘는 기능**을 추가한다면 서버 측 권한 검증을 반드시 넣어야 한다.

권한 검증의 기본 형태는 다음이다.

```php
session_start();
$currentUserId = $_SESSION['id'] ?? '';
$num = $_GET['num'] ?? '';

require_once __DIR__ . '/../config.php';
$link = care_db_connect() or die('연결 실패');

$query = "SELECT id FROM center WHERE num='$num'";
$result = mysqli_query($link, $query);
$row = mysqli_fetch_assoc($result);

if(! $row || $row['id'] !== $currentUserId){
    http_response_code(403);
    echo "permission denied";
    exit;
}
```

이 예시는 현재 CARE 스타일에 맞춘 최소 예시다. 실제 조치에서는 SQL Injection 방어를 위해 Prepared Statement까지 같이 적용하는 것이 맞다.

권한 검증에서 중요한 원칙은 다음이다.

| 원칙 | 설명 |
|---|---|
| 클라이언트 값을 신뢰하지 않기 | hidden input, disabled input, URL 파라미터의 사용자 ID를 그대로 믿지 않음 |
| 세션 사용자 기준으로 판단 | 현재 로그인한 사용자는 서버 세션에서 가져옴 |
| 리소스 소유자와 비교 | DB에서 조회한 글/파일/문의의 소유자와 세션 사용자를 비교 |
| 불일치 시 차단 | 권한이 없으면 `403 Forbidden` 또는 오류 처리 |

## 6. 조치 후 점검

현재는 별도 조치 대상이 아니므로, 조치 후 점검보다는 **회귀 방지 점검**으로 보는 것이 맞다.

추후 게시글 수정/삭제 기능이나 비공개 리소스를 추가한다면 다음을 확인한다.

- 사용자 A로 로그인한 뒤 사용자 B의 리소스 번호를 요청했을 때 차단되는지 확인한다.
- 요청에 `id=B` 같은 파라미터를 추가해도 서버가 세션 사용자 A 기준으로 처리하는지 확인한다.
- 목록에 공개되지 않은 리소스 번호를 직접 입력했을 때 접근이 차단되는지 확인한다.
- 권한이 없는 접근은 조용히 성공하지 않고 `403`, alert, redirect 등 명확한 차단 결과를 남기는지 확인한다.

### 후속 연계 포인트

11번은 지금 당장 B-L을 따로 만드는 것보다, 다른 취약점 점검과 조치 과정에서 함께 확인하기 쉽다.

| 연계 항목 | 11번과 연결되는 방식 |
|---|---|
| 13. 프로세스 검증 누락 | URL 직접 접근, 단계 생략, 비정상 요청을 확인할 때 현재 사용자가 그 리소스에 접근할 권한이 있는지도 같이 확인 |
| 14. 악성 파일 업로드 | 업로드 파일을 사용자별로 나누거나 게시글 첨부파일 소유자를 검증할 때, 다른 사용자의 파일에 접근 가능한지 확인 |
| 15. 파일 다운로드 | `filename` 경로 조작을 막는 과정에서 해당 파일이 현재 사용자에게 허용된 파일인지 권한 검증을 같이 적용 |
| 19. 관리자 페이지 노출 | 관리자/일반 사용자 권한을 나누는 기능을 만들 경우, 일반 사용자가 관리자 기능에 접근 가능한지 확인 |
| 게시글 수정/삭제 기능 추가 시 | `num` 값을 바꿔 다른 사용자의 글을 수정하거나 삭제할 수 있는지 확인 |

따라서 현재 단계에서는 11번을 별도 B-L로 억지로 약화하지 않는다. 이후 게시글 수정/삭제, 비공개 게시글, 파일 소유자 검증, 관리자 기능처럼 **권한 경계가 생기는 작업**을 할 때 11번을 함께 재점검한다.

## 7. 증거

현재 이 항목은 실행 증거를 수집하지 않았다. 대신 소스 기준으로 다음 근거를 확인했다.

- [ ] `modifyModel.php`가 POST `id`를 사용하지 않고 `$_SESSION['id']` 기준으로 `UPDATE member` 수행
- [ ] `deleteModel.php`가 POST `id`를 사용하지 않고 `$_SESSION['id']` 기준으로 `DELETE member` 수행
- [ ] `deleteModel.php`는 비밀번호를 다시 확인한 뒤 탈퇴 처리
- [ ] `center` 폴더에 실제 게시글 수정/삭제 처리 파일이 없음
- [ ] `view.php?num=`은 공개 게시글 조회 구조라 11번 권한 검증이 아니라 공개 리소스 조회로 판단
- [ ] `download.php?filename=`은 15번 파일 다운로드 항목으로 분리

### B 분류 근거 코드

`B`로 분류한 이유는 “권한 검증을 의심할 만한 기능은 있으나, 현재 CARE 원본 코드에서는 클라이언트가 넘긴 사용자 ID를 신뢰하지 않고 세션 사용자 기준으로 처리하기 때문”이다.

회원정보 수정 처리 파일은 POST 요청에 `id=admin`, `id=victim` 같은 값을 추가하더라도 그 값을 읽지 않는다. 서버 세션의 로그인 ID를 `$id`로 사용한다.

```php
// D:\care\member\modifyModel.php
session_start();
if($_SESSION['id'] == ""){
    echo "<script>location.href='login.php';</script>";
    exit;
}
$id = $_SESSION['id'];
```

수정 쿼리도 클라이언트 입력 ID가 아니라 위에서 세션으로 정한 `$id`를 기준으로 수행된다.

```php
// D:\care\member\modifyModel.php
$query = "UPDATE member SET pw='$pw', name='$name', mobile='$mobile',
email='$email', address='$address' WHERE id='$id'";
```

회원 탈퇴 처리도 같은 구조다. 삭제 대상 ID를 POST나 GET에서 받지 않고 세션에서 가져온다.

```php
// D:\care\member\deleteModel.php
session_start();
$id = $_SESSION['id'];
$pw = $_POST['pw'];
$pwCheck = $_POST['pwCheck'];
```

삭제 전에는 세션 사용자 ID의 비밀번호를 조회하고, 입력한 비밀번호가 일치할 때만 같은 세션 사용자 ID를 삭제한다.

```php
// D:\care\member\deleteModel.php
$query = "SELECT pw FROM member WHERE id='$id'";
$result = mysqli_query($link, $query);
$row = mysqli_fetch_assoc($result);

if($pw == $row['pw']){
    $query = "DELETE FROM member WHERE id='$id'";
    mysqli_query($link, $query);
    mysqli_close($link);
}
```

게시판 쪽도 11번 권한 검증으로 바로 확정하기 어렵다. 현재 `center` 폴더에는 게시글 작성·조회·다운로드 파일은 있지만, 다른 사용자의 글을 수정/삭제하는 처리 파일이 없다.

```text
D:\care\center
- center(table).txt
- download.php
- list.php
- view.php
- write.php
- writeModel.php
```

따라서 현재 CARE 원본 기준으로는 “다른 사용자의 비공개 리소스 ID를 변조해 조회·수정·삭제하는 흐름”이 바로 확인되지 않는다. 이 때문에 취약한 `A`가 아니라, 공격면은 있으나 원본 코드에서 막혀 있는 `B`로 둔다.

실행 증거를 찍는다면 다음 정도면 충분하다.

- [ ] `modifyModel.php` 요청에 `id=victim`을 추가해도 로그인한 자기 계정만 수정되는 화면 또는 DB 결과
- [ ] `deleteModel.php` 요청에 `id=victim`을 추가해도 로그인한 자기 계정 기준으로만 처리되는 화면 또는 DB 결과

## 8. 판단

이 항목은 **B: 최초 상태에 권한 검증을 의심할 만한 공격면은 있지만, 원본 코드 기준으로는 타 사용자 권한 탈취가 바로 성립하지 않는 상태**로 판단한다.

| 항목 | 판단 |
|---|---|
| 1차 분류 | B |
| 판정 상태 | 확정 |
| 조치 상태 | 해당 없음 |
| 증거 상태 | 없음 |

의외로 보일 수 있는 이유는 CARE에 회원정보 수정, 회원 탈퇴, 게시글 번호 조회, 파일명 다운로드처럼 권한 검증과 비슷해 보이는 기능이 있기 때문이다.

하지만 PDF p.733-734가 보는 핵심은 **타 사용자 접근이 제한된 리소스의 ID나 일련번호를 바꿔서 남의 권한을 탈취할 수 있는지**다.

CARE 회원정보 수정/탈퇴는 서버에서 세션 ID를 기준으로 처리하므로, 클라이언트가 `id`를 변조해도 다른 사용자의 계정을 직접 수정하거나 삭제하는 흐름이 확인되지 않는다. 게시글 조회는 공개 목록 기반이고, 파일명 다운로드 문제는 15번 파일 다운로드 취약점으로 분리하는 것이 맞다.

따라서 11번은 현재 CARE 원본 기준으로 취약하다고 보기 어렵고, `B` 판정을 유지한다.
