# 웹 취약점 - 17 데이터 평문 전송

source: [[40_자료/주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드.pdf|주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드]], p.766-768.

## 1. 개요

**데이터 평문 전송**은 로그인, 회원가입, 회원정보 수정, 게시글 작성처럼 사용자가 중요한 값을 보내는 구간이 암호화되지 않은 HTTP로 처리되어, 네트워크 중간에서 요청 내용을 그대로 볼 수 있는 취약점이다.

PDF p.766의 판단 기준은 다음처럼 정리할 수 있다.

| 판단 | 기준 |
|---|---|
| 양호 | 중요 정보 전송구간에 암호화 통신이 적용됨 |
| 취약 | 중요 정보 전송구간에 암호화 통신이 이루어지지 않음 |

현재 CARE 실습 환경은 WEB 서버가 Apache `VirtualHost *:80`으로 구성되어 있고, LB 스크립트도 nginx `LISTEN_PORT=80`을 기본값으로 사용한다. HTTPS 또는 443 포트 설정은 현재 스크립트에서 확인되지 않는다. 따라서 이 항목은 CARE PHP 코드의 특정 함수 취약점이라기보다 **웹 서비스 전송구간 설정 취약점**으로 본다.

즉, 17번은 구현이 까다롭다고 해서 D가 아니다. 로그인과 회원정보 수정 같은 중요 정보 전송 기능이 이미 있고, 그 전송구간이 HTTP이므로 **A-3 환경 설정형 취약점**으로 분류한다.

## 2. 문제가 되는 부분

### 2.1 중요 정보 송수신 페이지

PDF Step 1은 먼저 중요 정보를 송수신하는 페이지가 있는지 확인한다. CARE에는 다음 기능들이 있다.

| 기능 | URL / 처리 파일 | 전송 방식 | 전송되는 주요 값 | 판단 |
|---|---|---|---|---|
| 로그인 | `/member/login.php` -> `loginModel.php` | POST | `id`, `pw` | 인증정보 전송 |
| 회원가입 | `/member/register.php` -> `registerModel.php` | POST | `id`, `pw`, `name`, `mobile`, `address`, `email` | 계정정보와 개인정보 전송 |
| 회원정보 수정 | `/member/modify.php` -> `modifyModel.php` | POST | `pw`, `name`, `mobile`, `address`, `email` | 개인정보 변경 |
| 게시글 작성 | `/center/write.php` -> `writeModel.php` | POST | `subject`, `content`, `upfile` | 사용자 입력값과 첨부파일 전송 |
| 게시글 검색 | `/center/list.php` | GET | `find`, `data` | URL에 검색어 노출 |

로그인, 회원가입, 회원정보 수정은 PDF가 말하는 중요 정보 전송구간에 직접 해당한다. 게시글 작성과 검색은 인증정보보다는 민감도가 낮지만, PDF 조치 예시가 게시판 구간도 함께 언급하므로 점검 범위에 포함한다.

### 2.2 현재 전송구간

현재 프로젝트 스크립트 기준 전송구간은 다음과 같다.

| 구성요소 | 확인 내용 | 의미 |
|---|---|---|
| WEB 서버 | `care-web-bootstrap.sh`가 Apache `VirtualHost *:80`을 생성 | CARE가 HTTP 80번 포트로 서비스됨 |
| LB 서버 | `LB.sh`, `lb-upstream-update.sh`의 기본 `LISTEN_PORT=80` | LB도 HTTP 80번 포트로 요청을 받음 |
| HTTPS 설정 | `ssl_certificate`, `a2enmod ssl`, `443` 설정 확인 안 됨 | 현재 스크립트만으로는 HTTPS 전송구간이 구성되지 않음 |

따라서 브라우저에서 `http://172.168.10.10/` 또는 LB를 통해 CARE에 접속하면 로그인 요청과 회원정보 요청이 HTTP 평문으로 전송된다.

## 3. 악용 흐름

공격자는 서버를 직접 침입하지 않아도, 같은 네트워크 구간에서 HTTP 요청을 관찰할 수 있으면 중요 정보를 읽을 수 있다.

```text
사용자
-> http://172.168.10.10/member/login.php 접속
-> id / pw 입력 후 로그인 요청 전송
-> 네트워크 중간에서 Paros, Wireshark 등으로 HTTP 요청 확인
-> POST body에 id와 pw가 평문으로 노출
-> 공격자가 계정 탈취 또는 재사용 공격에 활용
```

이 실습에서는 실제 비밀번호가 아니라 실습용 계정으로만 확인한다. 증거로는 “HTTP 요청 본문에 실습 계정의 `id`, `pw`가 그대로 보인다”는 사실만 남기면 충분하다.

## 4. 점검 방법

PDF p.766-767의 Step 1~3에 맞춰 확인한다.

### PDF Step 1. 중요 정보 송수신 페이지 존재 여부 확인

다음 URL 또는 화면을 확인한다.

| 확인 대상 | 예시 URL | 확인할 점 |
|---|---|---|
| 로그인 | `http://172.168.10.10/member/login.php` | `id`, `pw` 입력칸이 있음 |
| 회원가입 | `http://172.168.10.10/member/register.php` | 계정정보와 개인정보 입력칸이 있음 |
| 회원정보 수정 | `http://172.168.10.10/member/modify.php` | 로그인 후 개인정보 수정 가능 |
| 게시글 작성 | `http://172.168.10.10/center/write.php` | 제목, 내용, 파일 업로드 입력 가능 |

이 단계는 “중요 정보가 오가는 기능이 있는가”를 확인하는 단계다. CARE에는 로그인과 회원정보 기능이 있으므로 Step 1은 충족된다.

### PDF Step 2. 암호화 통신 적용 여부 확인

Paros 또는 Wireshark로 로그인 요청을 잡는다.

Paros 기준 흐름은 다음과 같다.

```text
1. 브라우저 프록시를 Paros로 연결한다.
2. CARE 로그인 페이지에 접속한다.
3. 실습용 계정으로 로그인한다.
4. Paros에서 `/member/loginModel.php` 요청을 찾는다.
5. 요청 URL이 `http://...`인지 확인한다.
6. POST body에 `id=...&pw=...`가 평문으로 보이는지 확인한다.
```

Wireshark를 쓴다면 다음처럼 필터링할 수 있다.

```text
http.request.method == "POST"
```

또는 WEB 서버 IP를 알고 있다면 다음처럼 좁힌다.

```text
http && ip.addr == 172.168.10.10
```

로그인 요청의 URL scheme이 `http`이고, 요청 본문에 `id`, `pw`가 그대로 보이면 PDF Step 2 기준으로 취약이다.

### PDF Step 3. 취약한 SSL/TLS 프로토콜 사용 여부 확인

이 단계는 HTTPS가 이미 적용된 사이트에서 SSL 2.0, SSL 3.0, 구버전 TLS 사용 여부를 확인하는 절차다.

현재 CARE 실습 환경은 HTTPS 자체가 구성되지 않은 상태이므로, Step 3은 바로 수행하는 핵심 증거가 아니다. 먼저 Step 2에서 **암호화 통신 미적용**이 확인되면 취약 판정에 충분하다.

나중에 HTTPS를 구성한 뒤에는 다음을 확인한다.

```bash
openssl s_client -connect 172.168.10.10:443 -tls1
openssl s_client -connect 172.168.10.10:443 -tls1_1
openssl s_client -connect 172.168.10.10:443 -tls1_2
```

TLS 1.0, TLS 1.1 연결은 실패하고 TLS 1.2 이상 연결만 성공하는 방향이 바람직하다.

## 5. 조치 방안

PDF p.768의 조치 방법은 크게 3개이고, 실제 적용 관점에서는 아래처럼 나누어 본다.

### PDF 조치 1. 전송 정보 제한과 암호화 통신 적용

#### PDF 조치 1-1. 불필요한 중요정보 전송 최소화

불필요하게 비밀번호, 주민등록번호, 계좌정보 같은 값을 반복 전송하지 않는다.

CARE 기준으로는 다음을 확인한다.

| 확인 대상 | 조치 방향 |
|---|---|
| 로그인 | `id`, `pw`는 로그인 시 한 번만 전송 |
| 회원정보 수정 | 변경에 필요한 값만 전송 |
| URL | 비밀번호나 개인정보를 GET 파라미터에 넣지 않음 |
| 로그 | Apache access log에 비밀번호가 남는 구조를 만들지 않음 |

이 조치는 전송량을 줄이는 보조 조치다. 하지만 HTTP 자체가 평문이면 로그인 비밀번호는 여전히 노출되므로, 핵심 조치는 1-2의 HTTPS 적용이다.

#### PDF 조치 1-2. 중요 정보 전송구간에 HTTPS 적용

로그인, 회원가입, 회원정보 수정처럼 중요 정보가 오가는 구간은 HTTPS로 처리한다.

현재 프로젝트에서는 두 방향이 있다.

| 방식 | 설명 | 이번 프로젝트 적합성 |
|---|---|---|
| WEB 서버 Apache에 HTTPS 적용 | Apache에 인증서와 443 VirtualHost를 설정 | VM 단일 WEB 확인에는 적합 |
| LB/nginx에서 HTTPS 종료 | LB가 443을 받고 WEB에는 내부 HTTP로 전달 | LB를 보고서 구조에 포함할 때 적합 |
| AWS ALB + ACM | AWS에서 인증서와 HTTPS Listener를 구성 | AWS 재사용 시 적합 |

Apache에 적용한다면 구조는 대략 다음과 같다.

```apache
<VirtualHost *:443>
    ServerName care.local
    DocumentRoot /var/www/html/care

    SSLEngine on
    SSLCertificateFile /etc/ssl/certs/care.crt
    SSLCertificateKeyFile /etc/ssl/private/care.key

    <Directory /var/www/html/care>
        Options -Indexes +FollowSymLinks
        AllowOverride All
        Require all granted
    </Directory>
</VirtualHost>
```

LB/nginx에서 HTTPS를 처리한다면 구조는 대략 다음과 같다.

```nginx
server {
    listen 443 ssl;
    server_name _;

    ssl_certificate /etc/ssl/certs/care.crt;
    ssl_certificate_key /etc/ssl/private/care.key;

    location / {
        proxy_pass http://backend_nodes;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-Proto https;
    }
}
```

#### PDF 조치 1-3. HTTP 요청을 HTTPS로 전환

사용자가 실수로 `http://`로 들어와도 HTTPS로 이동시키는 설정을 둔다.

Apache 예시는 다음과 같다.

```apache
<VirtualHost *:80>
    ServerName care.local
    Redirect permanent / https://care.local/
</VirtualHost>
```

nginx 예시는 다음과 같다.

```nginx
server {
    listen 80;
    server_name _;
    return 301 https://$host$request_uri;
}
```

실습 환경에서 자체 서명 인증서를 쓰면 브라우저 경고가 뜰 수 있다. 그래도 HTTP 평문 전송보다는 전송 내용이 암호화된다는 점이 핵심이다.

### PDF 조치 2. 클라이언트 측 노출 위치에서 중요정보 제거

PDF는 쿠키 등 클라이언트 측에 비밀번호, 인증 식별값, 개인정보를 저장하지 말라고 한다.

CARE에서는 다음을 확인한다.

| 확인 위치 | 확인할 점 |
|---|---|
| URL | `?pw=...`, `?email=...` 같은 민감정보가 GET 파라미터로 노출되지 않는가 |
| Cookie | 비밀번호나 개인정보가 쿠키에 직접 저장되지 않는가 |
| Hidden field | 서버가 검증해야 할 인증값이나 권한값을 hidden input에만 의존하지 않는가 |
| LocalStorage / SessionStorage | 비밀번호나 개인정보를 브라우저 저장소에 넣지 않는가 |

이 조치는 HTTPS와 별개로 필요하다. HTTPS를 적용해도 브라우저 저장소나 URL에 민감정보를 남기면 다른 방식으로 유출될 수 있다.

### PDF 조치 3. 취약한 SSL/TLS 프로토콜 비활성화

HTTPS를 구성한 뒤에는 구버전 SSL/TLS를 막아야 한다.

Apache 예시는 다음과 같다.

```apache
SSLProtocol all -SSLv2 -SSLv3 -TLSv1 -TLSv1.1
```

nginx 예시는 다음과 같다.

```nginx
ssl_protocols TLSv1.2 TLSv1.3;
```

현재 CARE 환경은 HTTPS 자체가 없으므로 이 조치는 아직 직접 검증 대상이 아니다. HTTPS를 적용한 뒤에 추가로 확인한다.

## 6. 조치 후 점검

조치 후에는 조치 전과 같은 절차를 반복한다.

| 확인 항목 | 기대 결과 |
|---|---|
| `http://172.168.10.10/member/login.php` 접속 | HTTPS로 redirect되거나 HTTP 접근이 차단됨 |
| HTTPS 로그인 요청 | 브라우저 주소창이 `https://`이고, 네트워크에서 평문 `id`, `pw`가 보이지 않음 |
| 쿠키 확인 | 비밀번호나 개인정보가 쿠키에 직접 저장되지 않음 |
| URL 확인 | 로그인, 회원정보 수정 과정에서 민감정보가 URL에 노출되지 않음 |
| TLS 프로토콜 확인 | TLS 1.2 이상만 허용됨 |

Paros를 프록시로 사용할 경우, HTTPS 트래픽은 별도 인증서 신뢰 설정을 하지 않으면 본문이 그대로 보이지 않는 것이 정상이다. 이 항목의 핵심은 “HTTP에서 보이던 `id`, `pw`가 전송구간에서 평문으로 보이지 않는다”는 차이를 캡처하는 것이다.

#### 7-4-1. HTTPS 적용 전 스냅샷

HTTPS 조치 전 상태로 되돌릴 수 있도록 VM 스냅샷을 먼저 생성했다.

```text
snapshot: HTTPS 하기 전
```

#### 7-4-2. Apache self-signed HTTPS 구성

실습 환경은 AWS ALB, ACM, nginx LB 없이 VM의 Apache만 사용하는 구조다. 따라서 공인 인증서가 아니라 self-signed 인증서로 HTTPS를 구성했다.

먼저 SSL 모듈을 활성화했다.

```bash
sudo a2enmod ssl
```

출력에서 `ssl`과 의존 모듈 `socache_shmcb`가 활성화되었다.

```text
Enabling module socache_shmcb.
Enabling module ssl.
To activate the new configuration, you need to run:
  systemctl restart apache2
```

첫 번째 인증서 생성 시도에서는 `Country Name` 입력 프롬프트에 명령어를 잘못 붙여 넣어 실패했다.

```text
Country Name (2 letter code) [AU]:sudo a2enmod ssl
String too long, must be at most 2 bytes long
```

다시 실행해서 다음 값으로 self-signed 인증서를 생성했다.

```text
Country Name: kr
State or Province Name: seoul
Locality Name: noryangjin
Organization Name: CARE
Organizational Unit Name: Lab
Common Name: 172.168.10.10
Email Address: blank
```

생성된 파일은 다음 두 개다.

```text
/etc/ssl/private/care-selfsigned.key
/etc/ssl/certs/care-selfsigned.crt
```

그 다음 Apache HTTPS site 설정을 추가했다.

```apache
<VirtualHost *:443>
    ServerName care.local
    DocumentRoot /var/www/html/care
    DirectoryIndex index.php index.html

    SSLEngine on
    SSLCertificateFile /etc/ssl/certs/care-selfsigned.crt
    SSLCertificateKeyFile /etc/ssl/private/care-selfsigned.key

    <Directory /var/www/html/care>
        Options -Indexes +FollowSymLinks
        AllowOverride All
        Require all granted
    </Directory>

    ErrorLog ${APACHE_LOG_DIR}/care_ssl_error.log
    CustomLog ${APACHE_LOG_DIR}/care_ssl_access.log combined
</VirtualHost>
```

설정 파일 경로는 다음이다.

```text
/etc/apache2/sites-available/care-ssl.conf
```

마지막으로 site를 활성화하고 Apache 설정 검사를 통과했다.

```bash
sudo a2ensite care-ssl.conf
sudo apache2ctl configtest
sudo systemctl reload apache2
```

확인된 출력은 다음이다.

```text
Enabling site care-ssl.
Syntax OK
```

`AH00558: Could not reliably determine the server's fully qualified domain name` 경고도 출력되었지만, `Syntax OK`가 나왔으므로 설정 문법 자체는 통과했다. 이 경고는 전역 `ServerName`이 없을 때 Apache가 흔히 출력하는 경고이며, 현재 HTTPS VirtualHost 문법 오류와는 별개다.

#### 7-4-3. 서버 내부 HTTPS 확인

Apache를 재시작한 뒤 서비스 상태를 확인했다.

```bash
sudo systemctl status apache2 --no-pager
```

상태는 `active (running)`이었다.

서버 내부에서 HTTPS 요청을 직접 확인했다.

```bash
curl -vk https://127.0.0.1/
```

확인된 핵심 출력은 다음이다.

```text
SSL connection using TLSv1.3 / TLS_AES_256_GCM_SHA384
Server certificate:
  subject: C=kr; ST=seoul; L=noryangjin; O=CARE; OU=Lab; CN=172.168.10.10
HTTP/1.1 200 OK
Set-Cookie: PHPSESSID=...
Content-Type: text/html; charset=UTF-8
```

또한 443 포트가 Apache에 의해 열려 있음을 확인했다.

```bash
sudo ss -ltnp | grep ':443'
```

출력에서 `*:443`이 `apache2` 프로세스에 의해 `LISTEN` 중이었다.

서버 방화벽도 확인했다.

```bash
sudo ufw status
```

결과는 다음과 같았다.

```text
Status: inactive
```

따라서 WEB 서버 내부 기준으로는 Apache HTTPS 설정과 443 포트 리슨이 동작한다.

#### 7-4-4. VM Windows에서 HTTPS 확인

VM Windows PowerShell에서 443 포트 연결을 확인했다.

```powershell
Test-NetConnection 172.168.10.10 -Port 443
```

결과는 다음과 같았다.

```text
ComputerName     : 172.168.10.10
RemoteAddress    : 172.168.10.10
RemotePort       : 443
InterfaceAlias   : Ethernet0
SourceAddress    : 172.168.10.100
TcpTestSucceeded : True
```

즉 VM Windows에서 WEB 서버의 443 포트까지 TCP 연결은 가능했다.

이어서 Windows의 `curl.exe`로 HTTPS 요청을 확인했다.

```powershell
curl.exe -vk https://172.168.10.10/
```

확인된 핵심 출력은 다음이다.

```text
Connected to 172.168.10.10 port 443
schannel: SSL/TLS handshake complete
GET / HTTP/1.1
HTTP/1.1 200 OK
Server: Apache
Set-Cookie: PHPSESSID=...
Content-Type: text/html; charset=UTF-8
```

Windows CLI 기준으로도 HTTPS 연결, TLS handshake, CARE HTML 응답은 성공했다.

#### 7-4-5. 브라우저 접속 실패

브라우저에서는 다음 URL로 접속을 시도했다.

```text
https://172.168.10.10/index.php
```

하지만 화면은 다음 오류로 실패했다.

```text
지금 이 페이지가 작동하지 않습니다.
172.168.10.10은(는) 데이터를 보내지 않았습니다.
ERR_EMPTY_RESPONSE
```

`Ctrl + Shift + Delete`로 브라우저 데이터를 삭제한 뒤에도 같은 증상이 지속되었다.

이 결과는 서버 자체의 HTTPS 실패라기보다 브라우저와 self-signed 인증서 처리, 캐시, 또는 실습 브라우저 환경 문제로 남겼다. 같은 VM Windows에서 `curl.exe -vk https://172.168.10.10/`는 `HTTP/1.1 200 OK`를 받았기 때문이다.

#### 7-4-6. VM self-signed HTTPS 시도 결론

VM 환경에서 self-signed HTTPS를 구성한 결과는 다음처럼 정리한다.

| 확인 항목 | 결과 |
|---|---|
| Apache SSL module | 활성화 |
| self-signed certificate | 생성 |
| Apache 443 VirtualHost | 생성 및 활성화 |
| Apache configtest | `Syntax OK` |
| 서버 내부 HTTPS 요청 | `HTTP/1.1 200 OK` |
| Windows TCP 443 연결 | 성공 |
| Windows `curl.exe` HTTPS 요청 | `HTTP/1.1 200 OK` |
| 브라우저 HTTPS 접속 | `ERR_EMPTY_RESPONSE`로 실패 |
| HTTP -> HTTPS redirect | 미구성 |
| Paros/Wireshark 조치 후 재검증 | 미수행 |

따라서 이번 VM 실습에서는 HTTPS 조치가 **서버와 CLI 기준으로는 동작**했지만, 브라우저 기반 증거와 Paros/Wireshark 재검증까지 완료하지 못했다.

이 항목은 VM self-signed 방식으로 더 시간을 쓰지 않고, AWS 환경에서 ALB/ACM 또는 EC2 HTTPS 구성으로 다시 확인한다.

## 8. 판단

| 항목 | 판단 |
|---|---|
| 1차 분류 | A-3 |
| 판정 상태 | 확정 |
| 조치 상태 | 부분 수행. VM Apache self-signed HTTPS는 CLI 기준 성공했으나 브라우저 검증 실패로 AWS에서 후속 |
| 증거 상태 | 조치 전 HTTP 평문 전송 증거 확보, 조치 후 서버/Windows CLI HTTPS 성공, 브라우저·Paros 재검증 미완료 |

17번은 CARE에 로그인, 회원가입, 회원정보 수정 기능이 이미 있고, 현재 서비스 구성이 HTTP 80번 포트 중심이므로 PDF p.766의 취약 기준에 해당한다.

조치가 SSL/TLS, Apache, nginx, LB 설정까지 이어져서 코드 취약점보다 번거로운 것은 맞다. 하지만 이것은 “실습 기능이 없다”는 뜻이 아니라 “전송구간 보안 설정이 빠져 있다”는 뜻이다. 따라서 D가 아니라 **A-3 환경 설정형 취약점**으로 정리한다.

최소 증거는 조치 전 HTTP 로그인 요청에서 `id`, `pw`가 평문으로 보이는 화면이다. 이 증거로 CARE가 데이터 평문 전송 취약 상태라는 점은 확인했다.

조치 후 검증은 VM Apache self-signed HTTPS 방식으로 시도했다. 서버 내부 `curl`과 VM Windows `curl.exe`에서는 TLS handshake와 `HTTP/1.1 200 OK`가 확인되었으므로 Apache HTTPS 설정 자체는 동작했다. 다만 브라우저에서는 `ERR_EMPTY_RESPONSE`가 계속 발생했고, 캐시 삭제 후에도 해결되지 않았다.

따라서 이번 VM 실습에서는 17번을 다음 상태로 종결한다.

```text
취약 여부: 확인
조치 설계: 확인
VM self-signed HTTPS: CLI 기준 부분 성공
브라우저/Paros 조치 후 증거: 미완료
후속: AWS 환경에서 HTTPS 구성 후 재검증
```

보고서에는 “HTTP 평문 전송 취약 확인”을 주 증거로 쓰고, self-signed HTTPS 시행착오는 조치 시도 및 후속 과제로 정리한다.
