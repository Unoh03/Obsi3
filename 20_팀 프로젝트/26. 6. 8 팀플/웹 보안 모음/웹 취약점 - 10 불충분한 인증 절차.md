# 웹 취약점 - 10 불충분한 인증 절차

source: [[40_자료/주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드.pdf|주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드]], p.729-732.

## 1. 개요

**불충분한 인증 절차**는 개인정보 변경, 본인인증, OTP 인증처럼 중요한 기능에 접근할 때 추가 인증 절차가 없거나, 인증 여부를 서버에서 제대로 검증하지 않아 중요 정보가 노출되거나 변조될 수 있는 취약점이다.

PDF p.729의 판단 기준은 다음처럼 정리할 수 있다.

| 판단 | 기준 |
|---|---|
| 양호 | 중요 정보 페이지 접근 시 추가 인증 절차가 있고, 인증 로직이 서버 사이드에서 구현되어 우회가 불가능함 |
| 취약 | 중요 정보 페이지 접근 시 추가 인증 절차가 없거나, 인증 로직 우회가 가능하여 비인가자가 접근 가능함 |

이번 CARE 실습에서는 회원정보 수정 기능을 기준으로 확인한다. CARE는 로그인만 되어 있으면 현재 비밀번호 재확인 없이 비밀번호와 개인정보를 바로 수정할 수 있으므로, 최초 상태 기준 **A-1 자연 취약**으로 분류했다.

## 2. 문제가 되는 부분

CARE의 회원정보 수정 기능은 로그인 세션만 확인한다.

| 파일 | 문제 지점 |
|---|---|
| `member/modify.php` | 로그인 여부만 확인하고 회원 수정 화면을 바로 보여줌 |
| `member/modifyModel.php` | 현재 비밀번호나 최근 재인증 여부를 확인하지 않고 회원정보를 수정함 |

`modify.php`는 세션에 로그인 ID가 있는지만 확인한다.

```php
if($_SESSION['id'] == ""){
    echo "<script>location.href='login.php';</script>";
    exit;
}
```

수정 화면에는 새 비밀번호, 비밀번호 확인, 이름, 휴대폰, 주소, 이메일 입력칸이 있다. 하지만 **현재 비밀번호를 다시 입력하는 칸은 없다.**

```php
<label>패스워드</label><input type="password" name="pw" id="pw" >
<label>패스워드 확인</label><input type="password" name="pwCheck" id="pwCheck">
<label>이름</label><input type="text" name="name" id="name" value="<?=$_SESSION['name'] ?>">
```

`modifyModel.php`도 마찬가지로 세션 ID만 확인한 뒤, 전달받은 값으로 바로 `UPDATE`를 수행한다.

```php
$id = $_SESSION['id'];

$pw = $_POST['pw'];
$pwCheck = $_POST['pwCheck'];
$name = $_POST['name'];
$mobile = $_POST['mobile'];
$address = $_POST['address'];
$email= $_POST['email'];
```

```php
$query = "UPDATE member SET pw='$pw', name='$name', mobile='$mobile',
email='$email', address='$address' WHERE id='$id'";

mysqli_query($link, $query);
```

이 코드에서 막는 것은 다음 정도다.

| 검증 항목 | 현재 처리 |
|---|---|
| 로그인 여부 | `$_SESSION['id']`가 비어 있으면 차단 |
| 새 비밀번호 입력 여부 | 비어 있으면 차단 |
| 새 비밀번호 / 확인값 일치 여부 | 다르면 차단 |
| 현재 비밀번호 재확인 | 없음 |
| 최근 본인인증 여부 | 없음 |
| OTP / 2차 인증 | 없음 |

따라서 이 취약점의 핵심은 **비밀번호 변경이라는 중요 기능에 추가 인증 절차가 없다는 점**이다.

## 3. 악용 흐름

이번 항목은 SQL Injection이나 XSS처럼 별도 payload를 넣는 방식이 아니다. 이미 로그인된 사용자의 세션을 가진 상태에서 중요 정보 수정 절차가 너무 쉽게 열리는지가 핵심이다.

```text
사용자 로그인
-> 메인 화면에서 Modify 클릭
-> 현재 비밀번호 재확인 없이 회원 수정 페이지 진입
-> 새 비밀번호와 개인정보 입력
-> 서버가 현재 비밀번호를 확인하지 않고 UPDATE 실행
-> 회원정보 수정 완료
```

공격자가 사용자의 브라우저를 잠깐 사용할 수 있거나, 사용자가 로그인한 상태에서 자리를 비운 경우를 생각하면 위험이 더 분명하다. 현재 구조에서는 공격자가 기존 비밀번호를 몰라도 새 비밀번호와 개인정보를 바꿀 수 있다.

## 4. 점검 방법

PDF p.729-730의 점검 방법은 두 단계다.

| 단계 | PDF 기준 | CARE에서의 해석 |
|---|---|---|
| Step 1 | 중요 정보 페이지 접근 시 재인증 절차 존재 여부 확인 | 회원 수정 페이지 접근 시 현재 비밀번호, OTP, 본인인증 같은 추가 인증이 있는지 확인 |
| Step 2 | 로직 변조, 파라미터 변조로 인증 로직 우회 가능 여부 확인 | 우회할 인증 로직이 있는지 확인 |

CARE에서는 Step 1에서 바로 취약하다.

```text
로그인 상태
-> Modify 클릭
-> 현재 비밀번호 재확인 없이 회원 수정 페이지 접근 가능
```

Step 2는 별도로 깊게 진행할 인증 로직이 없다. 우회할 인증 로직이 존재하는데 뚫리는 것이 아니라, **추가 인증 로직 자체가 없기 때문**이다. 따라서 이 항목은 Step 1에서 취약으로 판단한다.

확인 URL은 다음과 같다.

```text
http://172.168.10.10/member/modify.php
http://172.168.10.10/member/modifyModel.php
```

## 5. 조치 방안

PDF p.730-732의 조치 방향은 다음과 같다.

1. 중요 정보를 다루는 페이지 접근 시 본인인증을 재확인한다.
2. 인증 과정은 클라이언트 사이드가 아니라 서버 사이드에서 구현한다.
3. 로그인 및 권한 체크는 모든 보호 페이지에서 일관되게 수행한다.

CARE에서는 회원정보 수정 전에 **현재 비밀번호 재확인**을 추가하는 방식이 가장 단순하다.

### 5.1 `modify.php`에 현재 비밀번호 입력칸 추가

회원 수정 form에 현재 비밀번호 입력칸을 추가한다.

```php
<label>현재 비밀번호</label><input type="password" name="currentPw" id="currentPw">
<div class="clear"></div>
```

클라이언트 측에서는 입력 누락을 먼저 안내할 수 있다.

```javascript
var currentPw = document.getElementById('currentPw');
if(currentPw.value == "" ){
    alert('현재 비밀번호를 입력하세요');
    currentPw.focus();
    return;
}
```

다만 이 검사는 편의 기능일 뿐이다. 사용자는 JavaScript를 우회할 수 있으므로, 실제 방어는 반드시 서버에서 해야 한다.

### 5.2 `modifyModel.php`에서 서버 측 재인증

`modifyModel.php`에서 사용자가 입력한 현재 비밀번호를 받는다.

```php
$currentPw = $_POST['currentPw'] ?? '';
```

그 다음 세션 ID에 해당하는 기존 비밀번호를 DB에서 조회하고, 현재 비밀번호와 비교한다.

```php
if($currentPw == ''){
    echo "<script>alert('현재 비밀번호를 입력하세요'); history.go(-1); </script>";
    mysqli_close($link);
    exit;
}

$reauthQuery = "SELECT pw FROM member WHERE id='$id'";
$reauthResult = mysqli_query($link, $reauthQuery);
$reauthRow = mysqli_fetch_assoc($reauthResult);

if(! $reauthRow || $currentPw != $reauthRow['pw']){
    echo "<script>alert('현재 비밀번호가 일치하지 않습니다.'); history.go(-1); </script>";
    mysqli_close($link);
    exit;
}
```

현재 CARE는 교육용으로 비밀번호가 평문 저장되어 있으므로 위 예시는 평문 비교를 사용한다. 실무에서는 비밀번호를 평문 저장하면 안 되고, `password_hash()`와 `password_verify()`를 사용해야 한다.

### 5.3 현재 프로젝트 코드 상태

현재 로컬 `D:\care\member\modify.php`와 `D:\care\member\modifyModel.php`에는 현재 비밀번호 재확인 코드가 적용되어 있다.

| 파일 | 현재 상태 |
|---|---|
| `modify.php` | `currentPw` 입력칸과 JavaScript 입력 확인이 추가됨 |
| `modifyModel.php` | `$_POST['currentPw']`를 받고, DB의 기존 비밀번호와 비교한 뒤 `UPDATE`를 수행함 |

따라서 조치 전 취약 상태는 기존 코드와 조치 전 스크린샷으로 설명하고, 조치 후 상태는 현재 코드와 조치 후 요청 스크린샷으로 확인한다.

## 6. 조치 후 점검

조치 후에는 다음을 확인한다.

| 점검 항목 | 현재 확인 |
|---|---|
| 현재 비밀번호 입력칸이 비어 있으면 회원정보 수정이 차단되어야 함 | 코드 기준 확인 |
| 현재 비밀번호가 틀리면 회원정보 수정이 차단되어야 함 | 코드 기준 확인, 화면 증거는 미수집 |
| 현재 비밀번호가 맞고 새 비밀번호와 비밀번호 확인값이 일치할 때만 수정되어야 함 | 요청 스크린샷으로 확인 |
| 로그인하지 않은 사용자는 `modify.php`, `modifyModel.php`에 접근할 수 없어야 함 | 기존 로그인 세션 검증 코드 유지 |
| JavaScript를 우회해 `modifyModel.php`에 직접 요청하더라도 서버 측 현재 비밀번호 검증이 동작해야 함 | `modifyModel.php`의 서버 측 검증 코드로 확인 |

## 7. 증거

아래 스크린샷과 코드 조각을 증거로 사용한다.

- [x] 조치 전: 현재 비밀번호 입력 없이 새 비밀번호와 개인정보 수정 요청이 전송됨

![[Pasted image 20260616104700.png]]

요청 body는 다음처럼 `pw`, `pwCheck`, `name`, `mobile`, `address`, `email`만 포함한다.

```text

```

기존 비밀번호가 `123`인 상태에서도 기존 비밀번호를 모른 채 새 비밀번호 `1234`와 개인정보 값을 전송할 수 있었다. 이 증거는 조치 전 `modifyModel.php`가 현재 비밀번호 재확인 없이 수정 요청을 처리할 수 있음을 보여준다.

- [x] 취약 코드: `modify.php`에 현재 비밀번호 입력칸이 없음

`패스워드`와 `패스워드 확인`은 새 비밀번호와 새 비밀번호 확인값일 뿐이고, 기존 비밀번호인지 확인하는 입력칸은 없다.

- [x] 취약 코드: `modifyModel.php`가 현재 비밀번호 확인 없이 `UPDATE member` 실행

```php
//로그인 안했으면 로그인부터.
```

조치 전 검증은 로그인 여부, 필수값 여부, 새 비밀번호와 확인값 일치 여부뿐이다. 현재 비밀번호를 DB의 기존 비밀번호와 비교하는 로직은 없다.

이 증거는 조치 후 회원정보 수정 요청이 현재 비밀번호 재확인 값을 함께 보내는 구조로 바뀌었음을 보여준다. 현재 비밀번호 불일치 차단 화면은 별도 스크린샷으로 남기지 않았지만, `modifyModel.php`의 서버 측 비교 로직으로 차단 조건은 확인된다.

## 8. 판단

이 항목은 조치 전 기준 **취약**으로 판단한다. 현재 로컬 코드에는 조치가 적용되어 있으며, 조치 후에는 현재 비밀번호 재확인을 거쳐야 회원정보 수정이 진행되는 구조로 바뀌었다.

| 항목 | 판단 |
|---|---|
| 1차 분류 | A-1 |
| 판정 상태 | 확정 |
| 조치 상태 | 로컬 코드 조치 적용 |
| 증거 상태 | 조치 전 요청, 취약 코드, 조치 코드, 조치 후 요청 확인 |

CARE 기본 기능에 회원정보 수정 페이지가 있고, 로그인만 되어 있으면 추가 인증 없이 비밀번호와 개인정보를 바꿀 수 있다.

PDF 기준으로는 Step 1에서 이미 취약하다. Step 2의 인증 로직 우회는 별도로 깊게 진행할 대상이 없다. 우회할 인증 로직이 있는 것이 아니라, **추가 인증 로직 자체가 없기 때문**이다.

따라서 이 취약점은 실습용 기능을 새로 만든 것이 아니라, CARE 원래 기능만으로 확인 가능한 자연 취약점이다.

조치 방향은 PDF p.730-732와 일치한다. 중요 정보 수정 시 현재 비밀번호를 다시 입력받고, `modifyModel.php`에서 DB의 기존 비밀번호와 비교한 뒤에만 `UPDATE member`를 실행한다. 클라이언트 입력 확인은 편의 기능이고, 실제 방어는 서버 측 재인증 로직이다.
