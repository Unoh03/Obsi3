# 웹 취약점 - 12·13 비밀번호 복구 절차와 프로세스 검증 누락

source: [[40_자료/주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드.pdf|주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드]], p.735-743.

## 1. 개요

**취약한 비밀번호 복구 절차**는 비밀번호 찾기나 재설정 과정에서 인증값이 너무 단순하거나, 웹 화면에 바로 노출되거나, 인증된 사용자 메일/SMS가 아닌 공격자 쪽으로 전달될 수 있는지 확인하는 항목이다.

PDF p.735의 판단 기준은 다음처럼 정리할 수 있다.

| 판단  | 기준                                                            |
| --- | ------------------------------------------------------------- |
| 양호  | 비밀번호 재설정 시 난수를 사용하고, 인증된 사용자 메일이나 SMS로 임시 비밀번호 또는 재설정 링크를 전송함 |
| 취약  | 비밀번호 재설정 값이 일정한 패턴으로 만들어지거나, 웹 사이트 화면에 바로 출력됨                 |

CARE 원본에는 비밀번호 복구 기능이 없다. 따라서 원래 CARE 기능만 보면 12번은 `D`에 가깝다. 하지만 13번 `프로세스 검증 누락`과 함께 보기 위해 `/vuln/password-recovery/` 아래에 실습용 비밀번호 복구 흐름을 만들었으므로, 현재 프로젝트에서는 **C: 실습용 기능 추가 재현**으로 처리한다.

이 문서는 12번과 13번을 분리된 실습 두 개로 보지 않는다. 같은 비밀번호 복구 기능 안에서 12번은 **인증번호 생성·전달·검증의 취약성**을 보고, 13번은 **인증번호 검증 단계를 건너뛰고 마지막 단계만 직접 호출할 수 있는지**를 본다.

## 2. CARE 원본 적용 가능성

CARE 원본에서 확인되는 계정 관련 기능은 다음 정도다.

| 기능 | 판단 |
|---|---|
| 로그인 | 있음 |
| 회원가입 | 있음 |
| 로그인 후 비밀번호 변경 | 있음 |
| 비밀번호 찾기 | 없음 |
| 임시 비밀번호 발급 | 없음 |
| 이메일/SMS 인증번호 전송 | 없음 |
| 재설정 링크 발급 | 없음 |

따라서 원본 CARE만으로는 PDF p.735-738의 점검 절차를 직접 수행할 수 없다.

이 항목을 억지로 원본 취약점이라고 쓰면 안 된다. 비밀번호 복구 기능이 없기 때문이다. 이번 실습에서는 원본과 분리된 `/vuln/password-recovery/` 기능을 만들어, PDF에서 말하는 취약한 복구 로직과 조치 로직을 비교한다.

## 3. 실습용 기능 구성

현재 로컬 CARE에는 다음 파일이 준비되어 있다.

```text
/vuln/password-recovery/
├─ request.php
├─ verify.php
└─ reset.php
```

각 파일의 역할은 다음과 같다.

| 파일 | 역할 | 12번 관련 지점 | 13번 관련 지점 |
|---|---|---|---|
| `request.php` | 아이디와 이메일 입력 후 인증번호 발급 | 취약 모드에서 고정 인증번호 `1234`를 웹 화면에 출력 | 복구 절차 시작 상태 생성 |
| `verify.php` | 인증번호 확인 | 취약 모드에서 고정 인증번호를 사용 | 조치 모드에서는 검증 성공 상태를 세션에 저장 |
| `reset.php` | 새 비밀번호 설정 | 복구 결과로 비밀번호 변경 | 취약 모드에서 `verify.php`를 건너뛰어도 변경 가능 |

현재는 서버에서 일부 실행 검증까지 진행했다. 스크린샷 기준으로 취약 모드의 인증번호 화면 출력, 고정 인증번호 검증, `reset.php` 직접 접근, 조치 모드의 인증번호 비노출, 서버 mailbox 기록, 올바른 인증번호 검증, 실패 횟수 제한을 확인했다.

## 4. 취약 모드 설계

취약 모드의 핵심은 인증번호가 고정되어 있고, 웹 화면에 직접 출력된다는 점이다.

`request.php`의 취약 모드는 다음 값을 세션과 화면에 사용한다.

```php
$_SESSION['pr_vuln_code'] = '1234';
$issuedCode = '1234';
$message = '취약 모드: 인증번호가 화면에 직접 출력되었습니다.';
```

이 구조에서는 사용자가 실제 메일이나 SMS를 확인하지 않아도 웹 화면에서 인증번호를 바로 볼 수 있다.

취약 흐름은 다음과 같다.

```text
request.php?mode=vuln
-> 아이디와 이메일 입력
-> 서버가 고정 인증번호 1234 발급
-> 인증번호가 웹 화면에 출력됨
-> 공격자가 인증번호를 확인하거나 추측 가능
```

이 흐름은 PDF의 취약 기준인 “일정 패턴으로 재설정” 또는 “웹 사이트 화면에 바로 출력”에 해당한다.

## 5. 조치 모드 설계

조치 모드의 핵심은 인증번호를 예측하기 어렵게 만들고, 웹 화면에 출력하지 않으며, 인증 상태와 만료 시간을 서버가 관리하는 것이다.

`request.php`의 조치 모드는 `random_int()`로 인증번호를 만든다.

```php
$code = (string)random_int(100000, 999999);
```

그리고 인증번호 원문을 그대로 세션에 저장하지 않고 hash로 저장한다.

```php
$_SESSION['pr_safe'] = [
    'user_id' => $id,
    'email' => $email,
    'code_hash' => password_hash($code, PASSWORD_DEFAULT),
    'expires_at' => time() + 300,
    'attempts' => 0,
    'verified' => false,
];
```

실제 운영이라면 이 인증번호는 등록된 이메일이나 SMS로 보내야 한다. 현재 프로젝트에서는 실제 메일/SMS 연동까지 하지 않으므로, 실습 확인용으로만 서버의 다음 파일에 기록한다.

```text
/tmp/care-password-recovery-mailbox.log
```

이 파일은 웹으로 직접 보여주는 화면이 아니라, 실습자가 SSH로 서버에 들어가 확인하는 임시 mailbox 역할이다.

## 6. 점검 방법

서버와 DB VM이 준비된 상태에서 다음 순서로 확인한다.

### 6.1 사전 준비

테스트용 회원이 필요하다.

```text
id: victim
email: victim@example.com 또는 DB에 실제 등록된 victim 이메일
```

DB에 해당 회원이 없다면 먼저 회원가입 또는 DB insert로 테스트 계정을 준비한다.

### 6.2 취약 모드 점검

1. 브라우저에서 취약 모드 요청 페이지에 접근한다.

```text
http://172.168.10.10/vuln/password-recovery/request.php?mode=vuln
```

2. 테스트 계정의 `id`, `email`을 입력한다.
3. 인증번호 `1234`가 웹 화면에 직접 출력되는지 확인한다.
4. `verify.php?mode=vuln`에서 `1234`를 입력했을 때 인증 성공 처리되는지 확인한다.

### 6.3 조치 모드 점검

1. 브라우저에서 조치 모드 요청 페이지에 접근한다.

```text
http://172.168.10.10/vuln/password-recovery/request.php?mode=safe
```

2. 테스트 계정의 `id`, `email`을 입력한다.
3. 웹 화면에 인증번호가 직접 출력되지 않는지 확인한다.
4. 서버에서 실습용 mailbox를 확인한다.

```bash
sudo find /tmp /var/tmp -name 'care-password-recovery-mailbox.log' -print
sudo cat "find 로 찾은 경로 복붙"
```

Apache가 systemd `PrivateTmp`를 사용하면 PHP의 `/tmp`는 SSH에서 바로 보는 `/tmp`와 다를 수 있다. 실제 캡처에서는 `systemd-private-...apache2.service.../tmp/` 아래에서 mailbox 파일을 확인했다.

5. `verify.php?mode=safe`에서 mailbox에 기록된 인증번호를 입력한다.
6. 틀린 인증번호를 여러 번 입력했을 때 실패 횟수 제한이 동작하는지 확인한다.
7. 인증번호 발급 후 시간이 오래 지나면 만료되는지 확인한다.

## 7. 조치 방안

PDF p.736-738의 조치 방향을 CARE 실습에 맞추면 다음과 같다.

| PDF 조치 | CARE 적용 방식 |
|---|---|
| 추측하기 어려운 정보 사용 | 단순 이름, 아이디, 생년월일 같은 공개 정보를 보안 질문으로 쓰지 않음 |
| 메일/SMS 인증 코드 전송 | 웹 화면에 인증번호를 출력하지 않고 등록된 연락 수단으로만 전송 |
| 난수 기반 임시값 생성 | `random_int()` 같은 암호학적으로 안전한 난수 사용 |
| 최소 길이와 복잡도 적용 | 새 비밀번호나 임시 비밀번호에 길이·복잡도 정책 적용 |
| 검증 실패 임계값 설정 | 일정 횟수 이상 실패하면 복구 절차 중단 또는 다른 방식 요구 |
| 임시 비밀번호 재사용 방지 | 임시값은 일회성으로 사용하고, 사용 직후 새 비밀번호 설정 요구 |

현재 조치 모드 코드에는 다음 요소가 반영되어 있다.

| 요소 | 현재 코드 상태 |
|---|---|
| `random_int()` 인증번호 생성 | 있음 |
| 인증번호 hash 저장 | 있음 |
| 만료 시간 | 300초 |
| 실패 횟수 제한 | 5회 |
| 웹 화면 직접 출력 차단 | safe 모드에서 차단 |
| 실제 이메일/SMS 전송 | 미구현. 실습용 mailbox로 대체 |

## 8. 13번 프로세스 검증 누락 통합

12번은 복구 절차의 인증번호 발급과 검증이 취약한지를 본다.

13번은 이 복구 절차에서 **이전 단계를 건너뛰고 마지막 단계만 직접 호출할 수 있는지**를 본다.

따라서 두 항목은 같은 `/vuln/password-recovery/` 기능 안에서 다음처럼 나눠 기록한다.

| 번호 | 핵심 질문 | 확인 파일 |
|---:|---|---|
| 12 | 인증번호가 안전하게 생성·전달·검증되는가 | `request.php`, `verify.php` |
| 13 | 인증번호 검증 단계를 거치지 않고 비밀번호를 바꿀 수 있는가 | `reset.php` |

13번의 핵심 증거는 `request.php`와 `verify.php`를 정상적으로 거치지 않고도 `reset.php`를 직접 호출해 새 비밀번호를 설정할 수 있는지다.

### 8.1 취약 모드 직접 호출

홈 화면에서 URL을 직접 입력해 `reset.php`로 바로 이동했다.

```text
http://172.168.10.10/vuln/password-recovery/reset.php
```

취약 모드에서는 인증번호 검증 단계를 거치지 않아도 새 비밀번호 입력 화면에 접근할 수 있다.

![[Pasted image 20260616114037.png]]

새 비밀번호를 입력하면 서버가 이전 단계 완료 여부를 확인하지 않고 비밀번호 변경 성공 메시지를 출력한다.

![[Pasted image 20260616114133.png]]

Burp 요청에서도 `reset.php`에 직접 POST로 `mode=vuln`, `id`, `new_password`, `new_password_check`만 전달되는 흐름을 확인했다.

![[Pasted image 20260616114155.png]]

즉 취약 모드의 문제는 단순히 입력값 검증이 약한 것이 아니라, `reset.php`가 “이 사용자가 인증번호 검증 단계를 통과했는가”를 서버 상태로 확인하지 않는다는 점이다.

### 8.2 조치 모드 직접 호출 차단

조치 모드에서는 `verify.php`가 남긴 서버 세션 상태가 없으면 `reset.php`가 비밀번호 변경을 차단한다.

## 9. 증거

현재 확보한 증거는 다음과 같다.

- [x] 취약 모드: 인증번호 `1234`가 웹 화면에 출력됨
- [x] 취약 모드: 고정 인증번호로 인증 성공
- [x] 취약 모드: 인증번호 검증 없이 `reset.php` 직접 호출 시 비밀번호 변경 가능
- [x] 조치 모드: 인증번호가 웹 화면에 출력되지 않음
- [x] 조치 모드: 인증번호가 서버의 실습용 mailbox에만 기록됨
- [x] 조치 모드: 올바른 인증번호로 검증 성공
- [x] 조치 모드: 잘못된 인증번호 반복 입력 제한
- [x] 조치 모드: 인증 완료 상태 없이 `reset.php` 직접 호출 시 차단

추가로 확보하면 좋은 증거는 다음이다.

- [ ] 취약 모드에서 변경된 비밀번호로 실제 로그인 성공
- [ ] 조치 모드에서 인증 완료 후에만 비밀번호 변경 성공

CARE 원본에는 비밀번호 복구 기능이 없으므로 12번과 13번을 원래 CARE의 자연 취약점으로 쓰면 안 된다.

다만 PDF p.735-743의 점검 흐름을 재현하기 위해 `/vuln/password-recovery/` 아래에 실습용 복구 기능을 만들었다. 취약 모드는 고정 인증번호와 화면 출력으로 12번을 보여주고, `reset.php` 직접 호출로 13번을 보여준다. 조치 모드는 `random_int()`, 만료 시간, 실패 횟수 제한, 서버 측 인증 완료 상태를 통해 두 항목의 방어 방향을 함께 보여준다.

따라서 현재 상태는 **12번과 13번을 하나의 비밀번호 복구 실습 흐름으로 통합해 설명하는 것이 더 정확하다.** 별도 문서로 완전히 분리하면 같은 기능의 증거가 둘로 찢어져 흐름이 약해진다.
