# 웹 취약점 - 19 관리자 페이지 노출

source: [[40_자료/주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드.pdf|주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드]], p.775-776.

## 1. 개요

**관리자 페이지 노출**은 관리자 로그인 페이지나 관리자 기능 URL이 외부에서 쉽게 접근 가능한 위치에 노출되어, 비인가 사용자가 관리자 인증을 시도하거나 관리자 기능을 탐색할 수 있는 취약점이다.

PDF p.775-776의 판단 기준은 다음처럼 정리할 수 있다.

|판단|기준|
|---|---|
|양호|관리자 페이지가 일반 사용자에게 노출되지 않고, 접근 가능한 IP·경로·인증 수단이 제한되어 있음|
|취약|관리자 페이지 URL이 유추 가능하거나 외부에 노출되어 비인가 사용자가 접근 및 로그인을 시도할 수 있음|

이 항목의 핵심은 관리자 계정 비밀번호가 강한지만 보는 것이 아니다. 관리자 페이지 자체가 일반 사용자 접근 범위에 노출되어 있는지, 접근 위치가 제한되어 있는지, 하위 관리자 기능이 직접 URL 입력으로 우회되지 않는지를 함께 확인한다.

```text
외부 사용자
-> /admin, /administrator, /manager 같은 관리자 URL 추측
-> 관리자 로그인 페이지 접근
-> 계정 추측, brute force, credential stuffing, 취약한 인증 절차 공격으로 확장
```

## 2. CARE 적용 가능성 분석

CARE 원본 기능을 기준으로 보면 관리자 페이지는 확인되지 않는다.

확인한 기준은 다음과 같다.

|확인 대상|확인 내용|판단|
|---|---|---|
|CARE 원본 화면|관리자 메뉴나 관리자 로그인 링크가 확인되지 않음|원본 공격면 없음|
|CARE 원본 소스|관리자 전용 디렉터리, 관리자 로그인 처리 파일이 확인되지 않음|원본 공격면 없음|
|서버 관리 도구|phpMyAdmin, Adminer 같은 관리 도구 endpoint를 CARE 기능으로 확인하지 않음|원본 공격면 없음|

따라서 원본 CARE 기준으로 19번은 **D**로 판단한다.

```text
원본 CARE 관리자 페이지: 없음
원본 CARE 관리자 URL 노출: 없음
원본 CARE 관리자 로그인 페이지 유추 가능성: 확인 근거 없음
```

다만 프로젝트 실습에서는 PDF 19번의 점검 방법과 조치 방법을 확인하기 위해, 별도 실습용 관리자 페이지를 구성한다.

이 경우 분류는 다음처럼 분리한다.

|기준|분류|
|---|---|
|CARE 원본 평가|D|
|실습용 관리자 페이지를 새로 만들어 재현|C|

즉, 이 노트는 “CARE 원본에서 자연 발생한 관리자 페이지 노출 취약점”이 아니라, **PDF 19번의 점검·조치 방법을 재현하기 위해 구성한 실습용 관리자 페이지**를 다룬다.

## 3. 실습용 관리자 페이지 구성

실습용 관리자 페이지는 일반적인 `/admin`, `/manager`, `/administrator` 같은 유추 가능한 경로가 아니라, 유추하기 어려운 파일명을 사용한다.

구성 내용은 다음과 같다.

|항목|계획|
|---|---|
|관리자 페이지 파일명|`gkfn8tlrkstnausqhwkd.php`|
|서비스 포트|`8888`|
|접근 허용 IP|`172.168.10.10`, `172.168.10.100`|
|접근 제한 방식|Apache `Require ip` 기반 IP allowlist|
|인증서 기반 접근|이번 실습에서는 구현하지 않고 조치안으로만 제시|
|하위 페이지 세션 검증|이번 실습은 단일 관리자 로그인 파일로 진행. 하위 페이지를 만들 경우 별도 세션 검증 필요|

권장 서버 배치 예시는 다음과 같다.

```text
/var/www/html/care-admin/gkfn8tlrkstnausqhwkd.php
```

접근 URL 예시는 다음과 같다.

```text
http://172.168.10.10:8888/gkfn8tlrkstnausqhwkd.php
```

이 경로는 서버에 배치되는 운영/실습 경로다. GitHub 저장소에는 이 결과 경로 자체를 그대로 올리는 것이 아니라, 실습용 PHP 파일 원본, Apache 설정 예시, 배포 스크립트, 노트를 올리는 방식으로 관리한다.
```bash
sudo mkdir -p /var/www/html/care-admin
sudo nano /var/www/html/care-admin/gkfn8tlrkstnausqhwkd.php
```
```php
<?php
session_start();

$message = '';

if ($_SERVER['REQUEST_METHOD'] === 'POST') {
    $id = $_POST['id'] ?? '';
    $pw = $_POST['pw'] ?? '';

    if ($id === 'admin' && $pw === 'admin1234') {
        $_SESSION['admin_login'] = true;
        $message = '관리자 로그인 성공';
    } else {
        $message = '관리자 로그인 실패';
    }
}
?>
<!doctype html>
<html lang="ko">
<head>
    <meta charset="utf-8">
    <title>관리자 로그인</title>
</head>
<body>
    <h1>CARE Admin Login</h1>

    <?php if ($message !== ''): ?>
        <p><?=htmlspecialchars($message, ENT_QUOTES, 'UTF-8')?></p>
    <?php endif; ?>

    <form method="post">
        <label>ID: <input name="id"></label><br>
        <label>PW: <input type="password" name="pw"></label><br>
        <button type="submit">Login</button>
    </form>
</body>
</html>
```
```bash
sudo grep -R "Listen 8888" /etc/apache2/ports.conf /etc/apache2/sites-available /etc/apache2/sites-enabled
(없으면)
echo "Listen 8888" | sudo tee -a /etc/apache2/ports.conf

sudo nano /etc/apache2/sites-available/care-admin-8888.conf
```
```conf
<VirtualHost *:8888>
    ServerName care-admin.local
    DocumentRoot /var/www/html/care-admin

    <Directory /var/www/html/care-admin>
        Options -Indexes +FollowSymLinks
        AllowOverride None

        Require ip 172.168.10.10
        Require ip 172.168.10.100
    </Directory>

    ErrorLog ${APACHE_LOG_DIR}/care-admin-8888-error.log
    CustomLog ${APACHE_LOG_DIR}/care-admin-8888-access.log combined
</VirtualHost>
```
```bash
sudo a2ensite care-admin-8888.conf
sudo apache2ctl configtest
sudo systemctl reload apache2
```
## 4. 문제가 되는 부분

관리자 페이지가 외부에 노출되면 다음 문제가 생길 수 있다.

|문제|설명|
|---|---|
|URL 유추 가능성|`/admin`, `/manager`, `/administrator` 같은 흔한 경로는 자동화 도구로 쉽게 탐색됨|
|로그인 시도 가능성|관리자 로그인 페이지가 보이면 계정 추측, 비밀번호 대입, credential stuffing 시도가 가능해짐|
|IP 제한 부재|내부 관리자나 지정된 관리망이 아닌 외부 사용자도 접근 가능|
|하위 페이지 직접 접근|로그인 페이지를 거치지 않고 `/admin/dashboard.php` 같은 하위 페이지를 직접 요청할 수 있음|
|추가 인증 부재|비밀번호가 유출되면 관리자 페이지 접근을 막을 2차 방어가 없음|

원본 CARE에는 관리자 페이지가 없으므로 이 문제는 원본 취약점으로 확정하지 않는다. 실습에서는 위 조건을 재현하기 위해 별도 관리자 페이지를 만든 뒤, IP 제한 조치 전후를 비교한다.

## 5. 점검 방법

PDF p.775-776의 점검 흐름은 다음처럼 정리할 수 있다.

|단계|내용|CARE 실습 적용|
|---|---|---|
|Step 1|관리자 페이지 URL이 외부에서 접근 가능한지 확인|`:8888/gkfn8tlrkstnausqhwkd.php` 직접 접근|
|Step 2|관리자 페이지 URL이 유추 가능한지 확인|원본 CARE에는 없음, 실습용 페이지는 유추 어려운 이름으로 구성|
|Step 3|지정 IP 외 접근이 차단되는지 확인|Apache `Require ip` 적용 전후 비교|
|Step 4|관리자 하위 페이지 직접 접근이 가능한지 확인|하위 페이지를 만들 경우 세션 검증 여부 확인|

### 5.1 관리자 페이지 접근 확인

조치 전에는 브라우저에서 다음 URL에 접근한다.

```text
http://172.168.10.10:8888/gkfn8tlrkstnausqhwkd.php
```

관리자 로그인 화면이 출력되면, 해당 관리자 페이지는 네트워크에서 접근 가능한 상태다.

### 5.2 IP allowlist 적용 전후 비교

허용할 IP는 다음 두 개로 제한한다.

```text
172.168.10.10
172.168.10.100
```

Apache 설정 적용 전에는 다른 IP에서도 관리자 페이지가 보일 수 있다.

Apache 설정 적용 후에는 다음처럼 판단한다.

|요청 출발지|기대 결과|
|---|---|
|`172.168.10.10`|접근 허용|
|`172.168.10.100`|접근 허용|
|그 외 IP|403 Forbidden 또는 사용자 정의 에러 페이지|

### 5.3 하위 페이지 직접 접근 확인

관리자 페이지가 여러 파일로 구성되어 있다면, 로그인 페이지뿐 아니라 하위 페이지도 확인해야 한다.

예시는 다음과 같다.

```text
/gkfn8tlrkstnausqhwkd.php
/admin-dashboard.php
/admin-user-list.php
```

취약한 구조는 다음과 같다.

```text
관리자 로그인 페이지는 인증 검사함
하지만 admin-dashboard.php는 세션 검사 없이 직접 접근 가능
```

이 경우 사용자가 로그인 페이지를 거치지 않고 하위 페이지 URL을 직접 입력해 접근할 수 있다.

따라서 모든 관리자 하위 페이지에는 다음과 같은 세션 검증이 필요하다.

```php
<?php
session_start();

if (($_SESSION['admin_login'] ?? false) !== true) {
    http_response_code(403);
    echo 'admin login required';
    exit;
}
```

이번 실습에서 관리자 페이지를 단일 파일로만 구성한다면, 하위 페이지 직접 접근 검증은 범위 밖으로 둔다. 하위 페이지를 추가로 만들 경우에만 이 조치를 확인한다.

## 6. 조치 방안

PDF p.776의 조치 방향을 CARE 실습에 맞추면 다음과 같다.

|PDF 조치|CARE 실습 적용|
|---|---|
|관리자 로그인 페이지 주소를 유추하기 어려운 URL 및 포트로 변경|`gkfn8tlrkstnausqhwkd.php`, 포트 `8888` 사용|
|관리자 페이지에 접근 가능한 IP 지정|Apache `Require ip`로 `172.168.10.10`, `172.168.10.100`만 허용|
|외부 노출이 불가피하면 2차 인증 적용|이번 실습에서는 구현하지 않고 조치안으로 제시|
|하위 페이지 직접 접근 방지|관리자 하위 페이지별 세션 검증 적용|

### 6.1 유추하기 어려운 URL 및 포트 사용

관리자 페이지를 흔한 경로에 두지 않는다.

피해야 할 예시는 다음과 같다.

```text
/admin/
/administrator/
/manager/
/admin/login.php
```

이번 실습에서는 다음 파일명과 포트를 사용한다.

```text
/gkfn8tlrkstnausqhwkd.php
port 8888
```

이 조치는 관리자 페이지를 완전히 보호하는 보안 수단은 아니다. URL 은닉은 보조 조치일 뿐이다. 핵심은 IP 제한, 인증, 하위 페이지 세션 검증이다.

### 6.2 IP allowlist 적용

Apache에서 관리자 페이지가 있는 경로에 IP 제한을 적용한다.

예시 설정은 다음과 같다.

```apache
<VirtualHost *:8888>
    ServerName care-admin.local
    DocumentRoot /var/www/html/care-admin

    <Directory /var/www/html/care-admin>
        Options -Indexes +FollowSymLinks
        AllowOverride None

        Require ip 172.168.10.10
        Require ip 172.168.10.100
    </Directory>

    ErrorLog ${APACHE_LOG_DIR}/care-admin-8888-error.log
    CustomLog ${APACHE_LOG_DIR}/care-admin-8888-access.log combined
</VirtualHost>
```

설정 적용 전에는 `Listen 8888`도 필요하다.

```apache
Listen 8888
```

설정 후에는 문법 검사를 먼저 수행한다.

```bash
sudo apache2ctl configtest
sudo systemctl reload apache2
```

정상이라면 허용된 IP 외에는 관리자 페이지 접근이 차단되어야 한다.

### 6.3 2차 인증 또는 인증서 기반 접근

PDF의 “OTP, VPN, 인증서”는 관리자 페이지가 외부에 노출될 수밖에 없는 경우 추가 인증 수단을 붙이라는 의미다.

이번 실습에서는 인증서 기반 접근까지 구현하지 않는다.

구현하지 않는 이유는 다음과 같다.

|항목|판단|
|---|---|
|TLS 서버 인증서|17번 데이터 평문 전송과 더 직접적으로 연결됨|
|클라이언트 인증서|별도 CA, 클라이언트 인증서 발급, Apache SSL 설정이 필요해 실습 범위가 커짐|
|OTP|별도 OTP 라이브러리 또는 TOTP 앱 연동이 필요함|
|VPN|네트워크 구성 항목에 가까움|

따라서 이번 실습에서는 2차 인증은 조치안으로만 제시하고, 실제 구현은 IP allowlist까지로 제한한다.

운영 환경에서는 다음 중 하나 이상을 적용한다.

```text
- VPN 접속 후에만 관리자 페이지 접근 허용
- 관리자 로그인에 OTP/TOTP 적용
- 클라이언트 인증서 기반 mTLS 적용
- 관리자 계정에 강한 비밀번호와 로그인 실패 제한 적용
```

### 6.4 관리자 하위 페이지 세션 검증

관리자 로그인 페이지를 보호해도, 하위 페이지가 세션 검증 없이 열리면 직접 URL 입력으로 우회될 수 있다.

예를 들어 다음 구조가 있다고 가정한다.

```text
/gkfn8tlrkstnausqhwkd.php
/admin-dashboard.php
```

`admin-dashboard.php` 맨 위에는 다음과 같은 검증이 필요하다.

```php
<?php
session_start();

if (($_SESSION['admin_login'] ?? false) !== true) {
    http_response_code(403);
    echo 'admin login required';
    exit;
}
```

이 조치는 IP allowlist와 별개다.

```text
IP allowlist: 관리자 페이지에 접근할 수 있는 네트워크 위치 제한
세션 검증: 접근한 사용자가 관리자 로그인을 완료했는지 확인
```

둘 중 하나만으로는 부족하다. 관리자 페이지를 실제 운영한다면 둘을 함께 적용하는 것이 맞다.

## 7. 조치 후 점검

조치 후에는 다음을 확인한다.

|점검|기대 결과|
|---|---|
|`172.168.10.10`에서 `:8888/gkfn8tlrkstnausqhwkd.php` 접근|허용|
|`172.168.10.100`에서 `:8888/gkfn8tlrkstnausqhwkd.php` 접근|허용|
|허용되지 않은 IP에서 같은 URL 접근|403 Forbidden 또는 사용자 정의 에러 페이지|
|흔한 관리자 URL 접근|존재하지 않음 또는 차단|

## 9. 판단

| 기준 | 판단 |
|---|---|
| CARE 원본 분류 | D |
| 실습 구현 분류 | C |
| 판정 상태 | 원본 D 확정, 실습 구현 검증 완료 |
| 조치 상태 | 실습용 조치 적용 및 검증 완료 |
| 증거 상태 | 원본 증거와 실습 증거 확보 |

CARE 원본에는 관리자 페이지가 확인되지 않는다. 따라서 원본 CARE 기준으로 19번 관리자 페이지 노출은 D로 판단한다.

다만 PDF p.775-776의 관리자 페이지 노출 점검과 조치 방법을 실습하기 위해, 별도 관리자 페이지를 구성했다. 이 경우 실습용 관리자 페이지는 원본 CARE 취약점이 아니라, PDF 조치 방법을 재현하기 위한 C 유형 실습으로 분리한다.

이번 실습에서는 유추하기 어려운 파일명 `gkfn8tlrkstnausqhwkd.php`, 별도 포트 `8888`, Apache IP allowlist를 사용했다. 허용 IP에서는 관리자 로그인 페이지가 출력되고, 비허용 IP에서는 403 Forbidden으로 차단되는 것을 확인했다. 따라서 PDF 19번의 핵심 조치인 URL/포트 은닉과 지정 IP 제한은 검증했다.

4번 조치인 “하위 페이지 URL 직접 접근 차단”은 관리자 로그인 이후의 dashboard, user-list 같은 하위 페이지에 각각 세션 검증을 넣으라는 의미다. 관리자 기능을 여러 파일로 구성할 경우 각 파일에서 `$_SESSION['admin_login']` 같은 관리자 인증 상태를 확인해야 한다.

이번 실습은 단일 관리자 로그인 파일만 구성했으므로 하위 페이지 직접 접근 차단은 해당 없음으로 처리한다. 추측 가능한 관리자 계정 로그인 시도와 2차 인증 적용은 이번 증거 범위에 포함하지 않고, 운영 환경에서 추가할 조치안으로 남긴다.
