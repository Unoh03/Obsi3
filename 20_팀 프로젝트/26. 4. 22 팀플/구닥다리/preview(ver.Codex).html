2조 NAT 포함 네트워크 토폴로지

ver.Codex | 기준 파일: A-R.ios, B-R.ios, C-R.ios

이 문서는 라우터 IOS 설정을 기준으로 토폴로지, NAT, 라우팅, 접근 제어를 보고서/발표용으로 정리한 것이다. 실제 동작 판단은 HTML 원본이 아니라 .ios 설정을 우선한다.

Transit LAN 1.2.123.0/24 A-R, C-R, B-R이 같은 Transit-SW 구간을 공유한다.
NAT 방향 LAN = inside
Transit = outside 내부 사설 IP를 Transit 쪽 서비스 IP로 변환한다.
DHCP A-R이 직접 제공 별도 DHCP 서버 192.168.1.2는 IOS에서 확인되지 않는다.
Monitor Static NAT 없음 192.168.3.4는 outbound PAT로 1.2.123.3을 사용한다.

전체 구조

세 라우터는 각자의 내부 LAN을 가지고 있고, 바깥쪽 인터페이스는 모두 1.2.123.0/24 Transit LAN에 연결된다. 내부망끼리는 직접 192.168.x.0/24로 통신하지 않고, 각 라우터의 NAT 주소 대역인 1.2.1.0/24, 1.2.2.0/24, 1.2.3.0/24를 통해 접근하도록 구성되어 있다.

[A LAN / EDGE-DNS-DHCP] [Transit-SW / 1.2.123.0/24] [B LAN / DATA-MGMT] 192.168.1.0/24 | 192.168.3.0/24 Client, DNS1, DNS2 | DB1, DB2, Log, Monitor | | | A-R f0/1: 1.2.123.1 -------------------+------------------- B-R f0/1: 1.2.123.3 A-R f0/0: 192.168.1.254 | B-R f0/0: 192.168.3.254 | | C-R f0/1: 1.2.123.2 C-R f0/0: 192.168.2.254 | [C APP LAN / 192.168.2.0/24] LB1, LB2, WEB1, WEB2, NFS, Cache
주의: 기존 HTML에는 A/B 연결 방식이 UDP Tunnel로 표현되어 있었다. 그러나 세 개의 .ios 파일 안에는 interface tunnel 같은 터널 설정이 없다. 따라서 라우터 설정 기준으로는 세 라우터가 같은 Transit LAN에 붙은 구조까지만 확정할 수 있다.

B-R 접근 제어 정책

B-R.ios에는 B_ZONE_IN ACL이 f0/1 inbound 방향에 적용되어 있다. 목적은 B 구역의 DB/Log 공개 NAT 주소에 대해 WEB 서버만 접근하게 제한하는 것이다.

정책 허용/차단 의미
1.2.2.3, 1.2.2.4 -> 1.2.3.1 허용 WEB1/WEB2에서 DB1 접근
1.2.2.3, 1.2.2.4 -> 1.2.3.2 허용 WEB1/WEB2에서 DB2 접근
1.2.2.3, 1.2.2.4 -> 1.2.3.3 허용 WEB1/WEB2에서 Log 접근
any -> 1.2.3.0/24 차단 후 로그 허용된 WEB 트래픽 외의 B 공개 서비스 대역 직접 접근 차단
any -> 그 외 목적지 허용 B 서비스 대역 외 일반 트래픽은 통과

발표용 핵심 문장

  • 이 토폴로지는 A, C, B 세 구역을 Transit LAN 1.2.123.0/24로 연결한다.
  • 각 구역의 실제 서버 IP는 192.168.x.0/24 사설망에 있고, 외부 접근은 NAT 주소 1.2.x.0/24를 사용한다.
  • A-R은 클라이언트 DHCP와 DNS Static NAT를 담당한다.
  • C-R은 LB/WEB 서비스를 Static NAT로 공개하고, NFS/Cache는 공개 NAT 없이 내부 전용으로 둔다.
  • B-R은 DB/Log를 Static NAT로 공개하지만, ACL로 WEB1/WEB2의 접근만 허용한다.
  • Monitor는 외부에서 접속받는 서버가 아니라, 내부에서 밖으로 조회하는 Pull 방식 장비로 보고 Static NAT를 만들지 않는다.
