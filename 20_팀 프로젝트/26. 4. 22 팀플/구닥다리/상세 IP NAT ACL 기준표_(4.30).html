2조 상세 IP NAT ACL 기준표

기준: 현재 A-R.ios, B-R.ios, 최신 C-R(4.28.1).ios, LB.sh, dns1.sh, dns2.sh, nfs-ha(5.6).sh, web-nfs(5.6).sh. 이 문서는 실제 설정 담당자가 IP, NAT, 정적 라우트, ACL 의도를 확인하는 상세 기준표다.

문서 역할: 한눈에 보는 그림은 최종 토폴로지 및 IP 설정.html을 사용한다. 이 문서는 VM별 IP/Gateway/DNS, NAT 매핑, 라우터별 설정, ACL 정책, 검증 기준을 확인하는 작업 기준표다.
운영 의도: 일반 사용자는 LB VIP로만 접근한다. WEB은 B Zone의 DB/Log와 통신하기 위해 Static NAT를 갖지만, 일반 직접 접근은 C-R ACL로 차단한다. Monitor는 예외적으로 WEB1/WEB2를 직접 점검할 수 있다. WEB upload는 NFS VIP를 바라본다.

전체 구조

핵심은 A-R, C-R, B-R 세 라우터가 같은 Transit LAN 1.2.123.0/24을 공유한다는 점이다.

[A-PC / EDGE-DNS-DHCP] [C-PC / APP + Transit-SW] [B-PC / DATA-MGMT] Client / DHCP / DNS LB / WEB / NFS / Cache DB / Log / Monitor 사설망: 192.168.1.0/24 사설망: 192.168.2.0/24 사설망: 192.168.3.0/24 | | | Switch-A APP-SW Switch-B | | | A-R -------- UDP Tunnel A 12357 / C 54378 -------- Transit-SW -------- UDP Tunnel B 37898 / C 15428 -------- B-R | | | A-R Transit: 1.2.123.1 C-R Transit: 1.2.123.2 B-R Transit: 1.2.123.3 / C-R C-R LAN: 192.168.2.254 Transit LAN: 1.2.123.0/24 - A-R: 1.2.123.1 - C-R: 1.2.123.2 - B-R: 1.2.123.3 VIP: - LB VIP: 192.168.2.10 -> 1.2.2.10 - NFS VIP: 192.168.2.50, WEB mount source 192.168.2.50:/share_directory
해석: A-R → C-R → B-R로 일렬 라우팅되는 구조가 아니다. 세 라우터가 Transit-SW에 함께 연결되어 있고, 각 Zone의 공인 역할 IP 대역으로 정적 라우팅한다.

NAT inside / outside

구분이번 프로젝트에서의 의미예시
ip nat inside 사설 IP를 쓰는 서버/클라이언트가 있는 쪽 A LAN 192.168.1.0/24, C APP LAN 192.168.2.0/24, B DATA/MGMT LAN 192.168.3.0/24
ip nat outside NAT 변환된 IP가 나가는 Transit 쪽 Transit LAN 1.2.123.0/24

예를 들어 DNS1은 실제 서버 IP가 192.168.1.2이고, 다른 Zone에서는 1.2.1.2로 접근한다. 현재 DNS 서버 자체의 NAT는 1.2.1.2, 1.2.1.3이고, 서비스 도메인 zzaphub.com / www.zzaphub.com은 LB 공인 VIP 1.2.2.10을 가리킨다.

DNS 주의: dns2.sh의 A 레코드는 현재 dns1.sh와 같이 1.2.2.10을 바라본다. 다만 slave zone의 실제 서비스 파일은 /var/cache/bind/db.zzaphub.com이고, 스크립트가 점검하는 파일은 /etc/bind/db.zzaphub.com이라 두 파일 기준이 달라질 수 있다.
DNS1 실제 IP: 192.168.1.2 | A-R ← 여기서 NAT 변환 | 외부에서 보는 IP: 1.2.1.2

보류 / 개선점

항목현재 결정이유
LB 포트 단위 제한보류초기 구축 중 변수를 줄이기 위해 permit ip any any는 유지한다. 추후 서비스 포트가 확정되면 제한한다.
DNS 포트 단위 ACL보류A Zone DNS는 Static NAT만 적용한다. DNS UDP/TCP 53 제한은 추후 보안 개선점으로 분리한다.
DNS2 zone 점검 기준주의 필요dns2.sh의 A 레코드는 1.2.2.10으로 맞춰졌다. 다만 slave 실제 파일은 /var/cache/bind/db.zzaphub.com이고 점검 파일은 /etc/bind/db.zzaphub.com이라 검증 기준이 갈릴 수 있다.
UDP Tunnel 포트 표기반영A-C는 192.168.33.121:12357 ↔ 192.168.33.129:54378, B-C는 192.168.33.122:37898 ↔ 192.168.33.129:15428로 문서화한다.
VIP 표기반영LB VIP 192.168.2.10 / 1.2.2.10, NFS VIP 192.168.2.50는 이중화 이후 추가된 논리 IP로 분리 표기한다.
라우터 ping 테스트참고용라우터 자체 ping은 실제 WEB/Client/Monitor 흐름을 대표하지 않는다. 최종 검증은 VM 기준으로 한다.
Monitor inbound차단 유지Monitor는 Pull 방식으로 동작한다. 외부에서 Monitor로 직접 들어오는 Static NAT는 만들지 않는다.
