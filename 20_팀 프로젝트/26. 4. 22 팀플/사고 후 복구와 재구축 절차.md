# 사고 후 복구와 재구축 절차

## 1. 이 문서의 목적

이 문서는 우리 프로젝트에서 장애나 침해가 났을 때 무엇을 확인하고, 어떤 순서로 격리하고, 어떻게 복구할지 정리한 문서다.

이번 목표는 완전한 클라우드 자동복구가 아니다. 현재 수업 환경에서는 AWS Auto Scaling처럼 서버를 자동 생성하고 LB에 자동 등록하는 구조까지 만들기 어렵다. 대신 다음 목표를 잡는다.

- 장애 위치를 빠르게 구분한다.
- 살아 있는 서버로 서비스를 우회한다.
- 문제가 있는 서버를 함부로 고치기 전에 증거를 남긴다.
- 스크립트와 매뉴얼로 재구축 가능한 부분을 명확히 한다.
- 수작업으로 만든 부분은 복구 때 위험하므로 따로 표시한다.

## 2. 기준 파일

장애 대응에서 기준이 흔들리면 복구 중에 더 큰 사고가 난다. 이 문서에서는 아래 파일을 기준으로 삼는다.

| 구분 | 기준 |
|---|---|
| 네트워크 지도 | `토폴로지와 IP NAT ACL 기준표_(4.30).html` |
| LB 구축 후보 | `쉘 스크립트/LB.sh` |
| DNS 구축 후보 | `쉘 스크립트/dns1.sh`, `쉘 스크립트/dns2.sh` |
| WEB 구축 후보 | `쉘 스크립트/web.sh`, `쉘 스크립트/web-nfs(5.6).sh` |
| NFS 운영 기준 | `NFS 운영 매뉴얼(4.29.1).md` |
| NFS 구축 후보 | `쉘 스크립트/nfs-ha(5.6).sh` |
| DB 구축 | 현재는 조원이 수작업 구축한 상태. `db.sh`를 실제 운영 기준으로 단정하지 않는다. |

`C-R.ios`는 구버전으로 보고 이번 복구 기준에서 제외한다. C Zone 기준은 `C-R(4.28.1).ios`와 4.30 HTML 기준표를 따른다.

## 3. 현재 구조 요약

서비스 흐름은 아래와 같다.

```text
사용자
  -> DNS1/DNS2
  -> zzaphub.com
  -> 1.2.2.10
  -> LB VIP 192.168.2.10
  -> LB1/LB2 nginx
  -> WEB1 192.168.2.3:8080 또는 WEB2 192.168.2.4:8080
  -> DB 1.2.3.1 또는 Log 1.2.3.3
```

업로드 파일 흐름은 아래와 같다.

```text
WEB1/WEB2
  -> /opt/tomcat/tomcat-10/webapps/upload
  -> NFS VIP 192.168.2.50:/share_directory
  -> 현재 VIP 소유 NFS
  -> rsync로 반대편 NFS에 복제
```

관측성 흐름은 아직 완성 기준이 아니라 보류 기준으로 본다.

```text
Monitor 192.168.3.4
  -> Prometheus Pull
  -> WEB/LB/DB/Log 등 node_exporter 확인

WEB1/WEB2
  -> Loki 1.2.3.3:3100 또는 192.168.3.3:3100으로 로그 Push
```

## 4. 사고 대응의 기본 원칙

장애와 침해는 대응 방식이 다르다.

장애는 서비스를 살리는 것이 먼저다. 예를 들어 WEB1만 죽었으면 WEB2와 LB로 우회하고 WEB1을 복구한다.

침해는 증거 보존과 확산 차단이 먼저다. 예를 들어 WEB1이 웹쉘에 털린 것으로 의심되면 바로 파일을 지우거나 서버를 재설치하지 않는다. 먼저 로그와 상태를 남기고, LB나 방화벽에서 WEB1을 제외한 뒤 복구한다.

기본 순서는 아래처럼 잡는다.

```text
1. 증상 확인
2. 어느 계층 장애인지 분류
3. 현재 상태와 로그 저장
4. 필요하면 문제 서버 격리
5. 살아 있는 서버로 서비스 유지
6. 스크립트 또는 수작업 절차로 복구
7. 복구 후 검증
8. 원인과 시행착오 기록
```

## 5. 공통 사전 확인 명령

어느 서버든 장애 대응 전에 시간, IP, 서비스 상태를 먼저 본다.

```bash
hostname
date
timedatectl
ip -br addr
ip route
df -h
free -h
uptime
```

서비스 상태 확인은 서버 역할에 맞게 실행한다.

```bash
systemctl status nginx --no-pager
systemctl status keepalived --no-pager
systemctl status tomcat --no-pager
systemctl status nfs-kernel-server --no-pager
systemctl status mariadb --no-pager
systemctl status named --no-pager
```

로그는 바로 삭제하지 말고 먼저 저장한다.

```bash
journalctl -u nginx -n 100 --no-pager
journalctl -u keepalived -n 100 --no-pager
journalctl -u tomcat -n 100 --no-pager
journalctl -u nfs-kernel-server -n 100 --no-pager
journalctl -u mariadb -n 100 --no-pager
journalctl -u named -n 100 --no-pager
```

## 6. DNS 장애

### 증상

- `zzaphub.com` 접속이 안 된다.
- IP로 직접 접속하면 되는데 도메인으로는 안 된다.
- DNS1 또는 DNS2 중 하나만 응답한다.

### 확인

```bash
dig @192.168.1.2 zzaphub.com
dig @192.168.1.3 zzaphub.com
dig @1.2.1.2 zzaphub.com
dig @1.2.1.3 zzaphub.com
```

정상 기준은 `zzaphub.com`이 `1.2.2.10`으로 응답하는 것이다.

DNS 서버에서 확인한다.

```bash
systemctl status named --no-pager
sudo named-checkconf
```

DNS1은 master zone 파일을 확인한다.

```bash
sudo named-checkzone zzaphub.com /etc/bind/db.zzaphub.com
```

DNS2는 slave이므로 실제 운영 파일 위치가 다를 수 있다. `dns2.sh`는 slave zone을 `/var/cache/bind/db.zzaphub.com`으로 설정한다. 따라서 DNS2는 아래 확인이 더 중요하다.

```bash
sudo ls -l /var/cache/bind/db.zzaphub.com
dig @127.0.0.1 zzaphub.com
```

### 복구

DNS1이 깨졌으면 `dns1.sh`를 기준으로 재구축한다.

```bash
bash dns1.sh
```

DNS2가 깨졌으면 `dns2.sh`를 기준으로 재구축한다.

```bash
bash dns2.sh
```

### 복구 후 검증

```bash
dig @192.168.1.2 zzaphub.com
dig @192.168.1.3 zzaphub.com
```

둘 다 `1.2.2.10`을 반환해야 한다.

## 7. LB 장애

### 증상

- `zzaphub.com` 접속이 안 된다.
- `1.2.2.10` 또는 `192.168.2.10` 접속이 안 된다.
- LB1/LB2 중 하나가 죽었는데 VIP가 이동하지 않는다.
- nginx는 살아 있는데 실제 WEB으로 연결되지 않는다.

### 확인

LB1/LB2에서 VIP를 확인한다.

```bash
ip a | grep 192.168.2.10
systemctl status nginx --no-pager
systemctl status keepalived --no-pager
journalctl -u keepalived -n 100 --no-pager
```

nginx 설정을 확인한다.

```bash
sudo nginx -t
cat /etc/nginx/conf.d/load-balancer.conf
```

WEB backend를 직접 확인한다.

```bash
curl -I http://192.168.2.3:8080/
curl -I http://192.168.2.4:8080/
curl -I http://192.168.2.10/
```

### 중요한 한계

현재 `LB.sh`의 keepalived health check는 `nginx`가 살아 있는지만 본다. 즉, WEB1/WEB2가 모두 죽어도 nginx가 살아 있으면 LB VIP는 정상처럼 남을 수 있다.

그래서 LB 장애 대응에서는 반드시 backend 확인을 따로 해야 한다.

```bash
curl -I http://192.168.2.3:8080/
curl -I http://192.168.2.4:8080/
```

### 복구

LB1 또는 LB2를 다시 구축해야 하면 해당 서버에서 실행한다.

```bash
# LB1
bash LB.sh MASTER

# LB2
bash LB.sh BACKUP
```

인터페이스 자동 탐지가 실패하면 프로젝트망 인터페이스를 지정한다.

```bash
IFACE=ens37 bash LB.sh MASTER
IFACE=ens37 bash LB.sh BACKUP
```

### 복구 후 검증

```bash
ip a | grep 192.168.2.10
curl -I http://192.168.2.10/
curl -I http://1.2.2.10/
```

## 8. WEB 장애 또는 침해

### 증상

- 특정 WEB만 응답하지 않는다.
- LB 접속은 되지만 간헐적으로 502 또는 5xx가 난다.
- 업로드가 안 된다.
- 이상한 JSP, WAR, class 파일이 생겼다.
- `application.properties`나 환경 파일 접근 흔적이 있다.

### 확인

WEB 서버에서 확인한다.

```bash
hostname
ip -br addr
systemctl status tomcat --no-pager
journalctl -u tomcat -n 100 --no-pager
curl -I http://localhost:8080/
```

NFS mount를 확인한다.

```bash
findmnt -T /opt/tomcat/tomcat-10/webapps/upload
mount | grep upload
```

정상 기준은 source가 `192.168.2.50:/share_directory`인 것이다.

### 침해 의심 시 격리

WEB 침해가 의심되면 먼저 LB에서 해당 WEB을 제외한다. 가장 단순한 방법은 LB의 nginx upstream에서 해당 WEB을 주석 처리하고 nginx를 reload하는 것이다.

```bash
sudo cp /etc/nginx/conf.d/load-balancer.conf /etc/nginx/conf.d/load-balancer.conf.bak.$(date +%Y%m%d-%H%M%S)
sudo vi /etc/nginx/conf.d/load-balancer.conf
sudo nginx -t
sudo systemctl reload nginx
```

이때 문제 WEB에 바로 접속해서 파일을 지우지 않는다. 먼저 로그와 파일 목록을 남긴다.

```bash
sudo find /opt/tomcat/tomcat-10/webapps -maxdepth 3 -type f -printf '%TY-%Tm-%Td %TH:%TM %p\n' | sort | tail -n 100
journalctl -u tomcat -n 300 --no-pager
```

### 복구

WEB 서버를 새로 만들거나 스냅백한 뒤 `web.sh`와 `web-nfs(5.6).sh`를 실행하는 흐름이 기본이다.

```bash
bash web.sh
bash 'web-nfs(5.6).sh'
```

단, 현재 DB는 조원이 수작업으로 구성한 상태다. 따라서 `web.sh` 안의 DB 접속 설정이 실제 운영 DB 설정과 다를 수 있다. WEB 재구축 전에는 반드시 현재 정상 WEB의 DB 접속 방식과 DB 계정 허용 Host를 확인해야 한다.

DB 서버에서 확인한다.

```bash
sudo mariadb -uroot -e "SELECT User, Host FROM mysql.user WHERE User='web';"
```

비밀번호 값은 출력하지 않는다.

### 복구 후 검증

```bash
systemctl status tomcat --no-pager
curl -I http://localhost:8080/
findmnt -T /opt/tomcat/tomcat-10/webapps/upload
echo "recovery test $(hostname) $(date)" | sudo tee /opt/tomcat/tomcat-10/webapps/upload/recovery-check-$(hostname)-$(date +%Y%m%d-%H%M%S).txt
```

LB에서 WEB backend를 확인한다.

```bash
curl -I http://192.168.2.3:8080/
curl -I http://192.168.2.4:8080/
curl -I http://192.168.2.10/
```

## 9. NFS 장애

NFS는 별도 운영 매뉴얼이 이미 있으므로 여기서는 사고 대응 관점만 요약한다.

### 증상

- WEB upload 경로 접근이 느리거나 멈춘다.
- `Stale file handle`이 보인다.
- NFS VIP가 사라졌다.
- NFS1/NFS2 파일이 서로 다르다.
- split-brain이 의심된다.

### 확인

WEB에서 확인한다.

```bash
findmnt -T /opt/tomcat/tomcat-10/webapps/upload
df -h /opt/tomcat/tomcat-10/webapps/upload
```

NFS1/NFS2에서 확인한다.

```bash
ip a | grep 192.168.2.50
sudo exportfs -v
systemctl status nfs-kernel-server --no-pager
systemctl status keepalived --no-pager
tail -n 50 /var/log/nfs-ha-sync.log
df -h /share_directory
```

### 복구

NFS 복구는 `NFS 운영 매뉴얼(4.29.1).md`를 따른다.

중요한 원칙은 아래와 같다.

- WEB은 NFS1/NFS2 실제 IP가 아니라 `192.168.2.50:/share_directory`만 mount해야 한다.
- 자동 delete sync는 기본 꺼짐 상태를 유지한다.
- 장애 직후에는 어느 쪽 데이터가 최신인지 확인하기 전까지 삭제 동기화를 켜지 않는다.
- 양쪽 NFS가 모두 VIP를 가지고 있으면 split-brain으로 보고 즉시 수동 정리한다.

### 복구 후 검증

WEB에서 테스트 파일을 만든다.

```bash
echo "nfs recovery test $(hostname) $(date)" | sudo tee /opt/tomcat/tomcat-10/webapps/upload/nfs-recovery-$(hostname)-$(date +%Y%m%d-%H%M%S).txt
```

NFS1/NFS2에서 같은 파일이 보이는지 확인한다.

```bash
ls -la /share_directory | tail
tail -n 50 /var/log/nfs-ha-sync.log
```

## 10. DB 장애

### 현재 주의점

DB는 현재 `db.sh`만으로 구축된 상태가 아니라 조원이 수작업으로 구성한 상태다. 따라서 `db.sh`를 그대로 실행한다고 현재 DB와 같은 상태가 된다고 단정하면 안 된다.

이것은 사고 후 복구에서 중요한 위험이다. DB는 상태를 가진 서버이고, 데이터가 핵심이다. WEB이나 LB처럼 쉽게 밀고 다시 만들면 안 된다.

### 증상

- WEB에서 DB 연결 오류가 난다.
- 로그인, 게시글, 업로드 메타데이터 등 DB가 필요한 기능이 실패한다.
- MariaDB 서비스가 내려가 있다.

### 확인

DB 서버에서 확인한다.

```bash
systemctl status mariadb --no-pager
sudo ss -lntp | grep 3306
sudo mariadb -uroot -e "SHOW DATABASES;"
sudo mariadb -uroot -e "SELECT User, Host FROM mysql.user;"
```

WEB에서 DB 연결을 확인한다. 실제 DB 계정 정보는 노출하지 않는다.

```bash
journalctl -u tomcat -n 100 --no-pager
```

### 복구

DB는 아래 순서를 지킨다.

```text
1. DB 서버 디스크와 서비스 상태 확인
2. MariaDB 로그 확인
3. 데이터 디렉터리 삭제 금지
4. 백업 존재 여부 확인
5. 조원이 수작업으로 만든 DB 계정/권한/스키마 확인
6. 필요 시 수작업 절차를 문서화한 뒤 복구
```

현재 상태에서는 `db.sh`를 바로 복구 명령으로 쓰지 않는다. 먼저 실제 운영 DB 설정을 문서화해야 한다.

### 복구 후 검증

```bash
sudo mariadb -uroot -e "SHOW DATABASES;"
sudo mariadb -uroot -e "SELECT User, Host FROM mysql.user WHERE User='web';"
```

WEB에서 실제 기능을 확인한다.

```bash
curl -I http://192.168.2.3:8080/
curl -I http://192.168.2.4:8080/
curl -I http://192.168.2.10/
```

## 11. Log / Monitor 장애

### 증상

- Grafana 대시보드가 안 열린다.
- Prometheus scrape target이 down이다.
- WEB 로그가 Loki에 안 들어온다.
- 장애가 났는데 확인할 로그가 없다.

### 확인

Monitor 서버에서 확인한다.

```bash
systemctl status grafana-server --no-pager
systemctl status prometheus --no-pager
ss -lntp | grep -E '3000|9090'
```

Log 서버에서 확인한다.

```bash
systemctl status loki --no-pager
ss -lntp | grep 3100
```

WEB에서 Log 서버로 접근 가능한지 확인한다.

```bash
curl -I http://1.2.3.3:3100/ready
```

### 현재 한계

4.30 기준표에서도 관측성 ACL과 로그 agent는 보류 항목이다. 즉, 아직 모든 서버의 로그와 메트릭이 자동으로 중앙 수집된다고 보면 안 된다.

이번 복구 문서에서는 Log/Monitor를 완성된 자동복구 시스템으로 보지 않는다. 장애 분석을 돕는 보조 시스템으로 본다.

## 12. 우리 수준에서 가능한 자동화

자동복구는 한 번에 완성하는 것이 아니라 단계가 있다.

| 단계 | 이름 | 우리 수준 가능성 | 설명 |
|---|---|---|---|
| 0 | 수동 복구 | 가능 | 사람이 명령을 보고 복구한다. 이 문서의 기본 목표다. |
| 1 | 서비스 자동 재시작 | 가능 | `systemd Restart=on-failure` 같은 설정으로 Tomcat/nginx 죽음을 자동 재시작한다. |
| 2 | 헬스체크 스크립트 | 가능 | 주기적으로 LB, WEB, NFS mount, DB 연결을 확인하고 로그를 남긴다. |
| 3 | 원격 복구 실행 | 부분 가능 | 관리 서버가 SSH로 접속해 스크립트를 실행한다. 단, SSH key와 권한 관리가 필요하다. |
| 4 | 서버 자동 생성 | 현재 어려움 | 클라우드 API, VM 템플릿, 이미지, 자동 네트워크 등록이 필요하다. |

이번 프로젝트에서 현실적인 목표는 0단계와 1단계, 가능하면 2단계까지다.

## 13. 미니 자동복구 아이디어

완전한 Auto Scaling은 어렵지만, 아래 정도는 수업 환경에서도 가능하다.

### WEB 로컬 헬스체크

WEB 서버에서 주기적으로 확인한다.

```bash
systemctl is-active --quiet tomcat
findmnt -T /opt/tomcat/tomcat-10/webapps/upload
curl -fsS http://localhost:8080/ >/dev/null
```

실패하면 로그를 남기고 Tomcat을 재시작한다.

```bash
sudo systemctl restart tomcat
```

단, mount 문제를 무조건 자동 복구하려고 `umount -f` 같은 명령을 쓰면 더 위험할 수 있다. NFS mount 문제는 우선 알림과 수동 절차로 처리한다.

### LB backend 헬스체크

LB에서 WEB1/WEB2를 확인한다.

```bash
curl -fsS http://192.168.2.3:8080/ >/dev/null
curl -fsS http://192.168.2.4:8080/ >/dev/null
```

이 결과를 보고 사람이 nginx upstream에서 장애 WEB을 제외한다.

자동으로 upstream을 고치는 것은 가능하지만, 잘못 만들면 정상 서버까지 빼거나 설정 파일을 망가뜨릴 수 있다. 현재 수준에서는 수동 변경이 더 안전하다.

## 14. 복구 후 공통 검증 체크리스트

복구가 끝났다고 말하려면 아래를 확인한다.

| 항목 | 확인 명령 |
|---|---|
| DNS | `dig @192.168.1.2 zzaphub.com`, `dig @192.168.1.3 zzaphub.com` |
| LB VIP | `ip a | grep 192.168.2.10` |
| LB 접속 | `curl -I http://192.168.2.10/` |
| WEB1 | `curl -I http://192.168.2.3:8080/` |
| WEB2 | `curl -I http://192.168.2.4:8080/` |
| NFS mount | `findmnt -T /opt/tomcat/tomcat-10/webapps/upload` |
| NFS VIP | `ip a | grep 192.168.2.50` |
| DB | `systemctl status mariadb --no-pager` |
| Log | `systemctl status loki --no-pager` |
| Monitor | `systemctl status prometheus --no-pager`, `systemctl status grafana-server --no-pager` |

## 15. 보고서에 넣을 수 있는 시행착오

처음에는 스크립트가 있으면 서버를 쉽게 다시 만들 수 있다고 생각했다. 하지만 검토하면서 스크립트와 실제 운영 상태가 항상 같지는 않다는 것을 알게 되었다. 특히 DB는 조원이 수작업으로 구성했기 때문에 `db.sh`를 그대로 복구 기준으로 삼으면 실제 운영 상태와 다른 서버가 만들어질 수 있다.

또한 LB가 이중화되어 있어도 모든 장애가 자동으로 해결되는 것은 아니다. 현재 LB의 keepalived는 nginx 프로세스 상태를 확인하지만, nginx 뒤의 WEB1/WEB2가 정상인지까지 판단하지 않는다. 따라서 LB VIP가 살아 있어도 실제 서비스가 실패할 수 있다.

NFS도 VIP와 rsync를 사용해 가용성을 높였지만 완전한 무손실 스토리지는 아니다. 장애 직전 파일이 반대편 NFS에 아직 복제되지 않았을 수 있고, split-brain이 생기면 어느 쪽 데이터가 맞는지 사람이 판단해야 한다.

이번 정리의 결론은 “자동복구를 못 한다”가 아니다. 우리 수준에서는 먼저 수동 복구 절차를 정확히 만들고, 그 다음 서비스 자동 재시작과 헬스체크 스크립트 같은 작은 자동화부터 붙이는 것이 현실적이다.

## 16. 남은 작업

- 실제 DB 수작업 설정 문서화
- `db.sh`를 현재 토폴로지 기준에 맞게 고칠지, 미사용 파일로 표시할지 결정
- `dns2.sh`의 slave zone 검증 방식 정리
- LB backend 헬스체크 절차 추가
- WEB 침해 시 LB에서 해당 WEB을 제외하는 절차를 더 쉽게 만들기
- Log/Monitor 설치 스크립트와 ACL 기준 확정
- 각 장애 시나리오별 실제 테스트 결과 추가
